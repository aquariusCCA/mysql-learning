## 本节导读

这一节是 `REGEXP` 运算符的延伸笔记，重点不是再解释什么是比较运算符，而是集中解决一个更实际的问题：

> **当你已经知道 `LIKE` 不够用了，接下来该怎么把“我要查什么”翻成正则模式。**

很多人学正则时最容易卡住的地方，不是某个符号没见过，而是：

1. 知道 `^`、`$`、`.`、`[]` 这些符号，但一到实际需求就不会拼。
2. 只会背符号，却不知道什么时候该用 `LIKE`，什么时候才该用 `REGEXP`。
3. 一看到复杂模式就直接乱写，没有先把需求拆成“开头 / 结尾 / 包含 / 次数 / 字符集合”。

所以这篇真正要掌握的，不只是“符号是什么意思”，而是：

> **先用自然语言说清楚需求，再把需求拆成模式，最后再写成 `WHERE ... REGEXP 'pattern'`。**

第一次学习时，建议先抓住三条主线：

- 先分清 `LIKE` 和 `REGEXP` 的使用边界
- 先会看“需求 → 模式”的转换，而不是一开始就死背符号
- 正则优先解决“字符集合、位置、次数、排除条件”这类 `LIKE` 不好表达的需求

复习时，这篇最适合作为“速查 + 反推模式”的页面：当你知道自己想查什么，但一时写不出模式时，就回到这里找对应写法。

## 先记 8 句话

- 简单前缀、后缀、包含匹配，优先先想 `LIKE`。
- 当需求变成“字符集合、排除字符、连续次数、复杂规则”时，再切到 `REGEXP`。
- `REGEXP` 的返回结果通常是 `1`、`0` 或 `NULL`。
- `^` 看开头，`$` 看结尾，`.` 看任意单个字符。
- `[]` 表示字符集合，`[^]` 表示排除集合。
- `*`、`+`、`?`、`{n,m}` 主要解决“出现多少次”的问题。
- `\|` 表示“多个模式选一个”。
- 真正写正则时，先用自然语言拆需求，再翻成模式，通常比直接硬写更稳。

## 关键字

- `REGEXP`：在 MySQL 中使用正则表达式做模式匹配的运算符。
- `正则表达式`：用一套特殊符号描述字符串匹配规则的写法。
- `pattern`：正则表达式模式。
- `^`：匹配字符串开头。
- `$`：匹配字符串结尾。
- `.`：匹配任意单个字符。
- `[]`：匹配字符集合中的任意一个字符。
- `[^]`：匹配不在字符集合中的字符。
- `*`、`+`、`?`：控制前一个字符出现的次数。
- `{n}`、`{n,}`、`{n,m}`：指定前一个字符或模式出现的精确次数或范围。
- `\|`：多个模式择一匹配。

## 先搞懂：什么时候用 `LIKE`，什么时候用 `REGEXP`

这是整篇最值得先分清楚的一件事。

### 优先用 `LIKE` 的场景

如果你的需求只是：

- 以什么开头
- 以什么结尾
- 包含什么字符串
- 某一位是什么字符

通常先用 `LIKE` 会更直观。

例如：

```sql
WHERE first_name LIKE 'S%'
WHERE last_name LIKE '_o%'
```

### 该切到 `REGEXP` 的场景

如果你的需求变成：

- 某个位置必须满足字符集合规则
- 不能出现某些字符
- 某个字符要连续出现几次
- 多种模式择一匹配
- 规则已经不是 `%` 和 `_` 能清楚表达的

这时再切到 `REGEXP`。

### 你可以这样记

> **`LIKE` 适合简单模式，`REGEXP` 适合规则型模式。**

### 回查提示

如果你的需求还能用“前缀 / 后缀 / 包含 / 某一位是什么”说清楚，先试 `LIKE`；如果已经变成“字符集合、次数、排除条件”，再切到 `REGEXP`。

## `REGEXP` 基本语法

MySQL 中的基本写法如下：

```sql
expr REGEXP pattern
```

返回结果通常为：

- `1`：表示符合正则表达式 `pattern`
- `0`：表示不符合
- `NULL`：如果 `expr` 或 `pattern` 为 `NULL`

### 最基础的理解

`REGEXP` 的意思不是“做复杂字符串处理”，而是：

> **判断某个字符串是否符合某个模式。**

例如：

```sql
SELECT 'shkstart' REGEXP '^s';
SELECT 'test' REGEXP 't$';
SELECT 'apple' REGEXP 'pp';
```

### 回查提示

如果你一时看不懂某条 `REGEXP` SQL，先把它翻译成一句中文：

- “这个字符串，是否符合这个模式？”

这样通常会清楚很多。

| `.` | 匹配任意单个字符 | `'a.b'` | “这个位置随便一个字符都行” |

| `a{n,}` | 至少匹配 `n` 次 `a` | `'a{2,}'` | “至少几次” |

如果你只记得“`REGEXP` 可以写很复杂的模式”，但忘了某个符号具体是什么意思，先回到这张表。

## 1. 查询以特定字符或字符串开头的记录

字符 `^` 匹配以特定字符或者字符串开头的文本。

在 `fruits` 表中，查询 `f_name` 字段以字母 `b` 开头的记录：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP '^b';
```

### 你可以这样理解

- `^b` 不是“包含 `b`”
- 而是“开头必须是 `b`”

### 回查提示

只要你的需求里出现“以某个字母开头”或“前缀必须是某段文本”，优先想到 `^`。

## 2. 查询以特定字符或字符串结尾的记录

字符 `$` 匹配以特定字符或者字符串结尾的文本。

在 `fruits` 表中，查询 `f_name` 字段以字母 `y` 结尾的记录：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP 'y$';
```

### 你可以这样理解

- `y$` 不是“包含 `y`”
- 而是“最后必须是 `y`”

### 回查提示

如果你的条件是“后缀必须是什么”，就检查模式结尾有没有写 `$`。

## 3. 用符号 `.` 来替代字符串中的任意一个字符

字符 `.` 匹配任意一个字符。

在 `fruits` 表中，查询 `f_name` 字段值包含字母 `a` 与 `g`，且两个字母之间只有一个字符的记录：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP 'a.g';
```

### 你可以这样理解

`a.g` 的意思是：

- 先出现 `a`
- 中间可以是任意一个字符
- 然后出现 `g`

### 回查提示

当你确定某个位置“有一个字符，但不在乎它是谁”时，就看 `.`。

## 4. 使用 `*`、`+`、`?` 与 `{n,m}` 来控制出现次数

这一组符号，主要解决的不是“字符是什么”，而是：

> **前一个字符或模式，出现几次。**

### 4.1 `*`：0 次或多次

- 星号 `*` 匹配前面的字符任意多次，包括 `0` 次。

在 `fruits` 表中，查询 `f_name` 字段值以字母 `b` 开头，且 `b` 后面出现字母 `a` 的记录：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP '^ba*';
```

这里的重点不是“必须很多个 `a`”，而是：

- 可以没有 `a`
- 也可以有一个或多个 `a`

### 4.2 `+`：至少 1 次

- 加号 `+` 匹配前面的字符至少一次。

在 `fruits` 表中，查询 `f_name` 字段值以字母 `b` 开头，且 `b` 后面出现字母 `a` 至少一次的记录：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP '^ba+';
```

### 4.3 `?`：0 次或 1 次

- 问号 `?` 表示前一个字符可有可无。

例如：

```sql
SELECT 'color' REGEXP 'colou?r',
       'colour' REGEXP 'colou?r'
FROM DUAL;
```

这里 `u?` 表示：

- 可以没有 `u`
- 也可以正好有一个 `u`

### 4.4 `{n}`、`{n,}`、`{n,m}`：更精确的次数控制

- `{n}`：恰好匹配 `n` 次
- `{n,}`：至少匹配 `n` 次
- `{n,m}`：匹配 `n` 到 `m` 次

例如：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP 'x{2,}';
```

表示：字母 `x` 至少连续出现 `2` 次。

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP 'a{2,4}';
```

表示：字母 `a` 连续出现 `2` 到 `4` 次。

### 一眼看懂这组符号

| 写法 | 直觉 |
| --- | --- |
| `a*` | 可以没有，也可以很多个 |
| `a+` | 至少一个 |
| `a?` | 可有可无 |
| `a{2}` | 恰好两个 |
| `a{2,}` | 至少两个 |
| `a{2,4}` | 两到四个 |

### 回查提示

如果你的需求里出现“至少几次”“最多几次”“恰好几次”“可有可无”，直接想到这组次数控制符号。

## 5. 匹配指定字符串或多个字符串

正则表达式不只可以匹配单个字符，也可以匹配指定字符串。

### 5.1 匹配单个指定字符串

在 `fruits` 表中，查询 `f_name` 字段值包含字符串 `on` 的记录：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP 'on';
```

这表示：只要字段内容中出现 `on` 这个片段，就算匹配成功。

### 5.2 匹配多个字符串之一：`\|`

如果要匹配多个字符串，多个模式之间可以使用 `\|` 隔开。

在 `fruits` 表中，查询 `f_name` 字段值包含字符串 `on` 或者 `ap` 的记录：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP 'on|ap';
```

### 你可以这样理解

`on|ap` 的意思是：

- 命中 `on` 也可以
- 命中 `ap` 也可以

### 回查提示

如果你的需求是“命中这个词，或者命中另一个词”，就可以从 `\|` 开始想。

## 6. 匹配指定字符中的任意一个

方括号 `[]` 指定一个字符集合，只匹配其中任何一个字符。

在 `fruits` 表中，查找 `f_name` 字段中包含字母 `o` 或者 `t` 的记录：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP '[ot]';
```

在 `fruits` 表中，查询 `s_id` 字段中包含 `4`、`5` 或者 `6` 的记录：

```sql
SELECT *
FROM fruits
WHERE s_id REGEXP '[456]';
```

### 你可以这样理解

`[ot]` 不是在匹配字符串 `ot`，而是在说：

- 这里的这个位置
- 只要是 `o` 或 `t` 任意一个都可以

### 范围写法

如果字符是一个连续范围，也可以写成：

- `[a-z]`
- `[0-9]`

### 回查提示

只要你的条件是“几个字符里任意一个都可以”，就优先看 `[]`。

## 7. 匹配指定字符以外的字符

`[^字符集合]` 匹配不在指定集合中的任何字符。

在 `fruits` 表中，查询 `f_id` 字段中包含字母 `a~e` 和数字 `1~2` 以外字符的记录：

```sql
SELECT *
FROM fruits
WHERE f_id REGEXP '[^a-e1-2]';
```

### 你可以这样理解

- `[a-e1-2]`：这些字符都可以
- `[^a-e1-2]`：这些字符都不可以

### 回查提示

当需求变成“不要这些字符”，就从 `[^...]` 开始写，而不是继续堆普通字符集合。

## 8. 写正则前，先这样拆需求

这是这篇最值得你真正练起来的一部分。

很多人写正则时，会一上来就直接拼模式；但更稳的做法是先拆需求。

### 第一步：先问自己要控制哪一类信息

先判断你的需求属于哪一种：

- 开头
- 结尾
- 中间某个位置
- 字符集合
- 排除某些字符
- 连续出现次数
- 多种模式选一种

### 第二步：再决定对应符号

例如：

- 开头 -> `^`
- 结尾 -> `$`
- 任意一个字符 -> `.`
- 字符集合 -> `[]`
- 排除集合 -> `[^]`
- 次数 -> `*`、`+`、`?`、`{n,m}`
- 二选一 -> `\|`

### 第三步：最后再写 SQL

例如需求：

> 查询名字以 `b` 开头，后面至少有一个 `a` 的记录。

你可以这样拆：

- 以 `b` 开头 -> `^b`
- 后面至少一个 `a` -> `a+`
- 合起来 -> `^ba+`

最终 SQL：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP '^ba+';
```

再例如需求：

> 查询名字中包含 `on` 或 `ap` 的记录。

拆法：

- 命中 `on` -> `on`
- 或命中 `ap` -> `|ap`
- 合起来 -> `on|ap`

最终 SQL：

```sql
SELECT *
FROM fruits
WHERE f_name REGEXP 'on|ap';
```

### 最实用的一句话

> **先用中文把需求拆清楚，再把每一段翻成正则。**

这通常比你直接盯着符号乱拼更不容易写错。

### 回查提示

如果你知道自己想查什么，却写不出模式，不要立刻乱试，先回到这里拆需求。

## 常见混淆点

- `LIKE` 更适合简单前后缀或模糊匹配；当规则变成“字符集合、次数控制、排除字符”时，`REGEXP` 更合适。
- `REGEXP` 本质上是在判断“是否符合某个模式”，不是在做一般意义上的字符串拼接或替换。
- 正则里的很多符号都有特殊含义，阅读模式时不要把它们当普通字符。
- `.` 表示“任意单个字符”，不是“任意长度”。
- `*`、`+`、`?` 控制的是“前一个字符或模式”的次数，不是对整段字符串随便生效。
- `[ot]` 表示“`o` 或 `t` 任一字符”，不是字符串 `ot`。
- `[^...]` 是排除集合，不是“普通的 `^` 再加字符集合”。
- 写复杂模式时，建议先用自然语言说清楚需求，再翻成正则，这样比较不容易写错。

## 常见回查问题

- `LIKE` 和 `REGEXP` 到底什么时候该怎么选？
- `expr REGEXP pattern` 的返回结果是什么？
- `^` 和 `$` 分别控制什么位置？
- `.` 和 `%` 有什么本质差别？
- `*`、`+`、`?`、`{n,m}` 分别表示什么次数规则？
- `[ot]` 和 `on|ap` 的差别是什么？
- `[^a-e1-2]` 这种写法到底是什么意思？
- 为什么写正则前，最好先把需求拆成“位置 / 集合 / 次数 / 选择”？

## 一分钟自测

先不要往下看答案，自己判断下面这些模式各自表示什么：

1. `'^b'`
2. `'y$'`
3. `'a.g'`
4. `'^ba*'`
5. `'^ba+'`
6. `'on|ap'`
7. `'[ot]'`
8. `'[^a-e1-2]'`
9. `'x{2,}'`
10. `'colou?r'`

### 参考答案

1. 以 `b` 开头。
2. 以 `y` 结尾。
3. 出现 `a` 和 `g`，且中间恰好一个任意字符。
4. 以 `b` 开头，后面可以没有 `a`，也可以有多个 `a`。
5. 以 `b` 开头，后面至少有一个 `a`。
6. 命中 `on` 或命中 `ap`。
7. 匹配字符 `o` 或 `t` 任意一个。
8. 匹配不在 `a-e` 和 `1-2` 范围内的字符。
9. 字母 `x` 至少连续出现 `2` 次。
10. `u` 可有可无，所以可匹配 `color` 与 `colour`。

## 一句话抓核心

这篇扩展笔记真正要掌握的，不只是 `^`、`$`、`.`、`[]`、`*`、`+` 这些符号会背，而是：**先判断该不该用 `REGEXP`，再把自然语言需求拆成“位置 / 集合 / 次数 / 选择”，最后再翻成正则模式写进 SQL。**

## 小结

这一节你需要记住：

- `REGEXP` 用于使用正则表达式做模式匹配。
- 简单匹配优先考虑 `LIKE`；复杂规则再切到 `REGEXP`。
- 最常用的一组符号是 `^`、`$`、`.`、`[]`、`[^]`、`*`、`+`、`?`、`{n,m}` 与 `\|`。
- 写正则时，不要一上来就乱拼，先把需求拆成“开头 / 结尾 / 任意字符 / 字符集合 / 排除集合 / 次数控制 / 多选一”。
- 这篇最适合当成“需求反推模式”的速查页，而不只是单纯的符号说明表。
