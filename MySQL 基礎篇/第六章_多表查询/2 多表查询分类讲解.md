# 2 多表查询分类讲解

> - 所属章节：第六章_多表查询  
> - 关键字：等值连接、非等值连接、自连接、非自连接、内连接、外连接、左外连接、右外连接、满外连接、表别名、连接条件  
> - 建议回查情境：分不清多表查询有哪些分类、忘记等值连接和非等值连接的差别、不确定自连接是什么意思、想快速区分内连接与外连接，或看到一个 JOIN 查询时不知道该从哪些角度判断它属于哪一类时

## 本节导读

上一节已经说明：多表查询不能只把多个表写进 `FROM`，还必须写出有效的连接条件，否则容易产生笛卡尔积。

这一节继续往下建立「多表查询的分类地图」。

需要特别注意的是，多表查询的分类不是只有一种标准。  
同一条 SQL 可以同时从多个角度来理解，例如：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 可以同时说成：

* 从连接条件来看：它是等值连接。
* 从参与表来看：它是非自连接。
* 从结果保留规则来看：它是内连接。

所以学习这一节时，不要把这些名词当成互相排斥的类别，而要把它们看成不同观察角度。

本节重点不是死记名词，而是理解：

> 不同分类维度，分别在回答多表查询中的不同问题。

## 你会在这篇学到什么

* 多表查询可以从多个维度分类，而不是只有一种分法。
* 按连接条件的形式，可以分成等值连接和非等值连接。
* 按参与连接的表是否为同一张表，可以分成自连接和非自连接。
* 按结果是否保留不匹配记录，可以分成内连接和外连接。
* 同一条 SQL 可以同时属于多个分类维度。
* 表别名可以让多表查询更短、更清楚，在自连接中几乎必不可少。
* 连接 `n` 张表时，通常至少需要 `n - 1` 个连接条件。
* MySQL 不支持旧式 Oracle `(+)` 外连接写法，实务中应使用 SQL99 的 `LEFT JOIN` / `RIGHT JOIN`。

## 2.1 分类维度总览

多表查询可以从三个常见角度来分类。

### 2.1.1 维度一：连接条件是什么形式

这个维度关注的是：

> 两张表是通过什么条件对应起来的？

常见类型有：

* 等值连接
* 非等值连接

例如：

```sql
e.department_id = d.department_id
```

这是等值连接，因为连接条件使用的是 `=`。

再例如：

```sql
e.salary BETWEEN j.lowest_sal AND j.highest_sal
```

这是非等值连接，因为它不是通过两个字段相等来匹配，而是通过薪资是否落在某个区间内来匹配。

### 2.1.2 维度二：参与连接的是不是同一张表

这个维度关注的是：

> 是不同表之间的连接，还是同一张表自己和自己连接？

常见类型有：

* 自连接
* 非自连接

例如：

```sql
FROM employees worker, employees manager
```

虽然写了两次 `employees`，但本质上还是同一张表。
只是它通过两个别名，分别扮演“员工”和“经理”两个角色。

这就是自连接。

### 2.1.3 维度三：是否保留没有匹配上的记录

这个维度关注的是：

> 查询结果中，要不要保留没有匹配成功的数据？

常见类型有：

* 内连接
* 外连接

内连接只保留匹配成功的记录。
外连接除了保留匹配成功的记录，还会保留某一侧或两侧没有匹配上的记录。

### 2.1.4 三个维度可以同时存在

这三个分类维度不是互相排斥的。

同一条 SQL 可以同时属于多个分类。

例如：

```sql
SELECT
    worker.last_name AS worker_name,
    manager.last_name AS manager_name
FROM employees worker
JOIN employees manager
    ON worker.manager_id = manager.employee_id;
```

这条 SQL 可以这样判断：

| 分类维度    | 判断结果 | 原因                                               |
| ------- | ---- | ------------------------------------------------ |
| 按连接条件   | 等值连接 | `worker.manager_id = manager.employee_id` 使用 `=` |
| 按参与表    | 自连接  | 两边都是 `employees` 表                               |
| 按结果保留规则 | 内连接  | 使用 `JOIN`，只保留匹配成功的数据                             |

所以它可以说是：

> 一个使用等值条件的自连接内连接查询。

再看另一个例子：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 可以这样判断：

| 分类维度    | 判断结果 | 原因                                         |
| ------- | ---- | ------------------------------------------ |
| 按连接条件   | 等值连接 | `e.department_id = d.department_id` 使用 `=` |
| 按参与表    | 非自连接 | 连接的是 `employees` 和 `departments` 两张不同表     |
| 按结果保留规则 | 左外连接 | 使用 `LEFT JOIN`，保留左表全部记录                    |

所以它可以说是：

> 一个使用等值条件的非自连接左外连接查询。

## 2.2 分类1：等值连接 vs 非等值连接

这一类分类是从“连接条件长什么样”来区分。

它回答的问题是：

> 表与表之间，是通过相等关系连接，还是通过区间、大小比较等非相等关系连接？

## 2.2.1 等值连接

等值连接指的是：连接条件使用 `=`，把两个表中值相等的记录对应起来。

例如，员工表和部门表可以通过 `department_id` 关联：

```sql
SELECT
    e.employee_id,
    e.last_name,
    e.department_id,
    d.department_id,
    d.department_name
FROM employees e, departments d
WHERE e.department_id = d.department_id;
```

这里的连接条件是：

```sql
e.department_id = d.department_id
```

因为使用的是 `=`，所以这是等值连接。

这条 SQL 的含义是：

1. 从 `employees` 表中取员工资料。
2. 从 `departments` 表中取部门资料。
3. 只保留员工部门编号和部门表部门编号相等的记录。
4. 最后组合出员工与部门的对应结果。

### SQL99 写法

同样的逻辑，也可以写成 SQL99 风格：

```sql
SELECT
    e.employee_id,
    e.last_name,
    e.department_id,
    d.department_id,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这里依然是等值连接，因为判断标准不是看你用了 `WHERE` 还是 `ON`，而是看连接条件是否使用相等关系。

## 2.2.2 多个条件与 AND：连接条件和筛选条件要分清楚

实际查询中，`WHERE` 或 `ON` 中可能会出现多个条件。

需要注意：

> 不是所有用 `AND` 接在一起的条件都叫连接条件。

例如：

```sql
SELECT
    o.order_id,
    c.name,
    c.region,
    o.product_id,
    o.quantity
FROM orders o, customers c
WHERE o.customer_id = c.customer_id
  AND c.region = 'East';
```

这段 SQL 中有两个条件：

```sql
o.customer_id = c.customer_id
```

这是连接条件，因为它说明订单表和客户表如何对应。

```sql
c.region = 'East'
```

这是筛选条件，因为它不是用来说明两张表怎么连接，而是用来过滤客户地区。

所以这条 SQL 的执行逻辑可以理解为：

1. 先通过 `o.customer_id = c.customer_id` 建立订单与客户的对应关系。
2. 再通过 `c.region = 'East'` 筛选出 East 区域的客户。

### 多字段等值连接

有时两张表之间不只靠一个字段对应，而是需要多个字段同时相等。

例如：

```sql
SELECT
    od.order_id,
    od.product_id,
    s.shipment_id
FROM order_details od
JOIN shipments s
    ON od.order_id = s.order_id
   AND od.product_id = s.product_id;
```

这里有两个连接条件：

```sql
od.order_id = s.order_id
```

以及：

```sql
od.product_id = s.product_id
```

这仍然属于等值连接，因为连接关系都是通过 `=` 建立的。

## 2.2.3 区分重复的列名

多表查询中，如果多个表里存在同名列，就应该在列名前加上表名或表别名。

例如，`employees` 和 `departments` 表都可能有 `department_id` 字段。

如果直接写：

```sql
SELECT
    department_id
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

数据库可能无法判断你要的是：

```sql
e.department_id
```

还是：

```sql
d.department_id
```

因此，在多表查询中更建议明确写出字段来源：

```sql
SELECT
    e.last_name,
    d.department_name,
    e.department_id
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这样 SQL 的含义会更清楚，也能避免字段歧义。

## 2.2.4 表的别名

表别名的作用是让 SQL 更短、更清楚。

例如，不使用别名时：

```sql
SELECT
    employees.last_name,
    departments.department_name,
    employees.department_id
FROM employees, departments
WHERE employees.department_id = departments.department_id;
```

使用别名后：

```sql
SELECT
    e.last_name,
    d.department_name,
    e.department_id
FROM employees e, departments d
WHERE e.department_id = d.department_id;
```

这里：

* `employees e` 表示给 `employees` 表起别名 `e`。
* `departments d` 表示给 `departments` 表起别名 `d`。

使用别名后，字段来源更容易阅读。

### 使用表别名的注意点

如果已经给表起了别名，后续查询字段和条件中通常就应该继续使用别名，不要混用原表名。

例如：

```sql
FROM employees e
```

后面就应该写：

```sql
e.last_name
```

而不是：

```sql
employees.last_name
```

在多表查询中，推荐养成一个习惯：

> 只要 SQL 涉及多张表，字段名前尽量加上表别名。

这样可以避免字段歧义，也可以让 SQL 在后续维护时更安全。

## 2.2.5 连接多个表

当查询涉及三张或更多表时，核心原则是：

> 连接 `n` 张表，通常至少需要 `n - 1` 个连接条件。

例如，连接员工表、部门表和地点表：

```sql
SELECT
    e.last_name,
    d.department_name,
    l.city
FROM employees e, departments d, locations l
WHERE e.department_id = d.department_id
  AND d.location_id = l.location_id;
```

这条 SQL 中有三张表：

* `employees`
* `departments`
* `locations`

因此通常至少需要两个连接条件：

```sql
e.department_id = d.department_id
```

用来连接员工表和部门表。

```sql
d.location_id = l.location_id
```

用来连接部门表和地点表。

如果少了其中一个连接条件，就可能造成局部笛卡尔积，导致结果行数异常变多。

### SQL99 写法

同样的查询，使用 SQL99 风格可以写成：

```sql
SELECT
    e.last_name,
    d.department_name,
    l.city
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
JOIN locations l
    ON d.location_id = l.location_id;
```

这种写法的好处是：每个 `JOIN` 后面都紧跟着对应的 `ON` 条件，连接链路比较清楚。

## 2.2.6 非等值连接

非等值连接指的是：连接条件不是 `=`，而是范围、区间或其他比较关系。

例如，查询员工的薪资等级：

```sql
SELECT
    e.last_name,
    e.salary,
    j.grade_level
FROM employees e, job_grades j
WHERE e.salary BETWEEN j.lowest_sal AND j.highest_sal;
```

这里的连接条件是：

```sql
e.salary BETWEEN j.lowest_sal AND j.highest_sal
```

它不是判断两个字段是否相等，而是判断员工薪资是否落在某个薪资等级区间内。

所以这属于非等值连接。

### 非等值连接的理解重点

假设 `job_grades` 表内容如下：

| grade_level | lowest_sal | highest_sal |
| ----------- | ---------- | ----------- |
| A           | 1000       | 2999        |
| B           | 3000       | 5999        |
| C           | 6000       | 9999        |

如果某个员工薪资是 `5000`，那么它会匹配到：

```text
3000 <= 5000 <= 5999
```

也就是等级 `B`。

所以非等值连接常用于：

* 区间匹配
* 等级划分
* 范围查询
* 分段归类

## 2.3 分类2：自连接 vs 非自连接

这一类分类是从“参与连接的是不是同一张表”来区分。

它回答的问题是：

> 连接的双方，是不同表，还是同一张表扮演不同角色？

## 2.3.1 自连接

自连接指的是：同一张表通过不同别名，扮演不同角色后再进行连接。

最常见的例子是员工和经理。

员工来自 `employees` 表，经理其实也来自 `employees` 表。
因为经理本身也是员工，所以不需要另一张 `managers` 表。

假设 `employees` 表中有：

| employee_id | last_name | manager_id |
| ----------- | --------- | ---------- |
| 100         | King      | NULL       |
| 101         | Kochhar   | 100        |
| 102         | De Haan   | 100        |
| 103         | Hunold    | 102        |

如果要查询：

> 每个员工的名字，以及他的经理名字。

可以写成：

```sql
SELECT
    CONCAT(worker.last_name, ' works for ', manager.last_name) AS relation_info
FROM employees worker, employees manager
WHERE worker.manager_id = manager.employee_id;
```

这里虽然两边都是 `employees` 表，但它们的角色不同：

| 别名        | 角色 | 含义       |
| --------- | -- | -------- |
| `worker`  | 员工 | 当前被查询的员工 |
| `manager` | 经理 | 员工对应的上级  |

连接条件是：

```sql
worker.manager_id = manager.employee_id
```

意思是：

> 员工的经理编号，要等于另一个员工记录的员工编号。

这就是自连接。

### SQL99 写法

也可以写成：

```sql
SELECT
    worker.last_name AS worker_name,
    manager.last_name AS manager_name
FROM employees worker
JOIN employees manager
    ON worker.manager_id = manager.employee_id;
```

这条 SQL 仍然是自连接，因为判断自连接的关键不是语法形式，而是：

> 连接双方本质上是不是同一张表。

## 2.3.2 自连接练习：查询 Chen 的经理信息

例如，查询 `last_name` 包含 `Chen` 的员工，以及他的经理姓名：

```sql
SELECT
    worker.last_name AS worker_name,
    manager.last_name AS manager_name
FROM employees worker
JOIN employees manager
    ON worker.manager_id = manager.employee_id
WHERE worker.last_name LIKE '%Chen%';
```

这里：

* `worker` 和 `manager` 都来自 `employees` 表。
* `worker` 表示员工角色。
* `manager` 表示经理角色。
* `worker.manager_id = manager.employee_id` 建立员工与经理之间的对应关系。

所以这仍然是自连接。

## 2.3.3 非自连接

非自连接指的是：参与连接的是两张或多张不同的表。

例如：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这里连接的是：

* `employees` 表
* `departments` 表

两张表本身不同，所以这是非自连接。

非自连接是最常见的多表查询形式，例如：

* 员工表连接部门表。
* 订单表连接客户表。
* 订单明细表连接商品表。
* 学生表连接班级表。

## 2.3.4 自连接与非自连接怎么区分

可以用一句话判断：

> 如果连接双方本质上是同一张表，只是用了不同别名，就是自连接；如果连接的是不同表，就是非自连接。

| 查询场景      | 是否自连接 | 原因                     |
| --------- | ----- | ---------------------- |
| 员工查询自己的经理 | 是     | 员工和经理都来自 `employees` 表 |
| 商品分类查询父分类 | 是     | 当前分类和父分类都来自分类表         |
| 员工查询所属部门  | 否     | 员工和部门来自不同表             |
| 订单查询客户信息  | 否     | 订单和客户来自不同表             |

## 2.4 分类3：内连接 vs 外连接

这一类分类是从“结果集中是否保留没有匹配上的记录”来区分。

它回答的问题是：

> 如果某一行在另一张表中找不到匹配数据，要不要把它保留下来？

## 2.4.1 内连接

内连接只保留两张表中能够匹配成功的记录。

例如：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 只会返回：

> 员工表中 `department_id` 能在部门表中找到对应部门的员工记录。

如果某个员工没有部门编号，或者他的部门编号在 `departments` 表中找不到对应记录，那么这位员工不会出现在结果集中。

### 内连接可以和其他分类维度组合

内连接不是只能搭配等值连接。

例如：

```sql
SELECT
    e.last_name,
    e.salary,
    j.grade_level
FROM employees e
JOIN job_grades j
    ON e.salary BETWEEN j.lowest_sal AND j.highest_sal;
```

这条 SQL 是：

* 非等值连接，因为连接条件是区间匹配。
* 非自连接，因为连接的是 `employees` 和 `job_grades`。
* 内连接，因为只保留匹配到薪资等级的员工。

所以不要把“内连接”等同于“等值连接”。
它们属于不同分类维度。

## 2.4.2 外连接

外连接除了返回匹配成功的记录以外，还会保留某一侧或两侧没有匹配上的记录。

没有匹配上的那一侧，对应字段会用 `NULL` 填充。

外连接主要包括：

* 左外连接：`LEFT JOIN`
* 右外连接：`RIGHT JOIN`
* 满外连接：`FULL OUTER JOIN`

## 2.4.3 左外连接

左外连接会保留左表中的所有记录。

例如：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 的含义是：

> 无论员工是否能匹配到部门，员工资料都会保留下来。

如果某个员工没有对应部门，那么结果中：

```text
d.department_name
```

会显示为 `NULL`。

这里可以把左表 `employees` 理解为“保留表”。

## 2.4.4 右外连接

右外连接会保留右表中的所有记录。

例如：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 的含义是：

> 无论某个部门下是否有员工，只要它存在于部门表中，就会被保留下来。

如果某个部门没有员工，那么结果中：

```text
e.last_name
```

会显示为 `NULL`。

这里可以把右表 `departments` 理解为“保留表”。

## 2.4.5 满外连接

满外连接会保留左右两张表的全部记录。

它包含三部分：

1. 左右表匹配成功的记录。
2. 左表有、右表没有匹配的记录。
3. 右表有、左表没有匹配的记录。

标准 SQL 中可以写成：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
FULL OUTER JOIN departments d
    ON e.department_id = d.department_id;
```

但是需要注意：

> MySQL 不支持直接使用 `FULL OUTER JOIN`。

在 MySQL 中，通常需要用 `LEFT JOIN`、`RIGHT JOIN` 和 `UNION` 或 `UNION ALL` 来模拟满外连接。
这一部分会在后面的「7 种 SQL JOINS 的实现」中进一步说明。

## 2.4.6 外连接中的 WHERE 注意事项

外连接最容易出错的地方是：

> 明明写了 `LEFT JOIN`，但在 `WHERE` 中筛选右表字段，结果把未匹配的数据过滤掉了。

例如：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'IT';
```

这条 SQL 虽然写了 `LEFT JOIN`，但 `WHERE d.department_name = 'IT'` 会把右表为 `NULL` 的记录过滤掉。

也就是说，那些没有匹配到部门的员工，本来可以通过 `LEFT JOIN` 保留下来，但最后又被 `WHERE` 条件排除了。

所以，学习外连接时要先记住：

* `ON` 主要负责连接条件。
* `WHERE` 主要负责最终结果过滤。
* 外连接后如果在 `WHERE` 中过滤非保留表字段，要小心把外连接效果变成类似内连接。

初学阶段不需要一次掌握所有细节，但要知道这个问题存在。

## 2.4.7 旧式 Oracle `(+)` 外连接写法

有些旧 SQL 或教材中会看到类似写法：

```sql
SELECT
    last_name,
    department_name
FROM employees, departments
WHERE employees.department_id = departments.department_id(+);
```

这表示一种旧式外连接写法，常见于 Oracle 旧代码中。

需要注意：

> MySQL 不支持这种 `(+)` 外连接写法。

在 MySQL 中，应该使用 SQL99 风格：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

如果看到 `(+)`，你只需要知道它大致是在表达外连接概念即可。
实际写 MySQL 时，不要使用这种语法。

## 2.5 如何判断一条 SQL 属于哪几类

看到一条多表查询时，可以按下面三个问题判断。

### 第一步：看连接条件

先问：

> 它是用 `=` 连接，还是用区间、大小比较等条件连接？

如果是：

```sql
e.department_id = d.department_id
```

就是等值连接。

如果是：

```sql
e.salary BETWEEN j.lowest_sal AND j.highest_sal
```

就是非等值连接。

### 第二步：看参与连接的表

再问：

> 连接双方是不是同一张表？

如果是：

```sql
FROM employees worker
JOIN employees manager
```

就是自连接。

如果是：

```sql
FROM employees e
JOIN departments d
```

就是非自连接。

### 第三步：看是否保留不匹配记录

最后问：

> 查询是否保留没有匹配上的记录？

如果是：

```sql
JOIN
```

通常是内连接。

如果是：

```sql
LEFT JOIN
```

就是左外连接。

如果是：

```sql
RIGHT JOIN
```

就是右外连接。

如果是：

```sql
FULL OUTER JOIN
```

就是满外连接，但 MySQL 不直接支持。

## 2.6 综合判断示例

### 示例 1：普通员工部门查询

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

判断：

| 分类维度  | 类型   | 原因                                  |
| ----- | ---- | ----------------------------------- |
| 按连接条件 | 等值连接 | `e.department_id = d.department_id` |
| 按参与表  | 非自连接 | `employees` 和 `departments` 是不同表    |
| 按结果保留 | 内连接  | `JOIN` 只保留匹配成功的记录                   |

### 示例 2：员工经理查询

```sql
SELECT
    worker.last_name AS worker_name,
    manager.last_name AS manager_name
FROM employees worker
JOIN employees manager
    ON worker.manager_id = manager.employee_id;
```

判断：

| 分类维度  | 类型   | 原因                                        |
| ----- | ---- | ----------------------------------------- |
| 按连接条件 | 等值连接 | `worker.manager_id = manager.employee_id` |
| 按参与表  | 自连接  | 两边都是 `employees` 表                        |
| 按结果保留 | 内连接  | `JOIN` 只保留匹配成功的记录                         |

### 示例 3：员工薪资等级查询

```sql
SELECT
    e.last_name,
    e.salary,
    j.grade_level
FROM employees e
JOIN job_grades j
    ON e.salary BETWEEN j.lowest_sal AND j.highest_sal;
```

判断：

| 分类维度  | 类型    | 原因                              |
| ----- | ----- | ------------------------------- |
| 按连接条件 | 非等值连接 | 使用 `BETWEEN ... AND ...`        |
| 按参与表  | 非自连接  | `employees` 和 `job_grades` 是不同表 |
| 按结果保留 | 内连接   | `JOIN` 只保留匹配成功的记录               |

### 示例 4：保留所有员工的部门查询

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

判断：

| 分类维度  | 类型   | 原因                                  |
| ----- | ---- | ----------------------------------- |
| 按连接条件 | 等值连接 | `e.department_id = d.department_id` |
| 按参与表  | 非自连接 | `employees` 和 `departments` 是不同表    |
| 按结果保留 | 左外连接 | 保留左表 `employees` 的全部记录              |

## 常见混淆点

* 等值连接 / 非等值连接，关注的是“连接条件怎么写”。
* 自连接 / 非自连接，关注的是“参与连接的是不是同一张表”。
* 内连接 / 外连接，关注的是“结果是否保留不匹配记录”。
* 这三组分类不是同一层级，也不是互相排斥的关系。
* 一条 SQL 可以同时是等值连接、自连接、内连接。
* 一条 SQL 也可以同时是非等值连接、非自连接、内连接。
* 自连接不是特殊语法，本质上只是同一张表起了不同别名后再连接。
* 左外连接不等于左表独有；左外连接包含匹配成功的数据和左表未匹配的数据。
* 右外连接不等于右表独有；右外连接包含匹配成功的数据和右表未匹配的数据。
* 多表查询中，同名字段不加表名或别名，容易出现歧义错误。
* 连接多个表时，不是表越多越难，关键是理清每张表之间的关联链路。
* MySQL 不支持 `(+)` 外连接写法，这部分更适合作为旧代码阅读补充，不适合作为 MySQL 实战语法使用。

* 内连接和外连接最大的区别是什么？

## 一句话抓核心

多表查询的分类不是单一层级，而是三个观察维度：连接条件决定它是等值还是非等值，参与表决定它是自连接还是非自连接，结果保留规则决定它是内连接还是外连接。

## 小结

这一节需要记住：

* 按连接条件分，多表查询可以分为等值连接和非等值连接。
* 按参与表分，多表查询可以分为自连接和非自连接。
* 按结果保留规则分，多表查询可以分为内连接和外连接。
* 三种分类维度可以同时套用到同一条 SQL 上。
* 表别名是多表查询中的高频工具，尤其在自连接中几乎必不可少。
* 多表查询中字段同名时，建议使用表别名限定字段来源。
* 连接 `n` 张表，通常至少需要 `n - 1` 个连接条件。
* 外连接会保留某一侧或两侧未匹配的数据，并用 `NULL` 填充未匹配的一侧。
* MySQL 不支持旧式 Oracle `(+)` 外连接写法，实务中应使用 `LEFT JOIN`、`RIGHT JOIN` 等 SQL99 风格写法。
