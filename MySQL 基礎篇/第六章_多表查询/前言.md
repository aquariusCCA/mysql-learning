多表查询并不是简单地把多张表放进同一条 SQL 中，而是要根据表与表之间的业务关系，通过合适的关联字段，把原本分散在不同表中的数据组合成一个有意义的查询结果。

## 你会在这篇学到什么

- 多表查询也称为关联查询。
- 多表查询是让两个或更多个表一起完成查询操作。
- 多表查询的重点不是“表多”，而是“表之间要有可对应的关系”。
- 表与表之间通常会通过某个字段建立对应关系，这个字段称为关联字段。
- 关联字段可以是外键，也可以只是查询时用来建立对应关系的普通字段。
- 如果多表查询缺少有效的连接条件，就可能产生错误结果，例如笛卡尔积。

## 多表查询的定义

多表查询，也称为关联查询，指的是在一条 SQL 语句中同时使用两个或更多个表，并根据表与表之间的关联关系，把分散在不同表中的数据组合起来查询。

例如，员工姓名可能存放在 `employees` 表中，而部门名称可能存放在 `departments` 表中。  
如果想查询「员工姓名及其所在部门名称」，就不能只查单一张表，而需要把员工表和部门表一起查询。

这类查询就属于多表查询。

## 进行多表查询的前提

进行多表查询之前，必须先确认这些表之间是否存在可以对应起来的关系。

这种关系可能是一对一、一对多，或者多对多。例如：

- 一个员工通常属于一个部门。
- 一个部门可以拥有多个员工。
- 一个订单属于一个客户。
- 一个客户可以拥有多笔订单。

也就是说，多表查询的重点不是把多张表机械地写在一起，而是要先理解：

> 这些表之间到底靠什么字段对应起来？

如果表之间没有明确的对应关系，却直接把多张表放在一起查询，就很容易得到无意义的组合结果。

## 关联字段与外键

表与表之间通常会通过某个字段建立对应关系，这个字段可以称为关联字段。

例如：

- `employees.department_id`
- `departments.department_id`

这两个字段都表示部门编号，因此可以用来把员工表和部门表关联起来。

需要注意的是：

> 关联字段不一定等于外键。

外键是一种数据库层面的约束，用来保证数据之间的引用关系是合法的。  
而关联字段是查询层面的概念，只要两个字段在业务意义上可以对应，就可以在查询时用它们建立连接关系。

换句话说：

- 有外键，通常代表这个字段可以作为关联字段。
- 没有外键，也不代表两张表一定不能关联。
- 能不能关联，关键要看字段在业务语义上是否能对应。

不过在实际开发中，如果两个表之间存在稳定的主从关系，通常会建议设计清楚主键、外键或至少保持字段含义一致，这样可以降低查询错误和数据不一致的风险。

## 示例：员工表与部门表

员工表和部门表是一个典型的一对多关系：

- 一个部门可以有多个员工。
- 一个员工通常只属于一个部门。

假设有两张表：

`employees` 表：

| employee_id | last_name | department_id |
| --- | --- | --- |
| 100 | King | 90 |
| 101 | Kochhar | 90 |
| 103 | Hunold | 60 |

`departments` 表：

| department_id | department_name |
| --- | --- |
| 60 | IT |
| 90 | Executive |

这两张表可以通过 `department_id` 进行关联：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
````

这条 SQL 的含义是：

1. 从 `employees` 表中取得员工资料。
2. 从 `departments` 表中取得部门资料。
3. 通过 `department_id` 找出员工所属的部门。
4. 最后组合出「员工姓名 + 部门名称」的结果。

这里的核心不是 `JOIN` 这个关键字本身，而是：

```sql
e.department_id = d.department_id
```

这一句才是真正说明两张表如何对应的连接条件。

## 本章学习主线

学习多表查询时，可以按下面这条主线理解：

1. 先知道多表查询是为了把分散在不同表中的相关数据组合起来。
2. 再理解如果没有连接条件，就会出现笛卡尔积。
3. 接着学习连接条件的不同写法，例如 SQL92 的 `WHERE` 写法和 SQL99 的 `JOIN ... ON` 写法。
4. 然后区分不同类型的连接，例如等值连接、非等值连接、自连接、内连接和外连接。
5. 最后再学习 `UNION`、七种 JOIN 图示以及多表查询的性能注意事项。

只要抓住一个核心原则，后面的内容就会清楚很多：

> 多表查询的本质，是根据正确的关联字段和连接条件，把多张表中彼此相关的数据组合起来。

## 常见混淆点

* 多表查询不是把多个表写进 `FROM` 就结束了，还必须有合理的连接条件。
* 关联字段不一定必须建立外键，但字段的业务含义必须能对应。
* 外键偏向数据库约束，关联字段偏向查询逻辑。
* 多表查询如果缺少连接条件，可能会产生笛卡尔积，导致结果行数异常暴增。
* 表之间能不能关联，不只看字段名称是否相同，更要看字段代表的业务含义是否一致。
