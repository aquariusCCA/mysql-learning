# 第六章_多表查询

这一章开始把单表查询推进到「多个表如何一起查」。

本章重点不只是学会 `JOIN` 语法，而是建立一套多表查询的判断框架：

- 多张表为什么可以一起查询？
- 表与表之间应该靠什么字段对应？
- 缺少连接条件时，为什么会产生笛卡尔积？
- SQL92 和 SQL99 的多表查询写法有什么差别？
- `WHERE`、`ON`、`USING` 分别适合什么场景？
- 等值连接、非等值连接、自连接、内连接、外连接分别从什么角度分类？
- 常见 JOIN 图示分别应该如何落成 SQL？
- `UNION` 和 `UNION ALL` 是如何合并查询结果的？
- MySQL 不支持 `FULL OUTER JOIN` 时，应该如何模拟满外连接？
- 多表查询写出来后，还要注意哪些可读性和性能问题？

学完这一章后，目标不是只会背 `INNER JOIN`、`LEFT JOIN`、`RIGHT JOIN`，而是能够看懂一条复杂 SQL 中每张表为什么出现、每个连接条件的作用，以及最终结果集保留的是哪一部分数据。

## 本章在整体学习地图中的位置

- 第三章建立了 `SELECT`、`FROM`、`WHERE` 的基础查询骨架。
- 第四章补上了比较、范围、空值与逻辑判断所需的运算符。
- 第五章开始处理结果如何排序与分页。
- 第六章进一步进入「多个表之间怎么建立正确关系」，是后续子查询、聚合统计、复杂报表查询和业务 SQL 优化的重要前置基础。

如果前面章节解决的是「从一张表中取出想要的数据」，那么这一章解决的是：

> 如何把多张表中彼此相关的数据，组合成一个有意义的查询结果。

## 本章学习主线

建议按下面这条主线理解整章内容：

1. 先理解多表查询的本质：不是把多张表放在一起，而是通过关联字段建立正确关系。
2. 再通过错误案例理解笛卡尔积：缺少连接条件时，结果行数会异常暴增。
3. 接着建立分类地图：多表查询可以从连接条件、参与表、结果保留规则三个维度分类。
4. 然后学习 SQL99 的 `JOIN ... ON` 写法：把连接关系写得比 SQL92 更清楚。
5. 再学习外连接：理解保留左表、保留右表、保留未匹配数据的差异。
6. 接着学习 `UNION`：理解它是合并结果集，不是连接表。
7. 再用 JOIN 图示整理 7 种常见结果集。
8. 最后补充 `NATURAL JOIN` 与 `USING`，并在章节小结中回收全章重点。

## 本章核心提醒

- 多表查询的核心不是「表很多」，而是「表之间有正确关系」。
- 只把多张表写进 `FROM`，不代表完成了正确连接。
- 缺少连接条件时，很容易产生笛卡尔积。
- SQL99 的 `JOIN ... ON` 通常比 SQL92 逗号连接更清楚。
- `ON` 主要写连接条件，`WHERE` 主要写最终筛选条件。
- 外连接中，条件放在 `ON` 或 `WHERE` 可能影响最终结果。
- `JOIN` 默认就是 `INNER JOIN`。
- `LEFT JOIN` 保留左表全部记录，`RIGHT JOIN` 保留右表全部记录。
- 左表独有和右表独有都需要外连接配合 `IS NULL`。
- `UNION` 是合并结果集，不是连接表。
- `UNION` 会去重，`UNION ALL` 不会去重。
- `USING` 只能用于同名字段的等值连接。
- `NATURAL JOIN` 会自动使用所有同名字段，正式开发中要谨慎。
- 多表连接要控制表数量，明确字段来源，并关注关联字段索引。
