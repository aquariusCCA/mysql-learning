## 本节导读

前面已经学过 SQL99 最常见、也最推荐掌握的连接写法：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这种写法的优点是：连接条件完全写出来，语义清楚，也方便维护。

SQL99 还提供了两种简化写法：

1. `NATURAL JOIN`
2. `JOIN ... USING (...)`

它们都和「同名字段上的等值连接」有关。

不过要特别注意：

> `NATURAL JOIN` 和 `USING` 不是新的连接类型，而是对等值连接语法的简化。

也就是说，它们不是在发明一种新的 JOIN 逻辑，而是在特定条件下，让你少写一部分 `ON a.column = b.column`。

本节的重点不是鼓励大量使用这两种写法，而是让你看得懂它们，并知道什么时候可以用、什么时候不建议用。

## 你会在这篇学到什么

* SQL99 提供了 `NATURAL JOIN` 和 `USING` 这两种简化连接写法。
* `NATURAL JOIN` 会自动找出两张表中所有同名字段，并对这些字段做等值连接。
* `USING` 允许你明确指定某个同名字段作为等值连接依据。
* `NATURAL JOIN` 和 `USING` 本质上都属于等值连接的语法简化。
* `NATURAL JOIN` 自动化程度高，但可控性较差，实际开发中要谨慎使用。
* `USING` 比 `NATURAL JOIN` 更可控，但只能用于两张表中同名的字段。
* `JOIN ... ON ...` 最通用，也最清楚，通常仍然是实际开发中的优先写法。
* 看到 `NATURAL JOIN` 或 `USING` 时，要能还原成普通的 `JOIN ... ON ...` 来理解。

## 快速回查表

| 写法                          | 核心作用           | 字段选择方式        | 可控性 | 适合场景                  |
| --------------------------- | -------------- | ------------- | --- | --------------------- |
| `JOIN ... ON ...`           | 明确写出连接条件       | 手动指定任意连接条件    | 最高  | 最通用、最清楚、最推荐           |
| `NATURAL JOIN`              | 自动按所有同名字段做等值连接 | 数据库自动找出所有同名字段 | 最低  | 了解即可，适合非常简单且字段设计明确的场景 |

## 6.1 为什么 SQL99 还要提供简化写法

普通 SQL99 连接写法如下：

```sql id="sqj0ui"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 很清楚，但在某些情况下，连接条件会显得重复。

例如，两张表的关联字段都叫：

```sql id="jl057m"
department_id
```

于是连接条件写成：

```sql id="nykac6"
e.department_id = d.department_id
```

字段名称完全相同，只是表别名不同。

为了简化这类「同名字段上的等值连接」，SQL99 提供了：

```sql id="eejmou"
NATURAL JOIN
```

和：

```sql id="s6n858"
USING
```

不过要记住：

> 简化写法的前提，是字段名称和业务语义都要对应。

字段同名不一定代表业务含义一定相同。
这也是 `NATURAL JOIN` 和 `USING` 必须谨慎使用的原因。

## 6.2 NATURAL JOIN 自然连接

`NATURAL JOIN` 可以翻译成自然连接。

它的核心规则是：

> 数据库会自动找出两张表中所有同名字段，并把这些同名字段全部作为等值连接条件。

也就是说，写：

```sql id="i7xlvf"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees
NATURAL JOIN departments;
```

数据库会自动检查 `employees` 和 `departments` 两张表中有哪些同名字段。

假设两张表共同拥有：

```text id="v4vzoz"
department_id
```

那么它大致可以理解成：

```sql id="u9as9j"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这里的重点是：

```text id="b0t0g7"
自动寻找同名字段
+
自动做等值连接
```

## 6.2.1 NATURAL JOIN 的基本特点

`NATURAL JOIN` 有几个特点：

* 不需要手动写 `ON` 条件。
* 不需要手动写 `USING` 字段。
* 会自动使用两张表中所有同名字段。
* 本质上仍然是等值连接。
* 适用范围比 `JOIN ... ON ...` 窄。
* 依赖表结构，一旦表结构变化，查询结果可能跟着变化。

可以把它理解成：

```text id="gqzvap"
让我自动找同名字段，然后自动连接。
```

## 6.2.2 NATURAL JOIN 的普通写法对照

假设 `employees` 和 `departments` 都有 `department_id` 字段。

自然连接：

```sql id="i6x3oy"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees
NATURAL JOIN departments;
```

可以大致理解成：

```sql id="jc5w92"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

也就是说，`NATURAL JOIN` 自动帮你补上了：

```sql id="ij0p5u"
ON e.department_id = d.department_id
```

不过这只是两张表**只有一个合理同名字段**时的情况。

一旦同名字段不只一个，就要特别小心。

## 6.3 NATURAL JOIN 的风险

`NATURAL JOIN` 最大的问题是：它太自动了。

自动化会让 SQL 变短，但也会让连接条件变得不明显。

## 6.3.1 风险一：会自动使用所有同名字段

假设 `employees` 和 `departments` 中不只都有：

```text id="n9g6ut"
department_id
```

还都有：

```text id="v8shk0"
manager_id
```

那么：

```sql id="dd9c16"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees
NATURAL JOIN departments;
```

不只是按照 `department_id` 连接，而可能等价于：

```sql id="skyh2j"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
   AND e.manager_id = d.manager_id;
```

这里就出现一个问题：

> 你原本可能只想用 `department_id` 连接，但 `NATURAL JOIN` 会把所有同名字段都纳入连接条件。

这可能导致结果变少，甚至查不到预期数据。

## 6.3.2 风险二：表结构变化会影响查询结果

假设一开始两张表只有一个同名字段：

```text id="lob79t"
department_id
```

所以自然连接结果正常。

后来某一天，两个表都新增了一个同名字段：

```text id="cd2nl3"
status
```

那么原本的：

```sql id="x19b7a"
FROM employees
NATURAL JOIN departments
```

可能会自动变成类似：

```sql id="jnkrc3"
ON employees.department_id = departments.department_id
AND employees.status = departments.status
```

这会改变查询结果。

重点是：SQL 本身没有改，但表结构改变后，连接条件被自动改变了。

这就是 `NATURAL JOIN` 在实务开发中比较危险的地方。

## 6.3.3 风险三：可读性不一定真的更好

`NATURAL JOIN` 的 SQL 很短：

```sql id="el1lft"
FROM employees
NATURAL JOIN departments
```

但读代码的人必须先知道两张表有哪些同名字段，才能判断真实连接条件。

这会造成阅读成本。

相比之下：

```sql id="xd6qy3"
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
```

虽然多写了一点，但连接条件完全可见，维护时更安全。

## 6.3.4 NATURAL JOIN 的实务建议

建议这样看待 `NATURAL JOIN`：

* 学习时要知道它的规则。
* 阅读旧代码时要能看懂它。
* 简单练习中可以尝试使用。
* 正式项目中不建议优先使用。
* 需要明确、稳定、可维护的 SQL 时，优先使用 `JOIN ... ON ...`。

一句话：

```text id="zmc3jl"
NATURAL JOIN 可以看懂，但不建议依赖。
```

## 6.4 USING 连接

`USING` 也是 SQL99 提供的简化写法。

它的核心规则是：

> 当两张表中有同名字段时，可以用 `USING (字段名)` 指定这个同名字段作为等值连接条件。

例如：

```sql id="idvy4g"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees e
JOIN departments d
    USING (department_id);
```

这表示：

> 使用两张表中同名的 `department_id` 字段进行等值连接。

它大致等价于：

```sql id="hj5p2t"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

## 6.4.1 USING 的基本特点

`USING` 有几个特点：

* 必须配合 `JOIN` 使用。
* 括号中写的是字段名。
* 字段必须同时存在于两张表中。
* 两边字段名称必须相同。
* 本质上是等值连接。
* 比 `NATURAL JOIN` 更可控，因为连接字段由你指定。
* 不如 `ON` 通用，因为 `ON` 可以写不同名字段、多个条件、非等值条件。

可以把它理解成：

```text id="awecqx"
我明确指定要用哪个同名字段连接。
```

## 6.4.2 USING 的普通写法对照

`USING` 写法：

```sql id="j5n9ke"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees e
JOIN departments d
    USING (department_id);
```

对应的 `ON` 写法：

```sql id="h0ehix"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

所以：

```sql id="p6xcme"
USING (department_id)
```

可以理解成：

```sql id="s7x3oq"
ON e.department_id = d.department_id
```

前提是：

```text id="uo2r0s"
employees 和 departments 中都存在 department_id 字段
```

## 6.4.3 USING 可以指定多个字段

`USING` 也可以指定多个同名字段。

例如：

```sql id="j0iwqv"
SELECT
    *
FROM table_a a
JOIN table_b b
    USING (id, type);
```

大致等价于：

```sql id="n7x65c"
SELECT
    *
FROM table_a a
JOIN table_b b
    ON a.id = b.id
   AND a.type = b.type;
```

也就是说：

```sql id="iqd6q9"
USING (id, type)
```

代表：

```text id="xpdshf"
id 相等
并且
type 相等
```

这仍然是等值连接。

## 6.5 USING 和 ON 的对应关系

`USING` 可以看成 `ON` 的简写，但不是所有 `ON` 都能改写成 `USING`。

## 6.5.1 可以改成 USING 的情况

当连接条件是：

```sql id="iz7t3a"
a.department_id = b.department_id
```

而且两边字段同名时，可以写成：

```sql id="kvbf08"
USING (department_id)
```

例如：

```sql id="z7h5wp"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

可以简化为：

```sql id="gr71qm"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees e
JOIN departments d
    USING (department_id);
```

## 6.5.2 不能改成 USING 的情况一：字段不同名

例如：

```sql id="vi0qcw"
SELECT
    e.last_name,
    l.city
FROM employees e
JOIN locations l
    ON e.location_code = l.location_id;
```

这里两边字段名不同：

```text id="zovjsk"
location_code
location_id
```

所以不能写成：

```sql id="ofkhdp"
USING (location_id)
```

这时应该使用 `ON`。

## 6.5.3 不能改成 USING 的情况二：非等值连接

例如：

```sql id="e1ais4"
SELECT
    e.last_name,
    e.salary,
    j.grade_level
FROM employees e
JOIN job_grades j
    ON e.salary BETWEEN j.lowest_sal AND j.highest_sal;
```

这是非等值连接。

它不能写成 `USING`，因为 `USING` 只能表达同名字段的等值连接。

## 6.5.4 不能改成 USING 的情况三：连接条件包含额外逻辑

例如：

```sql id="o9jx87"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
   AND d.status = 'ACTIVE';
```

这里除了连接条件：

```sql id="sf62z6"
e.department_id = d.department_id
```

还有额外限制：

```sql id="utyeix"
d.status = 'ACTIVE'
```

这种情况下可以把等值部分改成 `USING`，但额外条件仍然要另外写：

```sql id="w89tja"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    USING (department_id)
WHERE d.status = 'ACTIVE';
```

或者继续使用 `ON`：

```sql id="d09rpy"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
   AND d.status = 'ACTIVE';
```

在需要强调连接逻辑完整性时，`ON` 通常更清楚。

## 6.6 NATURAL JOIN、USING、ON 的区别

三者都可以出现在 SQL99 多表查询中，但可控性不同。

| 比较点        | `JOIN ... ON ...` | `JOIN ... USING (...)` | `NATURAL JOIN` |
| ---------- | ----------------- | ---------------------- | -------------- |
| 是否需要手写连接条件 | 需要                | 需要指定字段名                | 不需要            |
| 字段是否必须同名   | 不需要               | 必须同名                   | 自动找所有同名字段      |
| 是否支持非等值连接  | 支持                | 不支持                    | 不支持            |
| 是否支持复杂条件   | 支持                | 有限制                    | 不适合            |
| 可控性        | 最高                | 中等                     | 最低             |
| 可读性        | 条件最明确             | 简洁但有限制                 | 短，但隐藏条件        |
| 实务推荐度      | 最高                | 可在简单场景使用               | 谨慎使用           |

## 6.6.1 用同一个例子比较

### 写法一：JOIN ... ON

```sql id="uu4rno"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

特点：

```text id="swowwu"
连接条件完全可见，最清楚。
```

### 写法二：JOIN ... USING

```sql id="jfbtjs"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees e
JOIN departments d
    USING (department_id);
```

特点：

```text id="u37r4u"
明确指定 department_id 这个同名字段，写法较短。
```

### 写法三：NATURAL JOIN

```sql id="cd7j4h"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees
NATURAL JOIN departments;
```

特点：

```text id="vz92el"
连接字段由数据库自动判断，写法最短，但连接条件最不明显。
```

## 6.7 实务建议

实际开发中，可以按下面优先级选择。

## 6.7.1 优先使用 JOIN ... ON

最推荐：

```sql id="k0x4g8"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

原因：

* 连接条件清楚。
* 不依赖字段是否同名。
* 可以处理复杂条件。
* 可以处理非等值连接。
* 表结构变化时，不容易悄悄改变连接条件。
* 代码审查和维护更容易。

## 6.7.2 简单同名字段连接可以使用 USING

可以接受：

```sql id="x1mcmt"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees e
JOIN departments d
    USING (department_id);
```

适合场景：

* 两张表确实有同名字段。
* 只需要基于这个同名字段做等值连接。
* 团队代码规范允许使用 `USING`。
* 查询逻辑简单，不会造成阅读困难。

## 6.7.3 谨慎使用 NATURAL JOIN

不优先推荐：

```sql id="e3j909"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees
NATURAL JOIN departments;
```

原因：

* 连接条件被隐藏。
* 会自动使用所有同名字段。
* 表结构变化可能改变查询结果。
* 阅读 SQL 时必须先知道两张表的字段结构。
* 维护风险较高。

更稳妥的写法是：

```sql id="o18ffd"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

## 6.8 如何阅读这类 SQL

看到 `NATURAL JOIN` 或 `USING` 时，不要只看语法表面，要把它还原成普通的 `JOIN ... ON ...` 来理解。

## 6.8.1 阅读 NATURAL JOIN

看到：

```sql id="nnnpwg"
FROM employees
NATURAL JOIN departments
```

要问：

1. `employees` 和 `departments` 有哪些同名字段？
2. 这些同名字段是不是都应该参与连接？
3. 这些字段的业务含义是否一致？
4. 表结构未来变化会不会影响结果？

可以在脑中还原成：

```sql id="ci2hgv"
FROM employees e
JOIN departments d
    ON e.同名字段1 = d.同名字段1
   AND e.同名字段2 = d.同名字段2
```

## 6.8.2 阅读 USING

看到：

```sql id="iajm4i"
FROM employees e
JOIN departments d
    USING (department_id)
```

要还原成：

```sql id="io1b57"
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
```

看到：

```sql id="j186o7"
USING (id, type)
```

要还原成：

```sql id="nwkvpa"
ON a.id = b.id
AND a.type = b.type
```

## 6.9 常见错误示例

### 错误 1：以为 NATURAL JOIN 只会连接一个字段

错误理解：

```text id="w4xar0"
NATURAL JOIN 会自动找一个最合适的字段连接。
```

正确理解：

```text id="pvobgq"
NATURAL JOIN 会自动使用所有同名字段做等值连接。
```

如果同名字段有多个，它们都会参与连接。

## 错误 2：同名字段就一定适合 USING

例如两张表都有：

```text id="t9bjj2"
status
```

但一个表示员工状态，一个表示部门状态。

这时虽然字段同名，但业务含义不同，不应该随便写：

```sql id="db9ea0"
USING (status)
```

字段能不能连接，不只看名字，还要看业务含义。

## 错误 3：不同名字段误用 USING

错误写法：

```sql id="scmy4r"
SELECT
    *
FROM employees e
JOIN locations l
    USING (location_id);
```

假设员工表字段叫：

```text id="wu9qkb"
location_code
```

地点表字段叫：

```text id="wy0or9"
location_id
```

字段不同名，就不能直接使用 `USING (location_id)` 表达这段连接。

应该写：

```sql id="y7ydko"
SELECT
    *
FROM employees e
JOIN locations l
    ON e.location_code = l.location_id;
```

## 错误 4：为了少写 SQL 而牺牲可读性

不推荐：

```sql id="d3d6if"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees
NATURAL JOIN departments;
```

更推荐：

```sql id="v5zr4n"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

少写几行 SQL，不一定代表更好。
多表查询最重要的是连接关系清楚。

## 常见混淆点

* `NATURAL JOIN` 和 `USING` 都不是新的连接类型，本质上仍然是在做等值连接。
* `NATURAL JOIN` 会自动使用所有同名字段，不是只挑一个字段。
* `USING` 必须指定同名字段，不能用于不同名字段。
* `USING` 只能表达同名字段上的等值连接，不能表达非等值连接。
* `JOIN ... ON ...` 可以处理同名字段，也可以处理不同名字段。
* `JOIN ... ON ...` 可以写等值连接，也可以写非等值连接。
* `NATURAL JOIN` 写法最短，但隐藏了连接条件。
* `USING` 比 `NATURAL JOIN` 更可控，但不如 `ON` 通用。
* 字段同名不代表业务含义一定相同。
* 表结构变化可能改变 `NATURAL JOIN` 的真实连接条件。
* 实际开发中，追求可维护性时，通常优先使用 `JOIN ... ON ...`。

## 一句话抓核心

SQL99 的 `NATURAL JOIN` 和 `USING` 都是在简化等值连接：`NATURAL JOIN` 自动使用所有同名字段，`USING` 由你指定同名字段，而 `JOIN ... ON ...` 虽然写法较长，但连接条件最清楚、最可控，也最适合实际开发。

## 小结

这一节需要记住：

* SQL99 提供了 `NATURAL JOIN` 和 `USING` 两种简化连接写法。
* `NATURAL JOIN` 会自动按所有同名字段做等值连接。
* `USING` 会按你指定的同名字段做等值连接。
* 两者本质上都属于等值连接的语法简化。
* `NATURAL JOIN` 自动化程度最高，但隐藏了连接条件，维护风险较高。
* `USING` 比 `NATURAL JOIN` 更可控，但只能用于同名字段的等值连接。
* `JOIN ... ON ...` 最通用，可以处理同名字段、不同名字段、等值连接、非等值连接和复杂条件。
* 实务开发中，最推荐优先使用 `JOIN ... ON ...`；`USING` 可以作为简单场景的简化写法；`NATURAL JOIN` 以理解和阅读为主，正式项目中要谨慎使用。
