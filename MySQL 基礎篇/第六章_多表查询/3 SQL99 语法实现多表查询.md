## 本节导读

前面已经学过 SQL92 风格的多表查询：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e, departments d
WHERE e.department_id = d.department_id;
````

这种写法可以完成多表查询，但它有一个明显问题：

> 连接条件和普通筛选条件都写在 `WHERE` 中，查询一复杂就不容易分清楚哪些条件是用来连接表，哪些条件是用来过滤结果。

SQL99 提供了更清楚的写法：

```sql
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

SQL99 的核心优势是：

> 使用 `JOIN` 明确表示“连接哪张表”，使用 `ON` 明确表示“表与表之间如何连接”。

这样一来，多表查询的结构会更清楚，也更适合实际开发中的复杂 SQL。

第一次阅读时，建议按下面顺序理解：

1. 先掌握 `JOIN ... ON` 的基本结构。
2. 再理解 `JOIN` / `INNER JOIN` 为什么表示内连接。
3. 接着理解 `LEFT JOIN` 与 `RIGHT JOIN` 各自保留哪一侧数据。
4. 最后记住：标准 SQL 支持 `FULL OUTER JOIN`，但 MySQL 不直接支持。

## 你会在这篇学到什么

* SQL99 常用 `JOIN ... ON` 来实现多表查询。
* `JOIN` 用来说明要连接哪张表。
* `ON` 用来说明表与表之间的连接条件。
* `WHERE` 更适合放连接完成后的普通筛选条件。
* `JOIN` 和 `INNER JOIN` 含义相同，都表示内连接。
* `LEFT JOIN` 会保留左表全部记录。
* `RIGHT JOIN` 会保留右表全部记录。
* `LEFT OUTER JOIN` 可以简写为 `LEFT JOIN`，`RIGHT OUTER JOIN` 可以简写为 `RIGHT JOIN`。
* SQL 标准支持 `FULL OUTER JOIN`，但 MySQL 不支持直接使用。
* 外连接后如果在 `WHERE` 中过滤非保留表字段，可能会让外连接效果变得像内连接。

## 3.1 从 SQL92 到 SQL99

SQL92 风格的多表查询通常写成：

```sql id="kdo7rp"
SELECT
    e.last_name,
    d.department_name
FROM employees e, departments d
WHERE e.department_id = d.department_id;
```

这条 SQL 中：

```sql id="bug9t0"
e.department_id = d.department_id
```

是连接条件，用来说明员工表和部门表如何对应。

这种写法的问题是：
如果 `WHERE` 后面同时出现很多条件，就不容易分清哪些是连接条件，哪些是普通筛选条件。

例如：

```sql id="ulgnyq"
SELECT
    e.last_name,
    d.department_name
FROM employees e, departments d
WHERE e.department_id = d.department_id
  AND d.department_name = 'IT'
  AND e.salary > 5000;
```

这段 SQL 中：

```sql id="gvfvqf"
e.department_id = d.department_id
```

是连接条件。

而：

```sql id="p3a5jl"
d.department_name = 'IT'
```

以及：

```sql id="mix5h9"
e.salary > 5000
```

是筛选条件。

SQL99 写法会把连接条件独立到 `ON` 中：

```sql id="f9f5cb"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'IT'
  AND e.salary > 5000;
```

这样就比较清楚：

* `ON`：负责说明表怎么连接。
* `WHERE`：负责说明连接完成后要保留哪些结果。

这就是 SQL99 写法更适合复杂多表查询的主要原因。

## 3.2 SQL99 基本语法

SQL99 使用 `JOIN ... ON` 子句来创建连接，基本结构如下：

```sql id="wr43u0"
SELECT
    表1.字段,
    表2.字段
FROM 表1
JOIN 表2
    ON 表1.关联字段 = 表2.关联字段;
```

如果连接三张表，可以继续往后写：

```sql id="yh5yls"
SELECT
    表1.字段,
    表2.字段,
    表3.字段
FROM 表1
JOIN 表2
    ON 表1 和 表2 的连接条件
JOIN 表3
    ON 表2 和 表3 的连接条件;
```

可以把这种写法理解成一条连接链路：

```text id="c2y27j"
表1
  JOIN 表2 ON 表1 和 表2 怎么连
  JOIN 表3 ON 已连接结果 和 表3 怎么连
```

这种写法的好处是：

* 每一张新加入的表都通过 `JOIN` 明确出现。
* 每一段连接关系都通过对应的 `ON` 说明。
* 表越多时，可读性通常越明显优于 SQL92 的逗号连接。
* 连接条件和普通筛选条件更容易分开维护。

### 基本语法说明

```sql id="wq4a0u"
SELECT 字段列表
FROM A表
JOIN B表
    ON A表.关联字段 = B表.关联字段
WHERE 普通筛选条件;
```

其中：

* `FROM A表`：指定主查询的起始表。
* `JOIN B表`：指定要连接的另一张表。
* `ON`：指定 A 表和 B 表之间的连接条件。
* `WHERE`：指定最终结果的筛选条件。

需要注意：

> 正常业务查询中，写 `JOIN` 时通常应该写清楚 `ON` 条件。
> 如果真的想要所有组合，应明确使用 `CROSS JOIN`，而不是故意省略连接条件。

## 3.3 ON 与 WHERE 的职责区别

SQL99 中最重要的观念之一是：

> `ON` 主要写连接条件，`WHERE` 主要写连接后的筛选条件。

例如：

```sql id="vylkg9"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'IT';
```

这里：

```sql id="ub0spw"
ON e.department_id = d.department_id
```

表示员工表和部门表通过部门编号连接。

```sql id="khmzk1"
WHERE d.department_name = 'IT'
```

表示连接完成后，只保留部门名称为 `IT` 的结果。

### 3.3.1 为什么不建议把所有条件都塞进 WHERE

虽然在内连接中，有些条件放在 `ON` 或 `WHERE` 里，结果可能一样，但为了可读性，仍然建议分清职责：

```sql id="l9hvb8"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'IT';
```

比下面这种写法更清楚：

```sql id="lf97gc"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
WHERE e.department_id = d.department_id
  AND d.department_name = 'IT';
```

第二种写法虽然在某些数据库中可能可以执行，但它削弱了 SQL99 的可读性优势，也容易让人看不出哪一个条件是真正的连接条件。

### 3.3.2 实务建议

建议养成下面习惯：

* 表与表如何对应，写在 `ON`。
* 对最终结果的筛选，写在 `WHERE`。
* 涉及多张表时，字段名前尽量加表别名。
* 每增加一张表，就检查是否有对应的连接条件。
* 外连接场景下，更要小心 `WHERE` 条件是否会过滤掉 `NULL`。

## 3.4 内连接 INNER JOIN

内连接只返回两张表中能够匹配成功的记录。

也就是说：

> 只有满足连接条件的行，才会出现在结果集中。

## 3.4.1 INNER JOIN 基本语法

```sql id="z8fuaf"
SELECT 字段列表
FROM A表
INNER JOIN B表
    ON A表.关联字段 = B表.关联字段
WHERE 普通筛选条件;
```

`INNER` 可以省略。

所以：

```sql id="ywtyzu"
FROM A表 INNER JOIN B表
```

和：

```sql id="snf5fh"
FROM A表 JOIN B表
```

含义相同，都表示内连接。

## 3.4.2 查询员工及其部门信息

```sql id="k6dj76"
SELECT
    e.employee_id,
    e.last_name,
    e.department_id,
    d.department_id,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 的含义是：

1. 从 `employees` 表中取得员工数据。
2. 连接 `departments` 表。
3. 通过 `e.department_id = d.department_id` 找到员工所属部门。
4. 只保留能匹配成功的员工与部门记录。

如果某个员工的 `department_id` 在部门表中找不到对应部门，那么这位员工不会出现在结果中。

这就是内连接的特点：

```text id="rehrzg"
只保留匹配成功的数据
```

## 3.4.3 JOIN 与 INNER JOIN 的关系

下面两段 SQL 含义相同。

写法一：省略 `INNER`

```sql id="w9bj4k"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

写法二：显式写出 `INNER JOIN`

```sql id="gsrrmz"
SELECT
    e.last_name,
    d.department_name
FROM employees e
INNER JOIN departments d
    ON e.department_id = d.department_id;
```

实际开发中，很多人会直接写 `JOIN`，因为它默认就是内连接。

但如果你是初学者，刚开始可以写完整的 `INNER JOIN`，帮助自己记住它的含义。

## 3.5 多表连续 JOIN

SQL99 的优势在三张表以上时会更明显。

例如，查询：

> 员工编号、城市、部门名称。

这个需求涉及三张表：

* `employees`：员工表。
* `departments`：部门表。
* `locations`：地点表。

它们之间的关联链路是：

```text id="eyxzaa"
employees.department_id = departments.department_id
departments.location_id = locations.location_id
```

SQL 可以写成：

```sql id="mra0oe"
SELECT
    e.employee_id,
    l.city,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
JOIN locations l
    ON d.location_id = l.location_id;
```

这条 SQL 的阅读方式是：

1. 先从 `employees e` 出发。
2. 通过 `e.department_id = d.department_id` 连接 `departments d`。
3. 再通过 `d.location_id = l.location_id` 连接 `locations l`。
4. 最后输出员工编号、城市、部门名称。

### 3.5.1 多表 JOIN 的理解重点

每一段 `JOIN ... ON` 都应该回答一个问题：

> 当前加入的这张表，靠什么条件和前面的查询结果连接起来？

例如：

```sql id="u4vhg7"
JOIN departments d
    ON e.department_id = d.department_id
```

回答的是：

> 员工表如何连接部门表？

```sql id="ltmdug"
JOIN locations l
    ON d.location_id = l.location_id
```

回答的是：

> 部门表如何连接地点表？

这样写比 SQL92 的逗号连接更容易阅读：

```sql id="ev8b9d"
SELECT
    e.employee_id,
    l.city,
    d.department_name
FROM employees e, departments d, locations l
WHERE e.department_id = d.department_id
  AND d.location_id = l.location_id;
```

SQL92 也能完成查询，但当表越来越多时，所有连接条件都挤在 `WHERE` 中，可读性会下降。

## 3.6 外连接 OUTER JOIN

外连接除了返回匹配成功的记录以外，还会保留某一侧没有匹配到的记录。

未匹配的一侧，会用 `NULL` 补齐。

常见外连接包括：

* 左外连接：`LEFT JOIN`
* 右外连接：`RIGHT JOIN`
* 满外连接：`FULL OUTER JOIN`

其中：

* `LEFT OUTER JOIN` 可以简写为 `LEFT JOIN`。
* `RIGHT OUTER JOIN` 可以简写为 `RIGHT JOIN`。
* `FULL OUTER JOIN` 是标准 SQL 语法，但 MySQL 不支持直接使用。

## 3.6.1 左外连接 LEFT JOIN

左外连接会保留左表中的所有记录。

语法：

```sql id="zeu6wv"
SELECT 字段列表
FROM A表
LEFT JOIN B表
    ON A表.关联字段 = B表.关联字段
WHERE 普通筛选条件;
```

它的结果可以理解为：

```text id="pc92tp"
左表全部记录 + 右表中能匹配上的数据
```

如果右表找不到对应数据，右表字段会显示为 `NULL`。

### 查询所有员工及其部门

```sql id="gltuhi"
SELECT
    e.last_name,
    e.department_id,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 的含义是：

> 无论员工是否能匹配到部门，员工记录都会被保留。

如果某个员工没有部门，或者部门编号在部门表中找不到对应记录，那么：

```sql id="oig7rz"
d.department_name
```

会显示为 `NULL`。

所以左外连接适合这种需求：

> 主表数据必须完整显示，另一张表有对应数据就补上，没有对应数据就显示 `NULL`。

## 3.6.2 右外连接 RIGHT JOIN

右外连接会保留右表中的所有记录。

语法：

```sql id="nqmmvn"
SELECT 字段列表
FROM A表
RIGHT JOIN B表
    ON A表.关联字段 = B表.关联字段
WHERE 普通筛选条件;
```

它的结果可以理解为：

```text id="ezqwi2"
右表全部记录 + 左表中能匹配上的数据
```

如果左表找不到对应数据，左表字段会显示为 `NULL`。

### 查询所有部门及其员工

```sql id="f21byb"
SELECT
    e.last_name,
    e.department_id,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 的含义是：

> 无论某个部门下是否有员工，只要它存在于 `departments` 表中，就会出现在结果里。

如果某个部门没有员工，那么：

```sql id="jh000v"
e.last_name
```

会显示为 `NULL`。

## 3.6.3 左外连接与右外连接的选择

左外连接和右外连接本质上是对称的。

下面这条右外连接：

```sql id="ar8k8p"
SELECT
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id;
```

可以改写成左外连接：

```sql id="pjb8sz"
SELECT
    e.last_name,
    d.department_name
FROM departments d
LEFT JOIN employees e
    ON e.department_id = d.department_id;
```

两者核心意思相同：保留 `departments` 表全部记录。

实际开发中，很多人更偏好使用 `LEFT JOIN`，因为从左到右阅读时比较直观：

```text id="xtu4io"
以左表为主，往右补数据
```

所以如果你可以调整表顺序，通常可以优先使用 `LEFT JOIN` 表达主表保留逻辑。

## 3.6.4 满外连接 FULL OUTER JOIN

满外连接会保留左右两张表的全部记录。

它包含三部分：

1. 左右表匹配成功的数据。
2. 左表有、右表没有匹配的数据。
3. 右表有、左表没有匹配的数据。

标准 SQL 写法如下：

```sql id="luxn5s"
SELECT 字段列表
FROM A表
FULL OUTER JOIN B表
    ON A表.关联字段 = B表.关联字段;
```

或者：

```sql id="yb61ts"
SELECT 字段列表
FROM A表
FULL JOIN B表
    ON A表.关联字段 = B表.关联字段;
```

但是需要特别注意：

> MySQL 不支持直接使用 `FULL JOIN` 或 `FULL OUTER JOIN`。

如果在 MySQL 中需要满外连接效果，通常会使用：

```sql id="vj3e4c"
LEFT JOIN
UNION
RIGHT JOIN
```

或者根据是否需要保留重复数据，使用：

```sql id="uvck14"
LEFT JOIN
UNION ALL
RIGHT JOIN
```

具体写法会在后面的「7 种 SQL JOINS 的实现」中详细整理。

## 3.7 外连接中的 WHERE 陷阱

外连接最容易出错的地方是：

> 明明写了 `LEFT JOIN`，但在 `WHERE` 中过滤右表字段，导致未匹配的数据被过滤掉。

例如：

```sql id="j6a3nr"
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'IT';
```

这条 SQL 虽然使用了 `LEFT JOIN`，但 `WHERE d.department_name = 'IT'` 会过滤掉 `d.department_name` 为 `NULL` 的记录。

也就是说：

* 原本 `LEFT JOIN` 会保留没有匹配到部门的员工。
* 但这些员工的 `d.department_name` 是 `NULL`。
* `WHERE d.department_name = 'IT'` 不会保留 `NULL`。
* 最后这些未匹配员工又被过滤掉了。

结果上看，这条 SQL 很可能接近内连接效果。

### 3.7.1 如果你的需求是只查询 IT 部门员工

如果需求是：

> 只查询 IT 部门员工。

可以写：

```sql id="l5fohq"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'IT';
```

这种需求本来就只要匹配成功且部门为 IT 的员工，用内连接更直观。

### 3.7.2 如果你的需求是保留所有员工，但只补 IT 部门信息

如果需求是：

> 所有员工都要显示，但只有 IT 部门员工显示部门名称，其他员工的部门信息显示为 `NULL`。

可以把条件放到 `ON` 中：

```sql id="d49add"
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id
   AND d.department_name = 'IT';
```

这样 `LEFT JOIN` 仍然会保留所有员工。

区别在于：

* 是不是 IT 部门，影响的是右表 `departments` 能不能匹配上。
* 不会影响左表 `employees` 是否被保留。

### 3.7.3 外连接条件放 ON 还是 WHERE

可以先用下面规则判断：

| 需求                 | 条件位置                 | 说明                         |
| ------------------ | -------------------- | -------------------------- |
| 只保留符合条件的最终结果       | `WHERE`              | 用来过滤整个结果集                  |
| 保留左表全部记录，只限制右表匹配内容 | `ON`                 | 用来影响右表如何匹配                 |
| 查询没有匹配到右表的数据       | `WHERE 右表字段 IS NULL` | 常用于找左表独有数据                 |
| 不确定语义时             | 先写清楚需求               | 外连接中 `ON` 和 `WHERE` 可能影响结果 |

这部分是 SQL99 多表查询中非常重要的实务细节。

## 3.8 MySQL 中如何理解 FULL OUTER JOIN

标准 SQL 支持满外连接，但 MySQL 不支持直接写：

```sql id="qsu1cg"
SELECT
    *
FROM A
FULL OUTER JOIN B
    ON A.id = B.id;
```

在 MySQL 中，这种写法会报错。

如果需要实现类似满外连接的效果，常见思路是：

```sql id="wj7qfy"
SELECT 字段列表
FROM A
LEFT JOIN B
    ON A.id = B.id

UNION

SELECT 字段列表
FROM A
RIGHT JOIN B
    ON A.id = B.id;
```

这里的逻辑是：

1. `LEFT JOIN` 取得左表全部记录，以及右表匹配数据。
2. `RIGHT JOIN` 取得右表全部记录，以及左表匹配数据。
3. 通过 `UNION` 合并结果，并去除重复。

如果你明确知道不会重复，或者业务上允许重复，可以考虑 `UNION ALL`。
不过满外连接的具体模拟方式，通常会放到后面的 JOIN 图示与 `UNION` 章节中进一步说明。

## 3.9 SQL92 与 SQL99 对照

### 3.9.1 两表内连接

SQL92 写法：

```sql id="zzkuqr"
SELECT
    e.last_name,
    d.department_name
FROM employees e, departments d
WHERE e.department_id = d.department_id;
```

SQL99 写法：

```sql id="tdzcty"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

两者表达的业务逻辑相同：

```text id="y6dfbw"
员工表通过 department_id 连接部门表
```

但 SQL99 把连接关系写得更清楚。

### 3.9.2 三表内连接

SQL92 写法：

```sql id="n84x3j"
SELECT
    e.employee_id,
    l.city,
    d.department_name
FROM employees e, departments d, locations l
WHERE e.department_id = d.department_id
  AND d.location_id = l.location_id;
```

SQL99 写法：

```sql id="f26xmu"
SELECT
    e.employee_id,
    l.city,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
JOIN locations l
    ON d.location_id = l.location_id;
```

SQL99 的优势是：
每张表加入时，都能马上看到它对应的连接条件。

### 3.9.3 左外连接

SQL99 写法：

```sql id="mng6rt"
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

含义：

```text id="upzdm2"
保留 employees 表全部记录，departments 表有匹配就补上，没有匹配就显示 NULL。
```

### 3.9.4 右外连接

SQL99 写法：

```sql id="c7vn5r"
SELECT
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id;
```

含义：

```text id="vvlv5i"
保留 departments 表全部记录，employees 表有匹配就补上，没有匹配就显示 NULL。
```

## 3.10 如何阅读一条 SQL99 多表查询

看到一条 SQL99 多表查询时，可以按下面顺序阅读。

### 第一步：先看 FROM，从哪张表出发

```sql id="bybhyv"
FROM employees e
```

表示查询从员工表开始。

### 第二步：看 JOIN，加入了哪些表

```sql id="i8njjx"
JOIN departments d
JOIN locations l
```

表示又加入了部门表和地点表。

### 第三步：看每个 ON，理解连接链路

```sql id="z8ufte"
ON e.department_id = d.department_id
ON d.location_id = l.location_id
```

表示：

1. 员工表通过部门编号连接部门表。
2. 部门表通过地点编号连接地点表。

### 第四步：看 WHERE，理解最终筛选条件

```sql id="ue9hjr"
WHERE d.department_name = 'IT'
```

表示只保留部门名称为 `IT` 的结果。

### 第五步：看 SELECT，确认最终输出字段

```sql id="sorhpq"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name,
    l.city
```

表示最终输出员工编号、员工姓名、部门名称和城市。

这个阅读顺序可以帮助你避免一开始就被 `SELECT` 字段列表干扰，而是先看懂 SQL 的数据来源和连接关系。

## 常见混淆点

* `JOIN` 默认就是 `INNER JOIN`，不写 `INNER` 不代表变成别的连接类型。
* `ON` 主要负责写连接条件，`WHERE` 更适合放连接完成后的筛选条件。
* 在内连接中，有些筛选条件放在 `ON` 或 `WHERE` 结果可能一样，但为了可读性，建议分清职责。
* 在外连接中，条件放在 `ON` 或 `WHERE` 可能明显影响结果。
* `LEFT JOIN` 保留左表全部记录，不代表只查询左表独有数据。
* `RIGHT JOIN` 保留右表全部记录，不代表只查询右表独有数据。
* 左外连接和右外连接的区别，不在于连接条件怎么写，而在于保留哪一侧的全部记录。
* `LEFT OUTER JOIN` 和 `LEFT JOIN` 含义相同，`OUTER` 可以省略。
* `RIGHT OUTER JOIN` 和 `RIGHT JOIN` 含义相同，`OUTER` 可以省略。
* SQL 标准支持 `FULL OUTER JOIN`，但 MySQL 不支持直接使用。
* SQL99 不是推翻 SQL92，而是把连接语法写得更结构化、更容易阅读。
* 如果只是想要所有组合，应明确使用 `CROSS JOIN`，不要用省略 `ON` 条件的方式表达。
* 多表查询中应尽量使用表别名限定字段来源，避免字段歧义。

## 一句话抓核心

SQL99 的核心优势，是用 `JOIN` 明确写出要连接的表，用 `ON` 明确写出连接条件，让多表查询的连接链路比 SQL92 更清楚；其中 `JOIN` 表示内连接，`LEFT JOIN` / `RIGHT JOIN` 表示外连接，而 `FULL OUTER JOIN` 虽然是标准 SQL 语法，但 MySQL 不直接支持。

## 小结

这一节需要记住：

* SQL99 常用 `JOIN ... ON` 来实现多表查询。
* `JOIN` 用来说明连接哪张表，`ON` 用来说明表与表如何匹配。
* `WHERE` 更适合放连接完成后的普通筛选条件。
* `JOIN` 和 `INNER JOIN` 含义相同，都会返回匹配成功的记录。
* 三张表连续连接时，建议每个 `JOIN` 后面都紧跟对应的 `ON` 条件。
* `LEFT JOIN` 保留左表全部记录，右表没有匹配时用 `NULL` 补齐。
* `RIGHT JOIN` 保留右表全部记录，左表没有匹配时用 `NULL` 补齐。
* 外连接中，条件放在 `ON` 或 `WHERE` 可能影响最终结果，尤其要注意 `WHERE` 过滤非保留表字段的问题。
* SQL 标准支持 `FULL OUTER JOIN`，但 MySQL 不支持直接使用。
* 在 MySQL 中，如果需要满外连接效果，通常需要使用 `LEFT JOIN`、`RIGHT JOIN` 和 `UNION` 组合模拟。
