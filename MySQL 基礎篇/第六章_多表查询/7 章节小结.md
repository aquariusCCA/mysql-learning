## 快速回查表一：本章核心概念

| 概念 | 核心含义 | 重点提醒 |
| --- | --- | --- |
| 多表查询 | 两个或更多表一起完成查询 | 重点不是表多，而是表之间有关系 |
| 关联字段 | 表与表之间可对应的字段 | 不一定必须建立外键，但业务语义要一致 |
| 连接条件 | 说明表与表如何匹配的条件 | 少了连接条件容易产生笛卡尔积 |
| 笛卡尔积 | 每一行和另一张表每一行都组合 | 普通业务查询中通常不是想要的结果 |
| 等值连接 | 使用 `=` 作为连接条件 | 常见于主键、外键或业务编号对应 |
| 非等值连接 | 使用区间、大小比较等连接条件 | 常见于薪资等级、范围匹配 |
| 自连接 | 同一张表通过不同别名连接自己 | 常见于员工与经理、分类与父分类 |
| 内连接 | 只保留匹配成功的数据 | `JOIN` 默认就是内连接 |
| 外连接 | 保留某一侧或两侧未匹配数据 | 未匹配一侧用 `NULL` 补齐 |
| UNION | 合并多个查询结果并去重 | 是纵向合并结果集，不是表连接 |
| UNION ALL | 合并多个查询结果但不去重 | 不需要去重时通常优先考虑 |
| USING | 指定同名字段做等值连接 | 比 `NATURAL JOIN` 更可控 |
| NATURAL JOIN | 自动按所有同名字段连接 | 条件隐藏，正式开发中要谨慎 |

## 快速回查表二：WHERE / ON / USING 怎么选

| 写法 | 使用范围 | 适用条件 | 优点 | 注意事项 |
| --- | --- | --- | --- | --- |
| `WHERE` | 适用于传统 SQL92 连接写法 | 手动写连接条件 | 最传统、容易看懂旧代码 | 连接条件和筛选条件容易混在一起 |
| `ON` | 只能配合 `JOIN` 使用 | 手动写连接条件 | 连接关系清楚，适合复杂多表查询 | 外连接中条件放 `ON` 或 `WHERE` 结果可能不同 |
| `USING` | 只能配合 `JOIN` 使用 | 两表关联字段同名，且是等值连接 | 写法较短 | 只能用于同名字段，不能表示非等值连接 |
| `NATURAL JOIN` | SQL99 简化写法 | 自动使用所有同名字段 | 最短 | 条件隐藏，表结构变化可能改变结果 |

## 快速回查表三：JOIN 结果怎么判断

| 需求 | 结果集 | 推荐写法 |
| --- | --- | --- |
| 只要两边匹配成功的数据 | `A ∩ B` | `A JOIN B ON ...` |
| 保留左表全部数据 | `A` | `A LEFT JOIN B ON ...` |
| 保留右表全部数据 | `B` | `A RIGHT JOIN B ON ...` |
| 只要左表有、右表没有的数据 | `A - A∩B` | `A LEFT JOIN B ON ... WHERE B.主键 IS NULL` |
| 只要右表有、左表没有的数据 | `B - A∩B` | `A RIGHT JOIN B ON ... WHERE A.主键 IS NULL` |
| 要左右两边全部数据 | `A ∪ B` | MySQL 中用 `LEFT JOIN` + `UNION` + `RIGHT JOIN` 模拟 |
| 只要左右两边各自独有数据 | `A ∪ B - A∩B` | 左独有 `UNION ALL` 右独有 |

## 7.1 本章主线总览

第六章可以按照下面这条主线理解。

### 第一步：先知道什么是多表查询

多表查询，也叫关联查询，指的是在一条 SQL 中同时使用两张或更多张表，并根据表之间的关联关系，把分散在不同表中的数据组合起来。

例如：

- 员工姓名在 `employees` 表。
- 部门名称在 `departments` 表。
- 想查询员工姓名和部门名称，就需要多表查询。

核心问题是：

```text id="kbn6oq"
employees 和 departments 靠什么字段对应？
```

通常答案是：

```sql id="uvtvgs"
employees.department_id = departments.department_id
```

### 第二步：理解缺少连接条件会产生笛卡尔积

错误写法：

```sql id="yjfdag"
SELECT
    last_name,
    department_name
FROM employees, departments;
```

这条 SQL 只把两张表放在一起，却没有说明两张表如何对应。

结果会变成：

```text id="br0eog"
每个员工 × 每个部门
```

如果员工表有 `107` 行，部门表有 `27` 行，结果就是：

```text id="vms58k"
107 × 27 = 2889 行
```

这就是笛卡尔积。

所以，多表查询第一条原则是：

> 只要查询涉及多张表，就必须检查是否写了正确的连接条件。

### 第三步：学习多表查询的分类

多表查询可以从三个维度分类：

| 分类维度   | 关注问题              | 常见类型       |
| ------ | ----------------- | ---------- |
| 连接条件   | 表与表靠什么条件对应？       | 等值连接、非等值连接 |
| 参与表    | 是不同表连接，还是同一张表连自己？ | 自连接、非自连接   |
| 结果保留规则 | 是否保留没有匹配上的记录？     | 内连接、外连接    |

同一条 SQL 可以同时属于多个分类。

例如：

```sql id="bekzfl"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

它同时是：

* 等值连接：因为使用 `=`
* 非自连接：因为连接的是不同表
* 内连接：因为 `JOIN` 默认只保留匹配成功的数据

### 第四步：从 SQL92 过渡到 SQL99

SQL92 风格：

```sql id="vh1wbf"
SELECT
    e.last_name,
    d.department_name
FROM employees e, departments d
WHERE e.department_id = d.department_id;
```

SQL99 风格：

```sql id="xsmrw1"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

两者都能表达多表连接。

差别在于：

* SQL92 把连接条件写在 `WHERE`。
* SQL99 把连接条件写在 `ON`。
* SQL99 的连接关系更清楚，更适合复杂多表查询。

### 第五步：学习外连接

内连接只保留匹配成功的数据。

外连接除了保留匹配成功的数据，还会保留某一侧或两侧没有匹配上的数据。

例如：

```sql id="s90h0r"
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

这条 SQL 会保留所有员工。
如果某个员工没有对应部门，部门字段会显示为 `NULL`。

### 第六步：学习 UNION

`UNION` 不是表连接，而是结果集合并。

```sql id="agichl"
SELECT employee_id, last_name
FROM employees
WHERE department_id = 60

UNION

SELECT employee_id, last_name
FROM employees
WHERE department_id = 90;
```

它表示：

```text id="c4xaz5"
第一批查询结果
+
第二批查询结果
=
合并后的结果集
```

其中：

* `UNION` 会去重。
* `UNION ALL` 不会去重。

### 第七步：用 JOIN、IS NULL、UNION 组合出不同结果集

例如左表独有：

```sql id="n4jnca"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_id IS NULL;
```

满外连接在 MySQL 中不能直接写 `FULL OUTER JOIN`，通常需要用：

```text id="c6udqd"
LEFT JOIN
UNION
RIGHT JOIN
```

或：

```text id="o0i0h6"
LEFT JOIN 全部
UNION ALL
右表独有
```

来模拟。

### 第八步：理解 SQL99 简化写法

`USING`：

```sql id="e9gnmv"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees e
JOIN departments d
    USING (department_id);
```

大致等价于：

```sql id="kwawfw"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

`NATURAL JOIN` 会自动使用所有同名字段做等值连接，但条件隐藏，实务中要谨慎。

## 7.2 多表查询的核心原则

多表查询可以用一句话概括：

> 根据正确的关联字段和连接条件，把多张表中彼此相关的数据组合成一个有意义的结果。

这句话里有三个关键词。

### 7.2.1 正确的关联字段

关联字段不一定必须建立外键，但业务含义必须能对应。

例如：

```sql id="xxva2c"
e.department_id = d.department_id
```

是合理的，因为两个字段都表示部门编号。

但如果随便拿两个字段连接，即使 SQL 能执行，结果也可能没有业务意义。

### 7.2.2 正确的连接条件

连接条件负责说明：

```text id="ktfcm4"
表 A 的哪一行
应该和
表 B 的哪一行
组合在一起
```

没有连接条件，就容易产生笛卡尔积。

连接条件写错，也会导致结果错误。

### 7.2.3 有意义的结果

多表查询不是为了把表越连越多，而是为了回答业务问题。

例如：

```text id="iniv5f"
查询员工及其部门名称
```

需要连接员工表和部门表。

但如果查询员工姓名，不需要部门信息，就没有必要连接部门表。

所以写多表查询前，应该先问：

> 当前查询真的需要这张表吗？

## 7.3 连接条件的三种写法

表连接的关联条件常见有三种写法：

1. `WHERE`
2. `ON`
3. `USING`

它们都能在特定场景表达连接关系，但适用范围不同。

## 7.3.1 WHERE 方式

SQL92 风格中，连接条件通常写在 `WHERE` 中。

```sql id="mzn5d5"
SELECT
    e.last_name,
    d.department_name
FROM employees e, departments d
WHERE e.department_id = d.department_id;
```

这里的：

```sql id="qpq2r8"
WHERE e.department_id = d.department_id
```

就是连接条件。

### WHERE 方式的特点

优点：

* 传统写法。
* 很多旧项目中能看到。
* 语法直观，容易入门。

缺点：

* 连接条件和普通筛选条件容易混在一起。
* 表越多时，可读性越差。
* 一旦少写某个连接条件，容易产生笛卡尔积。

例如：

```sql id="nghjsm"
SELECT
    e.last_name,
    d.department_name
FROM employees e, departments d
WHERE e.department_id = d.department_id
  AND d.department_name = 'IT';
```

其中：

```sql id="plw03p"
e.department_id = d.department_id
```

是连接条件。

```sql id="m8w7i0"
d.department_name = 'IT'
```

是筛选条件。

这两类条件都放在 `WHERE` 中，查询复杂后比较容易混淆。

## 7.3.2 ON 方式

SQL99 风格中，连接条件通常写在 `ON` 中。

```sql id="p39kz7"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

这里的：

```sql id="xa18wm"
ON e.department_id = d.department_id
```

表示员工表和部门表通过部门编号连接。

### ON 方式的特点

优点：

* 连接关系更清楚。
* 每个 `JOIN` 可以紧跟对应的 `ON`。
* 多表查询时更容易阅读。
* 连接条件和普通筛选条件可以分开。

例如：

```sql id="wh1trn"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'IT';
```

这里：

* `ON`：说明员工表和部门表如何连接。
* `WHERE`：说明最终只保留 IT 部门的数据。

实际开发中，建议优先使用这种写法。

## 7.3.3 USING 方式

`USING` 是 SQL99 提供的简化写法。

它适用于：

* 两张表有同名字段。
* 这个同名字段就是连接依据。
* 连接条件是等值连接。

例如：

```sql id="xeo0bi"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees e
JOIN departments d
    USING (department_id);
```

它大致等价于：

```sql id="n4e2ha"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

### USING 的限制

`USING` 不是所有连接都能用。

它不能用于：

#### 字段不同名

```sql id="b6om05"
ON e.location_code = l.location_id
```

因为字段名不同，不能直接写 `USING (location_id)`。

#### 非等值连接

```sql id="s1qjxq"
ON e.salary BETWEEN j.lowest_sal AND j.highest_sal
```

因为这不是等值连接。

#### 复杂连接逻辑

```sql id="fhs60p"
ON e.department_id = d.department_id
AND d.status = 'ACTIVE'
```

这种场景继续使用 `ON` 通常更清楚。

## 7.3.4 WHERE / ON / USING 选择建议

| 场景             | 建议                          |
| -------------- | --------------------------- |
| 阅读旧代码          | 要能看懂 `WHERE` 连接写法           |
| 自己写新的多表查询      | 优先使用 `JOIN ... ON`          |
| 同名字段等值连接，逻辑很简单 | 可以使用 `USING`                |
| 字段不同名          | 使用 `ON`                     |
| 非等值连接          | 使用 `ON`                     |
| 外连接            | 优先使用 `ON` 写连接条件，并小心 `WHERE` |
| 追求可维护性         | 优先使用 `JOIN ... ON`          |

一句话：

> `WHERE` 要看得懂，`ON` 要熟练写，`USING` 要知道限制。

## 7.4 JOIN 类型速查

## 7.4.1 INNER JOIN 内连接

内连接只保留匹配成功的数据。

```sql id="e5eet6"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

`JOIN` 默认就是 `INNER JOIN`。

所以下面两种写法等价：

```sql id="qn9f2o"
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

```sql id="zhz30u"
FROM employees e
INNER JOIN departments d
    ON e.department_id = d.department_id;
```

适合需求：

```text id="o0nv1f"
只查询两边都能匹配上的数据
```

## 7.4.2 LEFT JOIN 左外连接

左外连接保留左表全部记录。

```sql id="nzjo78"
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

含义：

```text id="l9yijh"
所有员工都显示；
有部门就补部门名称；
没有部门就显示 NULL。
```

适合需求：

```text id="tqgfgd"
主表数据必须完整显示
```

## 7.4.3 RIGHT JOIN 右外连接

右外连接保留右表全部记录。

```sql id="tdi9if"
SELECT
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id;
```

含义：

```text id="q2ob4t"
所有部门都显示；
有员工就补员工资料；
没有员工就显示 NULL。
```

右外连接通常可以通过调整表顺序改写成左外连接。

例如：

```sql id="ceufsv"
SELECT
    e.last_name,
    d.department_name
FROM departments d
LEFT JOIN employees e
    ON e.department_id = d.department_id;
```

很多开发者更偏好 `LEFT JOIN`，因为阅读方向更直观。

## 7.4.4 FULL OUTER JOIN 满外连接

满外连接保留左右两边全部记录。

它包含：

```text id="c28722"
匹配成功的数据
+
左表独有的数据
+
右表独有的数据
```

标准 SQL 写法：

```sql id="lu5zz4"
SELECT
    *
FROM A
FULL OUTER JOIN B
    ON A.id = B.id;
```

但是：

```text id="vf8fei"
MySQL 不支持直接使用 FULL OUTER JOIN。
```

MySQL 中通常用 `LEFT JOIN`、`RIGHT JOIN` 和 `UNION` / `UNION ALL` 模拟。

## 7.5 7 种 JOIN 结果集速查

以下假设：

```text id="mf1x2u"
A = employees
B = departments
```

连接条件：

```sql id="i6u8pi"
e.department_id = d.department_id
```

## 7.5.1 交集：A ∩ B

只保留匹配成功的数据。

```sql id="rrxetv"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

## 7.5.2 左表全部：A

保留所有员工。

```sql id="p5co1f"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

## 7.5.3 右表全部：B

保留所有部门。

```sql id="piqwwu"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id;
```

## 7.5.4 左表独有：A - A∩B

查询员工表中有，但部门表匹配不到的数据。

```sql id="vwcd3m"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_id IS NULL;
```

重点：

```text id="ei0ppr"
LEFT JOIN 之后，再筛右表主键为 NULL。
```

## 7.5.5 右表独有：B - A∩B

查询部门表中有，但员工表匹配不到的数据。

```sql id="ua48l7"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id
WHERE e.employee_id IS NULL;
```

重点：

```text id="wzuxfg"
RIGHT JOIN 之后，再筛左表主键为 NULL。
```

## 7.5.6 满外连接：A ∪ B

MySQL 中可以写：

```sql id="zai70p"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id

UNION

SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id;
```

这里用 `UNION`，是为了去掉左右外连接中重复出现的交集。

如果要使用 `UNION ALL`，可以写成：

```sql id="ei7y7x"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id

UNION ALL

SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id
WHERE e.employee_id IS NULL;
```

这表示：

```text id="gn59qq"
左表全部
+
右表独有
=
满外连接结果
```

因为第二段只取右表独有，不会重复交集，所以可以使用 `UNION ALL`。

## 7.5.7 左右独有：A ∪ B - A∩B

只查询两边不匹配的数据。

```sql id="ho91gs"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_id IS NULL

UNION ALL

SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d
    ON e.department_id = d.department_id
WHERE e.employee_id IS NULL;
```

这表示：

```text id="ul8i8g"
左表独有
+
右表独有
```

适合做数据一致性检查。

## 7.6 UNION 复习

`UNION` 和 `UNION ALL` 都是把多个查询结果合并成一个结果集。

它们不是表连接。

## 7.6.1 UNION

```sql id="nqsqwp"
SELECT employee_id, last_name
FROM employees
WHERE department_id = 60

UNION

SELECT employee_id, last_name
FROM employees
WHERE department_id = 90;
```

特点：

* 合并多个结果集。
* 会去除重复行。
* 去重的是最终结果中的整行记录。
* 有额外去重成本。

## 7.6.2 UNION ALL

```sql id="f8x7g2"
SELECT employee_id, last_name
FROM employees
WHERE department_id = 60

UNION ALL

SELECT employee_id, last_name
FROM employees
WHERE department_id = 90;
```

特点：

* 合并多个结果集。
* 不去重。
* 保留所有结果。
* 通常比 `UNION` 少一步去重成本。

## 7.6.3 UNION 和 JOIN 的区别

| 比较点  | JOIN    | UNION         |
| ---- | ------- | ------------- |
| 作用   | 表与表横向连接 | 多个结果集纵向合并     |
| 结果变化 | 列可能变多   | 行可能变多         |
| 核心要求 | 要有连接条件  | 输出列结构要对齐      |
| 常见用途 | 员工 + 部门 | 第一批员工 + 第二批员工 |

一句话：

> `JOIN` 是把相关资料接到同一行；`UNION` 是把多批查询结果合并成一个结果集。

## 7.7 NATURAL JOIN 与 USING 复习

## 7.7.1 USING

`USING` 用于同名字段的等值连接。

```sql id="dm42lp"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees e
JOIN departments d
    USING (department_id);
```

大致等价于：

```sql id="coj8y3"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

限制：

* 必须配合 `JOIN`。
* 字段必须同名。
* 只能表示等值连接。
* 不适合不同名字段。
* 不适合非等值连接。

## 7.7.2 NATURAL JOIN

`NATURAL JOIN` 会自动使用所有同名字段做等值连接。

```sql id="i2kjsg"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees
NATURAL JOIN departments;
```

它可能自动等价于：

```sql id="jj7dsy"
ON employees.department_id = departments.department_id
```

但如果两张表还有其他同名字段，它也会自动加入连接条件。

风险：

* 连接条件隐藏。
* 会使用所有同名字段。
* 表结构变化可能改变查询结果。
* 实务维护风险较高。

## 7.7.3 实务建议

| 写法                     | 推荐程度   | 原因         |
| ---------------------- | ------ | ---------- |
| `JOIN ... ON ...`      | 最推荐    | 条件明确、最可控   |
| `JOIN ... USING (...)` | 简单场景可用 | 适合同名字段等值连接 |
| `NATURAL JOIN`         | 谨慎使用   | 条件隐藏，依赖表结构 |

## 7.8 n 张表关联规则

连接 `n` 张表时，通常至少需要 `n - 1` 个连接条件。

例如连接三张表：

```sql id="pd578h"
SELECT
    e.last_name,
    d.department_name,
    l.city
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
JOIN locations l
    ON d.location_id = l.location_id;
```

这里有三张表：

* `employees`
* `departments`
* `locations`

至少需要两段连接条件：

```sql id="ng759d"
e.department_id = d.department_id
```

以及：

```sql id="qgphwu"
d.location_id = l.location_id
```

这条规则的本质是：

> 表越多，越要理清每一张表是通过哪一段关系加入查询的。

## 7.8.1 n - 1 只是最低要求，不代表条件越少越好

例如三张表通常至少需要两个连接条件，但实际业务可能还需要额外筛选：

```sql id="x4oosk"
SELECT
    e.last_name,
    d.department_name,
    l.city
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
JOIN locations l
    ON d.location_id = l.location_id
WHERE l.city = 'Seattle';
```

其中：

* `e.department_id = d.department_id` 是连接条件。
* `d.location_id = l.location_id` 是连接条件。
* `l.city = 'Seattle'` 是筛选条件。

不要把所有条件都叫连接条件。

## 7.8.2 少连接条件的风险

如果三张表少了一段连接条件，例如：

```sql id="fvhmc1"
SELECT
    e.last_name,
    d.department_name,
    l.city
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
JOIN locations l;
```

这时 `locations` 表没有说明如何连接，就可能产生局部笛卡尔积。

所以写多表查询时，每加入一张表都要问：

```text id="t4fq32"
这张表是靠什么条件接进来的？
```

## 7.9 性能与开发实践提醒

多表查询写得出来，不代表它在生产环境中就足够好。

多表连接越多，通常会带来：

* 查询逻辑更复杂。
* SQL 更难维护。
* 优化器需要考虑更多连接顺序。
* 数据量大时性能风险更高。
* 如果关联字段没有索引，查询可能明显变慢。

## 7.9.1 控制不必要的表连接

不要为了“顺手”连接不需要的表。

例如只查询员工姓名：

```sql id="mwygjb"
SELECT
    e.last_name
FROM employees e;
```

就不需要连接部门表。

如果需求变成查询员工姓名和部门名称，才需要：

```sql id="ua4maw"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

原则：

> 查询结果不需要的表，就不要参与连接。

## 7.9.2 关联字段类型要一致

例如：

```sql id="sq1ska"
e.department_id = d.department_id
```

两边字段类型最好一致。

如果一边是数字，一边是字符串，数据库可能需要隐式转换，影响索引使用，也容易造成结果异常。

## 7.9.3 关联字段最好建立索引

多表连接时，经常会在关联字段上做匹配。

例如：

```sql id="h92keo"
e.department_id = d.department_id
```

如果数据量较大，关联字段上有合适索引，通常更有利于查询效率。

常见需要关注的字段：

* 主键字段
* 外键字段
* 高频关联字段
* 高频筛选字段

## 7.9.4 外连接要小心 WHERE 条件

例如：

```sql id="b9oxkb"
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'IT';
```

这条 SQL 虽然写了 `LEFT JOIN`，但 `WHERE d.department_name = 'IT'` 会过滤掉右表为 `NULL` 的记录。

所以外连接中要特别注意：

* 条件放在 `ON` 还是 `WHERE`，结果可能不同。
* 如果要保留左表全部数据，不要随便在 `WHERE` 中过滤右表字段。
* 如果要查左表独有，才使用 `WHERE 右表主键 IS NULL`。

## 7.9.5 超过三张表要特别关注可读性

多表连接不是不能超过三张表，而是超过三张表后要更注意：

* 每张表是否真的必要。
* 每段连接条件是否完整。
* 字段别名是否清楚。
* SQL 是否可以拆分。
* 是否需要查看执行计划。
* 是否有合适索引。

建议写法：

```sql id="p9tssg"
SELECT
    e.last_name,
    d.department_name,
    l.city,
    j.job_title
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
JOIN locations l
    ON d.location_id = l.location_id
JOIN jobs j
    ON e.job_id = j.job_id;
```

不要写成连接链路不清楚的形式。

## 7.10 多表查询阅读顺序

看到一条复杂 SQL 时，不建议从 `SELECT` 字段开始硬读。

建议按下面顺序阅读。

## 7.10.1 第一步：看 FROM，确定主表或起始表

```sql id="u31q3j"
FROM employees e
```

先确认查询从哪张表开始。

## 7.10.2 第二步：看 JOIN，确认加入了哪些表

```sql id="zodz9t"
JOIN departments d
JOIN locations l
JOIN jobs j
```

确认查询涉及哪些表。

## 7.10.3 第三步：看 ON，确认每张表如何连接

```sql id="r7kjla"
ON e.department_id = d.department_id
ON d.location_id = l.location_id
ON e.job_id = j.job_id
```

把连接链路理清楚。

## 7.10.4 第四步：看 WHERE，确认筛选条件

```sql id="xrrfz9"
WHERE d.department_name = 'IT'
```

判断最终结果要过滤哪些数据。

## 7.10.5 第五步：看 SELECT，确认输出字段

```sql id="rv8lcr"
SELECT
    e.last_name,
    d.department_name,
    l.city
```

最后确认查询结果输出什么。

## 7.10.6 第六步：看 GROUP BY、HAVING、ORDER BY、LIMIT

如果 SQL 中有聚合、排序或分页，再继续看：

* `GROUP BY`
* `HAVING`
* `ORDER BY`
* `LIMIT`

这样读 SQL，会比从第一行开始硬读更稳。

## 7.11 常见错误检查清单

写完多表查询后，可以用下面清单检查。

## 7.11.1 是否少了连接条件

错误风险：

```sql id="xh6rtk"
SELECT
    e.last_name,
    d.department_name
FROM employees e, departments d;
```

检查问题：

```text id="kacxsf"
employees 和 departments 靠什么条件连接？
```

正确写法：

```sql id="cdl17f"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

## 7.11.2 是否把连接条件和筛选条件混在一起导致难读

可以执行，但可读性较差：

```sql id="y8cpb3"
SELECT
    e.last_name,
    d.department_name
FROM employees e, departments d
WHERE e.department_id = d.department_id
  AND d.department_name = 'IT';
```

更推荐：

```sql id="vn19ei"
SELECT
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'IT';
```

## 7.11.3 是否误把 LEFT JOIN 当成左表独有

错误理解：

```sql id="sfk2bg"
SELECT
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d
    ON e.department_id = d.department_id;
```

这不是左表独有。

它包含：

```text id="rm5f24"
交集
+
左表未匹配数据
```

如果只要左表独有，要加：

```sql id="otppp4"
WHERE d.department_id IS NULL
```

## 7.11.4 是否误把 RIGHT JOIN 当成右表独有

右外连接不是右表独有。

如果只要右表独有，要加：

```sql id="unb5fc"
WHERE e.employee_id IS NULL
```

## 7.11.5 是否在 MySQL 中直接写 FULL OUTER JOIN

MySQL 不支持：

```sql id="ef4q2p"
SELECT
    *
FROM employees e
FULL OUTER JOIN departments d
    ON e.department_id = d.department_id;
```

应改用：

```text id="tt3vhu"
LEFT JOIN
UNION
RIGHT JOIN
```

或：

```text id="wkmw46"
LEFT JOIN 全部
UNION ALL
右表独有
```

## 7.11.6 是否滥用 NATURAL JOIN

不建议：

```sql id="m91fvv"
SELECT
    employee_id,
    last_name,
    department_name
FROM employees
NATURAL JOIN departments;
```

更推荐：

```sql id="f7pcze"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

原因：

```text id="j9zw4m"
连接条件明确，可维护性更好。
```

## 7.11.7 UNION 查询字段是否对齐

错误风险：

```sql id="xha3hh"
SELECT employee_id, last_name
FROM employees

UNION

SELECT department_id
FROM departments;
```

问题：

```text id="zx5z13"
第一段 SELECT 返回 2 列，第二段 SELECT 返回 1 列。
```

使用 `UNION` 时要检查：

* 列数是否一致。
* 对应位置类型是否兼容。
* 对应位置业务含义是否一致。
* 是否需要去重。

## 7.11.8 关联字段是否有索引

如果大表连接：

```sql id="x6akuc"
ON e.department_id = d.department_id
```

要关注：

* `e.department_id` 是否常用于连接。
* `d.department_id` 是否是主键或索引字段。
* 数据类型是否一致。
* 执行计划是否合理。

## 7.12 多表查询实务建议

## 7.12.1 优先写清楚，而不是写短

不推荐为了少写几行而牺牲可读性。

比起：

```sql id="t5a0im"
FROM employees
NATURAL JOIN departments
```

更推荐：

```sql id="ho8jo2"
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id
```

因为连接条件完全可见。

## 7.12.2 表别名要稳定且有意义

推荐：

```sql id="awhhro"
employees e
departments d
locations l
jobs j
```

不要乱用难懂别名：

```sql id="rshzd4"
employees a
departments b
locations c
jobs x
```

如果 SQL 很复杂，有意义的别名会大幅提高可读性。

## 7.12.3 SELECT 字段尽量明确

不建议在生产查询中随便写：

```sql id="q5j6xn"
SELECT *
```

多表查询中 `SELECT *` 可能带来：

* 返回不必要字段。
* 字段重名。
* 网络传输变多。
* 后续代码依赖不稳定。
* 表结构变化影响结果。

更推荐：

```sql id="xegcm0"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

## 7.12.4 先写连接，再加筛选

写复杂 SQL 时，可以分步骤：

第一步，先写出连接关系：

```sql id="c75mfn"
SELECT
    e.employee_id,
    e.last_name,
    d.department_name
FROM employees e
JOIN departments d
    ON e.department_id = d.department_id;
```

第二步，再加筛选条件：

```sql id="iz9zh1"
WHERE d.department_name = 'IT'
```

第三步，再加排序、分页或聚合。

这样比较不容易把连接条件写错。

## 7.12.5 结果异常时先检查是否产生笛卡尔积

如果结果行数远超预期，优先检查：

* 是否少了 `ON`。
* 是否少了 `WHERE` 连接条件。
* 多张表是否少了某一段连接链路。
* 连接字段是否写错。
* 是否误用了 `CROSS JOIN`。
* 外连接后 `WHERE` 条件是否改变了结果。

## 常见混淆点

* 多表查询不是把多张表写进 SQL 就结束了，关键是要有正确连接条件。
* 笛卡尔积不是语法错误，但普通业务查询中通常不是想要的结果。
* `WHERE`、`ON`、`USING` 都能表达连接关系，但适用范围和可读性不同。
* `ON` 首先服务于连接条件，不是单纯的额外筛选条件。
* `WHERE` 更适合写连接完成后的普通筛选条件。
* 外连接中，条件放在 `ON` 或 `WHERE` 可能影响结果。
* `USING` 只能用于同名字段的等值连接。
* `NATURAL JOIN` 会自动使用所有同名字段，不是只挑一个字段。
* `JOIN` 默认就是内连接。
* `LEFT JOIN` 不等于左表独有。
* `RIGHT JOIN` 不等于右表独有。
* 左表独有和右表独有都需要外连接配合 `IS NULL`。
* MySQL 不支持直接写 `FULL OUTER JOIN`。
* `UNION` 是合并结果集，不是连接表。
* `UNION` 会去重，`UNION ALL` 不会去重。
* 连接 `n` 张表通常至少需要 `n - 1` 个连接条件，但这只是最低要求。
* 多表连接数量越多，越要关注可读性、索引和执行效率。
* 字段同名不代表业务含义一定相同。
* 字段能不能连接，不只看名字，更要看业务语义。

## 一句话抓核心

第六章的核心不是背多少种 `JOIN` 名词，而是掌握一条主线：先确认表之间的业务关系，再用正确的连接条件把表连起来，接着根据结果需求选择内连接、外连接或 `UNION` 组合，最后检查连接链路、索引与可维护性。

## 小结

这一章需要记住：

* 多表查询的本质是根据关联字段，把多张表中彼此相关的数据组合起来。
* 关联字段不一定必须建立外键，但业务含义必须能对应。
* 缺少连接条件会产生笛卡尔积。
* SQL92 常把连接条件写在 `WHERE`。
* SQL99 常使用 `JOIN ... ON`，连接关系更清楚。
* `USING` 是同名字段等值连接的简化写法。
* `NATURAL JOIN` 会自动使用所有同名字段，实务中要谨慎。
* 多表查询可以从连接条件、参与表、结果保留规则三个维度分类。
* `JOIN` 默认表示内连接，只保留匹配成功的数据。
* `LEFT JOIN` 保留左表全部记录。
* `RIGHT JOIN` 保留右表全部记录。
* 左表独有要用 `LEFT JOIN` 加 `WHERE 右表主键 IS NULL`。
* 右表独有要用 `RIGHT JOIN` 加 `WHERE 左表主键 IS NULL`。
* MySQL 不支持直接写 `FULL OUTER JOIN`。
* MySQL 可以用 `LEFT JOIN`、`RIGHT JOIN` 和 `UNION` / `UNION ALL` 模拟满外连接。
* `UNION` 和 `UNION ALL` 是合并查询结果，不是表连接。
* `UNION` 会去重，`UNION ALL` 不会去重。
* 连接 `n` 张表时，通常至少需要 `n - 1` 个连接条件。
* 多表查询要控制连接表数量，避免连接不必要的表。
* 被关联字段的数据类型应保持一致，并尽量建立合适索引。
* 写多表查询时，优先追求连接关系清楚、结果语义正确、后续容易维护。
