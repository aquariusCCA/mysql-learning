## 连接条件的三种写法

表连接的关联条件常见可以有三种写法：`WHERE`、`ON`、`USING`。

- `WHERE`：适用于所有关联查询。
- `ON`：只能和 `JOIN` 一起使用，通常只写关联条件；虽然关联条件也可以并到 `WHERE` 中，但分开写可读性通常更好。
- `USING`：只能和 `JOIN` 一起使用，而且要求两个表中的关联字段名称一致，并且只能表示等值连接。

### WHERE / ON / USING 的区别

如果从“表达方式”来理解：

- `WHERE` 是传统写法，把连接条件直接放在筛选条件里。
- `ON` 是 SQL99 更常见的写法，把“表怎么连”单独写出来。
- `USING` 则是对同名字段等值连接的进一步简化。

## WHERE 方式

把关联条件写在 `WHERE` 后面：

```sql
SELECT
    last_name,
    department_name
FROM
    employees,
    departments
WHERE
    employees.department_id = departments.department_id;
```

这种写法属于更传统的 SQL92 风格，适合所有关联查询，但当查询变复杂时，连接条件和普通筛选条件可能会混在一起。

## ON 方式

把关联条件写在 `ON` 后面，只能和 `JOIN` 一起使用：

```sql
SELECT
    last_name,
    department_name
FROM
    employees
INNER JOIN departments
ON
    employees.department_id = departments.department_id;
```

`INNER JOIN` 也可以简写为 `JOIN`：

```sql
SELECT
    last_name,
    department_name
FROM
    employees
JOIN departments
ON
    employees.department_id = departments.department_id;
```

这种写法的优势是：连接关系更清楚，也更适合多表查询时逐段阅读。

## USING 方式

把关联字段写在 `USING()` 中，只能和 `JOIN` 一起使用，并且要求两个表中的关联字段名称相同，而且只能表示 `=` 等值连接。

例如，查询员工姓名与职位名称：

```sql
SELECT
    last_name,
    job_title
FROM
    employees
INNER JOIN jobs
USING (job_id);
```

这里的 `USING (job_id)` 表示：直接使用两张表中同名的 `job_id` 字段完成等值连接。

## n 张表关联规则

连接 `n` 张表时，通常至少需要 `n - 1` 个关联条件。

例如，查询员工姓名、职位名称、部门名称：

```sql
SELECT
    last_name,
    job_title,
    department_name
FROM
    employees,
    departments,
    jobs
WHERE
    employees.department_id = departments.department_id
    AND employees.job_id = jobs.job_id;
```

也可以写成 `JOIN` 风格：

```sql
SELECT
    last_name,
    job_title,
    department_name
FROM
    employees
INNER JOIN departments
INNER JOIN jobs
ON
    employees.department_id = departments.department_id
    AND employees.job_id = jobs.job_id;
```

这条规则的核心是：表越多，必须理清每张表之间的关联链路，否则就容易出现错误连接或笛卡尔积。

## 性能提醒

多表连接本质上可以类比为多层嵌套匹配，连接的表越多，资源消耗通常越高。因此，写多表查询时要有明显的性能意识。

需要特别注意：

- 要控制连接表的数量，不要连接不必要的表。
- 多表连接越多，SQL 查询性能通常下降越明显。
- 很多数据库管理系统本身也会对可连接表数量设置限制。

## 开发实践提醒

根据常见开发规范，多表关联查询时还应注意：

- 超过三个表的 `JOIN` 要特别关注必要性与性能成本。
- 需要关联的字段，数据类型应保持一致。
- 被关联字段最好建立索引。
- 即使只是双表 `JOIN`，也应关注索引与执行效率。

这些提醒的核心不是禁止多表查询，而是强调：多表查询写得出来，不代表它在生产环境里就足够高效。

## 常见混淆点

- `WHERE`、`ON`、`USING` 都能表达连接关系，但适用范围和可读性并不相同。
- `USING` 不是任意连接都能写，它要求关联字段必须同名，而且只适合等值连接。
- `ON` 并不等于额外筛选条件，它首先服务于“连接条件”本身。
- 连接 `n` 张表至少需要 `n - 1` 个关联条件，不代表条件越多越好，而是要保证关联链路完整且正确。
- 表连接数量多，不仅会让 SQL 更难读，也往往会让性能变差。

## 一句话抓核心

第六章的核心不是背多少种 `JOIN` 名词，而是掌握：表要靠正确的关联条件连起来，连接写法要清楚，连接数量要克制，关联字段最好有索引。

## 小结

这一章需要记住：

- 连接条件常见写法有 `WHERE`、`ON`、`USING` 三种。
- `WHERE` 最通用，`ON` 更适合 `JOIN` 风格，`USING` 适合同名字段的等值连接。
- 连接 `n` 张表，通常至少需要 `n - 1` 个关联条件。
- 多表连接要尽量避免不必要的表参与查询。
- 被关联字段的数据类型应一致，并尽量建立索引。
