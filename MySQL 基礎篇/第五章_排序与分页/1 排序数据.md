## 本节导读

这一节主要说明如何使用 `ORDER BY` 子句对查询结果排序。

排序不会改变表中数据的存储顺序，它只影响本次查询结果的显示顺序。只要业务上关心“最新、最早、最高、最低、按编号顺序、固定分页顺序”，就不应该依赖数据库默认返回顺序，而应该显式写出 `ORDER BY`。

第一次阅读时，建议先掌握 `ORDER BY` 的位置与基本语法，再依次理解单列排序、别名排序、字段位置排序、多列排序。复习或实务使用时，重点关注「多列排序优先级」、「分页时的稳定排序」以及「`NULL` 值在排序中的位置」。

- 如何按多个字段组合排序。
- 为什么分页时最好加入唯一字段作为排序兜底。

## 关键字

- `ORDER BY`：用于指定查询结果排序规则的子句。
- `ASC`：升序排序，也是默认排序方向。
- `DESC`：降序排序。
- `AS`：定义列别名时常用的关键字。
- `单列排序`：按一个字段决定结果顺序。
- `多列排序`：按多个字段从左到右逐级决定顺序。
- `别名排序`：按 `SELECT` 中定义的列别名排序。
- `字段位置排序`：按查询结果中的列位置编号排序，例如 `ORDER BY 3`。
- `稳定排序`：在排序字段出现相同值时，仍能保证结果顺序可预期、可重复。
- `兜底排序字段`：通常是主键或唯一字段，例如 `id`、`employee_id`，用于让排序结果稳定。
- `NULL 排序`：空值参与排序时的默认位置与自定义位置。

## 1.1 为什么必须显式排序

如果没有使用排序操作，查询返回的数据可能看起来像是按添加数据的顺序显示，但不应该依赖这种默认顺序。

数据库表本质上不保证“自然顺序”。同一条 SQL 在不同执行计划、不同索引、不同数据量、不同版本、不同并发状态下，都可能返回不同的显示顺序。

只要业务上关心结果顺序，就应该显式使用 `ORDER BY`。

例如：

- 查最新订单：应按 `create_time DESC` 排序。
- 查薪资最高员工：应按 `salary DESC` 排序。
- 查最早入职员工：应按 `hire_date ASC` 排序。
- 做分页列表：应先明确排序，再分页。

### 回查提示

如果业务上在意“最新、最早、最高、最低、按编号顺序、分页顺序稳定”，就不要依赖数据库默认返回顺序，直接写 `ORDER BY`。

## 1.2 ORDER BY 的位置与基本语法

`ORDER BY` 通常写在查询语句靠后的位置。

常见 SQL 子句的书写顺序如下：

```sql
SELECT   查询字段
FROM     表名
WHERE    行过滤条件
GROUP BY 分组字段
HAVING   分组后过滤条件
ORDER BY 排序字段
LIMIT    偏移量, 行数;
```

因此，如果同时使用 `ORDER BY` 和 `LIMIT`，一般是：

```sql
SELECT 字段
FROM 表名
ORDER BY 排序字段
LIMIT 偏移量, 行数;
```

也就是：**先排序，再截取需要的记录范围**。

`ORDER BY` 的基本格式：

```sql
ORDER BY 字段名 [ASC | DESC];
```

也可以同时指定多个排序字段：

```sql
ORDER BY 字段1 [ASC | DESC], 字段2 [ASC | DESC], 字段3 [ASC | DESC];
```

### 回查提示

记住一个常用骨架：`WHERE -> GROUP BY -> HAVING -> ORDER BY -> LIMIT`。排序通常在过滤、分组之后，在分页之前。

## 1.3 单列排序

下面的例子查询员工的姓名、职位、部门和入职日期，并按 `hire_date` 升序排序。

![1554974260133.png](./images/1554974260133.png)

```sql
SELECT   last_name, job_id, department_id, hire_date
FROM     employees
ORDER BY hire_date;
```

如果没有显式写排序方向，`ORDER BY hire_date` 等价于：

```sql
ORDER BY hire_date ASC;
```

![1554974822229.png](./images/1554974822229.png)

![1554974827522.png](./images/1554974827522.png)

下面的例子改为按入职日期降序排序，也就是较晚入职的记录排在前面。

```sql
SELECT   last_name, job_id, department_id, hire_date
FROM     employees
ORDER BY hire_date DESC;
```

### 回查提示

如果你只需要一个明确顺序，先从单列排序开始。确定单列无法满足业务顺序时，再改成多列排序。

## 1.4 使用列别名排序

`ORDER BY` 可以使用 `SELECT` 列表中定义的列别名。

下面的例子把 `salary * 12` 命名为 `annsal`，再按 `annsal` 排序。

![1554974853194.png](./images/1554974853194.png)

![1554974858252.png](./images/1554974858252.png)

```sql
SELECT employee_id, last_name, salary * 12 AS annsal
FROM   employees
ORDER BY annsal;
```

如果需要按年薪由高到低排序，可以继续在别名后指定排序方向：

```sql
SELECT employee_id, last_name, salary * 12 AS annsal
FROM   employees
ORDER BY annsal DESC;
```

这种写法适合用于排序派生字段，例如：

- 年薪
- 折扣后金额
- 税后金额
- 计算后的统计值
- 拼接或转换后的显示字段

它能避免在 `ORDER BY` 中重复写同一段表达式。

例如下面这种写法，也能表达同样的排序意图：

```sql
SELECT employee_id, last_name, salary * 12 AS annsal
FROM   employees
ORDER BY salary * 12 DESC;
```

但当表达式变长时，`ORDER BY` 中重复写整段计算逻辑会让 SQL 更难读，也更不利于维护。只要排序对象是计算字段或较长表达式，通常更适合先定义别名，再按别名排序。

需要注意：

- 在 MySQL 中，`ORDER BY` 可以使用 `SELECT` 中定义的列别名。
- 但 `WHERE` 通常不能直接使用 `SELECT` 中定义的列别名，因为 `WHERE` 的逻辑处理阶段早于 `SELECT` 列表。
- 不同数据库对别名可用位置的支持可能略有差异，本节以 MySQL 为主。

### 回查提示

排序计算字段时，优先给它一个清楚的别名，再按别名排序，通常比重复写表达式更清楚。

## 1.5 使用查询列的位置排序（不推荐）

除了字段名称与列别名，`ORDER BY` 也可以使用查询结果中的列位置编号。

例如下面的 `ORDER BY 3 DESC`，表示按 `SELECT` 列表中的第三列，也就是 `annsal`，进行降序排序。

```sql
SELECT employee_id, last_name, salary * 12 AS annsal
FROM   employees
ORDER BY 3 DESC;
```

这种写法虽然简短，但不建议作为常规写法。

原因是它依赖 `SELECT` 列表的顺序：一旦前面的字段增删或调整位置，`ORDER BY 3` 指向的字段就可能变化，导致排序结果和原本预期不一致。

例如原本第三列是 `annsal`：

```sql
SELECT employee_id, last_name, salary * 12 AS annsal
FROM   employees
ORDER BY 3 DESC;
```

后来有人把 `department_id` 插入到第三列：

```sql
SELECT employee_id, last_name, department_id, salary * 12 AS annsal
FROM   employees
ORDER BY 3 DESC;
```

此时 `ORDER BY 3 DESC` 就变成按 `department_id` 排序，而不是按 `annsal` 排序。

为了可读性和维护性，实务中优先写字段名或列别名。

### 回查提示

看到 `ORDER BY 3` 这类写法时，先去数它对应的是 `SELECT` 里的第几列，再判断这种写法是否值得保留。自己写 SQL 时，除非是临时查询，否则不建议使用。

## 1.6 多列排序

当一个排序字段无法完全决定结果顺序时，可以使用多列排序。

排序字段的先后顺序很重要：

1. 先按第一个字段排序。
2. 第一个字段的值相同时，才继续按第二个字段排序。
3. 第二个字段的值也相同时，才继续按第三个字段排序。
4. 依此类推。

```sql
SELECT last_name, department_id, salary
FROM   employees
ORDER BY department_id, salary DESC;
```

上面的查询会先按 `department_id` 排序。只有在 `department_id` 相同的记录之间，才会继续按 `salary DESC` 排序。

![1554974901572.png](./images/1554974901572.png)

![1554974907498.png](./images/1554974907498.png)

使用多列排序时需要注意：

- 多列排序从左到右依次生效。
- 越靠左的字段，排序优先级越高。
- 如果第一列的值已经全部不同，后续排序字段通常就没有机会影响结果顺序。
- 每个排序字段都可以单独指定 `ASC` 或 `DESC`。
- 没有指定排序方向时，默认使用 `ASC`。
- 普通查询中，可以使用不在 `SELECT` 列表中的字段排序；但如果搭配 `DISTINCT`、复杂分组或某些数据库特性，可能会有额外限制。

例如：

```sql
SELECT last_name, department_id, salary
FROM   employees
ORDER BY department_id ASC, salary DESC, employee_id ASC;
```

含义是：

1. 先按部门编号升序排序。
2. 同部门内，再按薪资降序排序。
3. 如果部门和薪资都相同，再按员工编号升序排序。

### 回查提示

多列排序的关键不是字段数量，而是优先级顺序。越靠左的字段，越先决定结果顺序。

## 1.7 稳定排序与唯一兜底字段

在业务列表或分页查询中，常常需要“稳定排序”。

所谓稳定排序，是指同一批数据在没有变化的情况下，多次查询应尽量返回相同的顺序。

只按一个非唯一字段排序时，可能出现顺序不稳定的问题。

例如：

```sql
SELECT id, title, create_time
FROM articles
ORDER BY create_time DESC
LIMIT 0, 10;
```

如果多篇文章的 `create_time` 完全相同，数据库并不保证这些文章之间的相对顺序固定。这样在分页时，可能出现：

- 第一页和第二页之间有重复资料。
- 某些资料被跳过。
- 同一页刷新后顺序改变。

比较稳的写法是加入唯一字段作为兜底排序：

```sql
SELECT id, title, create_time
FROM articles
ORDER BY create_time DESC, id DESC
LIMIT 0, 10;
```

这样排序规则就变成：

1. 先按创建时间由新到旧排序。
2. 如果创建时间相同，再按 `id` 由大到小排序。

只要 `id` 是唯一的，就能让结果顺序更可预期。

### 常见兜底排序字段

| 场景 | 常见排序写法 |
| --- | --- |
| 最新资料列表 | `ORDER BY create_time DESC, id DESC` |
| 最早资料列表 | `ORDER BY create_time ASC, id ASC` |
| 金额由高到低 | `ORDER BY amount DESC, id DESC` |
| 部门内薪资排序 | `ORDER BY department_id ASC, salary DESC, employee_id ASC` |

### 回查提示

只要 SQL 会用于分页、列表页、导出顺序或对账结果，就建议让 `ORDER BY` 最后有一个唯一字段兜底。

## 1.8 NULL 值排序

排序字段可能出现 `NULL` 值。

在 MySQL 中，默认排序规则通常是：

- `ORDER BY 字段 ASC`：`NULL` 会排在前面。
- `ORDER BY 字段 DESC`：`NULL` 会排在后面。

例如：

```sql
SELECT employee_id, last_name, commission_pct
FROM   employees
ORDER BY commission_pct ASC;
```

如果 `commission_pct` 有 `NULL`，这些 `NULL` 通常会出现在结果前面。

如果希望升序排序时让 `NULL` 排在最后，可以写：

```sql
SELECT employee_id, last_name, commission_pct
FROM   employees
ORDER BY commission_pct IS NULL, commission_pct ASC;
```

解释：

- `commission_pct IS NULL` 对非空值返回 `0`，对 `NULL` 返回 `1`。
- MySQL 默认升序排序时，`0` 在前，`1` 在后。
- 所以非空值会先排，`NULL` 会被放到最后。

如果希望降序排序时让 `NULL` 排在最前，可以写：

```sql
SELECT employee_id, last_name, commission_pct
FROM   employees
ORDER BY commission_pct IS NOT NULL, commission_pct DESC;
```

### 回查提示

只要排序字段可能为空，就要确认业务想让 `NULL` 排前面还是后面。不要只记 `ASC`、`DESC`，也要考虑空值位置。

## 1.9 ORDER BY 与索引的基础注意点

`ORDER BY` 不只是显示顺序问题，也可能影响查询性能。

一般来说：

- 如果排序字段上有合适的索引，数据库可能直接利用索引顺序返回结果。
- 如果排序字段没有合适的索引，数据库可能需要额外排序。
- 如果在排序字段上使用函数或复杂表达式，可能使索引不容易被利用。
- 多列排序是否能有效利用索引，和索引字段顺序、排序方向、过滤条件都有关系。

例如下面的写法可能较容易利用 `create_time` 或组合索引：

```sql
SELECT id, title, create_time
FROM articles
ORDER BY create_time DESC, id DESC
LIMIT 10;
```

但下面这种写法，因为对字段做了函数处理，可能不利于直接利用索引排序：

```sql
SELECT id, title, create_time
FROM articles
ORDER BY DATE(create_time) DESC
LIMIT 10;
```

这不表示不能写函数排序，而是要知道：当数据量很大时，排序方式可能影响性能。

### 回查提示

小数据量先以可读性为主；大数据量列表查询要开始关注 `ORDER BY` 是否能配合索引，尤其是后台管理列表、交易记录、日志查询、订单查询。

## 常见混淆点

- 不写 `ORDER BY` 时，不要依赖查询结果的默认显示顺序。
- `ASC` 是升序，`DESC` 是降序；不写排序方向时默认是 `ASC`。
- `ORDER BY` 通常写在 `WHERE`、`GROUP BY`、`HAVING` 后面，在 `LIMIT` 前面。
- `ORDER BY` 可以使用 `SELECT` 中定义的列别名。
- `ORDER BY annsal` 是按列别名排序；`ORDER BY 3` 是按查询结果的第三列排序，后者更依赖列顺序，不推荐作为常规写法。
- 多列排序不是同时平均生效，而是按字段顺序逐级排序。
- 分页查询只写 `ORDER BY create_time DESC` 可能不够稳定，最好再加唯一字段，例如 `id DESC`。
- MySQL 中 `NULL` 在升序排序时通常排前面，在降序排序时通常排后面。
- `DESC employees;` 中的 `DESC` 是 `DESCRIBE` 的缩写；`ORDER BY salary DESC` 中的 `DESC` 才表示降序排序。
- 字符串排序结果可能受到字符集与排序规则，也就是 collation 的影响。

## 一句话抓核心

排序数据的核心是：只要查询结果需要稳定顺序，就用 `ORDER BY` 明确指定排序字段和方向；多列排序时，字段从左到右逐级决定结果顺序；分页或列表查询时，最好用唯一字段兜底，保证结果稳定。

## 小结

这一节你需要记住：

- 查询结果需要稳定顺序时，应显式使用 `ORDER BY`。
- `ASC` 表示升序，`DESC` 表示降序，默认方向是 `ASC`。
- `ORDER BY` 通常写在 `WHERE`、`GROUP BY`、`HAVING` 后面，在 `LIMIT` 前面。
- `ORDER BY` 可以按单列排序，也可以按多列排序。
- `ORDER BY` 可以使用 `SELECT` 中定义的列别名。
- `ORDER BY` 也可以使用查询列的位置编号，但这种写法依赖字段顺序，维护风险较高。
- 多列排序时，只有前一列出现相同值，后一列的排序才会进一步影响结果。
- 分页或列表查询中，建议加入主键或唯一字段作为兜底排序，避免结果不稳定。
- 排序字段可能有 `NULL` 时，要确认业务希望空值排前面还是后面。
- 大数据量查询中，`ORDER BY` 是否能配合索引，会影响查询性能。
