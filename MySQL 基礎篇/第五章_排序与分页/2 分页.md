## 本节导读

这一节主要说明如何在 MySQL 中使用分页查询。

分页的核心是：不要一次把所有记录都显示出来，而是先确定结果顺序，再只取当前需要查看的那一小段结果。

第一次阅读时，建议先理解“偏移量 + 行数”这组规则，再看常见写法、分页公式和 `LIMIT ... OFFSET ...` 的等价写法。复习或实务使用时，重点关注三个问题：

1. 分页前有没有先排序？
2. 排序规则是否稳定？
3. 页码很大时，是否会出现性能问题？

## 关键字

- `分页`：把一整批查询结果分成多段显示。
- `LIMIT`：MySQL 中限制返回行数的分页子句。
- `位置偏移量`：从第几条记录开始取数据，第一条记录的偏移量是 `0`。
- `行数`：本次查询最多要返回多少条记录。
- `OFFSET`：另一种显式表示偏移量的写法。
- `PageNo`：当前页码，常见业务习惯是从 `1` 开始。
- `PageSize`：每页显示的记录数。
- `稳定分页`：分页结果顺序可预期，不容易出现重复、跳页或漏资料。
- `大偏移量分页`：例如 `LIMIT 100000, 10`，偏移量很大的分页。
- `Keyset Pagination`：基于上一页最后一笔数据继续往后查的分页方式，也常被称为游标分页或 Seek Pagination。

## 快速回查表

| 场景 | 写法 | 说明 |
| --- | --- | --- |
| 取前 10 条记录 | `LIMIT 10` | 等价于 `LIMIT 0, 10` |
| 取第 11 至 20 条记录 | `LIMIT 10, 10` | 从偏移量 `10` 开始，再取 `10` 条 |
| 取第 5 条之后的 3 条记录 | `LIMIT 4, 3` | 偏移量从 `0` 开始计数 |
| 用显式偏移量写法取数据 | `LIMIT 3 OFFSET 4` | 与 `LIMIT 4, 3` 等价 |
| 写第 `PageNo` 页数据 | `LIMIT (PageNo - 1) * PageSize, PageSize` | 常见分页公式 |
| 稳定分页 | `ORDER BY create_time DESC, id DESC LIMIT 0, 10` | 时间相同时，再用唯一字段兜底 |
| 只需要一条记录 | `LIMIT 1` | 可减少返回量，也常用于“只取一条”场景 |
| 大偏移量分页 | `LIMIT 100000, 10` | 可能需要先扫描或处理大量前置记录，再丢弃 |
| Keyset 分页 | `WHERE id < 上一页最后id ORDER BY id DESC LIMIT 10` | 适合连续下一页，不适合任意跳页 |

## 2.1 分页背景

分页常见于两类场景：

- 查询返回的记录太多，一次全部显示出来不方便查看。
- 明明表里有很多数据，但当前页面只想显示其中一小段，例如第 `1` 页、第 `2` 页、第 `11` 至 `20` 条记录。

所谓分页显示，就是将数据库中的结果集按段切开，每次只显示当前需要的那一部分。

例如后台订单列表可能有 100 万笔订单，但用户当前页面只需要看最近的 20 笔：

```sql
SELECT id, order_no, amount, create_time
FROM orders
ORDER BY create_time DESC, id DESC
LIMIT 0, 20;
```

### 回查提示

当你已经会写查询，但结果太多不方便展示时，就应该想到分页，而不是把所有数据一次查出来再手动挑选。

## 2.2 MySQL 中使用 LIMIT 实现分页

MySQL 中常用 `LIMIT` 子句实现分页，基本格式如下：

```sql
LIMIT [位置偏移量,] 行数
```

含义：

- 第一个参数“位置偏移量”表示从哪一行开始取数据，是可选参数。
- 如果不指定偏移量，则默认从第一条记录开始返回。
- 第一条记录的位置偏移量是 `0`。
- 第二条记录的位置偏移量是 `1`。
- 第二个参数“行数”表示本次最多返回多少条记录。

例如：

```sql
SELECT *
FROM employees
LIMIT 0, 10;
```

表示从偏移量 `0` 开始，最多返回 `10` 条记录，也就是第一页常见写法。

`LIMIT 10` 是简写：

```sql
SELECT *
FROM employees
LIMIT 10;
```

它等价于：

```sql
SELECT *
FROM employees
LIMIT 0, 10;
```

### 回查提示

只要你分不清 `LIMIT 10, 20` 里的两个数字分别代表什么，就回到这里先确认：**偏移量在前，行数在后**。

## 2.3 ORDER BY 与 LIMIT 的先后关系

分页不是单纯把资料切开而已。更准确地说，分页应该是：

1. 先确定完整结果集的顺序。
2. 再从这个顺序中截取某一段资料。

因此，分页查询通常应该搭配 `ORDER BY`：

```sql
SELECT id, last_name, hire_date
FROM employees
ORDER BY hire_date DESC, employee_id DESC
LIMIT 0, 10;
```

SQL 书写顺序通常是：

```sql
SELECT   查询字段
FROM     表名
WHERE    行过滤条件
GROUP BY 分组字段
HAVING   分组后过滤条件
ORDER BY 排序字段
LIMIT    偏移量, 行数;
```

如果没有 `ORDER BY`，数据库不保证每次返回的资料顺序都一样。对于分页来说，这可能导致：

- 第一页和第二页出现重复资料。
- 某些资料被跳过。
- 使用者刷新页面后顺序变化。
- 同一套 SQL 在数据量变化后结果顺序改变。

### 回查提示

分页时不要只问“取第几页”，还要先问“按什么顺序分页”。没有排序规则的分页，通常是不可靠的。

## 2.4 常见分页写法

下面是几种最常见的分页写法。

```sql
-- 前 10 条记录
SELECT * FROM 表名 LIMIT 0, 10;

-- 前 10 条记录的简写
SELECT * FROM 表名 LIMIT 10;

-- 第 11 至 20 条记录
SELECT * FROM 表名 LIMIT 10, 10;

-- 第 21 至 30 条记录
SELECT * FROM 表名 LIMIT 20, 10;
```

在 MySQL 中，还可以写成更直观的 `OFFSET` 形式：

```sql
SELECT * FROM 表名 LIMIT 3 OFFSET 4;
```

这表示从偏移量 `4` 开始，返回后面的 `3` 条记录。

由于第一条记录的偏移量是 `0`，所以偏移量 `4` 对应的是第 `5` 条记录。

它和下面这条语句等价：

```sql
SELECT * FROM 表名 LIMIT 4, 3;
```

两种写法的差异是参数顺序：

| 写法 | 参数顺序 | 含义 |
| --- | --- | --- |
| `LIMIT 4, 3` | 先偏移量，再行数 | 从偏移量 `4` 开始，取 `3` 条 |
| `LIMIT 3 OFFSET 4` | 先行数，再偏移量 | 取 `3` 条，偏移量是 `4` |

### 回查提示

`LIMIT offset, size` 和 `LIMIT size OFFSET offset` 的参数顺序刚好不同，看到时要特别确认。

## 2.5 分页公式

分页最常用的公式是：

```sql
SELECT *
FROM table_name
ORDER BY 排序字段
LIMIT (PageNo - 1) * PageSize, PageSize;
```

这里：

- `PageNo` 表示当前页码。
- `PageSize` 表示每页显示多少条记录。
- `(PageNo - 1) * PageSize` 用来算出当前页第一条记录的偏移量。

例如：

| 页码 | 每页笔数 | 偏移量计算 | LIMIT 写法 |
| --- | --- | --- | --- |
| 第 1 页 | 10 | `(1 - 1) * 10 = 0` | `LIMIT 0, 10` |
| 第 2 页 | 10 | `(2 - 1) * 10 = 10` | `LIMIT 10, 10` |
| 第 3 页 | 10 | `(3 - 1) * 10 = 20` | `LIMIT 20, 10` |
| 第 5 页 | 20 | `(5 - 1) * 20 = 80` | `LIMIT 80, 20` |

实际查询示例：

```sql
SELECT id, order_no, amount, create_time
FROM orders
ORDER BY create_time DESC, id DESC
LIMIT 20, 10;
```

如果 `PageNo = 3`，`PageSize = 10`，则偏移量是：

```text
(PageNo - 1) * PageSize = (3 - 1) * 10 = 20
```

所以 SQL 是：

```sql
LIMIT 20, 10;
```

### 回查提示

当业务需求不是“前 10 条”，而是“第 N 页，每页 M 条”时，直接套公式比手算更稳。

## 2.6 稳定分页

分页最容易忽略的问题是：排序规则不稳定。

例如：

```sql
SELECT id, title, create_time
FROM articles
ORDER BY create_time DESC
LIMIT 0, 10;
```

如果多篇文章的 `create_time` 完全相同，数据库不保证这些资料之间的相对顺序固定。

比较稳的写法是加上唯一字段作为兜底排序：

```sql
SELECT id, title, create_time
FROM articles
ORDER BY create_time DESC, id DESC
LIMIT 0, 10;
```

这样排序规则变成：

1. 先按创建时间由新到旧排序。
2. 如果创建时间相同，再按 `id` 由大到小排序。

因为 `id` 通常是唯一的，所以最终排序结果更稳定。

### 常见稳定分页写法

| 场景 | 建议写法 |
| --- | --- |
| 最新列表 | `ORDER BY create_time DESC, id DESC` |
| 最早列表 | `ORDER BY create_time ASC, id ASC` |
| 金额由高到低 | `ORDER BY amount DESC, id DESC` |
| 分数排行榜 | `ORDER BY score DESC, id ASC` |
| 部门内薪资排序 | `ORDER BY department_id ASC, salary DESC, employee_id ASC` |

### 回查提示

只要分页 SQL 会给前端列表、后台管理页、报表导出使用，就尽量让排序规则稳定。最简单做法是最后补一个唯一字段，例如主键 `id`。

## 2.7 大偏移量分页的性能问题

`LIMIT` 很方便，但大偏移量分页可能有性能问题。

例如：

```sql
SELECT id, order_no, amount, create_time
FROM orders
ORDER BY id ASC
LIMIT 100000, 10;
```

这表示跳过前 `100000` 条，再取 `10` 条。

问题在于：数据库通常不能直接“瞬移”到第 `100001` 条。它可能需要先扫描、排序或定位前面大量资料，再丢弃前 `100000` 条，最后返回 `10` 条。

所以，页码越大，查询可能越慢。

尤其是以下场景要注意：

- 订单表、日志表、交易明细表数据量很大。
- 后台允许使用者直接跳到很后面的页码。
- 排序字段没有合适索引。
- `ORDER BY` 使用函数或复杂表达式。
- 查询同时带有复杂 `WHERE` 条件。

### 如何缓解

常见方向有：

- 给 `WHERE` 条件和 `ORDER BY` 字段建立合适索引。
- 限制最大可跳页数或最大导出数量。
- 对连续翻页场景使用 Keyset Pagination。
- 对报表类需求改成异步导出，不让用户在页面上无限翻页。

### 回查提示

`LIMIT 0, 10` 通常没问题；`LIMIT 100000, 10` 才是需要警觉的地方。偏移量越大，越应该思考索引和分页策略。

## 2.8 Keyset Pagination

Keyset Pagination 又常被称为游标分页或 Seek Pagination。

它的核心思想不是“跳过前面多少条”，而是“从上一页最后一笔资料之后继续查”。

例如按照 `id` 由大到小查询第一页：

```sql
SELECT id, title, create_time
FROM articles
ORDER BY id DESC
LIMIT 10;
```

假设第一页最后一笔资料的 `id = 900`，下一页可以这样查：

```sql
SELECT id, title, create_time
FROM articles
WHERE id < 900
ORDER BY id DESC
LIMIT 10;
```

这样数据库不需要跳过大量前置记录，而是从条件范围继续往后查。

如果排序规则是 `create_time DESC, id DESC`，下一页条件会稍微复杂：

```sql
SELECT id, title, create_time
FROM articles
WHERE
    create_time < '2026-04-26 10:00:00'
    OR (create_time = '2026-04-26 10:00:00' AND id < 900)
ORDER BY create_time DESC, id DESC
LIMIT 10;
```

这里的 `'2026-04-26 10:00:00'` 和 `900` 代表上一页最后一笔资料的排序值。

### Offset 分页与 Keyset 分页比较

| 比较点 | Offset 分页 | Keyset 分页 |
| --- | --- | --- |
| 典型写法 | `LIMIT 100000, 10` | `WHERE id < lastId LIMIT 10` |
| 是否适合跳到任意页 | 适合 | 不适合 |
| 是否适合连续下一页 | 可以，但大页码可能慢 | 很适合 |
| 大数据量性能 | 大 offset 可能变慢 | 通常较稳定 |
| 实作复杂度 | 简单 | 稍复杂，需要记录上一页最后一笔资料 |
| 常见场景 | 后台表格、传统分页 | 动态消息、订单流水、日志列表、无限滚动 |

### 回查提示

如果需求是“第 1 页、第 2 页、第 3 页、可跳页”，通常用 Offset 分页。如果需求是“下一页、再下一页、无限滚动”，可以考虑 Keyset 分页。

## 2.9 后端接口中常见的分页参数处理

在后端接口中，分页通常不会直接让前端传 `offset`，而是传：

- `pageNo`：第几页。
- `pageSize`：每页几笔。

后端再换算成：

```text
offset = (pageNo - 1) * pageSize
```

例如 Java 后端常见逻辑：

```java
int pageNo = Math.max(request.getPageNo(), 1);
int pageSize = Math.min(Math.max(request.getPageSize(), 1), 100);
int offset = (pageNo - 1) * pageSize;
```

这段逻辑表达的是：

- `pageNo` 最小为 `1`。
- `pageSize` 最小为 `1`。
- `pageSize` 最大限制为 `100`，避免一次查太多。
- 再用公式计算偏移量。

SQL 可以写成：

```sql
SELECT id, order_no, amount, create_time
FROM orders
ORDER BY create_time DESC, id DESC
LIMIT #{offset}, #{pageSize};
```

### 回查提示

实务上不要完全相信前端传来的页码和每页笔数。后端应该限制 `pageNo` 和 `pageSize` 的范围。

## 2.10 其他数据库的分页写法

不同数据库管理系统使用的分页关键字可能不同。本节重点是 MySQL 的 `LIMIT`，但下面这些写法也值得知道。

### MySQL / MariaDB

MySQL 和 MariaDB 常见写法：

```sql
SELECT name, hp_max
FROM heros
ORDER BY hp_max DESC
LIMIT 0, 5;
```

也可以写成：

```sql
SELECT name, hp_max
FROM heros
ORDER BY hp_max DESC
LIMIT 5 OFFSET 0;
```

### PostgreSQL / SQLite

PostgreSQL 和 SQLite 常见写法：

```sql
SELECT name, hp_max
FROM heros
ORDER BY hp_max DESC
LIMIT 5 OFFSET 0;
```

### SQL Server

SQL Server 可以使用 `TOP` 取得前几笔：

```sql
SELECT TOP 5 name, hp_max
FROM heros
ORDER BY hp_max DESC;
```

但 `TOP` 主要适合“取前 N 笔”，不是完整的分页语法。

SQL Server 2012 之后常见分页写法是：

```sql
SELECT name, hp_max
FROM heros
ORDER BY hp_max DESC
OFFSET 0 ROWS FETCH NEXT 5 ROWS ONLY;
```

### DB2

DB2 常见使用 `FETCH FIRST ... ROWS ONLY`：

```sql
SELECT name, hp_max
FROM heros
ORDER BY hp_max DESC
FETCH FIRST 5 ROWS ONLY;
```

### Oracle 12c 之后

Oracle 12c 之后可以使用类似标准 SQL 的写法：

```sql
SELECT last_name, salary
FROM employees
ORDER BY salary DESC
OFFSET 0 ROWS FETCH NEXT 5 ROWS ONLY;
```

### Oracle 传统 ROWNUM 写法

Oracle 传统写法常结合 `ROWNUM`，但要特别注意排序与取前几笔的顺序。

如果想表达“先按薪资降序排序，再取前 4 笔”，应使用子查询：

```sql
SELECT *
FROM (
    SELECT last_name, salary
    FROM employees
    ORDER BY salary DESC
)
WHERE rownum < 5;
```

不要直接写成：

```sql
SELECT rownum, last_name, salary
FROM employees
WHERE rownum < 5
ORDER BY salary DESC;
```

因为这种写法容易变成先取出前几笔，再对这几笔排序，不一定代表全表薪资最高的前几笔。

### 回查提示

如果你发现别人的分页 SQL 里根本没有 `LIMIT`，先别急着判断写错了，先确认对方用的是哪一种数据库。

## 使用 LIMIT 的价值

合理使用 `LIMIT` 有几个直接好处：

- 可以减少一次返回的数据量。
- 可以降低网络传输成本。
- 可以避免前端一次渲染过多资料。
- 如果业务上只需要少量记录，例如只取一条结果，`LIMIT 1` 往往能减少不必要的返回。

不过需要注意：

`LIMIT 1` 不代表查询一定很快。它是否能明显提升性能，仍然取决于：

- `WHERE` 条件是否能使用索引。
- `ORDER BY` 是否能使用索引。
- 是否需要先排序大量资料后才能取第一笔。

例如：

```sql
SELECT *
FROM users
WHERE email = 'test@example.com'
LIMIT 1;
```

如果 `email` 有索引，通常可以很快定位。

但下面这种查询：

```sql
SELECT *
FROM users
ORDER BY create_time DESC
LIMIT 1;
```

如果 `create_time` 没有合适索引，数据库仍可能需要处理大量资料才能找出最新一笔。

## 常见混淆点

- `LIMIT` 的第一个数字不是“第几条记录”，而是偏移量；它从 `0` 开始计数。
- `LIMIT 10` 和 `LIMIT 0, 10` 是等价写法。
- `LIMIT 4, 3` 表示偏移量 `4`，取 `3` 条。
- `LIMIT 3 OFFSET 4` 表示取 `3` 条，偏移量 `4`。
- `LIMIT offset, size` 和 `LIMIT size OFFSET offset` 的参数顺序不同。
- 如果查询结果需要稳定顺序，分页前应该先写 `ORDER BY`。
- 分页排序最好加唯一字段兜底，例如 `ORDER BY create_time DESC, id DESC`。
- `LIMIT` 在 MySQL 中通常写在 `SELECT` 语句最后。
- 页码很大时，Offset 分页可能变慢。
- `LIMIT 1` 可以减少返回数量，但不保证查询一定快。
- 不同数据库的分页语法不统一，跨数据库迁移时不要直接照抄。
- Oracle 传统 `ROWNUM` 取前几笔时，要注意先排序再取数通常需要子查询。

## 一句话抓核心

分页的核心是：先用 `ORDER BY` 确定稳定顺序，再用 `LIMIT` 指定“从哪一条开始、取多少条”；如果偏移量很大，要进一步考虑索引和 Keyset Pagination。

## 小结

这一节你需要记住：

- MySQL 中分页通常使用 `LIMIT [位置偏移量,] 行数`。
- 偏移量从 `0` 开始，第一页通常对应 `LIMIT 0, PageSize`。
- `LIMIT n OFFSET m` 是 `LIMIT m, n` 的另一种写法，但参数顺序不同。
- 业务上按页查询时，可以直接套用 `(PageNo - 1) * PageSize` 的公式。
- 如果分页结果需要稳定顺序，应和 `ORDER BY` 一起使用。
- 分页排序最好加唯一字段兜底，例如 `ORDER BY create_time DESC, id DESC`。
- 大偏移量分页可能变慢，尤其是数据量大、排序字段没有索引时。
- 连续下一页或无限滚动场景，可以考虑 Keyset Pagination。
- 后端接口应限制 `pageNo` 和 `pageSize`，避免异常参数造成性能问题。
- 跨数据库分页语法不同，Oracle 传统 `ROWNUM` 写法尤其要注意排序与取数顺序。
