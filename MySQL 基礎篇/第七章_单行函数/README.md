# 第七章_单行函数

这一章开始把学习重点从「SQL 查询语句怎么写」推进到「查询出来的值还能怎么加工、转换、判断和展示」。

本章的核心不是死记所有函数名称，而是先建立 MySQL 函数的整体使用框架：

```text
函数是什么
 -> SQL 为什么需要函数
 -> 单行函数与聚合函数有什么差异
 -> 每一类函数适合解决什么问题
 -> 实务中使用函数时要注意什么
```

如果只背函数名，很容易出现两个问题：

1. 知道函数怎么写，但不知道什么时候该用。
2. 写得出 SQL，但忽略 `NULL`、返回类型、索引影响、版本差异、时区、字符集与安全性问题。

所以这一章的学习目标不是「把所有函数背完」，而是建立一套判断方式：

> 遇到查询结果加工需求时，能判断它属于数值处理、字符串处理、日期时间处理、条件判断、安全处理、环境排查，还是其他辅助转换。

---

## 本章在整体学习地图中的位置

在进入本章前，你已经学习过：

- 数据库基础概念
- MySQL 环境
- 基础 `SELECT`
- 运算符
- 排序与分页
- 多表查询

这些内容主要解决的是：

```text
怎么把资料查出来？
```

从这一章开始，重点变成：

```text
查出来之后，怎么把值处理成业务真正需要的样子？
```

例如：

- 把金额保留两位小数。
- 把姓名、邮箱、编号格式统一。
- 把日期格式化成指定格式。
- 判断状态码并输出中文说明。
- 把 `NULL` 改成默认值。
- 查看当前数据库、连接用户、字符集和排序规则。
- 对 IP、进制、字符集等特殊值做转换。
- 理解哪些旧版安全函数不适合继续当成现代最佳实践。

本章也是后续学习聚合函数、分组查询、报表统计、数据清洗和 SQL 优化的重要前置基础。

---

## 你会在本章学到什么

- 函数的基本概念，以及 SQL 中为什么需要函数。
- 内置函数与自定义函数的区别。
- 单行函数与聚合函数的差异。
- MySQL 常见内置函数的分类方式。
- 数值函数如何处理取整、小数位、随机数、对数、进制转换。
- 字符串函数如何处理拼接、截取、替换、去空格、定位和清洗。
- 日期时间函数如何处理当前时间、时间戳、日期拆解、日期加减、格式化与解析。
- 流程控制函数如何处理 `IF()`、`IFNULL()`、`CASE WHEN`、状态码映射和分类输出。
- 加密与解密函数如何区分旧版教材函数、哈希函数、现代加密函数与安全注意事项。
- MySQL 信息函数如何排查当前数据库、版本、用户、字符集和排序规则。
- 其他函数如何处理数字格式化、进制转换、IP 转换、性能测试和字符集转换。
- 使用函数时需要注意的 `NULL`、返回类型、索引影响、版本差异、可移植性和可读性问题。

## 本章共同实务提醒

### 1. 注意 `NULL`

很多函数只要参数中出现 `NULL`，结果就可能变成 `NULL`。

例如：

```sql
SELECT CONCAT('A', NULL, 'B');
```

结果通常是：

```text
NULL
```

所以在实际查询中，经常需要搭配：

```sql
IFNULL()
COALESCE()
NULLIF()
CASE WHEN
```

---

### 2. 注意函数的返回类型

有些函数看起来在处理数字或日期，但返回结果可能是字符串。

例如：

| 函数 | 常见返回结果 |
| --- | --- |
| `FORMAT()` | 字符串 |
| `DATE_FORMAT()` | 字符串 |
| `CONCAT()` | 字符串 |
| `UNIX_TIMESTAMP()` | 数值时间戳 |
| `STR_TO_DATE()` | 日期或日期时间值 |
| `INET_ATON()` | IPv4 数字 |
| `INET6_ATON()` | 二进位字符串 |

返回类型会影响：

- 排序
- 比较
- Java 接值类型
- 是否还能继续计算
- 是否适合存回数据库

---

### 3. 注意索引影响

在 `WHERE` 条件中对索引字段套函数，可能影响索引使用。

例如：

```sql
WHERE YEAR(order_time) = 2026
WHERE LOWER(email) = 'test@example.com'
WHERE DATE_FORMAT(create_time, '%Y-%m') = '2026-04'
WHERE CONVERT(code USING utf8mb4) = 'A001'
```

这些写法不是绝对不能用，但在大表或正式系统中要谨慎。

更常见的思路是：

- 日期条件改成范围查询。
- 字符串格式在写入时就规范化。
- 字符集在表设计阶段就统一。
- 必要时使用合适的索引、生成列或函数索引。
- 最终以执行计划为准。

---

### 4. 注意 DBMS 可移植性

MySQL 函数不一定能直接搬到 Oracle、SQL Server、PostgreSQL 使用。

例如：

- 字符串拼接写法不同。
- 日期格式化函数不同。
- 时间戳处理方式不同。
- 加密函数支援情况不同。
- 类型转换语法不同。

所以笔记中看到的函数，应先默认是：

```text
MySQL 写法
```

跨数据库时要查目标 DBMS 官方文档。

---

### 5. 注意日期时间的时区与规则

日期时间函数容易踩到这些问题：

- `NOW()` 与 `SYSDATE()` 的执行语义可能不同。
- `UTC_DATE()` / `UTC_TIME()` 和本地时间不同。
- `UNIX_TIMESTAMP()` 与时区有关。
- `WEEK()` 的周起始规则会影响结果。
- `DATEDIFF()` 只看日期差，不看具体时分秒。
- `DATE_FORMAT()` 返回的是字符串，不是日期型别。
- `STR_TO_DATE()` 的格式必须和原字符串对应。

---

### 6. 注意字符集与排序规则

字符串函数的结果可能受字符集和排序规则影响，例如：

- 大小写比较是否敏感。
- 中文排序是否符合预期。
- `LENGTH()` 和 `CHAR_LENGTH()` 是否不同。
- `STRCMP()` 的比较结果。
- `CHARSET()` 和 `COLLATION()` 查到的实际规则。

遇到异常时，可以回查：

- [3 字符串函数](./3%20字符串函数.md)
- [7 MySQL 信息函数](./7%20MySQL%20信息函数.md)
- [8 其他函数](./8%20其他函数.md)

### 7. 注意安全函数的版本差异

加密与解密函数不能只看旧教材。

需要特别注意：

- `PASSWORD()` 是旧版 MySQL 相关函数，不适合作为应用系统密码储存方案。
- `ENCODE()`、`DECODE()` 属于旧版教材常见函数，现代版本应注意兼容性与弃用情况。
- `MD5()`、`SHA()` 不适合当成高安全性的现代密码储存方案。
- 更应该认识 `SHA2()`、`AES_ENCRYPT()`、`AES_DECRYPT()`、`RANDOM_BYTES()` 等现代函数。
- 加密后的二进位结果应注意使用合适的储存类型，例如 `VARBINARY` 或 `BLOB`。

## 本章常见混淆点总览

| 混淆点 | 正确理解 |
| --- | --- |
| 单行函数只能处理一行 | 错。单行函数是对结果集中的每一行分别处理 |
| 聚合函数一定只返回一行 | 不一定。有 `GROUP BY` 时，每组返回一行 |
| `ROUND()` 和 `TRUNCATE()` 一样 | 不一样。前者四舍五入，后者直接截断 |
| `CHAR_LENGTH()` 和 `LENGTH()` 一样 | 不一样。前者看字符数，后者看字节数 |
| MySQL 字符串位置从 0 开始 | 通常从 1 开始 |
| `CONCAT()` 遇到 `NULL` 会自动忽略 | 不一定，常见结果是整串变 `NULL` |
| `NOW()` 和 `SYSDATE()` 完全一样 | 不完全一样，执行语义可能不同 |
| `DATE_FORMAT()` 回传日期 | 错，通常回传字符串 |
| `CASE WHEN` 可以不写 `ELSE` 没关系 | 可以不写，但没有命中时会返回 `NULL` |
| `CONNECTION_ID()` 是连接总数 | 错，它是当前连接的 connection ID / thread ID |
| `FORMAT()` 适合继续参与数值计算 | 不适合，通常只适合显示 |
| `INET_ATON()` 可处理所有 IP | 主要处理 IPv4，IPv6 应看 `INET6_ATON()` |
| `BENCHMARK()` 可代表完整 SQL 性能 | 错，它主要测试表达式重复执行成本 |
| 旧版加密函数可以直接当现代安全方案 | 不建议，要注意版本、安全性与应用层方案 |

## 自我检查题

### 问题 1

`ROUND(amount, 2)`、`DATE_FORMAT(create_time, '%Y-%m-%d')`、`AVG(salary)` 中，哪一个是聚合函数？

答案：`AVG(salary)` 是聚合函数。`ROUND()` 和 `DATE_FORMAT()` 是单行函数。

---

### 问题 2

为什么下面 SQL 在大表中要谨慎？

```sql
SELECT *
FROM orders
WHERE YEAR(order_time) = 2026;
```

答案：如果 `order_time` 有索引，对字段套 `YEAR()` 可能影响索引使用。更常见的写法是时间范围：

```sql
WHERE order_time >= '2026-01-01'
  AND order_time <  '2027-01-01'
```

---

### 问题 3

`CONCAT('A', NULL, 'B')` 的结果通常是什么？如何避免？

答案：结果通常是 `NULL`。可以搭配 `IFNULL()` 或改用合适的 `CONCAT_WS()` 写法。

---

### 问题 4

`DATE_FORMAT()` 和 `STR_TO_DATE()` 的方向有什么不同？

答案：

- `DATE_FORMAT()`：日期 / 日期时间 -> 字符串。
- `STR_TO_DATE()`：字符串 -> 日期 / 日期时间。

---

### 问题 5

`USER()` 和 `CURRENT_USER()` 为什么都值得知道？

答案：`USER()` 偏向客户端连接时提供的用户信息，`CURRENT_USER()` 是服务器实际用于权限判断的认证账号。排查权限问题时两者都可能有用。

---

### 问题 6

为什么 `FORMAT(total_amount, 2)` 不适合作为金额计算的中间值？

答案：因为 `FORMAT()` 通常返回字符串，适合显示，不适合继续做数值计算。

---

### 问题 7

如果要支持 IPv6，为什么只学 `INET_ATON()` / `INET_NTOA()` 不够？

答案：因为 `INET_ATON()` / `INET_NTOA()` 主要处理 IPv4。IPv6 应认识 `INET6_ATON()` / `INET6_NTOA()`、`IS_IPV4()`、`IS_IPV6()`。

---

### 问题 8

为什么《加密与解密函数》不能只记 `PASSWORD()`、`MD5()`、`SHA()`？

答案：因为这些函数涉及版本与安全性问题。现代应用中更要理解 `SHA2()`、`AES_ENCRYPT()`、`AES_DECRYPT()`、`RANDOM_BYTES()`，并知道密码储存通常还要考虑应用层哈希、盐值和安全策略。
