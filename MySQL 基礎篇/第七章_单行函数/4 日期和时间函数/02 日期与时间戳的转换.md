# 4.2 日期与时间戳的转换

> 所属章节：[第七章_单行函数 / 4 日期和时间函数](./README.md)
> 关键字：UNIX_TIMESTAMP、FROM_UNIXTIME、时间戳
> 建议回查情境：需要把普通日期转换成 Unix 时间戳，或把时间戳还原成可读日期时间时

## 本节导读

这一节整理日期时间和 Unix 时间戳之间的互相转换。它非常常见于日志、接口、缓存、跨系统传输和前后端时间交换场景。

## 你会在这篇学到什么

- `UNIX_TIMESTAMP()` 在不同参数形式下的行为
- `FROM_UNIXTIME()` 如何把时间戳转回普通时间
- 为什么时间戳适合做系统间时间传递

## 函数表

| 函数 | 用法 |
| --- | --- |
| `UNIX_TIMESTAMP()` | 以 Unix 时间戳的形式返回当前时间 |
| `UNIX_TIMESTAMP(date)` | 将时间 `date` 转成 Unix 时间戳 |
| `FROM_UNIXTIME(timestamp)` | 将 Unix 时间戳转换为普通格式的时间 |

## 示例

```sql
SELECT UNIX_TIMESTAMP(NOW());
+-----------------------+
| UNIX_TIMESTAMP(NOW()) |
+-----------------------+
|            1576380910 |
+-----------------------+
1 row in set (0.01 sec)

SELECT UNIX_TIMESTAMP(CURDATE());
+---------------------------+
| UNIX_TIMESTAMP(CURDATE()) |
+---------------------------+
|                1576339200 |
+---------------------------+
1 row in set (0.00 sec)

SELECT UNIX_TIMESTAMP(CURTIME());
+---------------------------+
| UNIX_TIMESTAMP(CURTIME()) |
+---------------------------+
|                1576380969 |
+---------------------------+
1 row in set (0.00 sec)

SELECT UNIX_TIMESTAMP('2011-11-11 11:11:11');
+---------------------------------------+
| UNIX_TIMESTAMP('2011-11-11 11:11:11') |
+---------------------------------------+
|                            1320981071 |
+---------------------------------------+
1 row in set (0.00 sec)
```

```sql
SELECT FROM_UNIXTIME(1576380910);
+---------------------------+
| FROM_UNIXTIME(1576380910) |
+---------------------------+
| 2019-12-15 11:35:10       |
+---------------------------+
1 row in set (0.00 sec)
```

## 使用提醒

- 时间戳本质上是一个整数，更适合系统传输和程序比较。
- 展示给使用者看时，通常还需要再用 `FROM_UNIXTIME()` 或 `DATE_FORMAT()` 转成可读格式。

# 4.2 日期与时间戳的转换

> - 所属章节：[第七章_单行函数 / 4 日期和时间函数](./README.md)  
> - 关键字：UNIX_TIMESTAMP、FROM_UNIXTIME、Unix 时间戳、timestamp、日期转换、时间转换、日志、接口、缓存、跨系统传输、时区  
> - 建议回查情境：需要把普通日期时间转换成 Unix 时间戳；需要把时间戳还原成可读日期时间；处理前后端时间交换、日志时间、缓存过期时间、跨系统传输时间时。

## 本节导读

这一节整理日期时间与 Unix 时间戳之间的互相转换。

Unix 时间戳可以理解为：

```text
从 1970-01-01 00:00:00 UTC 到某个时间点之间经过的秒数。
```

它的优点是本质上是数字，适合系统传输、比较、排序、缓存、日志与程序处理。  
但它不适合直接给人阅读，所以通常需要再转换成日期时间格式显示。

MySQL 中常用两个函数处理这件事：

| 方向 | 函数 |
| --- | --- |
| 日期时间 -> 时间戳 | `UNIX_TIMESTAMP()` |
| 时间戳 -> 日期时间 | `FROM_UNIXTIME()` |

---

## 你会在这篇学到什么

- Unix 时间戳的基本概念。
- `UNIX_TIMESTAMP()` 没有参数时返回什么。
- `UNIX_TIMESTAMP(date)` 如何把日期时间转成时间戳。
- `FROM_UNIXTIME(timestamp)` 如何把时间戳转回可读日期时间。
- 为什么时间戳适合跨系统传输，但显示给用户时通常还要格式化。
- 时间戳与时区之间有什么关系。
- 在接口、日志、缓存、查询条件中如何使用时间戳。

---

## 快速定位

| 想解决的问题 | 优先使用 |
| --- | --- |
| 想取得当前时间的 Unix 时间戳 | `UNIX_TIMESTAMP()` |
| 想把某个日期时间转成时间戳 | `UNIX_TIMESTAMP(date)` |
| 想把时间戳转成普通日期时间 | `FROM_UNIXTIME(timestamp)` |
| 想把时间戳转成指定显示格式 | `FROM_UNIXTIME(timestamp, format)` 或搭配 `DATE_FORMAT()` |
| 想比较两个时间先后 | 可直接比较时间戳数字，也可比较日期时间类型 |
| 想处理前后端传来的秒级时间戳 | `FROM_UNIXTIME()` |
| 想处理毫秒级时间戳 | 先除以 `1000` 再转换 |

---

## 快速回查表

| 函数 | 用法 | 返回结果 | 高频程度 |
| --- | --- | --- | --- |
| `UNIX_TIMESTAMP()` | 返回当前时间的 Unix 时间戳 | 秒级数字 | 高频 |
| `UNIX_TIMESTAMP(date)` | 将日期时间转成 Unix 时间戳 | 秒级数字 | 高频 |
| `FROM_UNIXTIME(timestamp)` | 将 Unix 时间戳转成日期时间 | `YYYY-MM-DD HH:MM:SS` | 高频 |
| `FROM_UNIXTIME(timestamp, fmt)` | 将时间戳按指定格式输出 | 字符串格式结果 | 中频 |

---

## 4.2.1 什么是 Unix 时间戳

Unix 时间戳是一个数字，表示从 UTC 时间 `1970-01-01 00:00:00` 到某个时间点经过了多少秒。

例如：

```text
1576380910
```

这对人来说不好读，但对系统来说很方便，因为数字很适合：

- 比较大小。
- 排序。
- 存入缓存。
- API 传输。
- 记录日志。
- 跨语言、跨系统交换。

### 时间戳的优点

| 优点 | 说明 |
| --- | --- |
| 好比较 | 数字越大，时间越晚 |
| 好传输 | 前后端、系统间传一个数字即可 |
| 好存储 | 日志、缓存、队列中都常见 |
| 语言通用 | Java、JavaScript、Python、MySQL 都能处理 |

### 时间戳的缺点

| 缺点 | 说明 |
| --- | --- |
| 不适合人直接阅读 | `1714111200` 不如 `2026-04-26 14:00:00` 直观 |
| 秒级与毫秒级容易混淆 | 后端常用秒或毫秒，前端 JavaScript 常用毫秒 |
| 时区要统一 | 同一个时间戳在不同时区显示可能不同 |

---

## 4.2.2 日期时间转时间戳：UNIX_TIMESTAMP()

### 没有参数

```sql
SELECT UNIX_TIMESTAMP();
```

表示取得当前时间的 Unix 时间戳。

也可以写：

```sql
SELECT UNIX_TIMESTAMP(NOW());
```

两者都常用于取得当前时间对应的秒级时间戳。

---

### 有日期时间参数

```sql
SELECT UNIX_TIMESTAMP('2011-11-11 11:11:11');
```

意思是把这个日期时间转换成 Unix 时间戳。

---

### 日期参数示例

```sql
SELECT UNIX_TIMESTAMP(CURDATE());
```

`CURDATE()` 只包含日期，所以概念上会转换成当天 `00:00:00` 对应的时间戳。

---

### 时间参数提醒

```sql
SELECT UNIX_TIMESTAMP(CURTIME());
```

这类写法容易让人误会，因为 `CURTIME()` 只有时间，没有年月日。实际解释会受到 MySQL 对时间值的转换规则影响。实务上如果要转换明确时间点，建议提供完整日期时间。

较清楚的写法：

```sql
SELECT UNIX_TIMESTAMP('2026-04-26 14:30:00');
```

---

## 4.2.3 时间戳转日期时间：FROM_UNIXTIME()

`FROM_UNIXTIME(timestamp)` 会把 Unix 时间戳转换成普通日期时间。

```sql
SELECT FROM_UNIXTIME(1576380910);
```

结果概念：

```text
2019-12-15 11:35:10
```

注意：显示结果会受到会话时区影响。也就是说，同一个时间戳，在不同时区显示出的本地日期时间可能不同。

---

## 4.2.4 指定格式输出时间戳

有些场景不只要把时间戳转成日期时间，还希望指定显示格式。

可以使用：

```sql
SELECT FROM_UNIXTIME(1576380910, '%Y-%m-%d');
```

结果概念：

```text
2019-12-15
```

也可以输出到秒：

```sql
SELECT FROM_UNIXTIME(1576380910, '%Y-%m-%d %H:%i:%s');
```

结果概念：

```text
2019-12-15 11:35:10
```

格式符会在 `4.7 日期格式化与解析` 中详细说明。

---

## 4.2.5 秒级时间戳与毫秒级时间戳

这是实际开发最容易出错的地方。

| 类型 | 位数概念 | 常见场景 |
| --- | --- | --- |
| 秒级时间戳 | 10 位左右 | MySQL `UNIX_TIMESTAMP()` 常见结果 |
| 毫秒级时间戳 | 13 位左右 | JavaScript `Date.now()` 常见结果 |

如果前端传来的是毫秒级时间戳，例如：

```text
1714111200000
```

不能直接写：

```sql
SELECT FROM_UNIXTIME(1714111200000);
```

因为 `FROM_UNIXTIME()` 通常期待秒级时间戳。应该先除以 `1000`：

```sql
SELECT FROM_UNIXTIME(1714111200000 / 1000);
```

---

## 4.2.6 常见实务场景

### 场景 1：把前端传来的秒级时间戳转成日期时间

```sql
SELECT FROM_UNIXTIME(1714111200) AS created_time;
```

---

### 场景 2：把毫秒级时间戳转成日期时间

```sql
SELECT FROM_UNIXTIME(1714111200000 / 1000) AS created_time;
```

---

### 场景 3：查询某个时间戳之后的资料

假设 API 传入秒级时间戳：

```sql
SELECT *
FROM orders
WHERE created_at >= FROM_UNIXTIME(1714111200);
```

---

### 场景 4：把日期时间输出成时间戳给系统使用

```sql
SELECT
    order_id,
    UNIX_TIMESTAMP(created_at) AS created_timestamp
FROM orders;
```

---

### 场景 5：日志报表中同时显示原时间与时间戳

```sql
SELECT
    log_id,
    created_at,
    UNIX_TIMESTAMP(created_at) AS created_at_ts
FROM system_log;
```

---

## 4.2.7 使用提醒

### 1. 秒级与毫秒级要先确认

这是最重要的提醒。

```text
MySQL UNIX_TIMESTAMP()：通常是秒级。
JavaScript Date.now()：通常是毫秒级。
```

如果时间差了几万年，通常就是秒、毫秒单位搞错了。

---

### 2. 时间戳显示会受时区影响

同一个 Unix 时间戳代表同一个绝对时间点，但转换成本地日期时间时，会受到时区影响。

例如同一个时间戳：

```text
UTC 显示：2026-04-26 06:00:00
台北显示：2026-04-26 14:00:00
```

因此跨系统时要统一：

- 存 UTC？
- 存本地时间？
- API 传秒级还是毫秒级？
- 前端显示时由谁负责转换？

---

### 3. 展示给使用者时通常要格式化

时间戳适合系统处理，不适合直接展示。

```sql
SELECT FROM_UNIXTIME(created_ts, '%Y-%m-%d %H:%i:%s') AS display_time
FROM logs;
```

---

### 4. 查询条件中注意栏位类型

如果资料表中存的是 `DATETIME`，查询时可以把传入时间戳转成日期时间再比较：

```sql
WHERE created_at >= FROM_UNIXTIME(:start_ts)
```

如果资料表中存的是数字时间戳，则可以直接比较数字：

```sql
WHERE created_ts >= :start_ts
```

不要混着写，否则可读性和性能都可能变差。

---

## 4.2.8 常见混淆点

### 混淆点 1：UNIX_TIMESTAMP() 是日期转数字

```sql
SELECT UNIX_TIMESTAMP('2026-04-26 14:30:00');
```

方向是：

```text
日期时间 -> 时间戳数字
```

---

### 混淆点 2：FROM_UNIXTIME() 是数字转日期

```sql
SELECT FROM_UNIXTIME(1714111200);
```

方向是：

```text
时间戳数字 -> 日期时间
```

---

### 混淆点 3：秒级和毫秒级不是同一个东西

```text
1714111200     秒级
1714111200000  毫秒级
```

毫秒级要先除以 `1000`。

---

### 混淆点 4：时间戳不是没有时区问题

时间戳本身代表绝对时间点，但转换成可读日期时间时，会按时区显示。

---

### 混淆点 5：时间戳适合传输，不一定适合报表显示

报表给人看时，通常需要转换成：

```text
YYYY-MM-DD HH:MM:SS
```

或业务指定格式。

---

## 4.2.9 自我检查题

### 问题 1

`UNIX_TIMESTAMP('2026-04-26 14:30:00')` 的作用是什么？

答案：把指定日期时间转换成 Unix 时间戳数字。

---

### 问题 2

`FROM_UNIXTIME(1714111200)` 的作用是什么？

答案：把秒级 Unix 时间戳转换成普通日期时间。

---

### 问题 3

前端传来 `1714111200000`，可以直接放进 `FROM_UNIXTIME()` 吗？

答案：通常不可以，因为它很可能是毫秒级时间戳。应该先除以 `1000`。

```sql
SELECT FROM_UNIXTIME(1714111200000 / 1000);
```

---

### 问题 4

为什么同一个时间戳在不同地区显示结果可能不同？

答案：因为时间戳转换成本地日期时间时，会受到时区影响。

---

## 4.2.10 本节总结

本节可以浓缩成下面几句话：

1. `UNIX_TIMESTAMP()` 用来把当前时间或指定日期时间转成 Unix 时间戳。
2. `FROM_UNIXTIME()` 用来把 Unix 时间戳转回日期时间。
3. 时间戳适合系统传输、比较、排序、日志和缓存。
4. 给使用者看时，通常要把时间戳转换成可读日期格式。
5. 秒级与毫秒级要特别小心，前端 JavaScript 常见毫秒级，MySQL 常见秒级。
6. 时间戳转换成日期时间时仍然会受到时区影响。

如果只记一句话：

> `UNIX_TIMESTAMP()` 是日期变数字，`FROM_UNIXTIME()` 是数字变日期；实务中最容易错的是秒级和毫秒级。
