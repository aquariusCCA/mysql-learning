# 4 日期和时间函数

> - 所属章节：[第七章_单行函数](../README.md)  
> - 章节定位：MySQL 单行函数中最常用于「时间取得、时间转换、日期拆解、时间计算、格式化显示、字符串解析」的一组函数。  
> - 建议回查情境：忘了当前时间函数、时间戳互转、日期字段提取、日期加减、时间差计算、秒数转换、日期格式化或字符串解析时。

## 本章导读

日期和时间函数是 MySQL 单行函数中非常实用的一组。它们不只用来「显示今天几号」或「取得当前时间」，更常出现在报表查询、订单筛选、日志记录、会员有效期、最近 N 天统计、时间戳转换、跨系统资料交换和前后端时间格式转换等场景中。

学习本章时，不建议直接死背所有函数，而是先建立一条主线：

```text
取得时间
 -> 转换时间
 -> 拆解时间
 -> 计算时间
 -> 格式化时间
 -> 解析时间字符串
```

只要这条主线清楚，遇到日期时间需求时，就能快速判断应该查哪一篇笔记。

## 你会在本章学到什么

- 如何取得当前日期、当前时间、当前日期时间与 UTC 时间。
- 如何在普通日期时间和 Unix 时间戳之间转换。
- 如何从日期中取出年份、月份、日期、星期、季度、周数等信息。
- 如何使用 `EXTRACT(type FROM date)` 统一提取日期时间字段。
- 如何在时间值和秒数之间互转。
- 如何进行日期加减、时间加减、日期差值和时间差值计算。
- 如何把日期时间格式化成指定字符串。
- 如何把字符串解析回日期时间值。
- 如何避免日期时间函数常见陷阱，例如时区、索引、周数规则、日期边界和格式不匹配。

## 本章实务提醒总览

### 1. 注意当前时间函数的语义

`NOW()`、`CURRENT_TIMESTAMP()`、`SYSDATE()` 都能取得日期时间，但执行语义不完全相同。入门阶段可以先理解为当前日期时间；进阶使用时要知道它们在语句执行过程中的取值时点可能不同。

---

### 2. 注意本地时间与 UTC 时间

`CURDATE()`、`NOW()` 等通常和当前连接或服务器时区有关；`UTC_DATE()`、`UTC_TIME()` 则是 UTC 标准时间。跨系统、跨地区、跨时区时，不要只凭显示结果判断时间是否正确。

---

### 3. 注意 Unix 时间戳与时区

`UNIX_TIMESTAMP()` 和 `FROM_UNIXTIME()` 很适合系统传输与日志处理，但它们和时区设定有关。前后端、数据库、服务器如果时区策略不一致，可能出现时间偏移。

---

### 4. 注意 `WEEK()`、`WEEKDAY()`、`DAYOFWEEK()` 的编号规则

日期拆解函数中，最容易混淆的是星期和周数：

| 函数 | 重点 |
| --- | --- |
| `WEEKDAY(date)` | 周一是 `0`，周日是 `6` |
| `DAYOFWEEK(date)` | 周日是 `1`，周六是 `7` |
| `WEEK(date)` | 周数规则会受模式影响，跨系统要谨慎 |

---

### 5. 注意 `DATEDIFF()` 与 `TIMEDIFF()` 的差别

`DATEDIFF()` 主要看日期差，通常返回相差天数。  
`TIMEDIFF()` 主要看时间差，返回时间间隔。

如果要精确到小时、分钟、秒，需要确认是否应该使用其他时间差计算方式或应用层处理。

---

### 6. 注意 `DATE_FORMAT()` 返回的是字符串

`DATE_FORMAT()` 很适合报表显示，但它的结果是字符串，不是日期型别。  
如果后续还要做日期比较、排序或计算，不能只看显示格式，要确认数据类型与排序逻辑是否正确。

---

### 7. 注意 `STR_TO_DATE()` 的格式必须对应

`STR_TO_DATE(str, fmt)` 的重点是：

```text
字符串长什么样，fmt 就要怎么对应。
```

例如：

```sql
STR_TO_DATE('2026-04-26', '%Y-%m-%d')
```

如果格式符和字符串不匹配，可能解析失败或得到非预期结果。

---

### 8. 注意在 `WHERE` 中对日期字段套函数可能影响索引

正式查询中要特别注意这类写法：

```sql
WHERE DATE(order_time) = '2026-04-26'
WHERE YEAR(order_time) = 2026
WHERE DATE_FORMAT(order_time, '%Y-%m') = '2026-04'
```

如果 `order_time` 是索引字段，对它套函数可能影响索引使用。更常见的做法是改成范围条件：

```sql
WHERE order_time >= '2026-04-26 00:00:00'
  AND order_time <  '2026-04-27 00:00:00'
```

或查某个月：

```sql
WHERE order_time >= '2026-04-01'
  AND order_time <  '2026-05-01'
```

实际是否使用索引要以执行计划为准，但工作中要先有这个意识：

> 不要随便在索引字段外层套日期函数。

## 常见混淆点总览

| 混淆点 | 正确理解 |
| --- | --- |
| `CURDATE()` 和 `NOW()` | 前者只返回日期，后者返回日期时间 |
| `CURTIME()` 和 `NOW()` | 前者只返回时间，后者包含日期和时间 |
| `NOW()` 和 `SYSDATE()` | 都能取得当前时间，但执行时点语义可能不同 |
| `UTC_DATE()` 和 `CURDATE()` | 前者是 UTC 日期，后者通常是本地/连接时区日期 |
| `UNIX_TIMESTAMP()` | 时间戳适合传输，但要注意时区 |
| `WEEKDAY()` 和 `DAYOFWEEK()` | 星期编号规则不同 |
| `DATEDIFF()` 和 `TIMEDIFF()` | 一个偏日期天数差，一个偏时间差 |
| `DATE_FORMAT()` | 结果是字符串，不是日期型别 |
| `STR_TO_DATE()` | 格式符必须和字符串结构对应 |
| 在日期字段上套函数 | 可能影响索引使用，正式查询要谨慎 |

## 自我检查题

### 问题 1

如果只想取得今天日期，不需要时分秒，应该优先使用哪个函数？

答案：`CURDATE()` 或 `CURRENT_DATE()`。

---

### 问题 2

如果想把 Unix 时间戳转成可读日期时间，应该使用哪个函数？

答案：`FROM_UNIXTIME(timestamp)`。

---

### 问题 3

`WEEKDAY()` 和 `DAYOFWEEK()` 的星期编号规则一样吗？

答案：不一样。`WEEKDAY()` 中周一是 `0`，周日是 `6`；`DAYOFWEEK()` 中周日是 `1`，周六是 `7`。

---

### 问题 4

如果想计算两个日期相差几天，应该优先看哪个函数？

答案：`DATEDIFF(date1, date2)`。

---

### 问题 5

`DATE_FORMAT(order_time, '%Y-%m')` 的结果是什么类型？

答案：字符串。它适合显示或分组，但后续比较、排序、索引使用都要谨慎。

---

### 问题 6

下面 SQL 有什么潜在问题？

```sql
SELECT *
FROM orders
WHERE DATE(order_time) = '2026-04-26';
```

答案：如果 `order_time` 是索引字段，对它套 `DATE()` 可能影响索引使用。可以改成时间范围：

```sql
WHERE order_time >= '2026-04-26 00:00:00'
  AND order_time <  '2026-04-27 00:00:00'
```

```text
取得、转换、拆解、提取、计算、格式化
```

> 日期和时间函数的核心不是背语法，而是判断当前需求属于「取得时间、转换时间、拆解时间、计算时间、格式化时间、解析时间」哪一种。
