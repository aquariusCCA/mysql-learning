# 4.1 获取当前日期与时间

> - 所属章节：[第七章_单行函数 / 4 日期和时间函数](./README.md)  
> - 关键字：CURDATE、CURRENT_DATE、CURTIME、CURRENT_TIME、NOW、SYSDATE、CURRENT_TIMESTAMP、LOCALTIME、LOCALTIMESTAMP、UTC_DATE、UTC_TIME、本地时间、UTC 时间、时区  
> - 建议回查情境：忘了当前日期、当前时间、当前日期时间该用哪个函数；分不清本地时间与 UTC 时间；不确定 `NOW()` 和 `SYSDATE()` 差异；需要在报表、日志、查询条件中取得当前时间时。

## 本节导读

这一节整理 MySQL 中最基础的一组日期时间函数。它们的共同点是：不需要你提供具体日期，MySQL 会根据当前连接环境、服务器时间或 UTC 标准时间返回对应结果。

学习这一组函数时，最重要的不是背函数名，而是先分清四个问题：

1. 只要日期，还是只要时间？
2. 要完整日期时间，还是只要其中一部分？
3. 要本地系统时间，还是 UTC 标准时间？
4. 这个时间是用于显示、记录、计算，还是跨系统传输？

只要这四个问题分清楚，`CURDATE()`、`CURTIME()`、`NOW()`、`SYSDATE()`、`UTC_DATE()`、`UTC_TIME()` 就不会混在一起。

- 为什么跨系统、跨时区场景要特别注意 UTC 与本地时间。
- 当前时间函数在查询、报表、日志与新增资料中的实务用法。

## 快速定位

| 想解决的问题 | 优先使用 |
| --- | --- |
| 只想取得当前日期 | `CURDATE()` / `CURRENT_DATE()` |
| 只想取得当前时间 | `CURTIME()` / `CURRENT_TIME()` |
| 想取得当前日期和时间 | `NOW()` / `CURRENT_TIMESTAMP()` |
| 想取得语句开始执行时的时间 | 优先理解 `NOW()` |
| 想取得函数实际执行当下的时间 | 了解 `SYSDATE()` |
| 想取得 UTC 日期 | `UTC_DATE()` |
| 想取得 UTC 时间 | `UTC_TIME()` |
| 想把当前时间存进新增资料 | 常用 `NOW()` 或栏位默认值 `CURRENT_TIMESTAMP` |
| 想做跨时区系统交换 | 优先考虑 UTC 时间或统一时区策略 |

## 4.1.1 只取得当前日期：CURDATE() / CURRENT_DATE()

`CURDATE()` 用来取得当前日期，只包含年、月、日，不包含时、分、秒。

```sql
SELECT CURDATE();
```

结果概念：

```text
2026-04-26
```

`CURRENT_DATE()` 是同义写法：

```sql
SELECT CURRENT_DATE();
```

### 适合场景

查询今天新增的资料：

```sql
SELECT *
FROM orders
WHERE order_date = CURDATE();
```

如果 `order_date` 是 `DATE` 类型，这种写法直观清楚。

如果栏位是 `DATETIME` 类型，不建议直接写：

```sql
WHERE order_time = CURDATE()
```

因为 `DATETIME` 包含时分秒，通常不会刚好等于当天 `00:00:00`。更常见的写法是日期范围：

```sql
SELECT *
FROM orders
WHERE order_time >= CURDATE()
  AND order_time <  CURDATE() + INTERVAL 1 DAY;
```

---

## 4.1.2 只取得当前时间：CURTIME() / CURRENT_TIME()

`CURTIME()` 用来取得当前时间，只包含时、分、秒，不包含日期。

```sql
SELECT CURTIME();
```

结果概念：

```text
14:30:25
```

`CURRENT_TIME()` 是同义写法：

```sql
SELECT CURRENT_TIME();
```

### 适合场景

- 只关心一天中的时间点。
- 比较营业时间、批次执行时间。
- 记录某个动作发生在当天几点。

示例：判断当前时间是否超过下午 5 点：

```sql
SELECT
    CASE
        WHEN CURTIME() >= '17:00:00' THEN '已超过下班时间'
        ELSE '仍在工作时间'
    END AS time_status;
```

---

## 4.1.3 取得当前日期时间：NOW() / CURRENT_TIMESTAMP()

`NOW()` 是最常用的当前日期时间函数，返回当前日期加时间。

```sql
SELECT NOW();
```

结果概念：

```text
2026-04-26 14:30:25
```

`CURRENT_TIMESTAMP()`、`LOCALTIME()`、`LOCALTIMESTAMP()` 也常被视为取得当前日期时间的相关写法。

```sql
SELECT
    NOW(),
    CURRENT_TIMESTAMP(),
    LOCALTIME(),
    LOCALTIMESTAMP();
```

### 适合场景

新增资料时记录建立时间：

```sql
INSERT INTO users(username, created_at)
VALUES ('tom', NOW());
```

查询最近 30 分钟资料：

```sql
SELECT *
FROM login_log
WHERE login_time >= NOW() - INTERVAL 30 MINUTE;
```

---

## 4.1.4 NOW() 与 SYSDATE() 的差异

入门阶段可以先把 `NOW()` 和 `SYSDATE()` 都理解成「取得当前日期时间」，但更精确地说，它们在执行时点上有差异。

| 函数 | 概念 |
| --- | --- |
| `NOW()` | 返回语句开始执行时的时间 |
| `SYSDATE()` | 返回函数实际执行当下的时间 |

这意味着在同一条 SQL 里，如果执行过程很长，`NOW()` 的值通常比较稳定，而 `SYSDATE()` 可能反映函数被调用当下的时间。

### 示例概念

```sql
SELECT NOW(), SYSDATE();
```

一般情况下两者看起来可能一样或非常接近。只有在需要精确理解执行时点时，才需要特别区分。

### 使用建议

大多数业务场景优先使用：

```sql
NOW()
```

除非你明确需要 `SYSDATE()` 的执行当下语义，否则不要为了「看起来更像系统时间」而随意替换。

---

## 4.1.5 取得 UTC 日期与时间：UTC_DATE() / UTC_TIME()

UTC 是世界标准时间。跨地区、跨系统、跨时区传输资料时，经常需要统一用 UTC 作为时间基准。

| 函数 | 返回内容 |
| --- | --- |
| `UTC_DATE()` | UTC 当前日期 |
| `UTC_TIME()` | UTC 当前时间 |

示例：

```sql
SELECT
    UTC_DATE(),
    UTC_TIME();
```

### 本地时间与 UTC 时间的差别

如果服务器或连接时区是台北时间，台北时间通常是 UTC+8。  
例如：

```text
本地时间：2026-04-26 14:30:00
UTC 时间：2026-04-26 06:30:00
```

这就是为什么跨时区系统不能只凭「看起来是当前时间」来处理。

---

## 4.1.6 常见实务场景

### 场景 1：查询今天的订单

如果 `order_time` 是 `DATETIME` 类型，建议使用范围查询：

```sql
SELECT *
FROM orders
WHERE order_time >= CURDATE()
  AND order_time <  CURDATE() + INTERVAL 1 DAY;
```

这比下面这种写法更适合实务场景：

```sql
WHERE DATE(order_time) = CURDATE()
```

因为对栏位套 `DATE()` 可能影响索引使用。

---

### 场景 2：新增资料时记录建立时间

```sql
INSERT INTO customer(name, created_at)
VALUES ('王小明', NOW());
```

如果栏位设计允许，也常见使用默认值：

```sql
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
```

---

### 场景 3：查询最近 7 天资料

```sql
SELECT *
FROM login_log
WHERE login_time >= NOW() - INTERVAL 7 DAY;
```

---

### 场景 4：比较当前时间是否在营业时间内

```sql
SELECT
    CASE
        WHEN CURTIME() BETWEEN '09:00:00' AND '17:30:00'
        THEN '营业时间'
        ELSE '非营业时间'
    END AS business_status;
```

---

## 4.1.7 使用提醒

### 1. DATE 与 DATETIME 不要混用判断

`CURDATE()` 返回的是日期，`NOW()` 返回的是日期时间。  
如果栏位是 `DATETIME`，通常不要用等号直接和 `CURDATE()` 比较。

推荐：

```sql
WHERE created_at >= CURDATE()
  AND created_at <  CURDATE() + INTERVAL 1 DAY
```

---

### 2. 查询条件中避免对索引栏位套函数

不推荐：

```sql
WHERE DATE(created_at) = CURDATE()
```

较推荐：

```sql
WHERE created_at >= CURDATE()
  AND created_at <  CURDATE() + INTERVAL 1 DAY
```

实际是否使用索引仍要看执行计划，但工作中要先建立这个意识。

---

### 3. 跨时区系统要统一策略

如果系统有跨国家、跨地区、前后端时间交换、日志汇整需求，要先决定：

- 数据库存本地时间还是 UTC 时间？
- 前端显示时是否转换时区？
- API 传输使用时间戳、UTC 字符串，还是本地时间字符串？

不要每个模块各自决定，否则后期很容易出现时间偏移问题。

---

### 4. NOW() 更常用于一般业务

一般新增时间、更新时间、查询最近 N 分钟资料，优先使用 `NOW()`。  
`SYSDATE()` 要在确实理解执行时点差异后再使用。

---

## 4.1.8 常见混淆点

### 混淆点 1：CURDATE() 不是完整日期时间

```sql
SELECT CURDATE();
```

只会得到日期，不包含时分秒。

---

### 混淆点 2：CURTIME() 不包含年月日

```sql
SELECT CURTIME();
```

只会得到时间，不包含日期。

---

### 混淆点 3：NOW() 与 CURDATE() 不是同一种结果

| 函数 | 结果概念 |
| --- | --- |
| `CURDATE()` | `2026-04-26` |
| `NOW()` | `2026-04-26 14:30:25` |

---

### 混淆点 4：UTC 时间不是本地时间

`UTC_DATE()`、`UTC_TIME()` 返回的是 UTC 标准时间，不一定等于你所在地区看到的本地时间。

---

### 混淆点 5：SYSDATE() 不是 NOW() 的简单别名

两者都能取得日期时间，但执行语义不同。一般业务优先使用 `NOW()`。

---

## 4.1.9 自我检查题

### 问题 1

`CURDATE()` 和 `NOW()` 的差别是什么？

答案：`CURDATE()` 只返回当前日期；`NOW()` 返回当前日期和时间。

---

### 问题 2

如果 `created_at` 是 `DATETIME` 类型，要查询今天资料，下面哪种写法更适合？

```sql
WHERE DATE(created_at) = CURDATE()
```

还是：

```sql
WHERE created_at >= CURDATE()
  AND created_at < CURDATE() + INTERVAL 1 DAY
```

答案：第二种更适合实务，因为它避免直接对栏位套 `DATE()`，通常更有利于索引使用。

---

### 问题 3

`UTC_TIME()` 返回的是本地时间吗？

答案：不是。它返回 UTC 标准时间。

---

### 问题 4

一般记录新增时间时，比较常用 `NOW()` 还是 `SYSDATE()`？

答案：一般优先使用 `NOW()`。

---

## 4.1.10 本节总结

本节可以浓缩成下面几句话：

1. `CURDATE()` / `CURRENT_DATE()` 只取当前日期。
2. `CURTIME()` / `CURRENT_TIME()` 只取当前时间。
3. `NOW()` / `CURRENT_TIMESTAMP()` 取得当前日期时间，是最常用的业务函数。
4. `SYSDATE()` 与 `NOW()` 在执行时点语义上有差异，一般业务优先使用 `NOW()`。
5. `UTC_DATE()`、`UTC_TIME()` 适合跨时区场景，但要和系统整体时间策略搭配。
6. 查询 `DATETIME` 类型的今天资料时，优先使用范围条件，而不是对栏位套 `DATE()`。

如果只记一句话：

> 只要日期用 `CURDATE()`，只要时间用 `CURTIME()`，完整当前时间用 `NOW()`，跨时区要考虑 UTC。
