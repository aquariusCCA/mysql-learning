## 1.1 什么是函数

函数可以理解为一段已经封装好的处理逻辑。它通常接收一个或多个输入值，经过内部处理后，返回一个结果。

用程序语言的角度来看，函数大致长这样：

```text
输入值 -> 函数处理 -> 返回结果
```

例如在 Java 中，我们可能会写一个方法来计算折扣价：

```java
BigDecimal calculateDiscountPrice(BigDecimal price, BigDecimal discountRate) {
    return price.multiply(discountRate);
}
```

在 SQL 中，函数的想法也类似，只是它通常直接作用在查询语句中，用来处理字段值、常量值或表达式结果。

例如：

```sql
SELECT ABS(-100);
```

这里的 `ABS()` 就是一个函数：

```text
输入：-100
处理：取绝对值
输出：100
```

再例如：

```sql
SELECT UPPER('mysql');
```

这里的 `UPPER()` 也是函数：

```text
输入：'mysql'
处理：转换成大写
输出：'MYSQL'
```

所以，函数的核心可以抓住三件事：

| 重点 | 说明                              |
| -- | ------------------------------- |
| 输入 | 函数接收的参数，可以是常量、字段、表达式，甚至另一个函数的结果 |
| 处理 | 函数内部已经定义好的规则                    |
| 输出 | 函数执行后返回的结果                      |

## 1.2 SQL 中为什么需要函数

SQL 不只是用来「把资料查出来」，还经常需要在查询过程中顺便对资料做处理。

例如：

* 把姓名统一转成大写或小写。
* 把日期格式化成指定显示格式。
* 计算金额的小数位。
* 把 `NULL` 改成默认值。
* 根据薪资区间输出不同文字标签。
* 统计每个部门的平均薪资。

这些需求如果全部交给 Java、Vue 或其他应用层处理，也可以做到，但有些数据处理直接放在 SQL 层会更方便，尤其是查询、筛选、统计和报表场景。

例如：

```sql
SELECT
    employee_id,
    last_name,
    UPPER(last_name) AS upper_name,
    ROUND(salary * 1.1, 2) AS adjusted_salary
FROM employees;
```

这段 SQL 做了几件事：

| 表达式                      | 作用            |
| ------------------------ | ------------- |
| `UPPER(last_name)`       | 将员工姓名转换成大写    |
| `salary * 1.1`           | 计算调薪后的薪资      |
| `ROUND(salary * 1.1, 2)` | 将调薪后的薪资保留两位小数 |
| `AS adjusted_salary`     | 给计算结果取别名      |

从这里可以看出，SQL 函数的价值不是「炫技」，而是让查询结果更接近业务真正需要的样子。

### SQL 函数常见用途

| 用途     | 示例函数                                       | 说明                 |
| ------ | ------------------------------------------ | ------------------ |
| 数值处理   | `ABS()`、`ROUND()`、`CEIL()`、`FLOOR()`       | 处理绝对值、取整、小数位       |
| 字符串处理  | `CONCAT()`、`SUBSTR()`、`TRIM()`、`REPLACE()` | 拼接、截取、去空格、替换文字     |
| 日期时间处理 | `NOW()`、`DATE_FORMAT()`、`DATEDIFF()`       | 取得当前时间、格式化日期、计算日期差 |
| 条件判断   | `IF()`、`IFNULL()`、`CASE WHEN`              | 根据条件返回不同结果         |
| 环境查询   | `VERSION()`、`DATABASE()`、`USER()`          | 查看当前 MySQL 环境或连接信息 |
| 汇总统计   | `COUNT()`、`SUM()`、`AVG()`、`MAX()`、`MIN()`  | 统计行数、总和、平均、最大最小值   |

## 1.3 内置函数与自定义函数

从函数来源来看，SQL 函数大致可以分成两类：

1. 内置函数
2. 自定义函数

### 1.3.1 内置函数

内置函数是数据库系统已经提供好的函数。我们不需要自己定义，只要按照语法调用即可。

例如：

```sql
SELECT
    ABS(-10),
    CONCAT('Hello', 'MySQL'),
    NOW();
```

这些函数都是 MySQL 已经提供好的能力。

内置函数的特点是：

* 学习成本较低。
* 稳定性较高。
* 常见需求通常已经覆盖。
* 适合查询、报表、数据清洗、格式转换等场景。

本章主要学习的就是 MySQL 的内置函数。

### 1.3.2 自定义函数

自定义函数是使用者根据业务需求自行编写的函数。它适合把某些重复出现的业务规则封装起来。

例如某个系统经常需要根据会员等级计算折扣，就可以考虑封装成自定义函数。

不过，自定义函数虽然可以提高复用性，但也要注意：

* 会增加数据库层的维护成本。
* 不同环境迁移时要额外处理函数定义。
* 权限、性能、可读性都需要评估。
* 业务逻辑放在数据库层还是应用层，要根据团队规范决定。

对初学阶段来说，优先掌握内置函数即可。自定义函数可以等到学习存储过程、函数、触发器等内容时再深入。

## 1.4 不同 DBMS 函数的差异

学习 SQL 函数时，一定要有一个观念：

> SQL 有标准语法，但不同数据库管理系统对函数的支持并不完全相同。

DBMS 是 Database Management System，也就是数据库管理系统。常见的 DBMS 包括：

* MySQL
* Oracle
* SQL Server
* PostgreSQL
* MariaDB

虽然它们都能执行 SQL，但函数名称、参数顺序、返回结果、日期格式、字符串拼接方式都可能不同。

例如，字符串拼接在不同 DBMS 中可能会看到不同写法：

| DBMS       | 常见字符串拼接写法                                 |   |                                 |
| ---------- | ----------------------------------------- | - | ------------------------------- |
| MySQL      | `CONCAT(str1, str2, ...)`                 |   |                                 |
| Oracle     | `str1                                     |   | str2`                           |
| SQL Server | `str1 + str2` 或 `CONCAT(str1, str2, ...)` |   |                                 |
| PostgreSQL | `str1                                     |   | str2`或`CONCAT(str1, str2, ...)` |

因此，函数 SQL 的可移植性通常比基础 `SELECT ... FROM ... WHERE ...` 更差。

### 为什么这件事很重要？

因为你在工作中可能会遇到：

* 旧系统用 Oracle，新系统用 MySQL。
* 报表 SQL 从 SQL Server 改写到 MySQL。
* 教材使用 MySQL，但公司环境使用 PostgreSQL。
* 同一个函数名在不同数据库中行为不同。

所以学习函数时，不只要知道「这个函数怎么用」，还要建立两个意识：

1. 这是 MySQL 的写法，还是 SQL 标准写法？
2. 如果换数据库，这段 SQL 是否需要改写？

### 学习建议

初学 MySQL 时，不需要一开始就比较所有 DBMS 的函数差异，但要养成习惯：

* 笔记中标明这是 MySQL 写法。
* 遇到跨数据库需求时，查目标 DBMS 的官方文档。
* 不要默认 MySQL 函数可以直接复制到其他数据库。

## 1.5 MySQL 内置函数的分类

MySQL 内置函数可以从两个角度分类：

1. 功能角度：这个函数用来解决什么类型的问题？
2. 使用方式角度：这个函数是一行一行处理，还是多行汇总处理？

这两个分类角度很重要，因为它们解决的问题不同。

---

### 1.5.1 从功能角度分类

从功能角度看，MySQL 内置函数可以分成以下几类：

| 函数类别       | 主要用途                   | 示例                                               |
| ---------- | ---------------------- | ------------------------------------------------ |
| 数值函数       | 处理数字、取整、四舍五入、随机数、数学运算  | `ABS()`、`ROUND()`、`TRUNCATE()`、`RAND()`          |
| 字符串函数      | 处理文本拼接、截取、替换、大小写、去空格   | `CONCAT()`、`SUBSTR()`、`REPLACE()`、`TRIM()`       |
| 日期和时间函数    | 取得日期、格式化日期、日期加减、计算时间差  | `NOW()`、`CURDATE()`、`DATE_FORMAT()`、`DATEDIFF()` |
| 流程控制函数     | 根据条件返回不同结果             | `IF()`、`IFNULL()`、`CASE WHEN`                    |
| 类型转换函数     | 转换数据类型或字符集             | `CAST()`、`CONVERT()`                             |
| 加密与哈希函数    | 对字符串做哈希或加解密处理          | `MD5()`、`SHA()`、`AES_ENCRYPT()`                  |
| MySQL 信息函数 | 查看当前数据库、版本、用户、字符集等环境信息 | `VERSION()`、`DATABASE()`、`USER()`                |
| 聚合函数       | 多行数据汇总统计               | `COUNT()`、`SUM()`、`AVG()`、`MAX()`、`MIN()`        |

功能分类的好处是：当你遇到需求时，可以快速知道该去哪个类别找函数。

例如：

* 想处理金额小数位：优先想到数值函数。
* 想处理姓名、编码、备注：优先想到字符串函数。
* 想处理注册时间、订单时间：优先想到日期时间函数。
* 想把状态码转成中文说明：优先想到流程控制函数。
* 想统计每个部门人数：优先想到聚合函数。

---

### 1.5.2 从使用方式角度分类

从使用方式角度看，可以先把函数分成两大类：

1. 单行函数
2. 聚合函数，也叫多行函数、分组函数

这条分类主线非常重要：

```text
单行函数：一行输入 -> 一行结果
聚合函数：多行输入 -> 一个汇总结果
```

更精确地说：

* 单行函数会对结果集中的每一行分别执行一次。
* 聚合函数会对一组数据进行汇总，返回一个统计结果。
* 如果没有 `GROUP BY`，聚合函数通常对整个查询结果汇总。
* 如果有 `GROUP BY`，聚合函数会对每个分组分别汇总。

## 1.6 单行函数（Single-Row Functions）

单行函数会对每一行输入分别执行操作，并为每一行返回一个结果。

它的重点不是「只能处理一行资料」，而是：

> 查询结果中有几行，它就对每一行分别处理一次。

例如：

```sql
SELECT
    employee_id,
    last_name,
    UPPER(last_name) AS upper_name
FROM employees;
```

如果 `employees` 表中有 100 笔员工资料，`UPPER(last_name)` 就会对这 100 笔资料分别执行一次，并产生 100 个转换后的结果。

### 1.6.1 单行函数的特点

| 特点           | 说明                       |
| ------------ | ------------------------ |
| 一行一行处理       | 每一行资料都会独立执行函数            |
| 不会主动合并多行     | 不会把多笔资料汇总成一个统计值          |
| 可处理字段、常量、表达式 | 参数可以是字段、数字、字符串、运算表达式     |
| 可嵌套使用        | 一个函数的结果可以再传给另一个函数        |
| 常用于资料转换      | 字符串处理、数值处理、日期处理、条件转换都很常见 |

### 1.6.2 单行函数的常见类别

| 类别     | 示例函数                                             | 常见用途               |
| ------ | ------------------------------------------------ | ------------------ |
| 字符串函数  | `UPPER()`、`LOWER()`、`CONCAT()`、`SUBSTR()`        | 处理姓名、编码、地址、备注等文本资料 |
| 数值函数   | `ABS()`、`CEIL()`、`FLOOR()`、`ROUND()`             | 处理金额、比例、分数、小数位     |
| 日期时间函数 | `NOW()`、`CURDATE()`、`DATE_FORMAT()`、`DATEDIFF()` | 处理订单日期、注册时间、到期日    |
| 流程控制函数 | `IF()`、`IFNULL()`、`CASE WHEN`                    | 根据条件输出不同结果         |
| 类型转换函数 | `CAST()`、`CONVERT()`                             | 转换数据类型或字符集         |
| 信息函数   | `DATABASE()`、`USER()`、`VERSION()`                | 查看当前数据库环境          |

### 1.6.3 单行函数示例

```sql
SELECT UPPER('mysql');
-- 结果：MYSQL

SELECT ROUND(3.14159, 2);
-- 结果：3.14

SELECT NOW();
-- 结果：当前日期时间
```

也可以用在表字段上：

```sql
SELECT
    employee_id,
    last_name,
    UPPER(last_name) AS upper_name,
    ROUND(salary, 2) AS salary_amount
FROM employees;
```

这类函数不会改变原表中的资料，只是改变查询结果的显示或计算结果。

### 1.6.4 单行函数可以放在哪里？

单行函数常见于以下位置：

| 位置         | 示例                          | 说明        |
| ---------- | --------------------------- | --------- |
| `SELECT`   | `SELECT UPPER(name)`        | 改变查询结果的显示 |
| `WHERE`    | `WHERE TRIM(code) = 'A001'` | 对筛选条件做处理  |
| `ORDER BY` | `ORDER BY LENGTH(name)`     | 根据函数结果排序  |
| `GROUP BY` | `GROUP BY DATE(order_time)` | 根据函数结果分组  |
| `HAVING`   | `HAVING COUNT(*) > 10`      | 搭配聚合结果筛选  |

不过要注意，在 `WHERE` 中对索引字段套函数，可能影响索引使用。这个问题会在 `1.9` 说明。

## 1.7 聚合函数（Aggregate Functions / Multiple-Row Functions）

聚合函数也叫多行函数或分组函数。它的作用是把多行数据汇总成一个统计结果。

常见聚合函数包括：

| 函数              | 作用                 | 备注           |
| --------------- | ------------------ | ------------ |
| `COUNT(*)`      | 统计行数               | 会计算符合条件的所有行  |
| `COUNT(column)` | 统计指定字段非 `NULL` 的行数 | 会忽略 `NULL`   |
| `SUM(column)`   | 计算总和               | 常用于金额、数量     |
| `AVG(column)`   | 计算平均值              | 通常会忽略 `NULL` |
| `MAX(column)`   | 取得最大值              | 可用于数值、日期等    |
| `MIN(column)`   | 取得最小值              | 可用于数值、日期等    |

### 1.7.1 没有 `GROUP BY` 的聚合

如果没有 `GROUP BY`，聚合函数会把整个查询结果当成一组来计算。

```sql
SELECT COUNT(*) AS total_count
FROM employees;
```

这会统计整张员工表共有多少笔资料。

```sql
SELECT AVG(salary) AS avg_salary
FROM employees;
```

这会计算所有员工的平均薪资。

### 1.7.2 搭配 `GROUP BY` 的聚合

如果搭配 `GROUP BY`，聚合函数会对每个分组分别计算。

```sql
SELECT
    department_id,
    COUNT(*) AS employee_count,
    AVG(salary) AS avg_salary
FROM employees
GROUP BY department_id;
```

这段 SQL 的意思是：

```text
先按照 department_id 分组，
再分别计算每个部门的人数和平均薪资。
```

所以聚合函数不是永远只返回整张表的一行结果。准确地说：

* 没有 `GROUP BY`：整批资料汇总成一行。
* 有 `GROUP BY`：每个分组汇总成一行。

## 1.8 单行函数与聚合函数对比总结

| 比较项目             | 单行函数                                   | 聚合函数                                      |
| ---------------- | -------------------------------------- | ----------------------------------------- |
| 处理方式             | 对每一行分别处理                               | 对多行资料做汇总                                  |
| 输入与输出关系          | 通常一行输入对应一行输出                           | 多行输入汇总成一个结果                               |
| 是否会合并多行          | 不会                                     | 会                                         |
| 常见用途             | 格式转换、字串处理、数值处理、日期处理、条件判断               | 统计行数、总和、平均、最大值、最小值                        |
| 常见函数             | `UPPER()`、`ROUND()`、`NOW()`、`IFNULL()` | `COUNT()`、`SUM()`、`AVG()`、`MAX()`、`MIN()` |
| 是否常搭配 `GROUP BY` | 不一定                                    | 经常搭配                                      |
| 示例               | `UPPER(last_name)`                     | `AVG(salary)`                             |

### 用一句话记忆

```text
单行函数：每一行各自处理。
聚合函数：多行资料一起统计。
```

### 用查询结果来理解

假设 `employees` 表中有 3 笔资料：

| employee_id | last_name | salary |
| ----------- | --------- | ------ |
| 1           | Chen      | 30000  |
| 2           | Lin       | 40000  |
| 3           | Wang      | 50000  |

使用单行函数：

```sql
SELECT UPPER(last_name)
FROM employees;
```

结果会有 3 行：

| UPPER(last_name) |
| ---------------- |
| CHEN             |
| LIN              |
| WANG             |

使用聚合函数：

```sql
SELECT AVG(salary)
FROM employees;
```

结果只有 1 行：

| AVG(salary) |
| ----------- |
| 40000       |

这就是单行函数和聚合函数最核心的差别。

## 1.9 函数使用时的实务提醒

这一节是把函数从「会写语法」提升到「工作中写得稳」的关键。

### 1.9.1 注意 `NULL` 的影响

很多函数遇到 `NULL` 时，结果也可能变成 `NULL`。

例如：

```sql
SELECT CONCAT('Hello', NULL);
```

在 MySQL 中，如果 `CONCAT()` 的参数中有 `NULL`，结果会变成 `NULL`。

所以处理可能为空的字段时，常会搭配 `IFNULL()`：

```sql
SELECT CONCAT('Hello', IFNULL(nickname, ''))
FROM users;
```

聚合函数也要注意 `NULL`：

| 函数              | 是否忽略 `NULL`         |
| --------------- | ------------------- |
| `COUNT(*)`      | 不管字段是否为 `NULL`，统计行数 |
| `COUNT(column)` | 只统计该字段非 `NULL` 的行   |
| `SUM(column)`   | 通常忽略 `NULL`         |
| `AVG(column)`   | 通常忽略 `NULL`         |

### 1.9.2 注意返回类型

函数返回的结果不一定和输入类型完全一样。

例如：

* `DATE_FORMAT()` 返回的是格式化后的字符串。
* `COUNT()` 返回的是数字。
* `CONCAT()` 返回的是字符串。
* `ROUND()` 返回的是数值处理后的结果。

这会影响后续比较、排序、计算和 Java 程序接值时的类型判断。

### 1.9.3 注意函数可能影响索引

在工作中，函数最常见的性能问题是：对索引字段套函数，可能让索引不容易被有效使用。

例如，如果 `order_time` 有索引，下面这种写法在大表中要谨慎：

```sql
SELECT *
FROM orders
WHERE YEAR(order_time) = 2025;
```

因为这里对 `order_time` 套了 `YEAR()`，数据库可能需要先对每一行计算年份，再判断是否等于 2025。

更常见的改写方式是使用范围条件：

```sql
SELECT *
FROM orders
WHERE order_time >= '2025-01-01'
  AND order_time <  '2026-01-01';
```

这类写法通常更有利于使用 `order_time` 上的索引。

### 1.9.4 注意可移植性

函数很方便，但越依赖某个 DBMS 的函数，SQL 跨数据库迁移时越可能需要改写。

例如：

```sql
SELECT CONCAT(first_name, last_name)
FROM employees;
```

这是 MySQL 中常见的字符串拼接写法，但换到某些数据库时，可能要改成 `||` 或其他语法。

所以在笔记中建议标明：

```text
这是 MySQL 写法。
```

### 1.9.5 注意可读性

函数可以嵌套，但不代表越嵌套越好。

例如：

```sql
SELECT IFNULL(TRIM(UPPER(REPLACE(name, ' ', ''))), 'UNKNOWN')
FROM users;
```

这种 SQL 虽然可以执行，但阅读成本很高。实务上可以考虑：

* 拆成多个字段别名。
* 使用子查询或 CTE。
* 把复杂业务逻辑放到应用层。
* 在笔记中说明每一层函数的目的。

## 1.10 自我检查题

学完本节后，可以用下面问题检查自己是否真的理解。

### 问题 1

`UPPER(name)` 是单行函数还是聚合函数？为什么？

答案：单行函数。因为它会对每一行的 `name` 分别处理，并为每一行返回一个转换后的结果。

### 问题 2

`COUNT(*)` 和 `COUNT(column)` 有什么差别？

答案：`COUNT(*)` 统计符合条件的行数，不关心某个字段是否为 `NULL`；`COUNT(column)` 只统计该字段非 `NULL` 的行数。

### 问题 3

为什么说函数会影响 SQL 可移植性？

答案：因为不同 DBMS 对函数名称、参数、语法和返回行为的支持不完全相同。MySQL 能执行的函数 SQL，换到 Oracle、SQL Server 或 PostgreSQL 时不一定能直接执行。

### 问题 4

下面 SQL 有什么潜在问题？

```sql
SELECT *
FROM orders
WHERE YEAR(order_time) = 2025;
```

答案：如果 `order_time` 是索引字段，对它套 `YEAR()` 可能影响索引使用。更常见的写法是使用时间范围：

```sql
WHERE order_time >= '2025-01-01'
  AND order_time <  '2026-01-01'
```

### 问题 5

为什么 `AVG(salary)` 是聚合函数？

答案：因为它不是对每一行分别返回一个结果，而是把多行薪资汇总后计算出一个平均值。

## 1.11 常见混淆点

### 混淆点 1：单行函数不是只能处理一条记录

单行函数的意思不是「只能处理一行资料」，而是「对结果集中的每一行分别处理」。

如果查询结果有 100 行，单行函数就会对这 100 行分别执行。

### 混淆点 2：聚合函数不是一定只返回一行

没有 `GROUP BY` 时，聚合函数通常对整批资料返回一行结果。

但如果使用 `GROUP BY`，聚合函数会对每个分组返回一行结果。

### 混淆点 3：功能分类和使用方式分类不是同一件事

MySQL 函数可以从功能上分成数值函数、字符串函数、日期函数等。

也可以从使用方式上分成单行函数和聚合函数。

这两个分类角度不冲突。

例如：

* `ROUND()` 是数值函数，也是单行函数。
* `DATE_FORMAT()` 是日期时间函数，也是单行函数。
* `SUM()` 是聚合函数，也是统计类函数。

### 混淆点 4：函数不会自动改变原表资料

像下面这种查询：

```sql
SELECT UPPER(last_name)
FROM employees;
```

只是把查询结果显示成大写，不会把 `employees` 表中的 `last_name` 真正改成大写。

如果要修改资料，需要使用 `UPDATE`。

### 混淆点 5：函数写得出来，不代表适合放在数据库层

有些处理适合放 SQL 层，例如查询、筛选、汇总、格式化。

但有些复杂业务逻辑，如果写在 SQL 函数中会降低可读性和可维护性，就要考虑放在 Java 应用层处理。

---

## 1.12 本节总结

本节的重点可以浓缩成下面几句话：

1. 函数是封装好的数据处理逻辑，有输入、有处理、有输出。
2. SQL 函数可以让查询结果更接近业务需要，例如格式转换、数值计算、日期处理、条件判断和统计汇总。
3. MySQL 函数可以从功能角度分类，也可以从使用方式角度分类。
4. 单行函数是对每一行分别处理，常用于字符串、数值、日期、条件转换。
5. 聚合函数是对多行资料做汇总，常用于 `COUNT()`、`SUM()`、`AVG()`、`MAX()`、`MIN()`。
6. 使用函数时要注意 `NULL`、返回类型、可移植性、索引影响和 SQL 可读性。

如果只记一句话：

> 单行函数负责「逐行处理」，聚合函数负责「汇总统计」。
