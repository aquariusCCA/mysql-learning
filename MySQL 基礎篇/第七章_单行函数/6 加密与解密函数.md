## 本节导读

这一节整理 MySQL 中与「加密、解密、哈希、随机字节」有关的函数。

这章和前面的数值函数、字符串函数、日期函数不太一样。其他函数通常只要知道「怎么用」就能解决大部分问题；但加密与解密函数必须先建立安全观念，否则很容易出现下面几种错误：

- 把 `MD5()` 当成安全的密码保存方式。
- 以为 `SHA()` / `SHA1()` 可以「解密」回原文。
- 看到 `ENCODE()` / `DECODE()` 就以为它们是现代 MySQL 推荐的加解密函数。
- 把 `AES_ENCRYPT()` 的结果存在 `VARCHAR`，导致二进制数据被字符集转换破坏。
- 直接把明文密码、密钥写在 SQL 语句中，留下日志泄露风险。
- 不区分「哈希」和「加密」，导致系统设计方向错误。

因此，本节的学习重点不是只背函数表，而是先分清三件事：

1. **哈希 / 摘要函数**：把原文变成固定长度摘要，通常不可逆，例如 `MD5()`、`SHA()`、`SHA2()`。
2. **可逆加密 / 解密函数**：使用密钥加密后，可以用密钥解回原文，例如 `AES_ENCRYPT()`、`AES_DECRYPT()`。
3. **旧版或不建议继续使用的函数**：例如 `PASSWORD()`、`ENCODE()`、`DECODE()`，适合认识旧教材与旧系统，但不建议作为现代新代码主力。

如果只记一句话：

> `MD5()`、`SHA()`、`SHA2()` 是哈希；`AES_ENCRYPT()`、`AES_DECRYPT()` 是可逆加解密；`PASSWORD()`、`ENCODE()`、`DECODE()` 主要用于理解旧版本或旧教材，不建议新项目继续使用。

## 6.1 先分清：哈希、加密、编码不是同一件事

这一章最重要的是先分清下面三个概念。

### 6.1.1 哈希 / 摘要

哈希函数会把输入内容转换成一段摘要值。它的重点是：

```text
原文 -> 哈希函数 -> 摘要值
```

一般情况下，摘要值不能直接还原成原文。

例如：

```sql
SELECT MD5('123');
```

结果类似：

```text
202cb962ac59075b964b07152d234b70
```

你可以用同样的原文再次计算，然后比较摘要是否相同，但不能把这段摘要直接「解密」回 `123`。

常见哈希函数：

- `MD5()`
- `SHA()` / `SHA1()`
- `SHA2()`

### 6.1.2 可逆加密 / 解密

可逆加密的重点是：

```text
原文 + 密钥 -> 加密函数 -> 密文
密文 + 同一密钥 -> 解密函数 -> 原文
```

例如：

- `AES_ENCRYPT()`：加密。
- `AES_DECRYPT()`：解密。

这种处理可以还原原文，所以密钥管理非常重要。如果密钥泄露，密文就可能被解开。

### 6.1.3 编码 / 压缩

编码和压缩不是加密。

例如：

- Base64 是编码，不是加密。
- `COMPRESS()` 是压缩，不是加密。
- 十六进制显示只是表示形式，不是安全保护。

所以不要看到资料变成一串看不懂的内容，就认为它已经安全加密。

## 6.2 函数总览与现代使用建议

### 6.2.1 现代学习建议

| 函数 | 类型 | 现代学习建议 |
| --- | --- | --- |
| `MD5(str)` | 哈希 / 摘要 | 了解即可，可用于普通校验，不建议用于密码安全 |
| `SHA(str)` / `SHA1(str)` | 哈希 / 摘要 | 了解即可，SHA-1 已不适合作为现代安全重点 |
| `SHA2(str, hash_length)` | 哈希 / 摘要 | 推荐重点学习，优先于 `MD5()`、`SHA1()` |
| `AES_ENCRYPT(str,key)` | 可逆加密 | 推荐认识，适合理解 MySQL 层可逆加密 |

| `COMPRESS()` / `UNCOMPRESS()` | 压缩 / 解压 | 不是加密，可放到其他函数或压缩函数理解 |

### 6.2.2 这章应该怎么定位

如果你是为了学习 MySQL 单行函数，本章的核心是：

```text

```

不建议把本章当成「密码学完整教程」。真正的密码保存、密钥管理、加密传输、访问控制，还需要搭配应用层框架、安全规范和基础设施设计。

对于 Java / Spring 项目来说，使用者密码通常更建议在应用层使用专门的密码哈希方案，例如 Spring Security 提供的 `BCryptPasswordEncoder`，而不是直接在 MySQL 中用 `MD5()` 或 `SHA()` 保存密码。

## 6.3 哈希与摘要函数

哈希函数的共同特点是：

- 输入相同，摘要结果通常相同。
- 输入稍微变化，摘要结果会明显不同。
- 通常不能从摘要值直接还原原文。
- 适合做校验、比对、去重、摘要键等场景。
- 不等于「安全的密码保存方案」。

---

### 6.3.1 MD5()

`MD5(str)` 会计算字符串的 MD5 摘要。

```sql
SELECT MD5('123') AS md5_value;
```

结果：

```text
202cb962ac59075b964b07152d234b70
```

### 函数特点

| 项目 | 说明 |
| --- | --- |
| 返回长度 | 32 个十六进制字符 |
| 是否可逆 | 不可逆 |
| `NULL` 行为 | 参数为 `NULL` 时返回 `NULL` |
| 适合场景 | 普通校验、旧系统兼容、非安全场景摘要 |
| 不适合场景 | 现代密码保存、安全敏感签名 |

### 使用提醒

`MD5()` 的常见误区是把它当成密码加密。更准确地说，`MD5()` 是摘要函数，不是加密函数；而且 MD5 已经不适合作为现代安全场景的主要算法。

---

### 6.3.2 SHA() / SHA1()

`SHA(str)` 和 `SHA1(str)` 在 MySQL 中通常表示 SHA-1 摘要函数。

```sql
SELECT SHA('Tom123') AS sha_value;
```

结果类似：

```text
c7c506980abc31cc390a2438c90861d0f1216d50
```

也可以写：

```sql
SELECT SHA1('abc') AS sha1_value;
```

### 函数特点

| 项目 | 说明 |
| --- | --- |
| 返回长度 | 40 个十六进制字符 |
| 是否可逆 | 不可逆 |
| `NULL` 行为 | 参数为 `NULL` 时返回 `NULL` |
| 相对 MD5 | 通常比 MD5 更强，但 SHA-1 也不适合作为现代安全重点 |
| 现代建议 | 新学习重点应转向 `SHA2()` |

---

### 6.3.3 SHA2()

`SHA2(str, hash_length)` 用来计算 SHA-2 家族摘要。

```sql
SELECT SHA2('abc', 256) AS sha2_256_value;
```

`hash_length` 常见取值包括：

| `hash_length` | 含义 |
| --- | --- |
| `224` | SHA-224 |
| `256` | SHA-256 |
| `384` | SHA-384 |
| `512` | SHA-512 |
| `0` | 等同于 `256` |

### 示例

```sql
SELECT
    SHA2('abc', 224) AS sha2_224,
    SHA2('abc', 256) AS sha2_256,
    SHA2('abc', 512) AS sha2_512;
```

### `NULL` 与无效参数

```sql
SELECT
    SHA2(NULL, 256) AS r1,
    SHA2('abc', NULL) AS r2,
    SHA2('abc', 123) AS r3;
```

概念上：

- 参数为 `NULL`，结果为 `NULL`。
- `hash_length` 不是允许值，结果为 `NULL`。

### 使用建议

如果只是从 MySQL 内置函数角度学习摘要函数，`SHA2()` 比 `MD5()` 和 `SHA1()` 更值得重点掌握。

不过，仍要注意：

> `SHA2()` 是哈希函数，不是可逆加密函数；它也不等于完整的密码保存方案。

---

### 6.3.4 哈希函数对比

| 函数 | 返回长度 | 是否可逆 | 安全建议 |
| --- | --- | --- | --- |
| `MD5(str)` | 32 个十六进制字符 | 不可逆 | 不建议用于现代安全密码 |
| `SHA(str)` / `SHA1(str)` | 40 个十六进制字符 | 不可逆 | SHA-1 也不建议作为现代安全重点 |
| `SHA2(str,256)` | 64 个十六进制字符 | 不可逆 | 比前两者更适合现代摘要场景 |
| `SHA2(str,512)` | 128 个十六进制字符 | 不可逆 | 摘要更长，存储空间也更大 |

## 6.4 AES_ENCRYPT() 与 AES_DECRYPT()

`AES_ENCRYPT()` 和 `AES_DECRYPT()` 是 MySQL 中常见的可逆加密与解密函数。

### 6.4.1 基本概念

```text
AES_ENCRYPT()：原文 + 密钥 -> 密文
AES_DECRYPT()：密文 + 同一密钥 -> 原文
```

可逆加密和哈希最大的差别是：

| 比较项目 | 哈希函数 | AES 加解密 |
| --- | --- | --- |
| 是否能还原原文 | 通常不能 | 可以 |
| 是否需要密钥 | 通常不需要 | 需要 |
| 密钥泄露风险 | 不适用或较低 | 很高 |
| 典型用途 | 摘要、比对、校验 | 保护需要还原的敏感资料 |

---

### 6.4.2 AES_ENCRYPT() 基本示例

```sql
SELECT AES_ENCRYPT('mysql', 'secret_key') AS encrypted_value;
```

结果是二进制字符串。不同客户端可能会用十六进制形式显示，看起来不像普通文字。

---

### 6.4.3 AES_DECRYPT() 基本示例

```sql
SELECT CAST(AES_DECRYPT(AES_ENCRYPT('mysql', 'secret_key'), 'secret_key') AS CHAR) AS plain_text;
```

结果：

```text
mysql
```

这里使用 `CAST(... AS CHAR)` 是为了把解密后的二进制结果转换成可读文字。

---

### 6.4.4 密钥错误时会怎样？

如果解密密钥错误：

```sql
SELECT CAST(AES_DECRYPT(AES_ENCRYPT('mysql', 'secret_key'), 'wrong_key') AS CHAR) AS plain_text;
```

结果可能是：

- `NULL`
- 或无法正确阅读的内容

所以可逆加密一定要注意：

- 密钥必须妥善管理。
- 加密和解密使用同一套密钥和参数。
- 不能把密钥随便写死在 SQL 或代码中。

---

### 6.4.5 初始化向量 IV 与加密模式

某些 AES 加密模式需要初始化向量，也就是 IV。

概念上：

```text
原文 + 密钥 + IV -> 密文
密文 + 同一密钥 + 同一 IV -> 原文
```

示例：

```sql
SET block_encryption_mode = 'aes-256-cbc';
SET @key_str = SHA2('My secret passphrase', 512);
SET @init_vector = RANDOM_BYTES(16);
SET @crypt_str = AES_ENCRYPT('text', @key_str, @init_vector);

SELECT CAST(AES_DECRYPT(@crypt_str, @key_str, @init_vector) AS CHAR) AS plain_text;
```

结果：

```text
text
```

### 使用提醒

- 使用需要 IV 的加密模式时，加密和解密必须使用同一个 IV。
- IV 不是密码，可以和密文一起保存，但要保证设计符合安全要求。
- 真正生产系统中，密钥、IV、salt、加密模式都应由团队安全规范决定。

---

## 6.5 RANDOM_BYTES()

`RANDOM_BYTES(len)` 会产生指定长度的随机二进制字符串。

### 6.5.1 基本用法

```sql
SELECT RANDOM_BYTES(16) AS random_value;
```

这个结果是二进制值，客户端可能显示为十六进制。

### 6.5.2 常见用途

| 用途 | 说明 |
| --- | --- |
| 产生 IV | AES 某些模式需要初始化向量 |
| 产生 salt | 增加哈希或密钥派生的随机性 |
| 产生随机 token | 需要随机二进制资料的场景 |

### 6.5.3 使用限制

`RANDOM_BYTES(len)` 的 `len` 有允许范围。常见学习重点是：

- `len` 为 `NULL` 时，结果为 `NULL`。
- 超出允许范围会报错。
- 用于 AES 初始化向量时，通常至少需要 16 bytes。

## 6.6 旧版函数与版本提醒

这一节是本章最需要更新旧教材观念的地方。

很多 MySQL 教材会列出：

- `PASSWORD()`
- `ENCODE()`
- `DECODE()`

这些函数在旧版本 MySQL 中确实存在，因此学习旧教材或维护旧系统时可能会遇到。  
但如果你使用的是 MySQL 8.x 或新项目，不建议把它们当成主力函数学习。

---

### 6.6.1 PASSWORD()

旧教材常写：

```sql
SELECT PASSWORD('mysql');
```

它会返回一段不可逆处理后的字符串。

但需要注意：

> `PASSWORD()` 主要是旧版 MySQL 内部密码处理相关函数，不适合作为应用系统保存用户密码的通用方案。

现代学习时应该把它定位为：

```text
旧版 / 教材 / 旧系统兼容函数
```

而不是：

```text
新项目密码保存推荐函数
```

---

### 6.6.2 ENCODE() 与 DECODE()

旧教材常写：

```sql
SELECT ENCODE('mysql', 'mysql');
```

再用：

```sql
SELECT DECODE(ENCODE('mysql', 'mysql'), 'mysql');
```

概念上，`ENCODE()` 与 `DECODE()` 是一组可逆处理函数：

```text
ENCODE(value, password_seed) -> 编码/加密后的值
DECODE(value, password_seed) -> 还原原值
```

但现代 MySQL 学习时，不建议继续把它们当成主要可逆加密方案。  
如果你要学习 MySQL 层可逆加密，应优先认识：

```sql
AES_ENCRYPT()
AES_DECRYPT()
```

---

### 6.6.3 旧函数应该怎么写进笔记？

建议在笔记中这样标记：

| 函数 | 笔记定位 |
| --- | --- |
| `PASSWORD()` | 旧版函数，了解旧教材和旧系统即可，不建议新项目使用 |
| `ENCODE()` | 旧版可逆处理函数，了解即可 |
| `DECODE()` | 旧版可逆处理函数，了解即可 |
| `AES_ENCRYPT()` | 现代 MySQL 可逆加密学习重点 |
| `AES_DECRYPT()` | 现代 MySQL 可逆解密学习重点 |
| `SHA2()` | 现代摘要函数学习重点 |

---

## 6.7 存储设计：CHAR、BINARY、VARBINARY、BLOB

加密与哈希函数不只要知道怎么调用，还要知道结果该怎么存。

### 6.7.1 哈希结果怎么存

`MD5()`、`SHA1()`、`SHA2()` 通常返回十六进制字符串。

例如：

| 函数 | 十六进制长度 | 二进制长度概念 |
| --- | --- | --- |
| `MD5()` | 32 个 hex 字符 | 16 bytes |
| `SHA1()` | 40 个 hex 字符 | 20 bytes |
| `SHA2(..., 256)` | 64 个 hex 字符 | 32 bytes |
| `SHA2(..., 512)` | 128 个 hex 字符 | 64 bytes |

如果只是学习或简单比对，可以用 `CHAR` 存十六进制字符串。  
如果考虑空间和比较效率，可以使用 `UNHEX()` 转成二进制，再用 `BINARY(N)` 存储。

示例：

```sql
CREATE TABLE hash_demo (
    id BIGINT PRIMARY KEY,
    md5_hex CHAR(32),
    md5_bin BINARY(16)
);

INSERT INTO hash_demo(id, md5_hex, md5_bin)
VALUES (1, MD5('abcdef'), UNHEX(MD5('abcdef')));
```

---

### 6.7.2 AES 加密结果怎么存

`AES_ENCRYPT()` 的结果是二进制字符串，不建议存进普通 `CHAR`、`VARCHAR`、`TEXT` 字段。

推荐使用：

- `VARBINARY`
- `BLOB`

示例：

```sql
CREATE TABLE secret_demo (
    id BIGINT PRIMARY KEY,
    secret_value VARBINARY(255)
);

INSERT INTO secret_demo(id, secret_value)
VALUES (1, AES_ENCRYPT('sensitive text', 'secret_key'));
```

查询解密：

```sql
SELECT CAST(AES_DECRYPT(secret_value, 'secret_key') AS CHAR) AS plain_text
FROM secret_demo
WHERE id = 1;
```

### 为什么不要直接用 VARCHAR？

因为加密结果可能包含任意 byte 值，如果经过字符集转换、尾端空白处理或文本编码变化，可能造成密文损坏，导致之后无法正确解密。

---

## 6.8 实务场景

### 6.8.1 普通校验：产生摘要

如果只是想对某段字符串产生摘要：

```sql
SELECT SHA2('hello mysql', 256) AS digest_value;
```

适合场景：

- 资料校验。
- 产生摘要键。
- 比对内容是否一致。
- 旧系统资料迁移时对照结果。

不适合场景：

- 直接保存用户密码。
- 安全等级要求高的签名机制。

---

### 6.8.2 使用 AES 加密敏感字段

示例：

```sql
CREATE TABLE customer_secret (
    customer_id BIGINT PRIMARY KEY,
    id_number_enc VARBINARY(255)
);

INSERT INTO customer_secret(customer_id, id_number_enc)
VALUES (1, AES_ENCRYPT('A123456789', 'secret_key'));
```

查询时解密：

```sql
SELECT
    customer_id,
    CAST(AES_DECRYPT(id_number_enc, 'secret_key') AS CHAR) AS id_number
FROM customer_secret;
```

### 实务提醒

这个示例只是帮助理解函数。真实系统中不要把密钥明文写在 SQL 里，应该由应用配置、密钥管理系统或安全规范统一处理。

---

### 6.8.3 使用随机 IV

```sql
SET block_encryption_mode = 'aes-256-cbc';
SET @key_str = SHA2('My secret passphrase', 512);
SET @iv = RANDOM_BYTES(16);
SET @cipher = AES_ENCRYPT('secret text', @key_str, @iv);

SELECT CAST(AES_DECRYPT(@cipher, @key_str, @iv) AS CHAR) AS plain_text;
```

这个示例说明：

- `RANDOM_BYTES(16)` 可以产生随机 IV。
- 解密时需要同一个 IV。
- IV 通常要和密文一起保存，否则之后无法解密。

---

### 6.8.4 不推荐：用 MD5 保存用户密码

不建议：

```sql
INSERT INTO users(username, password_hash)
VALUES ('tom', MD5('123456'));
```

原因：

- MD5 不适合现代密码保存。
- 没有 salt，容易被字典和彩虹表攻击。
- 速度太快，反而有利于暴力破解。
- 密码保存应该使用专门的密码哈希算法。

更合理的方向：

```text
在 Java / Spring Security 应用层使用 BCrypt、Argon2、PBKDF2 等专门方案。
数据库只保存最终 password_hash。
```

---

## 6.9 使用提醒

### 6.9.1 不要把哈希叫成解密

下面这句话是错的：

```text
把 MD5 解密回来。
```

更准确的说法是：

```text
重新计算原文的 MD5，然后比较两个摘要是否相同。
```

哈希不是加密，通常不能解密回原文。

---

### 6.9.2 不要把 MD5 / SHA1 当成现代密码保存方案

`MD5()` 和 `SHA1()` 可以产生摘要，但不适合现代密码保存。  
如果是用户密码，应优先使用应用层密码哈希方案，而不是简单 SQL 哈希函数。

---

### 6.9.3 明文参数可能出现在传输和日志中

如果你写：

```sql
SELECT AES_ENCRYPT('my_password', 'my_key');
```

这代表明文和密钥都出现在 SQL 语句里。它们可能暴露在：

- 客户端历史记录。
- SQL 日志。
- 慢查询日志。
- 网络传输过程。
- 监控系统。

所以生产系统要避免把敏感值直接暴露在 SQL 语句中。

---

### 6.9.4 AES 结果要用二进制字段保存

`AES_ENCRYPT()` 返回的是二进制字符串，建议使用：

- `VARBINARY`
- `BLOB`

不要随便用普通文本字段保存密文。

---

### 6.9.5 注意密钥管理比函数本身更重要

加密算法再强，如果密钥被写死、被提交到 Git、被打印到日志，系统仍然不安全。

密钥管理应考虑：

- 密钥不要硬编码。
- 密钥不要出现在日志。
- 密钥要限制访问权限。
- 密钥轮换机制。
- 开发、测试、生产环境分离。

---

### 6.9.6 加密字段通常不适合普通模糊查询

如果字段被加密，数据库看到的是密文，不是原文。  
因此下面这类需求会变复杂：

- 模糊查询姓名。
- 依身份证后四码查询。
- 对手机号排序。
- 对加密字段做范围查询。

加密前要先确认业务是否需要查询、排序、筛选，否则之后可能很难处理。

---

## 6.10 常见混淆点

### 混淆点 1：MD5、SHA、SHA2 不是加密函数，而是哈希 / 摘要函数

它们通常不能解密回原文，只能重新计算后比较摘要。

---

### 混淆点 2：SHA() 不是 SHA2()

| 函数 | 意义 |
| --- | --- |
| `SHA()` / `SHA1()` | SHA-1 |
| `SHA2()` | SHA-2 家族 |

现代学习时，`SHA2()` 更值得重点掌握。

---

### 混淆点 3：可逆加密一定要保存解密所需参数

如果使用 AES 加密时涉及：

- 密钥
- IV
- salt
- KDF 参数
- 加密模式

解密时必须能取得对应参数，否则无法还原原文。

---

### 混淆点 4：ENCODE() / DECODE() 和 AES_ENCRYPT() / AES_DECRYPT() 不应混为一谈

`ENCODE()` / `DECODE()` 是旧版教材常见函数。  
现代学习 MySQL 可逆加密时，应优先看 `AES_ENCRYPT()` / `AES_DECRYPT()`。

---

### 混淆点 5：PASSWORD() 不是应用系统保存密码的推荐方案

即使旧教材写了 `PASSWORD()`，也不代表它适合你的业务系统。  
它应该被归类为旧版本函数认识，而不是新项目方案。

---

### 混淆点 6：压缩不是加密

`COMPRESS()` 会让内容变成不可直接阅读的形式，但它的目的不是安全保护，而是压缩。  
能还原不代表它是安全加密。

---

## 6.11 自我检查题

### 问题 1

`MD5('123')` 的结果能不能直接解密回 `123`？

答案：不能。`MD5()` 是哈希 / 摘要函数，不是可逆加密函数。

---

### 问题 2

`SHA()` 和 `SHA2()` 是同一个函数吗？

答案：不是。`SHA()` 通常等同于 `SHA1()`，属于 SHA-1；`SHA2()` 是 SHA-2 家族函数，可以指定 224、256、384、512 等摘要长度。

---

### 问题 3

如果要把 AES 加密后的结果存入表中，字段类型优先考虑什么？

答案：优先考虑 `VARBINARY` 或 `BLOB`，因为 AES 加密结果是二进制字符串。

---

### 问题 4

为什么不建议用下面方式保存用户密码？

```sql
INSERT INTO users(username, password_hash)
VALUES ('tom', MD5('123456'));
```

答案：`MD5()` 不适合现代密码保存，缺少 salt，速度太快，容易被字典、彩虹表或暴力破解攻击。更建议在应用层使用专门密码哈希算法，例如 BCrypt、Argon2、PBKDF2。

---

### 问题 5

`AES_DECRYPT()` 一定会在密钥错误时返回 `NULL` 吗？

答案：不一定。它可能返回 `NULL`，也可能返回无法正确理解的内容。因此不能只靠是否为 `NULL` 判断密钥一定正确。

---

### 问题 6

`ENCODE()` / `DECODE()` 在现代 MySQL 学习中应该怎么定位？

答案：主要用于认识旧教材或旧系统，不建议新项目继续使用；现代可逆加密应优先认识 `AES_ENCRYPT()` 与 `AES_DECRYPT()`。

---

## 6.12 本节总结

本节可以浓缩成下面几条：

1. 哈希和加密不是同一件事：哈希通常不可逆，加密可以用密钥解回原文。
2. `MD5()`、`SHA()`、`SHA1()`、`SHA2()` 是摘要函数，不是解密函数。
3. `SHA2()` 比 `MD5()` 和 `SHA1()` 更值得作为现代摘要函数重点学习。
4. `AES_ENCRYPT()` 与 `AES_DECRYPT()` 是 MySQL 中更值得认识的可逆加解密函数。
5. AES 加密结果是二进制字符串，建议存入 `VARBINARY` 或 `BLOB`。
6. `RANDOM_BYTES()` 可以用于产生随机 IV、salt 或随机 token。
7. `PASSWORD()`、`ENCODE()`、`DECODE()` 属于旧教材 / 旧系统常见函数，不建议新项目作为主力使用。
8. 用户密码保存不建议直接使用 `MD5()`、`SHA1()` 或简单 `SHA2()`，更建议放在应用层使用专门密码哈希方案。
9. 加密函数的安全性不只取决于函数名称，还取决于密钥管理、传输加密、日志保护、字段类型和整体系统设计。

如果只记一句话：

> 哈希用于比对，加密用于还原；旧函数用于认识，现代实务更要注意版本、安全与密钥管理。
