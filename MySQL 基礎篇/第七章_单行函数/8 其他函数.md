## 本节导读

这一节整理 MySQL 中一些不容易直接归入「数值函数」「字符串函数」「日期时间函数」「流程控制函数」「信息函数」的常用函数。

虽然这一组函数看起来比较杂，但它们在实际开发和运维中并不少见，尤其常出现在下面几类场景：

- 报表显示：把数字格式化成带千分位和固定小数位的字符串。
- 进制转换：在二进制、八进制、十进制、十六进制之间转换。
- IP 地址处理：把 IPv4 地址和数字互转，或处理 IPv6 地址。
- 表达式性能测试：粗略观察某个表达式重复执行的耗时。
- 字符集转换：把字符串转换成指定字符集，辅助解决乱码、比较规则或迁移问题。

学习这一节时，不建议把它当成「冷门函数集合」来背。更好的方式是先把它分成四类：

```text
格式化显示
进制与值转换
网络地址处理
性能测试与字符集转换
```

这样你遇到需求时，才知道该去哪里找函数。

## 8.1 这一组函数该怎么分类

「其他函数」这个名称容易让人误以为它们不重要，但实际上它们只是比较难放进单一主题。

可以先把本节函数分成四组：

| 分类 | 代表函数 | 核心用途 |
| --- | --- | --- |
| 格式化显示 | `FORMAT()` | 把数字转换成适合展示的字符串 |
| 进制转换 | `CONV()` | 在不同进制之间转换数值表示 |
| 网络地址处理 | `INET_ATON()`、`INET_NTOA()`、`INET6_ATON()`、`INET6_NTOA()` | IP 地址与内部表示互转 |
| 性能与字符集辅助 | `BENCHMARK()`、`CONVERT()` | 测试表达式耗时、转换字符集或类型 |

### 这一章的主线

可以用一句话记：

> 这一章不是核心业务逻辑函数，而是处理显示格式、底层表示、网络地址、字符集与性能观察的辅助函数。

## 8.2 `FORMAT()`

`FORMAT()` 用来把数字格式化成适合显示的字符串。它常见于报表、金额展示、统计结果输出等场景。

### 8.2.1 基本语法

```sql
FORMAT(value, n)
FORMAT(value, n, locale)
```

| 参数 | 说明 |
| --- | --- |
| `value` | 要格式化的数字 |
| `n` | 小数点后保留几位 |
| `locale` | 可选，指定地区格式，例如小数点和千分位符号 |

### 8.2.2 重要特性

`FORMAT()` 有三个重点：

1. 会四舍五入到指定小数位。
2. 结果通常带有千分位格式。
3. 回传结果是字符串，不是数值。

### 8.2.3 示例

```sql
SELECT
    FORMAT(12332.123456, 4) AS f1,
    FORMAT(12332.1, 4)      AS f2,
    FORMAT(12332.2, 0)      AS f3;
```

结果概念：

| 表达式 | 结果 |
| --- | --- |
| `FORMAT(12332.123456, 4)` | `'12,332.1235'` |
| `FORMAT(12332.1, 4)` | `'12,332.1000'` |
| `FORMAT(12332.2, 0)` | `'12,332'` |

### 8.2.4 指定 locale

```sql
SELECT FORMAT(12332.2, 2, 'de_DE') AS result;
```

结果概念：

```text
12.332,20
```

这表示不同地区格式可能使用不同的千分位和小数点符号。

### 8.2.5 `FORMAT()` 和 `ROUND()` 的差别

| 比较项目 | `FORMAT()` | `ROUND()` |
| --- | --- | --- |
| 主要用途 | 显示格式 | 数值计算 |
| 是否加千分位 | 通常会 | 不会 |
| 回传类型 | 字符串 | 数值 |
| 适合场景 | 报表展示、画面输出 | 计算金额、比例、小数位 |
| 是否适合继续参与计算 | 不建议 | 适合 |

### 8.2.6 使用提醒

`FORMAT()` 很适合最终输出，但不适合拿来当计算中间值。

不建议这样写：

```sql
SELECT FORMAT(amount, 2) * 1.05
FROM orders;
```

更建议：

```sql
SELECT FORMAT(amount * 1.05, 2) AS display_amount
FROM orders;
```

先完成数值计算，最后再格式化显示。

## 8.3 `CONV()`

`CONV()` 用来在不同进制之间转换数值表示。

### 8.3.1 基本语法

```sql
CONV(value, from_base, to_base)
```

| 参数 | 说明 |
| --- | --- |
| `value` | 要转换的值，可以是数字或字符串 |
| `from_base` | 原本的进制 |
| `to_base` | 要转换成的进制 |

### 8.3.2 核心定义

```text
CONV(value, from_base, to_base)
= 把 value 当成 from_base 进制，再转换成 to_base 进制
```

这是最重要的观念。

### 8.3.3 示例

```sql
SELECT
    CONV(16, 10, 2)     AS c1,
    CONV(8888, 10, 16)  AS c2,
    CONV('FF', 16, 10)  AS c3,
    CONV('10', 2, 8)    AS c4;
```

结果概念：

| 表达式 | 结果 | 说明 |
| --- | --- | --- |
| `CONV(16, 10, 2)` | `'10000'` | 十进制 16 转二进制 |
| `CONV(8888, 10, 16)` | `'22B8'` | 十进制 8888 转十六进制 |
| `CONV('FF', 16, 10)` | `'255'` | 十六进制 FF 转十进制 |
| `CONV('10', 2, 8)` | `'2'` | 二进制 10 转八进制 |

### 8.3.4 `CONV(10, 2, 8)` 为什么容易误解

```sql
SELECT CONV(10, 2, 8);
```

这不是「十进制 10 转八进制」。

它的意思是：

```text
把 10 当成二进制，再转换成八进制。
```

二进制 `10` 等于十进制 `2`，所以转换成八进制也是：

```text
2
```

如果你想把十进制 `10` 转成二进制，应该写：

```sql
SELECT CONV(10, 10, 2);
```

结果：

```text
1010
```

### 8.3.5 使用提醒

- `from_base` 和 `to_base` 的范围通常是 `2` 到 `36`。
- 任一参数为 `NULL`，结果会返回 `NULL`。
- `CONV()` 的结果是字符串表示，不是单纯数值。
- 如果涉及负数、有符号与无符号解释，要特别小心。

## 8.4 `INET_ATON()`

`INET_ATON()` 用来把点分十进制 IPv4 地址转换成一个整数。

### 8.4.1 基本语法

```sql
INET_ATON(ipvalue)
```

### 8.4.2 示例

```sql
SELECT INET_ATON('192.168.1.100') AS ip_number;
```

结果概念：

```text
3232235876
```

计算方式：

```text
192 * 256^3
+ 168 * 256^2
+ 1   * 256
+ 100
= 3232235876
```

### 8.4.3 适合场景

`INET_ATON()` 适合用于：

- 把 IPv4 地址转换成数字储存。
- 对 IP 范围做比较。
- 做 IP 黑名单、白名单、区间判断。
- 把点分十进制字符串改成更容易排序或比较的形式。

### 8.4.4 IP 范围查询示例

假设 `login_ip_num` 存的是 `INET_ATON(ip)` 的结果：

```sql
SELECT *
FROM login_logs
WHERE login_ip_num BETWEEN INET_ATON('192.168.1.1')
                       AND INET_ATON('192.168.1.255');
```

这样可以查询 `192.168.1.1` 到 `192.168.1.255` 之间的 IPv4 地址。

### 8.4.5 使用提醒

如果要存 `INET_ATON()` 的结果，应优先考虑 `INT UNSIGNED`。

原因是 IPv4 最大值可能超过有符号 `INT` 的正数范围，例如 `255.255.255.255` 对应的数值是 `4294967295`。

## 8.5 `INET_NTOA()`

`INET_NTOA()` 是 `INET_ATON()` 的反向操作，用来把数字形式的 IPv4 地址还原成点分十进制字符串。

### 8.5.1 基本语法

```sql
INET_NTOA(value)
```

### 8.5.2 示例

```sql
SELECT INET_NTOA(3232235876) AS ip_text;
```

结果：

```text
192.168.1.100
```

### 8.5.3 与 `INET_ATON()` 对比

| 函数 | 输入 | 输出 | 方向 |
| --- | --- | --- | --- |
| `INET_ATON()` | IPv4 字符串 | 整数 | IP -> 数字 |
| `INET_NTOA()` | 整数 | IPv4 字符串 | 数字 -> IP |

### 8.5.4 查询时还原 IP

```sql
SELECT
    user_id,
    INET_NTOA(login_ip_num) AS login_ip,
    login_time
FROM login_logs;
```

这种写法适合报表展示或后台查询。

## 8.6 IPv6 与 IP 验证相关函数

`INET_ATON()` 和 `INET_NTOA()` 主要处理 IPv4。现代系统如果要兼容 IPv6，应该认识下面这些函数：

| 函数 | 用途 |
| --- | --- |
| `INET6_ATON(ip)` | 将 IPv6 或 IPv4 地址转换成内部二进制格式 |
| `INET6_NTOA(value)` | 将内部二进制格式转回 IPv6 或 IPv4 字符串 |
| `IS_IPV4(value)` | 判断字符串是否为有效 IPv4 |
| `IS_IPV6(value)` | 判断字符串是否为有效 IPv6 |
| `IS_IPV4_COMPAT(value)` | 判断是否为 IPv4-compatible IPv6 地址 |
| `IS_IPV4_MAPPED(value)` | 判断是否为 IPv4-mapped IPv6 地址 |

### 8.6.1 IPv6 转换示例

```sql
SELECT INET6_NTOA(INET6_ATON('fdfe::5a55:caff:fefa:9089')) AS ip_text;
```

结果概念：

```text
fdfe::5a55:caff:fefa:9089
```

`INET6_ATON()` 也可以处理 IPv4：

```sql
SELECT INET6_NTOA(INET6_ATON('10.0.5.9')) AS ip_text;
```

结果：

```text
10.0.5.9
```

### 8.6.2 IP 格式检查

```sql
SELECT
    IS_IPV4('10.0.5.9')    AS is_ipv4_1,
    IS_IPV4('10.0.5.256')  AS is_ipv4_2,
    IS_IPV6('::1')         AS is_ipv6_1;
```

结果概念：

| 表达式 | 结果 |
| --- | --- |
| `IS_IPV4('10.0.5.9')` | `1` |
| `IS_IPV4('10.0.5.256')` | `0` |
| `IS_IPV6('::1')` | `1` |

### 8.6.3 使用建议

如果系统只处理 IPv4，可以使用：

```text
INET_ATON() / INET_NTOA()
```

如果系统需要兼容 IPv6，优先考虑：

```text
INET6_ATON() / INET6_NTOA()
```

如果只是要验证输入是否合法，可以使用：

```text
IS_IPV4() / IS_IPV6()
```

## 8.7 `BENCHMARK()`

`BENCHMARK()` 用来重复执行某个表达式，粗略观察 MySQL 处理该表达式需要多少时间。

### 8.7.1 基本语法

```sql
BENCHMARK(count, expr)
```

| 参数 | 说明 |
| --- | --- |
| `count` | 表达式重复执行次数 |
| `expr` | 要重复执行的表达式 |

### 8.7.2 示例

```sql
SELECT BENCHMARK(1000000, MD5('mysql'));
```

结果通常是：

```text
0
```

但重点不是结果值，而是客户端显示的执行时间，例如：

```text
1 row in set (0.20 sec)
```

### 8.7.3 返回值说明

`BENCHMARK()` 的返回值通常是：

| 情况 | 返回值 |
| --- | --- |
| 正常执行 | `0` |
| `count` 为 `NULL` 或负数等不合适参数 | `NULL` |

### 8.7.4 适合用途

`BENCHMARK()` 适合用来：

- 粗略比较两个表达式的执行成本。
- 观察某个函数重复执行时的大致耗时。
- 教学或实验中感受函数执行成本差异。

例如：

```sql
SELECT BENCHMARK(1000000, SHA2('mysql', 256));
```

### 8.7.5 不适合用途

`BENCHMARK()` 不适合拿来当完整 SQL 查询优化工具。

原因是它主要测量的是「表达式重复执行」，不是完整查询。完整查询还涉及：

- SQL 解析
- 优化器
- 表扫描
- 索引选择
- 锁
- I/O
- 网络传输
- 缓存状态
- 并发负载

如果要分析真实查询性能，应使用：

```sql
EXPLAIN
EXPLAIN ANALYZE
SHOW PROFILES -- 旧版本场景
Performance Schema
慢查询日志
```

## 8.8 `CONVERT()`

`CONVERT()` 有两种常见语境，必须分清楚：

1. 字符集转换：`CONVERT(value USING charset)`
2. 类型转换：`CONVERT(value, type)`

本节原本讨论的是第一种，也就是字符集转换。

---

### 8.8.1 字符集转换语法

```sql
CONVERT(value USING char_code)
```

例如：

```sql
SELECT CONVERT('mysql' USING utf8mb4);
```

意思是把字符串转换成指定字符集。

### 8.8.2 搭配 `CHARSET()` 查看效果

```sql
SELECT
    CHARSET('mysql') AS before_charset,
    CHARSET(CONVERT('mysql' USING utf8mb4)) AS after_charset;
```

这类写法适合用来观察转换前后的字符集。

### 8.8.3 字符集转换实务场景

`CONVERT(value USING charset)` 常见于：

- 排查乱码问题。
- 处理旧系统迁移资料。
- 比较不同字符集的字符串。
- 把 binary 字符串转换成非 binary 字符串后再做大小写处理。
- 临时观察某个值转换后的字符集表现。

### 8.8.4 类型转换语法

`CONVERT()` 也可以做类型转换：

```sql
CONVERT(expr, type)
```

它和 `CAST(expr AS type)` 很接近。

例如：

```sql
SELECT CONVERT('123', UNSIGNED) AS number_value;
```

也可以写成：

```sql
SELECT CAST('123' AS UNSIGNED) AS number_value;
```

### 8.8.5 `CONVERT()` 和 `CAST()` 的关系

| 写法 | 用途 |
| --- | --- |
| `CONVERT(expr USING charset)` | 字符集转换 |
| `CONVERT(expr, type)` | 类型转换 |
| `CAST(expr AS type)` | 类型转换 |

### 8.8.6 使用提醒

如果你在字段上使用 `CONVERT()` 或 `CAST()`，可能会影响索引使用。

例如：

```sql
SELECT *
FROM users
WHERE CONVERT(email USING utf8mb4) = 'test@example.com';
```

如果 `email` 本身有索引，这种写法可能让 MySQL 不容易有效使用索引。

更常见的做法是：

- 写入资料前就统一字符集和格式。
- 数据库表字段本身使用正确字符集。
- 必要时通过迁移脚本修正旧资料。
- 查询时避免对索引字段临时套转换函数。

---

## 8.9 典型实务场景

### 8.9.1 报表金额显示

```sql
SELECT
    order_id,
    FORMAT(total_amount, 2) AS display_amount
FROM orders;
```

适合用于展示：

```text
12,332.10
```

但如果还要继续计算，应使用原始数值字段，不要拿 `FORMAT()` 后的字符串继续算。

---

### 8.9.2 十进制转十六进制

```sql
SELECT
    id,
    CONV(id, 10, 16) AS hex_id
FROM orders;
```

适合调试编号、编码显示、底层表示观察。

---

### 8.9.3 十六进制转十进制

```sql
SELECT CONV('FF', 16, 10) AS decimal_value;
```

结果：

```text
255
```

---

### 8.9.4 IPv4 地址存储为数字

写入时：

```sql
INSERT INTO login_logs(user_id, login_ip_num, login_time)
VALUES (1001, INET_ATON('192.168.1.100'), NOW());
```

查询时：

```sql
SELECT
    user_id,
    INET_NTOA(login_ip_num) AS login_ip,
    login_time
FROM login_logs;
```

---

### 8.9.5 查询某个 IP 区间

```sql
SELECT *
FROM login_logs
WHERE login_ip_num BETWEEN INET_ATON('10.0.0.1')
                       AND INET_ATON('10.0.0.255');
```

这种写法适合 IPv4 区间查询。

---

### 8.9.6 验证 IP 输入是否合法

```sql
SELECT
    input_ip,
    CASE
        WHEN IS_IPV4(input_ip) = 1 THEN 'IPv4'
        WHEN IS_IPV6(input_ip) = 1 THEN 'IPv6'
        ELSE '非法 IP'
    END AS ip_type
FROM ip_import_temp;
```

这类写法适合资料导入或后台检查。

---

### 8.9.7 粗略比较表达式耗时

```sql
SELECT BENCHMARK(1000000, MD5('mysql'));
SELECT BENCHMARK(1000000, SHA2('mysql', 256));
```

可以粗略比较两个表达式重复执行时的耗时差异。

注意：这不是完整查询性能测试。

---

### 8.9.8 转换字符集后比较

```sql
SELECT *
FROM old_users
WHERE CONVERT(old_name USING utf8mb4) = '张三';
```

这种写法可以用于临时排查，但正式查询中应谨慎，因为可能影响索引使用。

---

## 8.10 使用提醒

### 8.10.1 `FORMAT()` 适合显示，不适合计算

`FORMAT()` 会返回字符串。

因此它适合：

```text
最终输出
报表显示
画面展示
```

不适合：

```text
作为金额计算中间值
作为排序数值
作为正式财务计算规则
```

如果要排序，建议按原始数值排序：

```sql
SELECT
    order_id,
    FORMAT(total_amount, 2) AS display_amount
FROM orders
ORDER BY total_amount DESC;
```

不要按 `display_amount` 排序，否则可能变成字符串排序。

---

### 8.10.2 `CONV()` 的结果是字符串表示

```sql
SELECT CONV(16, 10, 2);
```

结果是：

```text
10000
```

它是转换后的字符串表示，不要直接把它误解成普通十进制数值 `10000`。

---

### 8.10.3 `INET_ATON()` 只适合 IPv4

`INET_ATON()` 和 `INET_NTOA()` 是 IPv4 相关函数。

如果系统需要处理 IPv6，应考虑使用：

```text
INET6_ATON()
INET6_NTOA()
IS_IPV4()
IS_IPV6()
```

---

### 8.10.4 存储 IPv4 数字要注意 unsigned

`INET_ATON('255.255.255.255')` 的结果是：

```text
4294967295
```

这个值超过有符号 `INT` 的正数范围。  
因此如果要用整数存 IPv4，字段应优先考虑：

```sql
INT UNSIGNED
```

---

### 8.10.5 `BENCHMARK()` 不是完整性能分析工具

`BENCHMARK()` 主要观察表达式重复执行成本，不适合直接代表完整查询性能。

完整 SQL 查询还要看：

```text
执行计划
索引
资料量
I/O
锁
缓存
并发
网络
```

---

### 8.10.6 `CONVERT()` 可能影响索引

对索引字段临时套 `CONVERT()`、`CAST()`、`BINARY` 这类函数，可能让 MySQL 无法有效使用索引。

例如：

```sql
WHERE CONVERT(code USING utf8mb4) = 'A001'
```

如果 `code` 是索引字段，正式系统中要特别谨慎。

---

### 8.10.7 字符集转换不等于修复所有乱码

`CONVERT()` 可以转换字符集，但如果数据在写入时已经被错误编码，单纯转换不一定能完全修复。

乱码问题通常要同时检查：

- 表字段字符集
- 连接字符集
- 程序端编码
- 数据原始来源
- 写入时是否已经损坏
- 排序规则是否正确

---

## 8.11 常见混淆点

### 混淆点 1：`FORMAT()` 和 `ROUND()` 不一样

```sql
SELECT
    FORMAT(12332.123456, 2),
    ROUND(12332.123456, 2);
```

概念差异：

| 函数 | 结果类型 | 主要用途 |
| --- | --- | --- |
| `FORMAT()` | 字符串 | 显示 |
| `ROUND()` | 数值 | 计算 |

---

### 混淆点 2：`CONV(10, 2, 8)` 不是十进制 10 转八进制

```sql
SELECT CONV(10, 2, 8);
```

意思是：

```text
把 10 当成二进制，再转换成八进制。
```

如果要十进制 10 转八进制，应写：

```sql
SELECT CONV(10, 10, 8);
```

---

### 混淆点 3：`INET_ATON()` 和 `INET_NTOA()` 是 IPv4 方向相反的一组函数

| 函数 | 方向 |
| --- | --- |
| `INET_ATON()` | IPv4 字符串 -> 数字 |
| `INET_NTOA()` | 数字 -> IPv4 字符串 |

---

### 混淆点 4：`INET_ATON()` 不是严格 IP 验证工具

有些短格式 IPv4 字符串可能产生非 `NULL` 结果，但这不代表它一定是你业务上想接受的合法格式。

如果要严格判断 IPv4，优先使用：

```sql
IS_IPV4(ip)
```

---

### 混淆点 5：`BENCHMARK()` 的返回值不是重点

```sql
SELECT BENCHMARK(1000000, MD5('mysql'));
```

返回值通常是：

```text
0
```

真正要看的通常是客户端显示的执行时间。

---

### 混淆点 6：`CONVERT(value USING charset)` 和 `CONVERT(value, type)` 不同

| 写法 | 作用 |
| --- | --- |
| `CONVERT(value USING utf8mb4)` | 字符集转换 |
| `CONVERT(value, UNSIGNED)` | 类型转换 |

---

### 混淆点 7：`CONVERT()` 和 `CAST()` 经常可以做类似的类型转换

```sql
SELECT CONVERT('123', UNSIGNED);
SELECT CAST('123' AS UNSIGNED);
```

两者都可以把字符串 `'123'` 转成无符号整数。

## 8.12 自我检查题

### 问题 1

`FORMAT(12332.123456, 2)` 的主要用途是什么？它返回数值还是字符串？

答案：主要用于显示格式化，返回字符串。

---

### 问题 2

`FORMAT()` 和 `ROUND()` 的核心差异是什么？

答案：

- `FORMAT()` 适合显示，通常会加千分位，返回字符串。
- `ROUND()` 适合数值计算，返回数值处理结果。

---

### 问题 3

`CONV('FF', 16, 10)` 的意思是什么？结果是什么？

答案：把十六进制 `FF` 转换成十进制，结果是 `255`。

---

### 问题 4

`CONV(10, 2, 8)` 是什么意思？

答案：把 `10` 当成二进制，再转换成八进制，所以结果是 `2`。

---

### 问题 5

`INET_ATON('192.168.1.100')` 的作用是什么？

答案：把点分十进制 IPv4 地址转换成整数。

---

### 问题 6

如果要把 `3232235876` 转回 IPv4 字符串，应该用哪个函数？

答案：

```sql
SELECT INET_NTOA(3232235876);
```

---

### 问题 7

为什么存放 `INET_ATON()` 的结果建议使用 `INT UNSIGNED`？

答案：因为某些 IPv4 地址转换后的数值会超过有符号 `INT` 的正数范围，例如 `255.255.255.255` 对应 `4294967295`。

---

### 问题 8

`BENCHMARK(1000000, MD5('mysql'))` 返回值通常是什么？真正该观察什么？

答案：返回值通常是 `0`，真正该观察的是客户端显示的执行时间。

---

### 问题 9

`CONVERT('abc' USING utf8mb4)` 和 `CONVERT('123', UNSIGNED)` 有什么差别？

答案：

- `CONVERT('abc' USING utf8mb4)` 是字符集转换。
- `CONVERT('123', UNSIGNED)` 是类型转换。

---

### 问题 10

下面 SQL 有什么潜在问题？

```sql
SELECT *
FROM users
WHERE CONVERT(email USING utf8mb4) = 'test@example.com';
```

答案：如果 `email` 字段有索引，对字段套 `CONVERT()` 可能影响索引使用。更好的方式通常是让字段本身使用正确字符集，并在写入时规范化数据。

---

## 8.13 本节总结

本节可以浓缩成下面几条：

1. 「其他函数」不是不重要，而是用途分散，主要用于显示格式、进制转换、网络地址处理、性能观察和字符集转换。
2. `FORMAT()` 用于数字显示格式化，会四舍五入并返回字符串，不适合替代数值计算。
3. `CONV(value, from_base, to_base)` 的关键是看清楚原始值被当成什么进制。
4. `INET_ATON()` 把 IPv4 字符串转成整数，`INET_NTOA()` 把整数转回 IPv4 字符串。
5. 存放 `INET_ATON()` 结果时，应注意使用 `INT UNSIGNED`。
6. 如果系统需要处理 IPv6，应认识 `INET6_ATON()`、`INET6_NTOA()`、`IS_IPV4()`、`IS_IPV6()`。
7. `BENCHMARK()` 用来粗略观察表达式重复执行耗时，返回值通常不是重点。
8. `CONVERT(value USING charset)` 用于字符集转换，`CONVERT(value, type)` 用于类型转换。
9. 在索引字段上套 `CONVERT()`、`CAST()` 或其他函数，可能影响索引使用。
10. 实务中要分清「显示格式」「储存格式」「计算格式」三件事。

如果只记一句话：

> 这一章的函数主要负责把资料转换成更适合显示、比较、储存、验证或观察的形式，但正式系统中要特别注意返回类型、IP 版本、字符集和索引影响。
