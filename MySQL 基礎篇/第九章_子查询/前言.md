## 本节导读

这一节是第九章的入口，重点不是马上背各种子查询语法，而是先建立一个核心画面：

> 一个查询可以先产生中间结果，另一个查询再使用这个中间结果继续判断、筛选、计算或返回数据。

前面几章学习的 `SELECT`、`WHERE`、`ORDER BY`、多表查询、聚合函数，主要是在学习如何写出一条完整查询。到了子查询这一章，思路会再往前推进一步：**查询条件不一定是固定值，也可以来自另一条查询的结果**。

如果你是第一次学习子查询，建议先抓住三件事：

1. 子查询是什么。
2. 外层查询为什么需要使用内层查询的结果。
3. 子查询返回的是一个值、一组值，还是一张临时结果表。

只要先建立这三个观念，后面学习单行子查询、多行子查询、相关子查询、`EXISTS`、`NOT EXISTS` 时，就会比较容易分辨它们各自适合解决什么问题。

## 你会在这篇学到什么

- 什么是子查询。
- 为什么有些查询需求需要先查出一个中间结果。
- 如何区分内层查询与外层查询。
- 子查询可能返回哪些类型的结果。
- 子查询通常可以出现在 `SELECT` 语句的哪些位置。
- 为什么学习阶段可以先用“内层结果给外层使用”来理解，但不能把它误解成数据库实际执行时一定永远先跑内层。
- 学习子查询时，后续章节应该按照什么顺序建立观念。

## 关键字

- `子查询`：嵌套在另一个 SQL 语句内部的查询。
- `内层查询`：被包在内部、负责提供中间结果的查询。
- `外层查询`：使用子查询结果继续筛选、比较、计算或返回数据的查询。
- `标量`：单一值，例如一个平均工资、一个最大值、一个员工编号。
- `集合`：多笔结果组成的一组值，例如多个部门编号、多个员工编号。
- `派生表`：子查询返回的临时结果表，通常出现在 `FROM` 子句中。
- `相关子查询`：子查询会引用外层查询当前行的数据，因此内层结果会随着外层当前行而变化。
- `不相关子查询`：子查询可以独立执行，不依赖外层查询当前行的数据。

## 1. 什么是子查询

子查询指的是：**一个查询语句嵌套在另一个 SQL 语句内部**。

例如：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
);
```

这条 SQL 可以拆成两层来看：

```sql
SELECT AVG(salary)
FROM employees;
```

这一段是内层查询，负责先算出员工平均工资。

```sql
SELECT last_name, salary
FROM employees
WHERE salary > 平均工资;
```

这一段是外层查询，负责找出工资高于平均工资的员工。

所以子查询的基本思路可以理解为：

1. 内层查询先提供一个中间结果。
2. 外层查询再使用这个中间结果继续完成最终查询。

### 注意：这是逻辑理解顺序，不一定是实际执行顺序

学习初期，可以先把子查询理解成“内层先算出结果，外层再使用结果”。

但要注意：这主要是帮助理解 SQL 语意的方式，不代表 MySQL 实际执行计划一定永远按照这个顺序执行。实际运行时，优化器可能会改写 SQL，例如把某些子查询转换成连接、半连接或其他更适合执行的形式。

所以在学习阶段可以先记：

> 逻辑上，子查询提供结果给外层查询使用；  
> 实务上，真实执行方式要看优化器和执行计划。

## 2. 为什么需要子查询

很多查询条件并不是一开始就知道的固定值，而是需要先查询、计算或筛选出来。

例如：

- 想查询工资高于公司平均工资的员工。
- 想查询工资高于某一位员工的员工。
- 想查询属于某几个部门的员工。
- 想查询至少有订单的客户。
- 想查询没有员工的部门。
- 想先算出每个部门的平均工资，再找出高于本部门平均工资的员工。

这些需求的共同点是：

> 外层查询要使用的条件，本身也是查询出来的结果。

如果不用子查询，可能要先手动执行一条 SQL，把结果复制出来，再写到另一条 SQL 里。这样不但麻烦，也不适合真实系统自动查询。

子查询可以把这两个步骤合并在同一条 SQL 中，让数据库自己完成中间结果的计算与后续筛选。

## 3. 子查询解决的核心问题

子查询本质上是在解决这类问题：

> 先得到一个中间结果，再让外层查询基于这个结果继续处理。

不同的中间结果，会对应不同类型的子查询。

| 需求类型 | 子查询返回结果 | 常见写法 | 后续对应章节 |
| --- | --- | --- | --- |
| 先查出一个值，再拿来比较 | 1 行 1 列 | `salary > (SELECT AVG(salary) ...)` | 单行子查询 |
| 先查出一组值，再判断是否属于其中之一 | 多行 1 列 | `department_id IN (SELECT id ...)` | 多行子查询 |
| 多个字段要作为一组一起比较 | 1 行多列或多行多列 | `(manager_id, department_id) IN (...)` | 成对比较 |
| 先产生一张临时结果表，再拿来连接 | 多行多列 | `FROM (SELECT ... GROUP BY ...) t` | 表子查询 / 派生表 |
| 针对外层每一行去检查是否存在对应资料 | 是否返回行 | `EXISTS (...)` / `NOT EXISTS (...)` | 相关子查询 |

学习子查询时，不要只看语法长相，更要问自己：

> 内层查询到底要返回什么？  
> 外层查询拿这个结果来做什么？

这个问题想清楚之后，才比较容易判断该用 `=`、`IN`、`ANY`、`ALL`、`EXISTS`，还是改用 `FROM` 子查询。

## 4. 子查询常见出现位置

在 `SELECT` 查询中，子查询可以出现在很多位置。入门阶段可以先记住几个最常见的位置：

### 4.1 出现在 `WHERE`

最常见，用来作为筛选条件。

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
);
```

适合处理：

- 比某个查询结果大；
- 等于某个查询结果；
- 属于某个查询结果集合；
- 是否存在对应资料。

### 4.2 出现在 `HAVING`

用于对分组后的结果继续筛选。

```sql
SELECT department_id, MIN(salary)
FROM employees
GROUP BY department_id
HAVING MIN(salary) > (
    SELECT MIN(salary)
    FROM employees
    WHERE department_id = 50
);
```

适合处理：

- 分组结果与某个统计值比较；
- 每组的平均值、最大值、最小值与另一个查询结果比较。

### 4.3 出现在 `FROM`

当子查询返回多行多列时，可以把它当成一张临时表使用，也叫派生表。

```sql
SELECT e.last_name, e.salary, dept_avg.avg_salary
FROM employees e
JOIN (
    SELECT department_id, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
) AS dept_avg
ON e.department_id = dept_avg.department_id;
```

注意：

> `FROM` 子查询通常必须取别名，否则外层查询无法引用这张派生表。

### 4.4 出现在 `SELECT`

可以把子查询结果当成查询字段的一部分。

```sql
SELECT last_name,
       salary,
       (
           SELECT AVG(salary)
           FROM employees
       ) AS company_avg_salary
FROM employees;
```

这种写法可以用来显示某个额外计算结果，但要注意子查询必须能返回适合当前位置使用的结果。

### 4.5 出现在 `ORDER BY`

可以用子查询产生排序依据。

```sql
SELECT employee_id, salary
FROM employees e
ORDER BY (
    SELECT department_name
    FROM departments d
    WHERE d.department_id = e.department_id
);
```

这种写法通常会涉及相关子查询，因为排序依据会随着外层每一行而变化。

### 4.6 关于 `GROUP BY` 和 `LIMIT`

入门阶段可以先保守记忆：

> 子查询最常练习的位置是 `SELECT`、`FROM`、`WHERE`、`HAVING`、`ORDER BY`。  
> `GROUP BY` 和 `LIMIT` 不是学习子查询时的主要使用位置，先不要把它们当成常规写法。

后续如果进入更细的语法限制或执行计划分析，再另外讨论不同数据库、不同版本与不同写法的具体限制。

## 5. 学习子查询时最重要的判断步骤

看到一道子查询题，不要急着写 SQL，可以先按下面顺序判断。

### 第一步：先确认最终要查什么

也就是外层查询要返回哪些资料。

例如：

- 要查员工姓名和薪资；
- 要查部门编号和平均工资；
- 要查没有订单的客户；
- 要查高于本部门平均工资的员工。

### 第二步：确认中间结果是什么

也就是内层查询要先提供什么。

例如：

- 公司平均工资；
- 某位员工的工资；
- 某些部门编号；
- 每个部门的平均工资；
- 是否存在对应订单。

### 第三步：确认子查询返回结果的形态

这是决定写法的关键。

| 子查询返回结果 | 常见类型 | 常见搭配 |
| --- | --- | --- |
| 1 行 1 列 | 标量子查询 | `=`、`>`、`<`、`>=`、`<=`、`<>` |
| 多行 1 列 | 列子查询 | `IN`、`NOT IN`、`ANY`、`ALL` |
| 1 行多列 | 行子查询 | `(col1, col2) = (...)` |
| 多行多列 | 表子查询 | `FROM (...) AS alias` |

### 第四步：判断是否依赖外层查询

如果子查询可以单独执行，不需要外层当前行的数据，通常是不相关子查询。

如果子查询中引用了外层查询的列，例如：

```sql
WHERE e1.department_id = e2.department_id
```

那么它就是相关子查询。

可以先记一句话：

> 子查询中使用了外层查询的列，就要高度怀疑它是相关子查询。

## 6. 初学者最容易混淆的地方

### 6.1 误以为子查询一定实际先执行

学习时可以用“内层结果给外层使用”来理解，但实际执行方式可能会被 MySQL 优化器改写。

所以不要把逻辑理解顺序直接等同于真实执行计划。

### 6.2 误以为 `=` 后面一定可以随便放子查询

如果使用 `=`、`>`、`<` 这类单行比较符，子查询必须稳定返回一个值。

如果子查询返回多行，就不能直接使用单行比较符，通常要改用 `IN`、`ANY`、`ALL` 等多行比较方式。

### 6.3 忽略 `NOT IN` 和 `NULL` 的问题

`NOT IN` 搭配子查询时，如果子查询结果中出现 `NULL`，可能会导致结果和直觉不一致，甚至查不到资料。

后面学习多行子查询时，需要特别注意这个问题。

### 6.4 误以为子查询一定比连接好，或连接一定比子查询好

同一个需求，有时可以用子查询，也可以用连接。

学习阶段应先关注：

- 哪一种写法更贴近题意；
- 哪一种写法更容易读懂；
- 哪一种写法更容易维护。

如果真的要比较性能，就要看具体数据库、资料量、索引设计与执行计划，而不是只凭 SQL 外观判断。

## 7. 本章建议学习顺序

建议按下面顺序学习这一章：

1. 先理解什么是子查询，以及为什么需要子查询。
2. 再学习单行子查询，掌握 `=`、`>`、`<` 等单行比较。
3. 接着学习多行子查询，分清 `IN`、`ANY`、`ALL`、`SOME`。
4. 再学习相关子查询，理解内层查询如何依赖外层当前行。
5. 最后比较子查询、自连接、派生表、`EXISTS` 等不同写法的取舍。

这样学习会比较顺，因为你会先从简单的“一个值”开始，再进入“一组值”，最后才进入“每一行都要重新判断”的相关子查询。
