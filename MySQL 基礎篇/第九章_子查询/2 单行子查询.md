## 本节导读

这一节聚焦在 **单行子查询**。

所谓单行子查询，最核心的前提是：

> 内层查询必须返回一个可以被外层当成「单一值」使用的结果。

因此，本节重点不只是会写：

```sql
=、>、>=、<、<=、<>
```

而是要看懂：

- 什么情况下子查询真的只会返回一个值；
- 什么情况下子查询语法看起来像单行子查询，但实际可能返回多行；
- 子查询返回 `0` 行、`NULL`、多行时，外层查询会出现什么结果；
- 为什么 `GROUP BY` 很容易让单行子查询变成多行子查询；
- 为什么 `CASE`、`HAVING`、成对比较中也可以出现子查询，但必须注意返回结果是否符合外层语法期待。

学习这一节时，建议抓住一句话：

> 单行子查询不是看 SQL 写得短不短，而是看内层查询最终能不能稳定提供一个单独值给外层使用。

## 关键字

- `单行子查询`：通常指返回 `1` 行 `1` 列结果，并能被外层当成单一值使用的子查询。
- `标量子查询`：返回单一值的子查询，常与单行子查询概念接近。
- `单行比较操作符`：`=`、`>`、`>=`、`<`、`<=`、`<>` 等比较符。
- `成对比较`：把多个字段当作一组整体进行比较。
- `不成对比较`：每个字段分开比较，彼此独立判断。
- `HAVING`：对分组后的结果继续筛选的子句。
- `CASE`：SQL 中的条件表达式。
- `NULL`：空值，代表未知或没有值，不能直接用 `=` 判断。
- `多行子查询误用`：子查询返回多行，却被外层当成单一值比较，通常会报错。

## 2.1 什么是单行子查询

单行子查询指的是：**内层查询最终只返回一个可以被外层当成普通值使用的结果**。

例如：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE employee_id = 149
);
```

这条 SQL 中：

```sql
SELECT salary
FROM employees
WHERE employee_id = 149
```

就是单行子查询。

因为 `employee_id` 通常是主键或唯一字段，所以这个子查询理论上只会返回一位员工的工资。

外层查询可以把它理解成：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > 149号员工的工资;
```

因此，单行子查询的本质是：

> 先查出一个值，再让外层查询拿这个值继续比较。

## 2.1.1 单行子查询的两个核心条件

一个子查询要能安全地当作单行子查询使用，通常要同时满足两个条件：

| 条件 | 说明 |
| --- | --- |
| 只返回一列 | 外层比较的是一个值，所以内层不能返回多个字段 |
| 最多返回一行 | 外层使用单行比较符时，不能接收多笔结果 |

例如下面这个子查询只返回一个值：

```sql
SELECT salary
FROM employees
WHERE employee_id = 149;
```

但是下面这个子查询可能返回多行：

```sql
SELECT salary
FROM employees
WHERE department_id = 50;
```

因为一个部门可能有多个员工，所以它不适合直接写成：

```sql
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE department_id = 50
);
```

如果子查询可能返回多行，就应该考虑改用 `IN`、`ANY`、`ALL`，或者先用聚合函数压成一个值。

## 2.1.2 单行子查询常见来源

下面几种写法通常比较容易得到单一值。

### 1. 使用主键或唯一字段筛选

```sql
SELECT salary
FROM employees
WHERE employee_id = 149;
```

因为 `employee_id` 通常唯一，所以结果稳定。

### 2. 使用聚合函数

```sql
SELECT MAX(salary)
FROM employees;
```

即使表中有很多员工，`MAX(salary)` 也只会返回一个最大工资。

常见聚合函数包括：

```sql
MAX()
MIN()
AVG()
SUM()
COUNT()
```

### 3. 使用明确只能返回一笔资料的条件

```sql
SELECT department_id
FROM departments
WHERE department_name = 'IT';
```

这要建立在 `department_name` 确实唯一的前提下。

如果资料表没有保证唯一，就不能只凭直觉认定它一定返回一行。

## 2.2 单行比较操作符

单行子查询常见会搭配以下比较操作符：

| 操作符 | 含义 | 示例语意 |
| --- | --- | --- |
| `=` | 等于 | 等于某个查询结果 |
| `>` | 大于 | 大于某个查询结果 |
| `>=` | 大于等于 | 大于或等于某个查询结果 |
| `<` | 小于 | 小于某个查询结果 |
| `<=` | 小于等于 | 小于或等于某个查询结果 |
| `<>` | 不等于 | 不等于某个查询结果 |

这些操作符可以直接搭配单行子查询，是因为外层查询期待右边是一个单独值。

例如：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
);
```

这里的子查询：

```sql
SELECT AVG(salary)
FROM employees
```

只返回一个平均工资，所以外层可以使用：

```sql
salary > 平均工资
```

## 2.2.1 单行比较符的使用前提

只要看到下面这些操作符：

```sql
=、>、>=、<、<=、<>
```

就要立刻检查右侧子查询是否真的只会返回一个值。

例如：

```sql
WHERE salary = (
    SELECT MIN(salary)
    FROM employees
)
```

这是合理的，因为 `MIN(salary)` 只会返回一个值。

但是：

```sql
WHERE salary = (
    SELECT MIN(salary)
    FROM employees
    GROUP BY department_id
)
```

这就有问题，因为 `GROUP BY department_id` 会让每个部门都返回一个最低工资，结果变成多行。

## 2.2.2 单行比较符与多行子查询不能混用

错误写法：

```sql
SELECT employee_id, last_name
FROM employees
WHERE salary = (
    SELECT salary
    FROM employees
    WHERE department_id = 50
);
```

如果 `department_id = 50` 的员工有很多位，子查询会返回多笔工资。

外层却使用：

```sql
salary = (...)
```

这表示外层期待右边只有一个值。

这时就会发生：

> 多行子查询误用单行比较符。

可以根据题意改成下面几种方式。

### 情况一：只要等于其中一个工资

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary IN (
    SELECT salary
    FROM employees
    WHERE department_id = 50
);
```

### 情况二：大于其中任意一个工资

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary > ANY (
    SELECT salary
    FROM employees
    WHERE department_id = 50
);
```

### 情况三：大于所有工资

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary > ALL (
    SELECT salary
    FROM employees
    WHERE department_id = 50
);
```

### 情况四：先压成一个值再比较

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
    WHERE department_id = 50
);
```

这时子查询返回的是 `50` 号部门的平均工资，所以又变回单行子查询。

## 2.3 单行子查询的结果情况

单行子查询最容易出问题的地方，不是语法，而是结果笔数。

可以用下面这张表判断：

| 子查询结果 | 外层行为 | 是否正常 |
| --- | --- | --- |
| 返回 `1` 行 `1` 列，且值不是 `NULL` | 可以正常比较 | 正常 |
| 返回 `1` 行 `1` 列，但值是 `NULL` | 比较结果通常不是 `TRUE` | 要小心 |
| 返回 `0` 行 | 标量子查询结果通常可理解为 `NULL` | 可能查不到结果 |
| 返回多行 | 单行比较符无法接收 | 通常报错 |
| 返回多列 | 单值比较无法接收 | 通常报错，除非使用行比较语法 |

## 2.3.1 返回一个正常值

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE employee_id = 149
);
```

如果 `employee_id = 149` 查到工资 `10500`，外层条件就相当于：

```sql
WHERE salary > 10500
```

这是最标准的单行子查询。

## 2.3.2 返回 `0` 行

```sql
SELECT last_name, job_id
FROM employees
WHERE job_id = (
    SELECT job_id
    FROM employees
    WHERE last_name = 'Haas'
);
```

如果表中没有 `last_name = 'Haas'` 的员工，子查询不会返回任何行。

这时外层条件无法得到实际可比较的值，可以先理解成：

```sql
WHERE job_id = NULL
```

而在 SQL 中，不能用 `=` 判断 `NULL`，所以结果通常不会被判断为 `TRUE`，最终可能查不到任何资料。

## 2.3.3 返回 `NULL`

如果子查询虽然返回一行，但该字段值本身是 `NULL`，也会出现类似问题。

例如：

```sql
SELECT last_name, commission_pct
FROM employees
WHERE commission_pct = (
    SELECT commission_pct
    FROM employees
    WHERE employee_id = 178
);
```

如果 `178` 号员工的 `commission_pct` 是 `NULL`，外层条件会变成：

```sql
commission_pct = NULL
```

这通常不会成立。

如果真的要判断 `NULL`，应该使用：

```sql
IS NULL
```

或者在 MySQL 中使用空值安全比较：

```sql
<=>
```

例如：

```sql
SELECT last_name, commission_pct
FROM employees
WHERE commission_pct <=> (
    SELECT commission_pct
    FROM employees
    WHERE employee_id = 178
);
```

不过 `<=>` 属于 MySQL 的空值安全比较符，学习初期先知道它可以处理 `NULL` 即可，不需要在所有场景都优先使用。

## 2.3.4 返回多行

```sql
SELECT employee_id, last_name
FROM employees
WHERE salary = (
    SELECT MIN(salary)
    FROM employees
    GROUP BY department_id
);
```

这里的子查询因为有：

```sql
GROUP BY department_id
```

所以会针对每个部门各返回一个最低工资。

结果变成：

```text
2500
3000
4200
...
```

这已经不是一个值，而是一组值。

外层却使用：

```sql
salary = (...)
```

因此会报错。

### 回查提示

只要子查询中出现 `GROUP BY`，就要特别检查它是否会返回多行。

如果你要的真的是一组值，就改用 `IN`、`ANY`、`ALL`。  
如果你要的仍然是一个值，就要再加一层聚合或限制条件，把结果压成单值。

## 2.4 典型代码示例

## 2.4.1 查询工资大于 149 号员工工资的员工

题目：查询工资大于 `149` 号员工工资的员工。

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE employee_id = 149
);
```

这条 SQL 可以拆成两步理解。

第一步，子查询先找出 `149` 号员工的工资：

```sql
SELECT salary
FROM employees
WHERE employee_id = 149;
```

第二步，外层查询找出工资高于这个值的员工：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > 149号员工的工资;
```

这个例子是标准的单行子查询，因为 `employee_id` 通常唯一，所以子查询只会返回一个工资值。

### 判断重点

| 判断点 | 结果 |
| --- | --- |
| 子查询返回几列 | 1 列，`salary` |
| 子查询返回几行 | 理论上 1 行，因为 `employee_id` 唯一 |
| 外层操作符 | `>` |
| 是否符合单行子查询 | 符合 |

## 2.4.2 返回 `job_id` 与 141 号员工相同，且 `salary` 比 143 号员工高的员工

题目：返回 `job_id` 与 `141` 号员工相同，且 `salary` 比 `143` 号员工高的员工。

```sql
SELECT last_name, job_id, salary
FROM employees
WHERE job_id = (
    SELECT job_id
    FROM employees
    WHERE employee_id = 141
)
AND salary > (
    SELECT salary
    FROM employees
    WHERE employee_id = 143
);
```

这条 SQL 同时使用两个单行子查询。

第一个子查询：

```sql
SELECT job_id
FROM employees
WHERE employee_id = 141
```

负责取得 `141` 号员工的职位编号。

第二个子查询：

```sql
SELECT salary
FROM employees
WHERE employee_id = 143
```

负责取得 `143` 号员工的工资。

外层查询同时满足两个条件：

```sql
job_id = 141号员工的job_id
salary > 143号员工的salary
```

### 判断重点

| 条件 | 子查询结果 | 外层比较 |
| --- | --- | --- |
| `job_id = (...)` | 141 号员工的 `job_id` | 等于 |
| `salary > (...)` | 143 号员工的 `salary` | 大于 |

这类题目要注意：

> 多个单行子查询可以同时出现在一条 SQL 中，但每一个子查询都必须各自满足单值前提。

## 2.4.3 返回公司工资最低的员工

题目：返回公司中工资最低的员工。

```sql
SELECT last_name, job_id, salary
FROM employees
WHERE salary = (
    SELECT MIN(salary)
    FROM employees
);
```

这条 SQL 的思路是：

1. 子查询先计算整张表的最低工资。
2. 外层查询找出工资等于这个最低值的员工。

内层查询：

```sql
SELECT MIN(salary)
FROM employees;
```

只会返回一个值。

所以外层可以使用：

```sql
salary = 最低工资
```

### 重要观念：子查询返回一行，不代表外层只返回一行

如果公司中有多位员工工资都等于最低工资，那么外层查询会返回多位员工。

这不是错误。

因为：

- 单行子查询限制的是内层查询结果；
- 外层查询可以返回多行。

也就是说：

> 单行子查询的「单行」指的是内层结果，不是最终查询结果。

## 2.5 成对比较和不成对比较

当题目需要同时比较多个字段时，容易混淆：

- 每个字段分别比较；
- 多个字段作为一组整体比较。

这就是不成对比较与成对比较的差异。

## 2.5.1 不成对比较

题目：查询与 `141` 号或 `174` 号员工的 `manager_id` 和 `department_id` 相同的其他员工的 `employee_id`、`manager_id`、`department_id`。

不成对比较可以写成：

```sql
SELECT employee_id, manager_id, department_id
FROM employees
WHERE manager_id IN (
    SELECT manager_id
    FROM employees
    WHERE employee_id IN (174, 141)
)
AND department_id IN (
    SELECT department_id
    FROM employees
    WHERE employee_id IN (174, 141)
)
AND employee_id NOT IN (174, 141);
```

这里的判断方式是：

```text
manager_id 落在目标 manager_id 集合中
department_id 落在目标 department_id 集合中
```

这两个条件是分开判断的。

也就是说：

> `manager_id` 和 `department_id` 不一定要来自同一位目标员工的原始组合。

## 2.5.2 成对比较

如果题目要求两个字段必须作为一组整体匹配，就要使用成对比较：

```sql
SELECT employee_id, manager_id, department_id
FROM employees
WHERE (manager_id, department_id) IN (
    SELECT manager_id, department_id
    FROM employees
    WHERE employee_id IN (141, 174)
)
AND employee_id NOT IN (141, 174);
```

这里的重点是：

```sql
(manager_id, department_id)
```

被当成一个整体。

外层会检查这个组合是否完整出现在子查询返回的组合中。

## 2.5.3 用例子理解成对与不成对

假设 `141` 号和 `174` 号员工的数据如下：

| employee_id | manager_id | department_id |
| --- | --- | --- |
| 141 | 100 | 10 |
| 174 | 200 | 20 |

### 不成对比较

不成对比较会形成两个集合：

```text
manager_id 集合：100、200
department_id 集合：10、20
```

所以以下组合都可能通过：

```text
(100, 10)
(100, 20)
(200, 10)
(200, 20)
```

因为它只要求两个字段分别落在各自集合中。

### 成对比较

成对比较只承认原本出现过的组合：

```text
(100, 10)
(200, 20)
```

所以：

```text
(100, 20)
(200, 10)
```

不会通过。

## 2.5.4 成对比较与单行子查询的关系

成对比较不一定都是单行子查询。

如果子查询只返回一组字段，可以写成：

```sql
WHERE (manager_id, department_id) = (
    SELECT manager_id, department_id
    FROM employees
    WHERE employee_id = 141
)
```

这时右边返回 `1` 行多列，是行子查询。

如果子查询返回多组字段，要写成：

```sql
WHERE (manager_id, department_id) IN (
    SELECT manager_id, department_id
    FROM employees
    WHERE employee_id IN (141, 174)
)
```

这时右边返回多行多列组合，已经更接近多行行比较。

所以学习这部分时，要抓住重点：

> 成对比较的关键不是返回几行，而是多个字段是否要作为同一组组合来判断。

## 2.5.5 成对比较与不成对比较总结

| 比较方式 | 判断方式 | 是否要求字段组合保持一致 | 适合场景 |
| --- | --- | --- | --- |
| 不成对比较 | 每个字段分开判断 | 不要求 | 只要各字段分别符合条件即可 |
| 成对比较 | 多个字段作为整体判断 | 要求 | 多字段组合必须完整匹配 |

## 2.6 单行子查询在 `HAVING` 中的用法

单行子查询也可以放在 `HAVING` 中，对分组后的结果继续筛选。

题目：查询最低工资大于 `50` 号部门最低工资的部门编号及其最低工资。

```sql
SELECT department_id, MIN(salary) AS min_salary
FROM employees
GROUP BY department_id
HAVING MIN(salary) > (
    SELECT MIN(salary)
    FROM employees
    WHERE department_id = 50
);
```

这条 SQL 可以拆成两层理解。

子查询：

```sql
SELECT MIN(salary)
FROM employees
WHERE department_id = 50
```

负责算出 `50` 号部门的最低工资。

外层查询：

```sql
SELECT department_id, MIN(salary)
FROM employees
GROUP BY department_id
```

负责算出每个部门的最低工资。

最后：

```sql
HAVING MIN(salary) > 50号部门最低工资
```

用来筛选出最低工资比 `50` 号部门还高的部门。

## 2.6.1 `WHERE` 和 `HAVING` 中子查询的差异

| 位置 | 作用对象 | 典型用途 |
| --- | --- | --- |
| `WHERE` | 分组前的每一笔资料 | 筛选员工、订单、明细资料 |
| `HAVING` | 分组后的结果 | 筛选部门、月份、类别等聚合结果 |

例如：

```sql
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
)
```

是筛选员工。

而：

```sql
HAVING MIN(salary) > (
    SELECT MIN(salary)
    FROM employees
    WHERE department_id = 50
)
```

是筛选分组后的部门。

### 回查提示

如果外层条件中出现聚合函数，例如：

```sql
MIN(salary)
AVG(salary)
COUNT(*)
```

通常就要考虑这个条件是不是应该写在 `HAVING`。

## 2.7 单行子查询在 `SELECT` 和 `CASE` 中的用法

单行子查询不只可以出现在 `WHERE` 和 `HAVING`，也可以出现在 `SELECT` 和 `CASE` 中。

## 2.7.1 在 `SELECT` 中使用单行子查询

例如：查询每位员工的工资，同时显示公司平均工资。

```sql
SELECT last_name,
       salary,
       (
           SELECT AVG(salary)
           FROM employees
       ) AS company_avg_salary
FROM employees;
```

这里的子查询：

```sql
SELECT AVG(salary)
FROM employees
```

只返回一个公司平均工资。

外层每一行员工资料都会显示这个平均工资。

### 适合场景

- 想在结果中显示某个统计值。
- 想让每一行资料都带上同一个参考值。
- 想比较员工工资与公司平均工资时，先把平均工资显示出来观察。

### 注意事项

如果子查询放在 `SELECT` 字段位置，它也必须返回适合当前字段位置使用的结果。

也就是说，通常要返回单一值。

## 2.7.2 在 `CASE` 中使用单行子查询

题目：显示员工的 `employee_id`、`last_name` 和 `location`。如果员工的 `department_id` 与 `location_id = 1800` 对应部门的 `department_id` 相同，则显示 `Canada`，否则显示 `USA`。

可以写成：

```sql
SELECT employee_id,
       last_name,
       CASE department_id
           WHEN (
               SELECT department_id
               FROM departments
               WHERE location_id = 1800
           )
           THEN 'Canada'
           ELSE 'USA'
       END AS location
FROM employees;
```

这里使用的是简单 `CASE` 写法：

```sql
CASE department_id
    WHEN 某个department_id THEN 'Canada'
    ELSE 'USA'
END
```

子查询负责返回 `location_id = 1800` 对应的 `department_id`。

## 2.7.3 `CASE` 子查询的重要限制

上面的写法成立有一个前提：

> `location_id = 1800` 只能查出一个 `department_id`。

如果这个条件会查出多个部门编号，下面这段子查询就会返回多行：

```sql
SELECT department_id
FROM departments
WHERE location_id = 1800
```

这时它不能直接放在：

```sql
CASE department_id
    WHEN (...)
```

因为 `WHEN` 后面期待的是一个单值。

## 2.7.4 如果 `CASE` 子查询可能返回多行，应该改用 `IN`

更安全的写法是使用搜索型 `CASE`：

```sql
SELECT employee_id,
       last_name,
       CASE
           WHEN department_id IN (
               SELECT department_id
               FROM departments
               WHERE location_id = 1800
           )
           THEN 'Canada'
           ELSE 'USA'
       END AS location
FROM employees;
```

这时语意变成：

> 如果员工的 `department_id` 属于 `location_id = 1800` 对应的部门集合，就显示 `Canada`。

这个写法可以处理子查询返回多个部门编号的情况。

## 2.7.5 简单 CASE 与搜索型 CASE 对比

| 写法 | 形式 | 适合场景 |
| --- | --- | --- |
| 简单 CASE | `CASE 字段 WHEN 值 THEN ...` | 和单一值做等值比较 |
| 搜索型 CASE | `CASE WHEN 条件 THEN ...` | 条件较复杂，或要搭配 `IN`、`>`、`EXISTS` 等判断 |

如果子查询一定只返回一个值，两种写法都可能使用。  
如果子查询可能返回多行，通常要改用搜索型 `CASE` 搭配 `IN` 或 `EXISTS`。

## 2.8 子查询不返回任何行与 NULL 问题

单行子查询常见的一个陷阱是：

> SQL 没有报错，但结果为空。

例如：

```sql
SELECT last_name, job_id
FROM employees
WHERE job_id = (
    SELECT job_id
    FROM employees
    WHERE last_name = 'Haas'
);
```

如果内层查询找不到 `last_name = 'Haas'` 的员工，就不会返回任何行。

此时外层条件无法得到实际的 `job_id` 值，所以通常查不到结果。

## 2.8.1 排查方式

遇到这种情况，可以先单独执行子查询：

```sql
SELECT job_id
FROM employees
WHERE last_name = 'Haas';
```

然后检查：

| 检查项目 | 说明 |
| --- | --- |
| 是否有结果 | 没有结果时，外层比较通常无法成立 |
| 是否只有一行 | 多行时不能使用单行比较符 |
| 是否为 `NULL` | `=`、`<>` 等比较遇到 `NULL` 通常不会成立 |
| 是否符合题目条件 | 可能是查询条件写错，或资料本来不存在 |

## 2.8.2 子查询返回 `NULL` 时的处理方式

假设你想查询佣金比例与某位员工相同的员工：

```sql
SELECT last_name, commission_pct
FROM employees
WHERE commission_pct = (
    SELECT commission_pct
    FROM employees
    WHERE employee_id = 178
);
```

如果 `178` 号员工的 `commission_pct` 是 `NULL`，这条 SQL 可能查不到你想要的结果。

因为：

```sql
commission_pct = NULL
```

不会被判断为 `TRUE`。

如果你的题意是「同样没有佣金」，应该写成：

```sql
SELECT last_name, commission_pct
FROM employees
WHERE commission_pct IS NULL;
```

如果你想让外层字段和子查询结果做空值安全比较，在 MySQL 中可以使用：

```sql
SELECT last_name, commission_pct
FROM employees
WHERE commission_pct <=> (
    SELECT commission_pct
    FROM employees
    WHERE employee_id = 178
);
```

不过这要确认题目真的需要比较 `NULL`，不能只是机械替换。

## 2.8.3 空结果与 NULL 的区别

| 情况 | 含义 | 常见结果 |
| --- | --- | --- |
| 子查询没有返回任何行 | 找不到符合条件的资料 | 外层比较通常不成立 |
| 子查询返回一行，但值为 `NULL` | 有资料，但该字段没有值 | 外层用 `=` 比较通常不成立 |
| 子查询返回多行 | 有多笔符合条件资料 | 单行比较符通常报错 |

### 回查提示

看到「SQL 可以执行，但没有查出结果」时，不要马上怀疑外层 SQL。  
先把子查询单独拿出来执行，检查它到底返回几行、几列、是否为 `NULL`。

## 2.9 非法使用子查询

单行子查询最大的前提是：

> 内层必须稳定返回一个单独值。

下面这段 SQL 是典型错误：

```sql
SELECT employee_id, last_name
FROM employees
WHERE salary = (
    SELECT MIN(salary)
    FROM employees
    GROUP BY department_id
);
```

错误原因：

```sql
SELECT MIN(salary)
FROM employees
GROUP BY department_id
```

会针对每个部门各返回一个最低工资。

也就是说，它返回的是：

```text
部门 10 的最低工资
部门 20 的最低工资
部门 30 的最低工资
...
```

这是多行结果。

外层却写：

```sql
salary = (...)
```

所以会报错。

## 2.9.1 修正方式一：如果要匹配任一部门的最低工资，用 `IN`

如果题意是：

> 查询工资等于任一部门最低工资的员工。

可以写成：

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary IN (
    SELECT MIN(salary)
    FROM employees
    GROUP BY department_id
);
```

因为子查询返回的是一组最低工资，所以外层要用 `IN`。

## 2.9.2 修正方式二：如果只要全公司最低工资，去掉 `GROUP BY`

如果题意是：

> 查询公司工资最低的员工。

应该写成：

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary = (
    SELECT MIN(salary)
    FROM employees
);
```

这里子查询只返回全公司一个最低工资。

## 2.9.3 修正方式三：如果要在每个部门内部比较，改用相关子查询

如果题意是：

> 查询每个部门中工资等于本部门最低工资的员工。

可以使用相关子查询：

```sql
SELECT e1.employee_id, e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary = (
    SELECT MIN(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

这里的子查询虽然会随着外层当前员工变化，但对每一位员工来说，内层都会返回该部门的一个最低工资。

所以它仍然可以搭配 `=` 使用。

## 2.9.4 修正方式四：如果要和一组值比较，改用 `ANY` 或 `ALL`

如果题意是：

> 查询工资大于任一部门最低工资的员工。

可以写成：

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary > ANY (
    SELECT MIN(salary)
    FROM employees
    GROUP BY department_id
);
```

如果题意是：

> 查询工资大于所有部门最低工资的员工。

可以写成：

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary > ALL (
    SELECT MIN(salary)
    FROM employees
    GROUP BY department_id
);
```

这部分会在多行子查询章节继续整理。

## 2.10 写题判断流程

遇到单行子查询题，可以按下面步骤判断。

## 2.10.1 第一步：确认外层要查什么

例如：

> 查询工资大于 149 号员工工资的员工。

外层要查：

```sql
SELECT last_name, salary
FROM employees
```

## 2.10.2 第二步：确认内层要先查什么

内层要查：

```text
149 号员工的工资
```

所以写成：

```sql
SELECT salary
FROM employees
WHERE employee_id = 149
```

## 2.10.3 第三步：确认内层是否稳定返回一个值

检查：

| 检查点 | 结果 |
| --- | --- |
| 查询列数 | 1 列，`salary` |
| 查询行数 | 理论上 1 行，因为 `employee_id` 唯一 |
| 是否可能多行 | 不太可能 |
| 是否可能没有资料 | 如果员工不存在，则可能 0 行 |

## 2.10.4 第四步：选择单行比较符

题目说「工资大于」，所以使用：

```sql
salary > (...)
```

完整写法：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE employee_id = 149
);
```

## 2.10.5 第五步：检查异常情况

写完后要检查：

1. 子查询是否只返回一列？
2. 子查询是否最多返回一行？
3. 子查询是否可能返回 `NULL`？
4. 子查询是否可能返回 `0` 行？
5. 如果子查询可能返回多行，是否应该改用 `IN`、`ANY`、`ALL`？
6. 如果多个字段要一起比较，是否应该使用成对比较？
7. 如果题目是分组后的条件，是否应该写在 `HAVING`？

## 2.11 题目语言与写法对照表

| 题目语言 | 子查询结果 | 优先写法 |
| --- | --- | --- |
| 高于某位员工的工资 | 一个工资值 | `salary > (子查询)` |
| 等于公司最低工资 | 一个聚合值 | `salary = (SELECT MIN(...))` |
| 高于公司平均工资 | 一个聚合值 | `salary > (SELECT AVG(...))` |
| 与某员工职位相同 | 一个 `job_id` | `job_id = (子查询)` |
| 某字段属于一组结果 | 多行一列 | `IN` |
| 比一组值中的任意一个大 / 小 | 多行一列 | `> ANY`、`< ANY` |
| 比一组值中的所有值大 / 小 | 多行一列 | `> ALL`、`< ALL` |
| 多个字段必须成组匹配 | 一组字段组合 | `(col1, col2) IN (...)` |
| 分组结果要和单一统计值比较 | 一个统计值 | `HAVING 聚合函数 > (子查询)` |
| 条件分支中比较单一查询结果 | 一个值 | `CASE ... WHEN (子查询)` |
| 条件分支中比较一组查询结果 | 多个值 | `CASE WHEN 字段 IN (子查询)` |

## 常见混淆点

- 单行子查询能用单行比较符的前提，是子查询真的只返回一个值。
- 单行子查询的「单行」指的是内层查询结果，不是最终查询结果。
- `MIN()`、`MAX()`、`AVG()` 这类聚合函数通常会把多行压成一个值。
- `GROUP BY` 很容易让原本看起来像单行子查询的写法变成多行结果。
- 子查询返回 `0` 行时，外层查询不一定报错，但可能查不到结果。
- 子查询返回 `NULL` 时，不能直接用 `=` 当成普通值比较。
- `CASE department_id WHEN (子查询)` 的前提是子查询只能返回一个值。
- 如果 `CASE` 中的子查询可能返回多个值，通常要改成 `CASE WHEN department_id IN (...)`。
- 不成对比较与成对比较的差别，不在字段数量，而在于是否把多个字段当作同一个组合来比较。
- 多行结果不一定是错误，错误在于外层使用了只接受单值的比较方式。

## 一句话抓核心

单行子查询的核心是：内层查询必须稳定提供一个单独值，外层才能用 `=`、`>`、`<` 等单行比较符把它当成普通常量继续判断；如果内层可能返回 `0` 行、`NULL` 或多行，就要重新检查条件、改用空值处理，或改成 `IN`、`ANY`、`ALL`、相关子查询等更合适的写法。
