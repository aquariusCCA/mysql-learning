## 本节导读

这一页不是新语法，而是 **子查询的书写技巧**。

前面几节已经学习了：

- 单行子查询；
- 多行子查询；
- 相关子查询；
- `EXISTS` / `NOT EXISTS`；
- `FROM` 子查询与派生表；
- 自连接与子查询的取舍。

但真正写题时，很多人卡住的地方不是不知道语法，而是不知道：

> 到底该先写里面，还是先写外面？

子查询看起来复杂，是因为一条 SQL 里同时出现了两个问题：

1. 外层最终要查什么？
2. 内层要先提供什么中间结果？

所以写子查询时，重点不是死记「一定先写内层」或「一定先写外层」，而是先判断题目的重心在哪里。

可以先记住一句话：

> 如果题目的关键是先算出一个中间结果，通常从里往外写；  
> 如果题目的最终输出很明确，通常从外往里写；  
> 如果是相关子查询，通常先搭外层，再写内层关联条件。

## 你会在这篇学到什么

- 子查询常见的两种书写顺序：从里往外写、从外往里写。
- 什么情况下适合先写内层查询。
- 什么情况下适合先写外层查询。
- 为什么相关子查询通常更适合先写外层。
- 写子查询前应该先问哪几个问题。
- 如何根据题目语意判断子查询返回一个值、一组值，还是一张临时表。
- 不同子查询类型的起手式。
- 写到一半卡住时，应该怎么排查。
- 写完子查询后，应该检查哪些地方。
- 为什么「书写顺序」不等于「数据库实际执行顺序」。

## 快速定位

- `1. 为什么需要子查询书写技巧`：理解为什么会卡在不知道先写哪一层。
- `2. 写子查询前先问四个问题`：建立通用拆题框架。
- `3. 从里往外写`：适合先算中间结果的题目。
- `4. 从外往里写`：适合最终输出很明确的题目。
- `5. 为什么相关子查询通常从外往里写`：理解外层别名和当前行的重要性。
- `6. 不同子查询类型的起手式`：单行、多行、行、表、相关、`EXISTS` 的写法入口。
- `7. 写题完整流程`：从题目到 SQL 的完整步骤。
- `8. 卡住时怎么排查`：整理常见卡点与解决方法。
- `9. 写完后的检查清单`：检查返回行数、操作符、别名、`NULL`、重复资料等问题。
- `10. 常见误区`：避免把书写顺序、逻辑顺序和实际执行顺序混在一起。

## 关键字

- `从里往外写`：先写子查询，确认中间结果，再把结果嵌回外层查询。
- `从外往里写`：先写主查询目标和外层骨架，再补上内层子查询条件。
- `中间结果`：内层查询先提供给外层使用的值、集合、字段组合或临时表。
- `外层骨架`：外层查询的 `SELECT ... FROM ... WHERE ...` 基本结构。
- `相关子查询`：内层查询依赖外层当前行资料的子查询。
- `外层当前行`：相关子查询中，外层查询正在处理的那一笔资料。
- `关联条件`：内层查询引用外层字段时，用来建立关系的条件。
- `派生表`：`FROM` 子查询产生的临时结果表。
- `书写顺序`：你写 SQL 时的思考顺序，不等于数据库实际执行顺序。
- `逻辑理解顺序`：为了理解 SQL 语意而使用的拆解顺序。
- `执行计划`：数据库优化器实际决定如何执行 SQL 的方式。

## 1. 为什么需要子查询书写技巧

子查询本身并不难，真正难的是：

> 一条 SQL 里同时有内层查询和外层查询，不知道该从哪一层开始写。

例如题目：

> 查询工资高于公司平均工资的员工。

你可能会想：

- 最终要查员工；
- 但条件里的平均工资要先算；
- 平均工资是子查询；
- 员工查询是外层查询。

如果没有拆题方法，就很容易在脑中混成一团。

所以写子查询时，应该先把题目拆成两块：

| 层次 | 要解决的问题 |
| --- | --- |
| 外层查询 | 最终要查什么资料 |
| 内层查询 | 要先提供什么中间结果 |

例如：

| 题目 | 外层查询 | 内层查询 |
| --- | --- | --- |
| 查询工资高于公司平均工资的员工 | 查询员工 | 计算公司平均工资 |
| 查询属于某些部门的员工 | 查询员工 | 查出部门编号集合 |
| 查询高于本部门平均工资的员工 | 查询员工 | 根据当前员工部门计算平均工资 |
| 查询没有订单的客户 | 查询客户 | 检查当前客户是否存在订单 |

只要先拆成这两块，就比较容易决定该从哪一层开始写。

## 2. 写子查询前先问四个问题

看到一道子查询题，不要马上写 SQL。  
建议先问四个问题。

## 2.1 第一个问题：最终要查什么？

这是外层查询的核心。

例如：

> 查询工资高于公司平均工资的员工。

最终要查的是员工：

```sql
SELECT last_name, salary
FROM employees
```

再例如：

> 查询没有员工的部门。

最终要查的是部门：

```sql
SELECT department_id, department_name
FROM departments
```

如果最终要查什么都没确定，就很容易不知道外层该怎么写。

## 2.2 第二个问题：中间要先得到什么？

这是内层查询的核心。

例如：

> 查询工资高于公司平均工资的员工。

中间结果是：

```text
公司平均工资
```

所以内层查询是：

```sql
SELECT AVG(salary)
FROM employees
```

再例如：

> 查询属于 `IT` 或 `HR` 部门的员工。

中间结果是：

```text
IT、HR 对应的部门编号集合
```

所以内层查询是：

```sql
SELECT id
FROM departments
WHERE name IN ('IT', 'HR')
```

## 2.3 第三个问题：中间结果返回几行几列？

这个问题决定你要用哪种子查询写法。

| 中间结果 | 返回形态 | 常见写法 |
| --- | --- | --- |
| 一个值 | `1` 行 `1` 列 | `=、>、<、>=、<=、<>` |
| 一组值 | 多行 `1` 列 | `IN、NOT IN、ANY、ALL` |
| 一组字段组合 | 一行多列或多行多列 | `(col1, col2) IN (...)` |
| 一张临时结果表 | 多行多列 | `FROM (...) AS alias` |
| 是否存在对应资料 | 是否返回行 | `EXISTS` / `NOT EXISTS` |

例如：

```sql
SELECT AVG(salary)
FROM employees
```

返回一个值，所以可以写：

```sql
salary > (...)
```

但是：

```sql
SELECT department_id
FROM departments
WHERE location_id = 1700
```

可能返回多个部门编号，所以应该写：

```sql
department_id IN (...)
```

## 2.4 第四个问题：内层是否依赖外层当前行？

如果子查询可以单独执行，通常是不相关子查询。

例如：

```sql
SELECT AVG(salary)
FROM employees;
```

可以独立执行。

但如果子查询中出现外层别名，例如：

```sql
WHERE e2.department_id = e1.department_id
```

就表示内层查询依赖外层当前行，这是相关子查询。

相关子查询通常要先写外层，因为内层需要引用外层表别名。

## 3. 从里往外写

## 3.1 什么是从里往外写

从里往外写，就是先写内层子查询，确认它能得到正确的中间结果，再把它放回外层查询中。

适合这种思路的题目通常是：

> 题目的关键在于先算出某个中间结果。

例如：

- 先算公司平均工资；
- 先查某位员工的工资；
- 先查某些部门编号；
- 先算每个部门的平均工资；
- 先产生一张统计结果表。

## 3.2 什么时候适合从里往外写

下面这些情况通常适合从里往外写：

| 情况 | 原因 |
| --- | --- |
| 子查询本身比较复杂 | 先确认中间结果，避免外层一起干扰 |
| 要先计算聚合值 | 例如平均工资、最高工资、最低工资 |
| 要先查出一组编号 | 例如部门编号、员工编号、订单编号 |
| 要先产生派生表 | 例如每个部门平均工资表 |
| 题目重点是“先求出某个结果” | 从内层开始比较符合思考顺序 |

## 3.3 从里往外写的步骤

可以按照下面步骤：

1. 先写出内层查询。
2. 单独执行内层查询，确认返回结果。
3. 判断内层结果是一个值、一组值，还是一张表。
4. 根据结果形态选择外层操作符。
5. 把内层查询放进外层 SQL。
6. 最后检查整个 SQL 的语意。

## 3.4 示例一：查询工资高于公司平均工资的员工

题目：

> 查询工资高于公司平均工资的员工。

### 第一步：先写内层查询

中间结果是公司平均工资：

```sql
SELECT AVG(salary)
FROM employees;
```

这个子查询返回 `1` 行 `1` 列。

### 第二步：再写外层查询

最终要查员工：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
);
```

### 为什么适合从里往外写

因为这题的关键是：

> 先知道公司平均工资是多少。

先把平均工资算出来，再嵌回外层条件，会很顺。

## 3.5 示例二：查询属于 `IT` 或 `HR` 部门的员工

题目：

> 查询所有在 `IT` 或 `HR` 部门的员工。

### 第一步：先写内层查询

先查出 `IT` 和 `HR` 的部门编号：

```sql
SELECT id
FROM departments
WHERE name IN ('IT', 'HR');
```

这个子查询可能返回多行一列。

### 第二步：再写外层查询

员工的 `department_id` 要属于这些部门编号：

```sql
SELECT name, department_id
FROM employees
WHERE department_id IN (
    SELECT id
    FROM departments
    WHERE name IN ('IT', 'HR')
);
```

### 为什么这里用 `IN`

因为内层查询返回的是：

```text
一组部门编号
```

所以外层不能写：

```sql
department_id = (...)
```

而应该写：

```sql
department_id IN (...)
```

## 3.6 示例三：先算每个部门平均工资，再查高于本部门平均工资的员工

题目：

> 查询工资高于本部门平均工资的员工。

如果你想用派生表 JOIN，也可以从里往外写。

### 第一步：先写部门平均工资表

```sql
SELECT department_id, AVG(salary) AS avg_salary
FROM employees
GROUP BY department_id;
```

这个结果是一张临时表：

| department_id | avg_salary |
| --- | --- |
| 10 | 5500 |
| 20 | 6200 |
| 30 | 7000 |

### 第二步：把它放到 `FROM` 中

```sql
SELECT e.last_name, e.salary, e.department_id, dept_avg.avg_salary
FROM employees e
JOIN (
    SELECT department_id, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
) AS dept_avg
ON e.department_id = dept_avg.department_id
WHERE e.salary > dept_avg.avg_salary;
```

### 为什么适合从里往外写

因为这题可以拆成：

1. 先得到每个部门平均工资表。
2. 再让员工表和这张表连接。

这时先把派生表写出来，会让整体结构更清楚。

## 4. 从外往里写

## 4.1 什么是从外往里写

从外往里写，就是先写外层查询骨架，确定最终要查什么，再补上子查询作为条件。

适合这种思路的题目通常是：

> 最终要输出什么很明确，只是条件中需要补一个子查询。

例如：

- 查询员工，但条件是高于某个子查询结果；
- 查询部门，但条件是没有对应员工；
- 查询客户，但条件是存在订单；
- 查询分组结果，但 `HAVING` 要和某个子查询比较。

## 4.2 什么时候适合从外往里写

下面这些情况通常适合从外往里写：

| 情况 | 原因 |
| --- | --- |
| 题目已经明确最终要输出哪些列 | 先搭外层骨架最快 |
| 外层筛选对象很明确 | 例如员工、部门、客户 |
| 子查询只是补充条件 | 外层可以先写出来 |
| 相关子查询需要引用外层别名 | 必须先有外层别名 |
| `EXISTS` / `NOT EXISTS` 题型 | 通常先确定外层当前行 |

## 4.3 从外往里写的步骤

可以按照下面步骤：

1. 先写外层 `SELECT ... FROM ...`。
2. 先写出外层要筛选的主对象。
3. 在 `WHERE` 或 `HAVING` 中留下子查询位置。
4. 再补内层查询。
5. 检查内层是否要引用外层字段。
6. 检查子查询返回结果是否符合外层操作符。

## 4.4 示例一：查询工资高于 149 号员工工资的员工

题目：

> 查询工资高于 `149` 号员工工资的员工。

### 第一步：先写外层骨架

最终要查员工：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (...);
```

### 第二步：补上内层查询

中间结果是 `149` 号员工的工资：

```sql
SELECT salary
FROM employees
WHERE employee_id = 149
```

### 第三步：组合完整 SQL

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE employee_id = 149
);
```

### 为什么适合从外往里写

因为题目一开始就明确：

> 查询员工。

外层目标很清楚，只差补上比较基准，所以从外层开始很自然。

## 4.5 示例二：查询最低工资大于 50 号部门最低工资的部门

题目：

> 查询最低工资大于 `50` 号部门最低工资的部门编号及其最低工资。

### 第一步：先写外层分组查询

最终要查每个部门的最低工资：

```sql
SELECT department_id, MIN(salary) AS min_salary
FROM employees
GROUP BY department_id
HAVING MIN(salary) > (...);
```

### 第二步：补上内层查询

中间结果是 `50` 号部门的最低工资：

```sql
SELECT MIN(salary)
FROM employees
WHERE department_id = 50
```

### 第三步：组合完整 SQL

```sql
SELECT department_id, MIN(salary) AS min_salary
FROM employees
GROUP BY department_id
HAVING MIN(salary) > (
    SELECT MIN(salary)
    FROM employees
    WHERE department_id = 50
);
```

### 为什么适合从外往里写

因为题目最终要的是：

```text
部门编号 + 每个部门最低工资
```

外层分组查询很明确，所以可以先写外层，再补 `HAVING` 中的子查询。

## 5. 为什么相关子查询通常从外往里写

相关子查询的特点是：

> 内层查询会引用外层当前行的数据。

因此，如果外层表别名还没定义，内层查询就不知道要引用谁。

例如：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

这里内层查询使用了：

```sql
e1.department_id
```

而 `e1` 是外层定义的表别名。

所以相关子查询通常要先写外层。

## 5.1 相关子查询的书写步骤

可以按下面步骤写：

1. 先确定外层当前行是谁。
2. 给外层表取清楚的别名。
3. 写出外层要返回的字段。
4. 在 `WHERE` 中留下子查询位置。
5. 写内层查询。
6. 在内层写出和外层当前行的关联条件。
7. 检查子查询返回值是否能被外层操作符接住。

## 5.2 示例：查询工资高于本部门平均工资的员工

题目：

> 查询工资高于本部门平均工资的员工。

### 第一步：先确定外层当前行

外层当前行是：

```text
每一位员工
```

所以先写：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (...);
```

### 第二步：补内层查询

内层要算：

```text
当前员工所在部门的平均工资
```

所以写：

```sql
SELECT AVG(e2.salary)
FROM employees e2
WHERE e2.department_id = e1.department_id
```

### 第三步：组合完整 SQL

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

### 这条 SQL 的关键

关键不是 `AVG()`，而是关联条件：

```sql
e2.department_id = e1.department_id
```

它表示：

> 内层查询只计算外层当前员工所在部门的平均工资。

## 5.3 相关子查询写法检查表

| 检查项目 | 说明 |
| --- | --- |
| 外层表是否有别名 | 例如 `employees e1` |
| 内层表是否有别名 | 例如 `employees e2` |
| 内层是否引用外层字段 | 例如 `e1.department_id` |
| 关联条件是否正确 | 例如部门对部门、客户对客户 |
| 子查询返回几行几列 | 决定外层能否用 `=`、`>`、`IN` |
| 是否可以改写成 JOIN | 有些题目派生表 JOIN 更清楚 |

## 6. 不同子查询类型的起手式

不同子查询类型，适合的下手方式不一样。

## 6.1 单行子查询

典型语意：

> 先查出一个值，再拿来比较。

常见起手式：

```sql
WHERE 字段 比较符 (
    SELECT 单一字段或聚合结果
    FROM 表
    WHERE 条件
)
```

示例：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
);
```

### 建议下手方式

如果中间值很关键，适合从里往外写。  
如果外层输出很明确，适合从外往里写。

### 检查重点

- 子查询是否只返回一列？
- 子查询是否最多返回一行？
- 是否可能返回 `NULL`？
- 是否可能返回 `0` 行？
- 是否误用了 `GROUP BY` 导致多行？

## 6.2 多行子查询

典型语意：

> 先查出一组值，再判断外层值是否属于其中，或和其中某些值比较。

常见起手式：

```sql
WHERE 字段 IN (
    SELECT 字段
    FROM 表
    WHERE 条件
)
```

或：

```sql
WHERE 字段 > ANY (
    SELECT 字段
    FROM 表
    WHERE 条件
)
```

或：

```sql
WHERE 字段 > ALL (
    SELECT 字段
    FROM 表
    WHERE 条件
)
```

### 建议下手方式

通常可以先写内层集合，再决定外层用 `IN`、`ANY`、`ALL`。

### 检查重点

- 子查询是否返回多行一列？
- 题目是「其中一个」还是「所有」？
- 是否需要 `IN`、`ANY`、`ALL`？
- 子查询结果是否可能包含 `NULL`？
- 如果使用 `NOT IN`，是否已经处理 `NULL`？

## 6.3 行子查询

典型语意：

> 多个字段要作为一组整体比较。

常见起手式：

```sql
WHERE (col1, col2) IN (
    SELECT col1, col2
    FROM 表
    WHERE 条件
)
```

示例：

```sql
SELECT employee_id, manager_id, department_id
FROM employees
WHERE (manager_id, department_id) IN (
    SELECT manager_id, department_id
    FROM employees
    WHERE employee_id IN (141, 174)
);
```

### 建议下手方式

先判断题目是否要求字段组合保持一致。

### 检查重点

- 是成对比较，还是不成对比较？
- 多个字段是否必须来自同一笔资料？
- 子查询返回的列数是否和外层括号中的列数一致？

## 6.4 表子查询 / 派生表

典型语意：

> 先产生一张临时结果表，再让外层继续连接、筛选或统计。

常见起手式：

```sql
FROM (
    SELECT ...
    FROM ...
    GROUP BY ...
) AS alias
```

示例：

```sql
SELECT e.last_name, e.salary, dept_avg.avg_salary
FROM employees e
JOIN (
    SELECT department_id, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
) AS dept_avg
ON e.department_id = dept_avg.department_id
WHERE e.salary > dept_avg.avg_salary;
```

### 建议下手方式

通常适合从里往外写。

先把派生表写出来，确认它长什么样，再和外层表连接。

### 检查重点

- 派生表是否有别名？
- 派生表字段是否有清楚的别名？
- 外层连接条件是否正确？
- 是否会因为连接造成重复资料？

## 6.5 相关子查询

典型语意：

> 每一笔外层资料，都要根据自己的某个字段去内层查询。

常见起手式：

```sql
SELECT ...
FROM table1 t1
WHERE 条件 (
    SELECT ...
    FROM table2 t2
    WHERE t2.col = t1.col
)
```

### 建议下手方式

通常从外往里写。

因为内层要引用外层当前行，必须先有外层别名。

### 检查重点

- 外层当前行是谁？
- 内层引用了外层哪个字段？
- 关联条件是否正确？
- 子查询返回的是值、集合，还是是否存在？
- 是否可以改写成派生表 JOIN？

## 6.6 `EXISTS` / `NOT EXISTS`

典型语意：

> 判断是否存在对应资料。

常见起手式：

```sql
WHERE EXISTS (
    SELECT 1
    FROM 关联表
    WHERE 关联表.字段 = 外层表.字段
)
```

或：

```sql
WHERE NOT EXISTS (
    SELECT 1
    FROM 关联表
    WHERE 关联表.字段 = 外层表.字段
)
```

示例：

```sql
SELECT c.id, c.name
FROM customers c
WHERE NOT EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

### 建议下手方式

通常从外往里写。

因为你要先确定：

```text
对外层哪一笔资料，检查是否存在对应记录
```

### 检查重点

- 子查询是否有关联外层字段？
- `EXISTS` 是否表达“存在”？
- `NOT EXISTS` 是否表达“不存在”？
- 是否误写成没有关联条件的 `EXISTS`？

## 7. 写题完整流程

遇到一道子查询题，可以按下面流程写。

## 7.1 第一步：圈出最终查询对象

问自己：

> 最后要查出来的是员工、部门、客户，还是统计结果？

例如：

```text
查询工资高于平均工资的员工
```

最终对象是：

```text
员工
```

外层通常就是：

```sql
SELECT ...
FROM employees
```

## 7.2 第二步：圈出中间结果

问自己：

> 条件中需要先查出什么？

例如：

```text
工资高于平均工资
```

中间结果是：

```text
平均工资
```

内层通常是：

```sql
SELECT AVG(salary)
FROM employees
```

## 7.3 第三步：判断中间结果形态

问自己：

> 中间结果是一个值、一组值、一组字段组合，还是一张表？

| 中间结果 | 代表类型 |
| --- | --- |
| 公司平均工资 | 单行子查询 |
| 多个部门编号 | 多行子查询 |
| `(manager_id, department_id)` 组合 | 行子查询 |
| 每个部门平均工资表 | 表子查询 |
| 是否有订单 | `EXISTS` / `NOT EXISTS` |

## 7.4 第四步：选择操作符

| 子查询结果 | 外层常用写法 |
| --- | --- |
| 一个值 | `=、>、<、>=、<=、<>` |
| 一组值 | `IN、NOT IN、ANY、ALL` |
| 一组字段组合 | `(col1, col2) IN (...)` |
| 一张临时表 | `JOIN (子查询) AS alias` |
| 是否存在资料 | `EXISTS` / `NOT EXISTS` |

## 7.5 第五步：选择书写顺序

| 情况 | 建议 |
| --- | --- |
| 中间结果复杂 | 从里往外写 |
| 外层输出明确 | 从外往里写 |
| 相关子查询 | 先写外层 |
| `EXISTS` / `NOT EXISTS` | 先写外层 |
| 派生表很复杂 | 先写派生表 |
| 不确定内层返回什么 | 先单独执行内层 |

## 7.6 第六步：组合 SQL 并检查

组合完成后，检查：

1. 子查询有没有括号。
2. 子查询能不能单独执行。
3. 子查询返回几行几列。
4. 外层操作符是否匹配子查询结果。
5. 字段来源是否清楚。
6. 表别名是否明确。
7. `NOT IN` 是否遇到 `NULL`。
8. `FROM` 子查询是否有别名。
9. 相关子查询是否有关联条件。
10. 是否可能产生重复资料。

## 8. 卡住时怎么排查

## 8.1 卡点一：不知道该写内层还是外层

先问：

```text
我现在比较清楚的是最终要查什么，还是中间要先算什么？
```

如果比较清楚最终要查什么，先写外层。  
如果比较清楚中间要先算什么，先写内层。

## 8.2 卡点二：不知道该用 `=` 还是 `IN`

看子查询返回结果。

| 子查询结果 | 操作符 |
| --- | --- |
| 一个值 | `=`、`>`、`<` 等 |
| 多个值 | `IN`、`ANY`、`ALL` |

如果不确定，就先单独执行子查询，看它返回几行。

## 8.3 卡点三：不知道该用 `ANY` 还是 `ALL`

看题目语意。

| 题目语意 | 写法 |
| --- | --- |
| 至少满足其中一个 | `ANY` |
| 必须满足全部 | `ALL` |

例如：

```sql
salary > ANY (...)
```

表示：

```text
工资大于其中至少一个值
```

```sql
salary > ALL (...)
```

表示：

```text
工资大于所有值
```

## 8.4 卡点四：相关子查询写不出来

先把外层写出来，并给表取别名：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE ...
```

然后问：

```text
内层要根据外层当前行的哪个字段查询？
```

如果是部门，就写：

```sql
WHERE e2.department_id = e1.department_id
```

相关子查询最关键的是：

> 先找出外层当前行，再写内外层关联条件。

## 8.5 卡点五：派生表不知道怎么接外层

先把派生表单独写出来：

```sql
SELECT department_id, AVG(salary) AS avg_salary
FROM employees
GROUP BY department_id
```

然后问：

```text
外层表要用哪个字段和派生表连接？
```

如果是部门编号，就写：

```sql
ON e.department_id = dept_avg.department_id
```

## 8.6 卡点六：SQL 能执行但结果为空

先单独执行子查询，检查：

- 是否返回 `0` 行；
- 是否返回 `NULL`；
- 是否返回多行；
- 是否条件写错；
- 是否 `NOT IN` 遇到 `NULL`；
- 是否相关子查询关联条件写错。

## 9. 写完后的检查清单

写完子查询后，可以用下面清单检查。

## 9.1 结构检查

| 检查项目 | 说明 |
| --- | --- |
| 子查询是否放在括号内 | `(...)` |
| 子查询是否是一条完整 `SELECT` | 能不能单独看懂 |
| 外层查询是否完整 | `SELECT ... FROM ... WHERE ...` |
| 表别名是否清楚 | 尤其是同一张表出现多次 |
| 派生表是否有别名 | `FROM (...) AS alias` |

## 9.2 结果形态检查

| 检查项目 | 说明 |
| --- | --- |
| 子查询返回几列 | 单值比较只能返回一列 |
| 子查询返回几行 | 决定用单行还是多行操作符 |
| 是否可能返回 `0` 行 | 外层可能查不到结果 |
| 是否可能返回 `NULL` | 比较结果可能变成未知 |
| 是否可能返回重复值 | 可能影响连接结果或性能 |

## 9.3 操作符检查

| 子查询结果 | 应搭配 |
| --- | --- |
| `1` 行 `1` 列 | `=、>、<、>=、<=、<>` |
| 多行 `1` 列 | `IN、NOT IN、ANY、ALL` |
| 多字段组合 | `(col1, col2) IN (...)` |
| 是否存在资料 | `EXISTS` / `NOT EXISTS` |
| 临时结果表 | `JOIN (子查询) AS alias` |

## 9.4 相关子查询检查

| 检查项目 | 说明 |
| --- | --- |
| 外层别名是否先定义 | 例如 `employees e1` |
| 内层是否引用外层字段 | 例如 `e1.department_id` |
| 关联条件是否正确 | 部门对部门、客户对客户 |
| 子查询是否真的需要相关 | 能否改成派生表 JOIN |
| 是否误把逻辑执行当成实际执行 | 性能要看 `EXPLAIN` |

## 9.5 风险检查

| 风险 | 处理方式 |
| --- | --- |
| 单行子查询返回多行 | 改用 `IN`、`ANY`、`ALL` 或聚合 |
| `NOT IN` 遇到 `NULL` | 加 `IS NOT NULL` 或改用 `NOT EXISTS` |
| 自连接产生重复资料 | 使用 `DISTINCT` 或重新确认题意 |
| 派生表没有别名 | 加上 `AS alias` |
| 相关子查询没关联外层 | 补上正确关联条件 |
| `UPDATE` / `DELETE` 范围不明 | 先用 `SELECT` 检查 |

## 10. 常见题型起手模板

## 10.1 高于某个统计值

题目：

> 查询工资高于公司平均工资的员工。

模板：

```sql
SELECT ...
FROM employees
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
);
```

起手建议：

> 先写内层 `AVG()`，再写外层比较。

## 10.2 属于某个查询结果集合

题目：

> 查询属于某些部门的员工。

模板：

```sql
SELECT ...
FROM employees
WHERE department_id IN (
    SELECT department_id
    FROM departments
    WHERE ...
);
```

起手建议：

> 先写内层集合，再决定外层用 `IN`。

## 10.3 比一组值中的任意一个大

题目：

> 查询工资高于任意一个部门平均工资的员工。

模板：

```sql
SELECT ...
FROM employees
WHERE salary > ANY (
    SELECT AVG(salary)
    FROM employees
    GROUP BY department_id
);
```

起手建议：

> 先确认题目是“至少一个”，再用 `ANY`。

## 10.4 比一组值中的所有值大

题目：

> 查询工资高于所有部门平均工资的员工。

模板：

```sql
SELECT ...
FROM employees
WHERE salary > ALL (
    SELECT AVG(salary)
    FROM employees
    GROUP BY department_id
);
```

起手建议：

> 先确认题目是“全部”，再用 `ALL`。

## 10.5 和自己所属群组比较

题目：

> 查询工资高于本部门平均工资的员工。

模板：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

起手建议：

> 先写外层当前员工，再写内层关联条件。

## 10.6 判断是否存在对应资料

题目：

> 查询有订单的客户。

模板：

```sql
SELECT c.id, c.name
FROM customers c
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

起手建议：

> 先写外层客户，再写内层检查订单是否存在。

## 10.7 判断是否不存在对应资料

题目：

> 查询没有订单的客户。

模板：

```sql
SELECT c.id, c.name
FROM customers c
WHERE NOT EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

起手建议：

> 先写外层客户，再写 `NOT EXISTS` 关联条件。

## 10.8 先统计再连接

题目：

> 查询工资高于本部门平均工资的员工。

模板：

```sql
SELECT e.last_name, e.salary, e.department_id, dept_avg.avg_salary
FROM employees e
JOIN (
    SELECT department_id, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
) AS dept_avg
ON e.department_id = dept_avg.department_id
WHERE e.salary > dept_avg.avg_salary;
```

起手建议：

> 先写派生表，再和外层表连接。

## 11. 书写顺序与执行顺序的区别

写子查询时要特别注意：

> 书写顺序不等于实际执行顺序。

例如，从里往外写只是你的思考方式：

```text
先写内层，再写外层
```

但数据库实际执行时，MySQL 优化器可能会改写 SQL。

例如：

- 把某些子查询改写成连接；
- 把某些 `IN` 子查询改写成半连接；
- 合并某些派生表；
- 调整连接顺序；
- 根据索引和统计信息选择执行策略。

所以应该区分三个概念：

| 概念 | 说明 |
| --- | --- |
| 书写顺序 | 你写 SQL 时先写哪一块 |
| 逻辑理解顺序 | 为了理解语意而拆解 SQL 的顺序 |
| 实际执行顺序 | 数据库优化器最终采用的执行计划 |

学习阶段重点是：

> 用书写顺序帮助自己写出正确 SQL。

实务优化阶段才需要进一步看：

```sql
EXPLAIN
```

或：

```sql
EXPLAIN ANALYZE
```

## 12. 常见误区

## 12.1 误区一：以为子查询一定要从里面开始写

不一定。

如果题目最终输出很明确，先写外层更顺。

例如：

> 查询工资高于 149 号员工工资的员工。

外层很清楚是查员工，所以可以先写：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (...);
```

再补内层。

## 12.2 误区二：以为相关子查询可以先随便写内层

相关子查询通常不能先随便写内层，因为内层要引用外层别名。

例如：

```sql
WHERE e2.department_id = e1.department_id
```

这里的 `e1` 必须先在外层定义。

所以相关子查询通常先写外层。

## 12.3 误区三：只会写子查询，不会先单独验证内层

如果子查询复杂，建议先单独执行内层。

例如：

```sql
SELECT department_id, AVG(salary) AS avg_salary
FROM employees
GROUP BY department_id;
```

确认结果正确后，再放进外层。

## 12.4 误区四：没有检查子查询返回几行几列

这是最常见错误。

例如：

```sql
WHERE salary = (
    SELECT MIN(salary)
    FROM employees
    GROUP BY department_id
)
```

子查询返回多行，却用 `=` 接，容易报错。

## 12.5 误区五：看到“没有”就直接用 `NOT IN`

题目出现「没有对应资料」时，不一定要用 `NOT IN`。

例如：

> 查询没有订单的客户。

通常更适合：

```sql
NOT EXISTS
```

因为 `NOT IN` 容易受到 `NULL` 影响。

## 12.6 误区六：派生表忘记取别名

错误写法：

```sql
SELECT ...
FROM (
    SELECT ...
)
```

正确写法：

```sql
SELECT ...
FROM (
    SELECT ...
) AS alias
```

`FROM` 子查询通常要有别名，外层才方便引用。

## 12.7 误区七：为了让 SQL 不报错，随便加 `MAX()` 或 `MIN()`

如果子查询返回多行，不要为了让它变成单值就随便加聚合函数。

例如：

```sql
SELECT MAX(salary)
FROM employees
WHERE last_name = 'Abel'
```

这会把语意变成：

> 叫 Abel 的员工中最高工资。

这可能符合题意，也可能不符合。

应该先确认题目真正要的是：

- 某一位 Abel；
- 任意一位 Abel；
- 所有 Abel；
- Abel 的最高工资；
- Abel 的最低工资。

## 13. 快速判断表

| 题目特征 | 建议起手方式 |
| --- | --- |
| 先算一个统计值 | 从里往外写 |
| 先查一组编号 | 从里往外写 |
| 最终输出字段很明确 | 从外往里写 |
| 外层要分组后筛选 | 先写外层分组，再补 `HAVING` 子查询 |
| 子查询要引用外层字段 | 从外往里写 |
| 判断是否存在对应资料 | 先写外层，再写 `EXISTS` |
| 判断是否不存在对应资料 | 先写外层，再写 `NOT EXISTS` |
| 先产生统计结果表 | 先写派生表 |
| 多字段要成组匹配 | 先确认字段组合 |
| 不确定用什么操作符 | 先检查子查询返回几行几列 |

## 14. 本节总结

子查询的书写顺序没有绝对规则。

真正重要的是先判断：

1. 最终要查什么？
2. 中间要先得到什么？
3. 中间结果返回几行几列？
4. 外层要如何使用这个结果？
5. 子查询是否依赖外层当前行？
6. 是否要用 `IN`、`ANY`、`ALL`、`EXISTS`、派生表或相关子查询？
7. 写完后是否检查了返回行数、`NULL`、别名、关联条件和重复资料？

可以先记这三个原则：

| 情况 | 建议 |
| --- | --- |
| 中间结果是题目重点 | 从里往外写 |
| 最终输出是题目重点 | 从外往里写 |
| 内层依赖外层当前行 | 通常先写外层 |

## 常见回查问题

- 子查询到底该先写内层还是外层？
- 什么情况下适合从里往外写？
- 什么情况下适合从外往里写？
- 为什么相关子查询通常先写外层？
- 为什么 `EXISTS` / `NOT EXISTS` 通常先写外层？
- 子查询写到一半卡住时该怎么办？
- 怎么判断子查询返回一个值还是一组值？
- 什么时候用 `=`，什么时候用 `IN`？
- 什么时候用 `ANY`，什么时候用 `ALL`？
- 派生表为什么要先单独写出来？
- 写完子查询后应该检查什么？
- 书写顺序和实际执行顺序有什么差别？

## 一句话抓核心

子查询的编写顺序不是固定规则；关键是先判断题目的重心是在“先算中间结果”，还是“先搭外层筛选骨架”，如果内层需要引用外层当前行，就通常先写外层并取好别名，再回头补内层查询与关联条件。
