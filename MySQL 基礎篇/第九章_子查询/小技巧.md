## 本节导读

这一页不是新语法，而是子查询的书写技巧。重点在于降低写 SQL 时的卡顿感：面对一条子查询，应该先从内层结果开始想，还是先从外层目标结果开始想。

如果题目已经很清楚告诉你最终要查什么，通常可以先从外层写起；如果子查询本身逻辑较复杂、你得先算出一个关键结果，通常更适合先从内层写起。相关子查询因为内层会依赖外层当前行，实务上也常从外层结构先搭起来。

## 你会在这篇学到什么

- 子查询常见的两种书写顺序。
- 什么情况下适合从里往外写。
- 什么情况下适合从外往里写。
- 为什么相关子查询通常更适合先写外层。

## 关键字

- `从里往外写`：先写子查询，再把结果嵌回外层查询。
- `从外往里写`：先写主查询目标，再补上内层子查询条件。
- `相关子查询`：内层查询依赖外层当前行数据的子查询。
- `书写顺序`：SQL 实际执行顺序之外的思考与编写顺序。

## 子查询的编写技巧

写子查询时，常见有两种下手方式：

- 从里往外写。
- 从外往里写。

这两种方法没有绝对对错，重点是看哪一种更容易让你先把问题拆开。

### 什么时候适合从里往外写

如果子查询本身相对复杂，或者题目的关键在于“先算出一个值 / 一组值”，通常适合从里往外写。

例如你可能会先问自己：

- 内层到底要先查出什么？
- 这个结果是单个值、一个集合，还是一张临时结果表？
- 外层接下来要怎么使用这个结果？

这种写法的好处是：先把最难的中间结果算清楚，再决定外层如何比较、筛选或连接。

### 什么时候适合从外往里写

如果题目相对直接，外层查询目标很明确，通常适合从外往里写。

例如题目已经明确要求：

- 最终要输出哪些列；
- 要筛掉哪些记录；
- 条件只是“和某个子查询结果比较”。

这时可以先把外层 `SELECT ... FROM ... WHERE ...` 骨架写出来，再补上内层子查询。

### 为什么相关子查询通常从外往里写

相关子查询的特点是：内层查询会依赖外层当前行的数据。

因此在书写时，通常会先明确外层正在处理哪张表、哪一行、要输出什么，再回头补内层条件，例如：

```sql
SELECT last_name, salary, department_id
FROM   employees e1
WHERE  salary > (
           SELECT AVG(salary)
           FROM   employees e2
           WHERE  e1.department_id = e2.department_id
       );
```

在这类题型里，如果外层别名和目标条件还没先定下来，内层很难写得稳定，所以相关子查询通常更适合从外往里写。

## 快速判断

可以用下面这组简单原则先决定下手方向：

| 情况 | 建议写法 |
| --- | --- |
| 子查询本身较复杂，得先算出中间结果 | 从里往外写 |
| 外层目标很明确，只差补一个子查询条件 | 从外往里写 |
| 相关子查询，需要引用外层当前行 | 通常从外往里写 |

## 常见回查问题

- 子查询到底该先写内层还是外层？
- 为什么有些题目从里往外写比较顺？
- 为什么相关子查询通常先写外层？

## 一句话抓核心

子查询的编写顺序不是固定规则；关键是先判断题目的重心是在“先算中间结果”，还是“先搭外层筛选骨架”，再决定从里往外写还是从外往里写。
