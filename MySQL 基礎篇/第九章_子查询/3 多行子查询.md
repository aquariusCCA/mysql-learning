## 本节导读

上一节整理的是 **单行子查询**，也就是内层查询只返回一个值，外层可以直接用：

```sql
=、>、>=、<、<=、<>
```

来比较。

这一节进入 **多行子查询**。多行子查询的重点是：

> 内层查询返回的是一组值，外层查询必须决定自己要和这组值中的「任意一个」比较，还是要和「全部值」比较。

例如：

```sql
salary IN (...)
salary < ANY (...)
salary < ALL (...)
```

这些写法看起来都在和子查询结果比较，但语意完全不同。

学习这一节时，不要只背操作符名称，而要反复问自己：

1. 子查询返回的是不是一组值？
2. 外层值要不要等于其中某一个？
3. 外层值只要满足其中一个比较就可以，还是必须满足全部比较？
4. 子查询结果中会不会出现 `NULL`？
5. 如果出现 `NULL`，`NOT IN`、`ANY`、`ALL` 的结果会不会和直觉不同？

只要能回答这些问题，多行子查询就不会只是死背语法，而是可以从题目语意推导出来。

- `IN`、`NOT IN`、`ANY`、`SOME`、`ALL` 分别在比较什么。

- `ANY` / `ALL` 如何用 `MIN()`、`MAX()` 的思维辅助理解。

- 多行子查询遇到空集合、`NULL`、重复值时要注意什么。

## 关键字

- `多行子查询`：内层查询返回多行结果的子查询。
- `列子查询`：返回多行 `1` 列结果的子查询，是本节最常讨论的形式。
- `集合比较子查询`：外层值与子查询返回的一组值进行比较。
- `IN`：判断外层值是否等于集合中的任意一个值。
- `NOT IN`：判断外层值是否不等于集合中的所有值，但会受到 `NULL` 影响。
- `ANY`：与比较符搭配使用，只要和集合中的某一个值比较成立即可。
- `SOME`：`ANY` 的同义词，作用相同。
- `ALL`：与比较符搭配使用，必须和集合中的所有值比较都成立。
- `NULL`：空值或未知值，会让比较结果变成未知。
- `空集合`：子查询没有返回任何行，`ANY` 与 `ALL` 会有不同语意。

## 3.1 什么是多行子查询

多行子查询指的是：**内层查询返回多行结果，外层查询要拿某个值与这一组结果进行比较**。

例如：

```sql
SELECT employee_id, last_name, department_id
FROM employees
WHERE department_id IN (
    SELECT department_id
    FROM departments
    WHERE location_id = 1700
);
```

这条 SQL 中，子查询：

```sql
SELECT department_id
FROM departments
WHERE location_id = 1700
```

可能返回多个部门编号：

```text
10
20
30
```

外层查询要判断：

```sql
department_id IN (10, 20, 30)
```

也就是：

> 员工的部门编号是否属于这一组部门编号。

所以，多行子查询的重点不是“里面有几条 SQL”，而是：

> 子查询返回的是一组值，外层必须用集合比较的方式处理。

## 3.1.1 多行子查询和单行子查询的差异

| 比较项目 | 单行子查询 | 多行子查询 |
| --- | --- | --- |
| 内层返回结果 | 一个值 | 一组值 |
| 常见结果形态 | `1` 行 `1` 列 | 多行 `1` 列 |
| 常见操作符 | `=`、`>`、`<`、`>=`、`<=`、`<>` | `IN`、`NOT IN`、`ANY`、`ALL`、`SOME` |
| 外层理解方式 | 和一个值比较 | 和一组值比较 |
| 常见错误 | 子查询意外返回多行 | 忽略 `NULL` 或误用 `ANY` / `ALL` |

单行子查询示例：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
);
```

多行子查询示例：

```sql
SELECT last_name, department_id
FROM employees
WHERE department_id IN (
    SELECT department_id
    FROM departments
    WHERE location_id = 1700
);
```

判断关键：

> 如果子查询返回一个值，用单行比较符；  
> 如果子查询返回一组值，用多行比较操作符。

## 3.1.2 本节主要讨论的是多行一列

严格来说，子查询可能返回：

- `1` 行 `1` 列；
- 多行 `1` 列；
- `1` 行多列；
- 多行多列。

本节重点是：

> 多行 `1` 列的子查询，也就是列子查询。

例如：

```sql
SELECT department_id
FROM departments
WHERE location_id = 1700
```

它返回的是一组部门编号。

如果子查询返回多行多列，通常要考虑：

- 行比较：`(col1, col2) IN (...)`
- 表子查询：`FROM (SELECT ...) AS alias`
- 派生表连接：`JOIN (SELECT ...) AS alias`

这些会和行子查询、表子查询、相关子查询一起理解。

## 3.2 多行比较操作符总览

多行子查询常见操作符如下：

| 操作符 | 基本语意 | 常见写法 |
| --- | --- | --- |
| `IN` | 等于集合中的任意一个值 | `x IN (子查询)` |
| `NOT IN` | 不等于集合中的所有值 | `x NOT IN (子查询)` |
| `ANY` | 只要和集合中的某一个值比较成立即可 | `x > ANY (子查询)` |
| `SOME` | `ANY` 的同义词 | `x > SOME (子查询)` |
| `ALL` | 必须和集合中的所有值比较都成立 | `x > ALL (子查询)` |

可以先这样理解：

```sql
x IN (子查询)
```

表示：

> `x` 是否等于集合中的某一个值。

```sql
x > ANY (子查询)
```

表示：

> `x` 是否大于集合中的至少一个值。

```sql
x > ALL (子查询)
```

表示：

> `x` 是否大于集合中的每一个值。

## 3.2.1 多行比较操作符的语法

`ANY`、`SOME`、`ALL` 通常要和比较符一起使用：

```sql
表达式 比较符 ANY (子查询)
表达式 比较符 SOME (子查询)
表达式 比较符 ALL (子查询)
```

比较符可以是：

```sql
=、<>、>、>=、<、<=
```

例如：

```sql
salary > ANY (
    SELECT salary
    FROM employees
    WHERE job_id = 'IT_PROG'
)
```

表示：

> 当前员工工资只要大于 `IT_PROG` 员工工资集合中的某一个值即可。

## 3.2.2 题目语言与操作符的第一层对应

| 题目语言 | 优先想到 |
| --- | --- |
| 属于其中一个 | `IN` |
| 不属于任何一个 | `NOT IN`，但要注意 `NULL` |
| 等于其中一个 | `IN` 或 `= ANY` |
| 大于其中至少一个 | `> ANY` |
| 小于其中至少一个 | `< ANY` |
| 大于所有值 | `> ALL` |
| 小于所有值 | `< ALL` |
| 至少满足一个 | `ANY` |
| 全部都要满足 | `ALL` |

### 回查提示

看到「其中一个」「至少一个」「任意一个可以成立」时，优先想到 `ANY`。  
看到「所有」「全部」「每一个都要成立」时，优先想到 `ALL`。  
看到「属于某个集合」时，优先想到 `IN`。

## 3.3 `IN` 与 `NOT IN`

## 3.3.1 `IN`：等于集合中的任意一个值

`IN` 用来判断某个值是否等于子查询结果中的任意一个值。

```sql
SELECT last_name, department_id
FROM employees
WHERE department_id IN (
    SELECT department_id
    FROM departments
    WHERE location_id = 1700
);
```

可以理解为：

```sql
department_id = 10
OR department_id = 20
OR department_id = 30
```

只要员工的 `department_id` 等于其中一个，就会被选出来。

### 适合场景

- 查询属于某几个部门的员工。
- 查询订单状态属于某些状态的资料。
- 查询员工编号是否在某个子查询结果中。
- 查询某字段是否落在另一张表查出的集合中。

## 3.3.2 `IN` 和 `= ANY`

在很多情况下：

```sql
x IN (子查询)
```

可以理解成：

```sql
x = ANY (子查询)
```

例如：

```sql
department_id IN (
    SELECT department_id
    FROM departments
    WHERE location_id = 1700
)
```

类似于：

```sql
department_id = ANY (
    SELECT department_id
    FROM departments
    WHERE location_id = 1700
)
```

不过实际写 SQL 时，如果只是要表达「属于集合中的某一个值」，通常直接用 `IN` 更直观。

## 3.3.3 `NOT IN`：不等于集合中的所有值

`NOT IN` 表示：

> 外层值不能等于子查询结果中的任何一个值。

例如：

```sql
SELECT employee_id, last_name
FROM employees
WHERE employee_id NOT IN (
    SELECT manager_id
    FROM employees
);
```

表面上看，这条 SQL 是要查询：

> 不是任何人经理的员工。

但这个写法有一个非常重要的风险：

> 如果子查询结果中包含 `NULL`，`NOT IN` 可能导致整条查询查不到预期结果。

这个问题会在 `3.8 NOT IN 的空值问题` 详细说明。

## 3.3.4 `NOT IN` 和 `<> ALL`

逻辑上可以先这样理解：

```sql
x NOT IN (集合)
```

类似于：

```sql
x <> ALL (集合)
```

也就是：

> `x` 必须不等于集合中的每一个值。

但因为 `NULL` 会让比较结果变成未知，所以 `NOT IN` 和 `<> ALL` 都要特别注意空值问题。

## 3.4 `ANY` 与 `SOME`

## 3.4.1 `ANY` 的核心语意

`ANY` 表示：

> 只要和子查询结果中的某一个值比较成立，整个条件就成立。

例如：

```sql
salary < ANY (
    SELECT salary
    FROM employees
    WHERE job_id = 'IT_PROG'
)
```

意思是：

> 当前员工工资只要小于 `IT_PROG` 员工工资集合中的至少一个值即可。

假设 `IT_PROG` 的工资集合是：

```text
4200
4800
9000
```

那么：

```sql
salary < ANY (4200, 4800, 9000)
```

只要满足下面任何一个即可：

```text
salary < 4200
salary < 4800
salary < 9000
```

所以如果某员工工资是 `5000`：

```text
5000 < 4200  false
5000 < 4800  false
5000 < 9000  true
```

只要有一个 `true`，整体就是 `true`。

## 3.4.2 `SOME` 是 `ANY` 的同义词

在 SQL 中：

```sql
SOME
```

和：

```sql
ANY
```

语意相同。

例如：

```sql
salary < SOME (
    SELECT salary
    FROM employees
    WHERE job_id = 'IT_PROG'
)
```

等同于：

```sql
salary < ANY (
    SELECT salary
    FROM employees
    WHERE job_id = 'IT_PROG'
)
```

实务和学习时，通常记 `ANY` 就够了。  
如果看到 `SOME`，知道它是 `ANY` 的别名即可。

## 3.4.3 `ANY` 常见写法

| 写法 | 语意 |
| --- | --- |
| `x = ANY (集合)` | `x` 等于集合中的至少一个值，类似 `IN` |
| `x > ANY (集合)` | `x` 大于集合中的至少一个值 |
| `x >= ANY (集合)` | `x` 大于等于集合中的至少一个值 |
| `x < ANY (集合)` | `x` 小于集合中的至少一个值 |
| `x <= ANY (集合)` | `x` 小于等于集合中的至少一个值 |
| `x <> ANY (集合)` | `x` 不等于集合中的至少一个值，通常不等于 `NOT IN` |

## 3.4.4 特别注意：`<> ANY` 通常不是你想的 `NOT IN`

很多人会误以为：

```sql
x <> ANY (1, 2, 3)
```

等于：

```sql
x NOT IN (1, 2, 3)
```

这是错误的。

因为：

```sql
x <> ANY (1, 2, 3)
```

表示：

> 只要 `x` 不等于其中至少一个值，就成立。

假设：

```text
x = 1
```

那么：

```text
1 <> 1 false
1 <> 2 true
1 <> 3 true
```

只要有一个 `true`，`<> ANY` 就成立。

所以 `x = 1` 时，`x <> ANY (1, 2, 3)` 仍然是 `true`。

如果你要表达：

> `x` 不等于集合中的任何一个值。

应该使用：

```sql
x NOT IN (...)
```

或者：

```sql
x <> ALL (...)
```

并且要注意 `NULL` 问题。

## 3.5 `ALL`

## 3.5.1 `ALL` 的核心语意

`ALL` 表示：

> 必须和子查询结果中的所有值比较都成立，整个条件才成立。

例如：

```sql
salary < ALL (
    SELECT salary
    FROM employees
    WHERE job_id = 'IT_PROG'
)
```

意思是：

> 当前员工工资必须小于所有 `IT_PROG` 员工的工资。

假设 `IT_PROG` 的工资集合是：

```text
4200
4800
9000
```

那么：

```sql
salary < ALL (4200, 4800, 9000)
```

表示必须同时满足：

```text
salary < 4200
salary < 4800
salary < 9000
```

也就是说，工资必须小于这组值中的最小值 `4200`。

## 3.5.2 `ALL` 常见写法

| 写法 | 语意 |
| --- | --- |
| `x = ALL (集合)` | `x` 必须等于集合中的所有值，比较少用 |
| `x > ALL (集合)` | `x` 大于集合中的所有值 |
| `x >= ALL (集合)` | `x` 大于等于集合中的所有值 |
| `x < ALL (集合)` | `x` 小于集合中的所有值 |
| `x <= ALL (集合)` | `x` 小于等于集合中的所有值 |
| `x <> ALL (集合)` | `x` 不等于集合中的所有值，类似 `NOT IN`，但同样要注意 `NULL` |

## 3.5.3 `ALL` 通常比 `ANY` 更严格

比较：

```sql
salary < ANY (...)
```

和：

```sql
salary < ALL (...)
```

差异如下：

| 写法 | 成立条件 | 严格程度 |
| --- | --- | --- |
| `salary < ANY (...)` | 只要小于其中一个值即可 | 较宽松 |
| `salary < ALL (...)` | 必须小于每一个值 | 较严格 |

如果集合是：

```text
4200
4800
9000
```

那么：

```sql
salary < ANY (...)
```

等价于：

```text
salary < 9000
```

而：

```sql
salary < ALL (...)
```

等价于：

```text
salary < 4200
```

所以 `ALL` 通常会筛出更少资料。

## 3.6 `ANY` / `ALL` 快速换算表

在子查询结果没有 `NULL`、而且集合不为空的情况下，可以先用 `MIN()` 和 `MAX()` 辅助理解 `ANY` / `ALL`。

假设集合是：

```text
10, 20, 30
```

| 写法 | 可以先理解成 | 原因 |
| --- | --- | --- |
| `x > ANY (集合)` | `x > MIN(集合)` | 只要大于最小值，就至少大于其中一个 |
| `x >= ANY (集合)` | `x >= MIN(集合)` | 只要大于等于最小值，就至少满足一个 |
| `x < ANY (集合)` | `x < MAX(集合)` | 只要小于最大值，就至少小于其中一个 |
| `x <= ANY (集合)` | `x <= MAX(集合)` | 只要小于等于最大值，就至少满足一个 |
| `x > ALL (集合)` | `x > MAX(集合)` | 必须大于所有值，就是大于最大值 |
| `x >= ALL (集合)` | `x >= MAX(集合)` | 必须大于等于所有值，就是大于等于最大值 |
| `x < ALL (集合)` | `x < MIN(集合)` | 必须小于所有值，就是小于最小值 |
| `x <= ALL (集合)` | `x <= MIN(集合)` | 必须小于等于所有值，就是小于等于最小值 |

### 重要提醒

这张表是帮助理解用的，不是所有情况下都能机械替换。

因为实际 SQL 中还要考虑：

- 子查询结果是否包含 `NULL`；
- 子查询是否返回空集合；
- 是否有重复值；
- 优化器是否会改写查询；
- 题目是否强调集合比较语意，而不是单纯极值比较。

学习阶段可以先用这张表帮助判断方向，但实务上仍要确认资料情况与题目语意。

## 3.7 典型代码示例

## 3.7.1 题型一：比 `IT_PROG` 中至少一个工资低

题目：返回其它 `job_id` 中，比 `job_id = 'IT_PROG'` 的员工里至少一个工资低的员工的 `employee_id`、`last_name`、`job_id` 和 `salary`。

```sql
SELECT employee_id, last_name, job_id, salary
FROM employees
WHERE job_id <> 'IT_PROG'
AND salary < ANY (
    SELECT salary
    FROM employees
    WHERE job_id = 'IT_PROG'
);
```

这里的子查询：

```sql
SELECT salary
FROM employees
WHERE job_id = 'IT_PROG'
```

会返回所有 `IT_PROG` 员工的工资集合。

外层条件：

```sql
salary < ANY (...)
```

表示：

> 外层员工的工资只要小于 `IT_PROG` 工资集合中的至少一个值即可。

如果 `IT_PROG` 工资集合是：

```text
4200
4800
9000
```

那么：

```sql
salary < ANY (...)
```

可以先理解成：

```sql
salary < 9000
```

因为只要小于最大值，就一定小于集合中的至少一个值。

### 语意提醒

中文题目中「任意一个」有时容易造成歧义。

- 如果题意是「小于其中至少一个」，用 `ANY`。
- 如果题意是「小于每一个，也就是比所有人都低」，用 `ALL`。

所以做题时不要只看「任意」两个字，要看题目真正想表达的是「至少一个」还是「全部」。

## 3.7.2 题型二：比 `IT_PROG` 中所有工资都低

题目：返回其它 `job_id` 中，比 `job_id = 'IT_PROG'` 的员工里所有工资都低的员工的 `employee_id`、`last_name`、`job_id` 和 `salary`。

```sql
SELECT employee_id, last_name, job_id, salary
FROM employees
WHERE job_id <> 'IT_PROG'
AND salary < ALL (
    SELECT salary
    FROM employees
    WHERE job_id = 'IT_PROG'
);
```

这里的重点是：

```sql
salary < ALL (...)
```

表示：

> 外层员工的工资必须小于每一位 `IT_PROG` 员工的工资。

如果 `IT_PROG` 工资集合是：

```text
4200
4800
9000
```

那么：

```sql
salary < ALL (...)
```

可以先理解成：

```sql
salary < 4200
```

也就是必须小于这组工资中的最小值。

### `ANY` 与 `ALL` 对比

| 写法 | 语意 | 如果集合是 `4200, 4800, 9000` |
| --- | --- | --- |
| `salary < ANY (...)` | 小于至少一个值 | 类似 `salary < 9000` |
| `salary < ALL (...)` | 小于所有值 | 类似 `salary < 4200` |

这两个条件只差一个关键字，但结果可能差很多。

## 3.7.3 题型三：查询平均工资最低的部门 ID

题目：查询平均工资最低的部门 ID。

### 写法一：两层聚合 + 最小值比较

可以先计算每个部门的平均工资，再从中找出最小平均工资。

```sql
SELECT department_id
FROM employees
GROUP BY department_id
HAVING AVG(salary) = (
    SELECT MIN(avg_sal)
    FROM (
        SELECT AVG(salary) AS avg_sal
        FROM employees
        GROUP BY department_id
    ) AS dept_avg_sal
);
```

这条 SQL 的逻辑是：

1. 内层派生表先计算每个部门的平均工资。
2. 外层子查询从这些平均工资中找出最小值。
3. 最外层 `HAVING` 找出平均工资等于这个最小值的部门。

### 写法二：使用 `ALL`

也可以改写成：

```sql
SELECT department_id
FROM employees
GROUP BY department_id
HAVING AVG(salary) <= ALL (
    SELECT AVG(salary)
    FROM employees
    GROUP BY department_id
);
```

这里的逻辑是：

> 当前部门的平均工资必须小于或等于所有部门的平均工资。

也就是说，当前部门平均工资必须是最低的。

### 为什么用 `<= ALL`

如果写成：

```sql
AVG(salary) < ALL (...)
```

会有问题。

因为子查询结果中包含当前部门自己的平均工资。  
当前部门平均工资不可能小于自己，所以应该用：

```sql
<= ALL
```

表示：

> 小于或等于所有部门平均工资。

这样才能保留平均工资最低的部门。

## 3.7.4 题型四：查询工资高于任一部门平均工资的员工

如果题目是：

> 查询工资高于任一部门平均工资的员工。

可以写成：

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary > ANY (
    SELECT AVG(salary)
    FROM employees
    GROUP BY department_id
);
```

意思是：

> 员工工资只要高于某一个部门的平均工资即可。

如果用快速换算辅助理解，在没有 `NULL` 且集合不为空时：

```sql
salary > ANY (...)
```

类似于：

```sql
salary > 最低的部门平均工资
```

## 3.7.5 题型五：查询工资高于所有部门平均工资的员工

如果题目是：

> 查询工资高于所有部门平均工资的员工。

可以写成：

```sql
SELECT employee_id, last_name, salary
FROM employees
WHERE salary > ALL (
    SELECT AVG(salary)
    FROM employees
    GROUP BY department_id
);
```

意思是：

> 员工工资必须高于每一个部门的平均工资。

在没有 `NULL` 且集合不为空时：

```sql
salary > ALL (...)
```

类似于：

```sql
salary > 最高的部门平均工资
```

这比 `> ANY` 严格很多。

## 3.8 `NOT IN` 的空值问题

多行子查询中最容易踩雷的地方之一，是：

```sql
NOT IN
```

搭配子查询结果中包含 `NULL`。

例如：

```sql
SELECT last_name
FROM employees
WHERE employee_id NOT IN (
    SELECT manager_id
    FROM employees
);
```

这条 SQL 表面上看是在查询：

> 不是任何人经理的员工。

但如果子查询：

```sql
SELECT manager_id
FROM employees;
```

返回结果中包含 `NULL`，外层查询可能得不到预期结果。

## 3.8.1 为什么 `NOT IN` 遇到 `NULL` 会出问题

假设子查询结果是：

```text
100
101
NULL
```

那么：

```sql
employee_id NOT IN (100, 101, NULL)
```

可以理解成：

```sql
employee_id <> 100
AND employee_id <> 101
AND employee_id <> NULL
```

前两个比较可能是真或假，但：

```sql
employee_id <> NULL
```

结果不是 `TRUE`，而是未知。

在 SQL 的三值逻辑中，`WHERE` 只会保留判断结果为 `TRUE` 的资料。  
如果条件结果是 `FALSE` 或未知，都不会被保留。

所以只要 `NOT IN` 的集合中出现 `NULL`，就可能让结果和直觉不同，甚至查不到资料。

## 3.8.2 修正方式一：在子查询中排除 `NULL`

如果仍然想使用 `NOT IN`，可以先排除 `NULL`：

```sql
SELECT employee_id, last_name
FROM employees
WHERE employee_id NOT IN (
    SELECT manager_id
    FROM employees
    WHERE manager_id IS NOT NULL
);
```

这样子查询结果中不会包含 `NULL`，`NOT IN` 的判断就比较符合预期。

## 3.8.3 修正方式二：改用 `NOT EXISTS`

也可以改用 `NOT EXISTS`：

```sql
SELECT e.employee_id, e.last_name
FROM employees e
WHERE NOT EXISTS (
    SELECT 1
    FROM employees m
    WHERE m.manager_id = e.employee_id
);
```

这条 SQL 的语意是：

> 对外层每一位员工，检查是否存在其他员工的 `manager_id` 等于他的 `employee_id`。  
> 如果不存在，就表示他不是任何人的经理。

`NOT EXISTS` 判断的是：

> 子查询有没有返回行。

它不像 `NOT IN` 那样直接把外层值和包含 `NULL` 的集合逐一比较，因此通常更适合处理这类「找不存在对应资料」的问题。

## 3.8.4 `NOT IN` 与 `NOT EXISTS` 对比

| 写法 | 判断方式 | 是否容易受 `NULL` 影响 | 适合场景 |
| --- | --- | --- | --- |
| `NOT IN` | 外层值不属于子查询结果集合 | 容易受影响 | 子查询结果确定不会有 `NULL` |
| `NOT EXISTS` | 不存在对应资料 | 较不容易受集合中 `NULL` 影响 | 找没有对应记录的资料 |

### 回查提示

看到：

```sql
NOT IN (子查询)
```

要立刻检查：

```sql
子查询返回结果中会不会包含 NULL？
```

如果可能包含 `NULL`，优先考虑：

1. 在子查询中加上 `IS NOT NULL`。
2. 改写成 `NOT EXISTS`。

## 3.9 多行子查询的结果情况

多行子查询不只要关心返回几行，也要关心里面有没有 `NULL`、有没有空集合。

## 3.9.1 返回正常多行值

例如：

```text
4200
4800
9000
```

这时 `IN`、`ANY`、`ALL` 都可以按集合比较来理解。

```sql
salary IN (...)
salary < ANY (...)
salary < ALL (...)
```

## 3.9.2 返回重复值

例如：

```text
4200
4200
4800
9000
```

通常不影响 `IN`、`ANY`、`ALL` 的语意判断。

因为集合比较关心的是值是否满足条件，不是重复出现几次。

不过从性能与可读性角度，有时可以使用：

```sql
SELECT DISTINCT ...
```

减少重复值。

例如：

```sql
SELECT employee_id, last_name
FROM employees
WHERE department_id IN (
    SELECT DISTINCT department_id
    FROM departments
    WHERE location_id = 1700
);
```

是否需要 `DISTINCT` 要看实际资料与执行计划，不是每次都必须加。

## 3.9.3 返回空集合

如果子查询没有返回任何行，就是空集合。

例如：

```sql
SELECT salary
FROM employees
WHERE job_id = 'NOT_EXIST_JOB';
```

如果没有这个职位，结果是空集合。

在空集合下：

| 写法 | 结果倾向 | 理解方式 |
| --- | --- | --- |
| `x IN (空集合)` | `FALSE` | `x` 不可能属于空集合 |
| `x = ANY (空集合)` | `FALSE` | 没有任何一个值可比较成立 |
| `x > ANY (空集合)` | `FALSE` | 没有任何一个值可比较成立 |
| `x < ANY (空集合)` | `FALSE` | 没有任何一个值可比较成立 |
| `x > ALL (空集合)` | `TRUE` | 没有任何值违反条件 |
| `x < ALL (空集合)` | `TRUE` | 没有任何值违反条件 |
| `x NOT IN (空集合)` | `TRUE` | `x` 确实不属于空集合 |

### 重要提醒

`ALL` 遇到空集合时容易让初学者意外。  
因为「必须满足全部值」在空集合中没有任何值可以反驳它，所以条件可能被视为成立。

因此，使用 `ALL` 时，如果子查询可能为空，要特别确认这是否符合题目本意。

## 3.9.4 返回包含 `NULL`

如果子查询结果包含 `NULL`，比较会变得复杂。

例如：

```text
4200
4800
NULL
```

可能影响：

- `IN`
- `NOT IN`
- `ANY`
- `ALL`

尤其是：

```sql
NOT IN
```

最容易出问题。

学习阶段可以先记：

> 子查询结果可能包含 `NULL` 时，不要机械使用 `NOT IN`；先排除 `NULL`，或改用 `NOT EXISTS`。

## 3.10 写题判断流程

遇到多行子查询题，可以按下面步骤判断。

## 3.10.1 第一步：确认外层要查什么

例如：

> 查询其它职位中，比 `IT_PROG` 任一员工工资低的员工。

外层要查：

```sql
SELECT employee_id, last_name, job_id, salary
FROM employees
WHERE job_id <> 'IT_PROG'
```

## 3.10.2 第二步：确认内层要返回什么集合

内层要查：

> 所有 `IT_PROG` 员工的工资。

```sql
SELECT salary
FROM employees
WHERE job_id = 'IT_PROG'
```

这个子查询会返回一组工资。

## 3.10.3 第三步：判断题目是“其中一个”还是“全部”

如果题目是：

> 比其中至少一个工资低。

用：

```sql
salary < ANY (...)
```

如果题目是：

> 比所有工资都低。

用：

```sql
salary < ALL (...)
```

## 3.10.4 第四步：检查是否有 `NULL`

如果比较字段可能包含 `NULL`，要检查是否需要过滤：

```sql
WHERE salary IS NOT NULL
```

例如：

```sql
SELECT salary
FROM employees
WHERE job_id = 'IT_PROG'
AND salary IS NOT NULL
```

对于 `salary` 这种通常不应为 `NULL` 的字段，风险较小。  
但对于 `manager_id`、`commission_pct` 这类可能为 `NULL` 的字段，就要特别小心。

## 3.10.5 第五步：确认是否可以改写成聚合

有时 `ANY` / `ALL` 可以改写成 `MIN()` / `MAX()`，让语意更直观。

例如：

```sql
salary < ANY (
    SELECT salary
    FROM employees
    WHERE job_id = 'IT_PROG'
)
```

在没有 `NULL` 且集合不为空时，类似于：

```sql
salary < (
    SELECT MAX(salary)
    FROM employees
    WHERE job_id = 'IT_PROG'
)
```

而：

```sql
salary < ALL (
    SELECT salary
    FROM employees
    WHERE job_id = 'IT_PROG'
)
```

类似于：

```sql
salary < (
    SELECT MIN(salary)
    FROM employees
    WHERE job_id = 'IT_PROG'
)
```

但是是否要改写，要看题目目标、可读性和执行计划。

## 3.11 题目语言与写法对照表

| 题目语言 | 子查询结果 | 优先写法 |
| --- | --- | --- |
| 属于某几个部门 | 多个部门编号 | `IN` |
| 不属于某几个部门 | 多个部门编号 | `NOT IN`，但检查 `NULL` |
| 等于其中一个值 | 一组值 | `IN` 或 `= ANY` |
| 大于其中至少一个值 | 一组值 | `> ANY` |
| 小于其中至少一个值 | 一组值 | `< ANY` |
| 大于所有值 | 一组值 | `> ALL` |
| 小于所有值 | 一组值 | `< ALL` |
| 大于等于所有值 | 一组值 | `>= ALL` |
| 小于等于所有值 | 一组值 | `<= ALL` |
| 找没有对应资料的记录 | 是否存在对应行 | `NOT EXISTS` 通常比 `NOT IN` 安全 |
| 找存在对应资料的记录 | 是否存在对应行 | `EXISTS` 或 `IN` |

## 3.12 常见误区

## 3.12.1 误区一：以为多行子查询可以直接用 `=`

错误写法：

```sql
SELECT employee_id, last_name
FROM employees
WHERE salary = (
    SELECT salary
    FROM employees
    WHERE department_id = 50
);
```

如果 `50` 号部门有多位员工，子查询会返回多笔工资。

这时应该根据题意改成：

```sql
salary IN (...)
```

或：

```sql
salary > ANY (...)
salary > ALL (...)
```

或者先用聚合函数压成单值。

## 3.12.2 误区二：混淆 `ANY` 和 `ALL`

```sql
salary < ANY (...)
```

表示：

> 小于至少一个值。

```sql
salary < ALL (...)
```

表示：

> 小于所有值。

两者结果差异很大，不能只靠中文「任意」两个字判断。

## 3.12.3 误区三：以为 `<> ANY` 等于 `NOT IN`

```sql
x <> ANY (1, 2, 3)
```

表示：

> `x` 只要不等于其中至少一个值即可。

这通常很容易成立。

如果要表达：

> `x` 不等于任何一个值。

应该使用：

```sql
x NOT IN (...)
```

或者：

```sql
x <> ALL (...)
```

并且注意 `NULL`。

## 3.12.4 误区四：忽略 `NOT IN` 中的 `NULL`

这是多行子查询最常见陷阱。

如果子查询结果中可能有 `NULL`，不要直接写：

```sql
NOT IN (子查询)
```

应先考虑：

```sql
WHERE 子查询字段 IS NOT NULL
```

或改用：

```sql
NOT EXISTS
```

## 3.12.5 误区五：忘记 `ANY` / `ALL` 要和比较符一起使用

`ANY` 和 `ALL` 不是单独使用的操作符，通常要搭配：

```sql
=、<>、>、>=、<、<=
```

例如：

```sql
salary > ANY (...)
salary < ALL (...)
```

而不是只写：

```sql
salary ANY (...)
```

## 3.12.6 误区六：没有检查空集合

如果子查询可能返回空集合，`ANY` 和 `ALL` 的结果可能和直觉不同。

尤其是：

```sql
x > ALL (空集合)
```

可能成立。

所以如果业务上不希望空集合导致条件成立，就要额外检查子查询是否有资料，或改写逻辑。

## 3.13 本节总结

多行子查询的核心不是操作符名称，而是集合比较思维。

看到多行子查询时，先问：

1. 子查询是不是返回多行一列？
2. 外层值是要等于其中一个，还是和其中一个比较？
3. 条件是至少一个成立即可，还是所有比较都要成立？
4. 子查询结果是否可能出现 `NULL`？
5. 子查询是否可能为空集合？
6. `NOT IN` 是否应该改成排除 `NULL` 或 `NOT EXISTS`？

只要这几个问题想清楚，`IN`、`ANY`、`ALL`、`SOME` 的选择就会稳定很多。

## 常见混淆点

- `IN` 更像“是否属于这个集合”。
- `ANY` 和 `ALL` 一定要搭配比较符一起理解。
- `SOME` 和 `ANY` 没有本质差别，`SOME` 只是别名。
- `ANY` 表示“至少满足一个”。
- `ALL` 表示“必须满足全部”。
- `x = ANY (...)` 通常类似 `x IN (...)`。
- `x <> ALL (...)` 通常类似 `x NOT IN (...)`，但要注意 `NULL`。
- `x <> ANY (...)` 通常不是 `NOT IN`。
- `NOT IN` 一旦碰到 `NULL`，结果常会和直觉不同。
- `ANY` 遇到空集合通常不成立。
- `ALL` 遇到空集合可能成立。
- `ANY` / `ALL` 可以用 `MIN()` / `MAX()` 辅助理解，但不能忽略 `NULL` 与空集合。

## 常见回查问题

- 什么是多行子查询？
- 多行子查询为什么也叫集合比较子查询？
- `IN`、`ANY`、`ALL`、`SOME` 到底差在哪里？
- `SOME` 和 `ANY` 有什么区别？
- “其中一个”应该用 `IN` 还是 `ANY`？
- “至少一个满足”为什么用 `ANY`？
- “所有值都满足”为什么用 `ALL`？
- `salary < ANY (...)` 和 `salary < ALL (...)` 差在哪里？
- `ANY` / `ALL` 如何用 `MIN()`、`MAX()` 辅助理解？
- `NOT IN` 为什么遇到 `NULL` 会查不到资料？
- `NOT IN` 应该怎么改写才安全？
- `NOT EXISTS` 和 `NOT IN` 什么时候该优先选哪个？
- 子查询返回空集合时，`ANY` 和 `ALL` 会怎样？

## 一句话抓核心

多行子查询的核心是：内层查询先返回一组值，外层查询再决定自己是要判断是否属于这个集合，还是要和集合中的至少一个值比较，或必须和集合中的所有值比较；写 SQL 时重点不是背 `IN`、`ANY`、`ALL` 的名称，而是判断题目语意属于“其中一个成立”还是“全部都要成立”，并特别注意 `NOT IN`、`NULL` 与空集合带来的陷阱。
