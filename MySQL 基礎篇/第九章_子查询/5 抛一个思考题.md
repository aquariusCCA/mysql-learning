## 本节导读

前面几节已经分别学习了：

- 单行子查询；
- 多行子查询；
- 相关子查询；
- `EXISTS` 与 `NOT EXISTS`；
- `FROM` 子查询与派生表。

这一页不再补充新语法，而是用一道简单题目训练一个更重要的能力：

> 同一个查询需求，通常不只一种 SQL 写法。  
> 会写只是第一步，真正重要的是知道为什么这样写，以及什么时候该换另一种写法。

本节题目是：

> 谁的工资比 `Abel` 高？

这道题可以写成：

- 分步查询；
- 自连接；
- 单行子查询；
- 多行子查询；
- 聚合子查询；
- 甚至在某些情况下改成 `ANY` / `ALL`。

所以学习这页时，不要只背“这题可以用自连接或子查询”，而要进一步思考：

1. `Abel` 是否只有一位？
2. 查询结果是否可能重复？
3. 题目说的是“比某一位 Abel 高”，还是“比所有叫 Abel 的人都高”？
4. 哪一种写法更贴近题意？
5. 哪一种写法更容易维护？
6. 如果要比较性能，应该看 SQL 外观，还是看执行计划？

## 你会在这篇学到什么

- 同一个查询需求为什么可能有多种 SQL 写法。
- 如何用分步查询拆解问题。
- 如何用自连接表达同一张表中两笔记录的比较。
- 如何用子查询表达“先求一个值，再拿这个值比较”。
- 为什么单行子查询要求 `Abel` 对应的结果必须是单一值。
- 如果 `Abel` 不唯一，应该如何改写 SQL。
- 自连接、子查询、`ANY`、`ALL` 在语意上的差异。
- 为什么不能简单认为“自连接一定比子查询快”。
- 实务中如何从可读性、维护性、资料唯一性、执行计划来选择写法。

## 快速定位

- `5.3 方式二：自连接`：把同一张表当成两份资料来比较。

- `5.9 常见误区`：整理这类题目最容易想错的地方。

## 关键字

- `自连接`：同一张表自己和自己连接，用来比较同表中不同记录。
- `子查询`：嵌套在另一个 SQL 语句中的查询。
- `单行子查询`：返回 `1` 行 `1` 列结果，可以当成单一值比较。
- `多行子查询`：返回多行结果，需要搭配 `IN`、`ANY`、`ALL` 等操作符。
- `ANY`：只要和子查询结果中的某一个值比较成立即可。
- `ALL`：必须和子查询结果中的所有值比较都成立。
- `资料唯一性`：查询条件是否保证只找到一笔资料。
- `重复资料`：连接查询时，因一对多关系导致外层结果重复出现。
- `执行计划`：数据库优化器实际决定如何执行 SQL 的计划。

## 5.1 题目理解

题目：

> 谁的工资比 `Abel` 高？

可以先拆成两层：

| 层次 | 问题 |
| --- | --- |
| 中间结果 | `Abel` 的工资是多少？ |
| 最终结果 | 哪些员工的工资比这个工资高？ |

也就是说，这道题的核心不是单纯查员工，而是：

> 外层查询的比较条件，需要先从资料表中查出来。

自然语言可以转成：

```text
先查 Abel 的工资
再查工资大于 Abel 工资的员工
```

这正是子查询适合表达的典型需求。

不过这题也可以从另一个角度理解：

> 同一张员工表中，拿 Abel 这笔员工资料，去和其他员工资料比较工资。

如果这样想，自连接也很自然。

所以这题的重点不是“哪一种写法唯一正确”，而是：

> 同一个需求，可以因为思考角度不同，而产生不同 SQL 写法。

## 5.2 方式一：分步查询

最直观的方式是分两步。

第一步，先查出 `Abel` 的工资：

```sql
SELECT salary
FROM employees
WHERE last_name = 'Abel';
```

假设查到：

```text
11000
```

第二步，再查询工资高于 `11000` 的员工：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > 11000;
```

## 5.2.1 分步查询的意义

分步查询通常不是最终要放进系统中的写法，但它很适合用来分析题目。

它可以帮助你看清楚：

1. 中间结果是什么；
2. 外层查询要拿这个结果做什么；
3. 子查询应该放在哪个位置；
4. 子查询应该返回一个值还是一组值。

## 5.2.2 分步查询的优缺点

| 项目 | 说明 |
| --- | --- |
| 优点 | 思路最直观，适合刚开始拆题 |
| 优点 | 可以先验证中间结果是否正确 |
| 缺点 | 需要人工或程序把第一步结果带入第二步 |
| 缺点 | 如果 Abel 的工资变动，第二条 SQL 中写死的数值就不可靠 |
| 缺点 | 不适合真实系统中需要自动执行的查询 |

所以分步查询适合用来理解问题，但如果要写成一条完整 SQL，通常会进一步改成自连接或子查询。

## 5.3 方式二：自连接

自连接写法：

```sql
SELECT e2.last_name, e2.salary
FROM employees e1
JOIN employees e2
ON e2.salary > e1.salary
WHERE e1.last_name = 'Abel';
```

这条 SQL 把同一张 `employees` 表当成两份资料：

| 表别名 | 角色 |
| --- | --- |
| `e1` | 代表 `Abel` 那一笔员工资料 |
| `e2` | 代表要被筛选出来的其他员工资料 |

连接条件：

```sql
e2.salary > e1.salary
```

表示：

> 找出工资比 `Abel` 高的员工。

筛选条件：

```sql
WHERE e1.last_name = 'Abel'
```

表示：

> `e1` 这份资料中，只拿 `Abel` 当作比较基准。

## 5.3.1 自连接的思考方式

自连接的思考方式是：

> 同一张表中，有一笔资料作为比较基准，另一批资料拿来和它比较。

也就是：

```text
employees e1：基准员工 Abel
employees e2：被比较的员工
```

这种写法很适合表达：

- 同表中员工之间比较；
- 同表中商品价格比较；
- 同表中订单日期比较；
- 同表中上下级关系；
- 同表中前后版本比较。

## 5.3.2 自连接写法的优点

自连接的优点是：

- 可以清楚表达同一张表中两批资料的关系。
- 当题目本质是“记录与记录之间互相比对”时很自然。
- 可以同时取出基准资料和被比较资料。
- 对某些复杂比较场景，连接写法比嵌套查询更容易扩展。

例如，如果你想同时显示 `Abel` 的工资和其他员工的工资，可以写：

```sql
SELECT e1.last_name AS base_employee,
       e1.salary AS base_salary,
       e2.last_name AS higher_employee,
       e2.salary AS higher_salary
FROM employees e1
JOIN employees e2
ON e2.salary > e1.salary
WHERE e1.last_name = 'Abel';
```

这样结果中可以清楚看到比较双方。

## 5.3.3 自连接写法的注意事项

### 1. 表别名一定要清楚

因为同一张表出现两次，所以必须清楚区分：

```sql
employees e1
employees e2
```

否则很容易搞不清楚哪个字段来自哪一份资料。

### 2. 如果 `Abel` 有多位，结果可能重复或语意改变

如果有多位员工的 `last_name` 都是 `Abel`，那么：

```sql
WHERE e1.last_name = 'Abel'
```

会匹配多笔基准资料。

此时 `e2` 中某些员工可能同时比多位 Abel 工资高，结果就可能重复。

如果只想去除重复员工，可以加：

```sql
SELECT DISTINCT e2.last_name, e2.salary
```

但更重要的是先确认题意：

> 是比哪一位 Abel 高？  
> 还是比所有叫 Abel 的员工都高？  
> 还是比任意一位叫 Abel 的员工高？

这三种语意对应的 SQL 不一样。

### 3. 自连接不等于一定比较快

自连接是一种表达方式，不代表它一定比子查询快。

是否更快要看：

- MySQL 版本；
- 优化器改写能力；
- 表资料量；
- 索引设计；
- 查询条件；
- 执行计划。

不能只凭 SQL 长相判断性能。

## 5.4 方式三：单行子查询

子查询写法：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE last_name = 'Abel'
);
```

这条 SQL 的思路是：

1. 子查询先找出 `Abel` 的工资。
2. 外层查询找出工资高于这个值的员工。

也就是把分步查询合并成一条 SQL。

## 5.4.1 子查询的思考方式

子查询的思考方式是：

> 先求出一个中间结果，再让外层查询使用这个中间结果。

在这题中，中间结果是：

```sql
SELECT salary
FROM employees
WHERE last_name = 'Abel'
```

外层使用这个结果：

```sql
WHERE salary > Abel的工资
```

所以完整语意是：

> 查询工资高于 Abel 工资的员工。

这和题目本身非常接近。

## 5.4.2 子查询写法的优点

子查询写法的优点是：

- 很贴近“先查 Abel 工资，再比较”的自然语言。
- 不需要把同一张表拆成两个别名。
- 对简单的“先求单一值，再比较”题型非常直观。
- 初学者容易从需求一步一步推导出来。

## 5.4.3 子查询写法的前提

这条 SQL 有一个重要前提：

> 子查询必须只返回一个工资值。

也就是说：

```sql
SELECT salary
FROM employees
WHERE last_name = 'Abel'
```

必须返回 `1` 行 `1` 列。

如果 `last_name = 'Abel'` 查到多位员工，子查询会返回多行。

外层却使用：

```sql
salary > (...)
```

这是单行比较符，不能接收多行结果。

所以这条 SQL 在资料不唯一时可能报错。

## 5.4.4 更严谨的单行子查询写法

如果题目其实是指某一位员工，最好使用唯一字段，例如 `employee_id`：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE employee_id = 174
);
```

这样子查询比较能稳定返回一个值。

如果一定要用 `last_name = 'Abel'`，就要确认资料表中 `last_name` 是否唯一。  
通常姓氏不适合当唯一条件。

## 5.5 如果 `Abel` 不唯一怎么办

这是这题最值得思考的地方。

如果资料表中有多位叫 `Abel` 的员工，那么题目：

> 谁的工资比 Abel 高？

就会变得不够精确。

你必须先确认题意。

## 5.5.1 语意一：比某一位 Abel 高

如果题目其实指的是某一位特定员工，就应该用唯一字段：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE employee_id = 174
);
```

这是最标准的单行子查询。

## 5.5.2 语意二：比任意一位 Abel 高

如果题意是：

> 工资比至少一位叫 Abel 的员工高。

可以使用 `ANY`：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > ANY (
    SELECT salary
    FROM employees
    WHERE last_name = 'Abel'
);
```

这表示：

> 只要工资大于其中某一位 Abel 的工资即可。

在没有 `NULL` 且集合不为空的情况下，可以先理解成：

```sql
salary > (
    SELECT MIN(salary)
    FROM employees
    WHERE last_name = 'Abel'
);
```

因为只要大于最低工资的 Abel，就至少大于一位 Abel。

## 5.5.3 语意三：比所有 Abel 都高

如果题意是：

> 工资必须比所有叫 Abel 的员工都高。

可以使用 `ALL`：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > ALL (
    SELECT salary
    FROM employees
    WHERE last_name = 'Abel'
);
```

这表示：

> 工资要大于每一位 Abel 的工资。

在没有 `NULL` 且集合不为空的情况下，可以先理解成：

```sql
salary > (
    SELECT MAX(salary)
    FROM employees
    WHERE last_name = 'Abel'
);
```

因为只要大于最高工资的 Abel，就一定大于所有 Abel。

## 5.5.4 语意四：比 Abel 的最高工资高

如果题目明确说：

> 比叫 Abel 的员工中最高工资还高。

可以直接使用聚合函数：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT MAX(salary)
    FROM employees
    WHERE last_name = 'Abel'
);
```

这会让子查询稳定返回一个值。

## 5.5.5 语意五：比 Abel 的最低工资高

如果题目明确说：

> 比叫 Abel 的员工中最低工资还高。

可以写：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT MIN(salary)
    FROM employees
    WHERE last_name = 'Abel'
);
```

这个语意比较接近：

```sql
salary > ANY (...)
```

但使用 `MIN()` 会让比较基准更明确。

## 5.5.6 不同语意对照表

| 题目真正语意 | 建议写法 |
| --- | --- |
| 比某一位特定 Abel 高 | 使用唯一字段，如 `employee_id` |
| 比至少一位 Abel 高 | `> ANY (子查询)` |
| 比所有 Abel 高 | `> ALL (子查询)` |
| 比 Abel 中最高工资高 | `> (SELECT MAX(salary) ...)` |
| 比 Abel 中最低工资高 | `> (SELECT MIN(salary) ...)` |

### 关键提醒

当查询条件不是唯一字段时，不要急着写单行子查询。

要先问：

> 子查询会不会返回多行？  
> 如果会，题目到底要 `ANY`、`ALL`，还是聚合成一个值？

## 5.6 自连接与子查询对比

同一题可以写成自连接，也可以写成子查询。

两者没有绝对优劣，重点是看题目语意和后续维护。

## 5.6.1 从思考方式比较

| 写法 | 思考方式 |
| --- | --- |
| 自连接 | 同一张表中，两批资料互相比对 |
| 子查询 | 先查出一个中间结果，再让外层使用 |

如果你脑中想的是：

> 先查 Abel 工资，再找更高工资的人。

子查询更直观。

如果你脑中想的是：

> 员工表中，拿 Abel 这笔记录和其他员工记录比较。

自连接也很自然。

## 5.6.2 从可读性比较

| 情况 | 可能更适合 |
| --- | --- |
| 子查询只返回一个简单值 | 子查询 |
| 需要同时显示比较基准和被比较对象 | 自连接 |
| 同表之间有上下级、前后版本、互相关系 | 自连接 |
| 题目语意是“是否高于某个查询结果” | 子查询 |
| 需要处理多位 Abel 的集合语意 | `ANY` / `ALL` / 聚合子查询 |

## 5.6.3 从重复资料风险比较

自连接在一对多情况下容易产生重复资料。

例如：

```sql
SELECT e2.last_name, e2.salary
FROM employees e1
JOIN employees e2
ON e2.salary > e1.salary
WHERE e1.last_name = 'Abel';
```

如果 `e1` 中有多位 Abel，或者一位 `e2` 员工符合多个 Abel 的比较条件，结果可能重复。

此时可以使用：

```sql
SELECT DISTINCT e2.last_name, e2.salary
```

但更根本的做法是先确认题意。

子查询如果写成单行比较，则不会重复，但如果子查询返回多行会报错。

所以两者的风险不同：

| 写法 | 主要风险 |
| --- | --- |
| 自连接 | 可能产生重复结果 |
| 单行子查询 | 子查询返回多行时报错 |
| 多行子查询 | 要正确选择 `ANY`、`ALL`、`IN` |
| 聚合子查询 | 要确认聚合语意是否符合题意 |

## 5.6.4 从维护性比较

维护性通常取决于：

- 题目语意是否清楚；
- SQL 是否容易读懂；
- 条件是否稳定；
- 是否能避免隐藏假设。

例如下面这条 SQL 有隐藏假设：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE last_name = 'Abel'
);
```

隐藏假设是：

> `last_name = 'Abel'` 只会查到一笔资料。

如果这个假设不成立，SQL 就有问题。

更容易维护的写法可能是：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE employee_id = 174
);
```

因为 `employee_id` 更能表达唯一性。

## 5.7 性能判断：不要只看 SQL 外观

课程或经验中常会听到：

> 自连接可能比子查询快。

这句话不能当成绝对规则。

更准确的说法是：

> 某些数据库、某些版本、某些资料量、某些索引条件下，自连接可能更容易被优化；但实际性能必须看执行计划。

## 5.7.1 为什么不能说自连接一定比较快

因为现代 MySQL 优化器可能会改写查询。

例如：

- 某些子查询可能被改写成连接。
- 某些 `IN` 子查询可能被优化成半连接。
- 某些派生表可能被合并。
- 某些相关子查询可能有特殊优化。
- 索引设计可能让原本较复杂的 SQL 反而执行得很好。

所以不能只看 SQL 长相就判断：

```text
自连接一定快
子查询一定慢
```

这是不严谨的。

## 5.7.2 实务上怎么判断性能

如果真的要比较性能，应该看：

```sql
EXPLAIN
```

或：

```sql
EXPLAIN ANALYZE
```

观察重点包括：

- 使用了哪些索引；
- 扫描了多少行；
- 表连接顺序；
- 子查询是否被优化器改写；
- 是否出现临时表；
- 是否出现 filesort；
- 实际执行时间；
- 估算行数是否合理。

学习阶段可以先关注 SQL 语意和可读性。  
等进入优化阶段，再用执行计划判断实际效率。

## 5.7.3 学习阶段的优先顺序

建议学习阶段先按这个顺序：

1. 先写出语意正确的 SQL。
2. 再让 SQL 变得清楚、容易读。
3. 再检查是否有多行、`NULL`、重复资料的问题。
4. 最后才讨论性能。
5. 真要讨论性能时，用执行计划说话。

不要一开始就陷入“哪种一定最快”。

## 5.8 写法选择流程

遇到类似题目时，可以按下面流程判断。

## 5.8.1 第一步：确认比较基准是否唯一

题目：

> 谁的工资比 Abel 高？

先问：

```text
Abel 是不是唯一？
```

如果是某一位员工，最好用：

```sql
employee_id
```

如果不是唯一，就不能直接写单行子查询。

## 5.8.2 第二步：确认题目语意

如果 `Abel` 不唯一，要继续问：

| 题意 | 写法 |
| --- | --- |
| 比某一位 Abel 高 | 用唯一字段 |
| 比任意一位 Abel 高 | `> ANY` |
| 比所有 Abel 高 | `> ALL` |
| 比 Abel 最高工资高 | `> MAX()` |
| 比 Abel 最低工资高 | `> MIN()` |

## 5.8.3 第三步：确认是否需要显示比较基准

如果只需要显示工资较高的员工：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (...);
```

子查询通常足够。

如果需要同时显示：

- Abel 的工资；
- 其他员工的工资；
- 两者差额；

自连接可能更适合：

```sql
SELECT e1.last_name AS base_employee,
       e1.salary AS base_salary,
       e2.last_name AS higher_employee,
       e2.salary AS higher_salary,
       e2.salary - e1.salary AS salary_diff
FROM employees e1
JOIN employees e2
ON e2.salary > e1.salary
WHERE e1.last_name = 'Abel';
```

## 5.8.4 第四步：确认是否会有重复资料

如果使用自连接，要检查：

- 基准资料是否多笔；
- 被比较资料是否会重复出现；
- 是否需要 `DISTINCT`；
- 是否应该先限定唯一基准。

## 5.8.5 第五步：必要时看执行计划

当资料量很大或查询很慢时，不要只凭感觉改 SQL。

应该比较不同写法的执行计划：

```sql
EXPLAIN
SELECT ...
```

再根据实际情况优化。

## 5.9 常见误区

## 5.9.1 误区一：以为同一题只有一种正确写法

SQL 通常有多种写法。

同一个需求可能可以用：

- 子查询；
- 自连接；
- 派生表；
- `EXISTS`；
- `IN`；
- `ANY` / `ALL`。

重点是语意是否正确、是否容易维护、是否符合资料特性。

## 5.9.2 误区二：直接用 `last_name = 'Abel'` 当成唯一条件

`last_name` 通常不是唯一字段。

如果它查到多笔资料，单行子查询可能会报错，自连接可能会产生重复或不同语意。

更严谨的做法是使用唯一字段，例如：

```sql
employee_id
```

## 5.9.3 误区三：子查询返回多行时随便加 `MAX()` 或 `MIN()`

如果子查询返回多行，不应该为了让 SQL 不报错就随便加聚合函数。

例如：

```sql
SELECT MAX(salary)
FROM employees
WHERE last_name = 'Abel'
```

这会把语意变成：

> 叫 Abel 的员工中最高工资。

这可能符合题意，也可能不符合。

加聚合函数前要先确认业务语意。

## 5.9.4 误区四：混淆 `ANY` 和 `ALL`

```sql
salary > ANY (...)
```

表示：

> 工资大于其中至少一个值。

```sql
salary > ALL (...)
```

表示：

> 工资大于所有值。

两者差异很大。

不要只看中文「任意」或「所有」两个字，要确认题目真正要的是“至少一个成立”还是“全部都要成立”。

## 5.9.5 误区五：以为自连接一定比子查询快

这不是绝对规则。

实务上要看：

- MySQL 版本；
- 优化器；
- 索引；
- 资料量；
- 执行计划。

如果没有执行计划，不应该直接下结论。

## 5.9.6 误区六：只关注性能，忽略可读性

一条 SQL 如果语意不清楚，即使能执行，也可能增加后续维护成本。

学习和实务中都应该先保证：

1. 语意正确；
2. 写法清楚；
3. 结果稳定；
4. 再讨论性能。

## 5.10 本节总结

这道题看起来只是：

> 谁的工资比 Abel 高？

但它背后训练的是 SQL 写法选择能力。

你应该学到的是：

1. 分步查询适合拆解问题。
2. 子查询适合表达“先求一个值，再拿来比较”。
3. 自连接适合表达“同一张表中不同记录互相比对”。
4. 如果比较基准不唯一，不能直接当成单行子查询。
5. `ANY`、`ALL`、`MAX()`、`MIN()` 都能处理多笔 Abel，但语意不同。
6. 自连接可能产生重复资料，子查询可能因为多行结果报错。
7. 性能不能只看 SQL 外观，要看执行计划。
8. 实务中应该先写出语意清楚、容易维护的 SQL，再根据执行计划优化。

## 写法选择对照表

| 需求语意 | 推荐写法 |
| --- | --- |
| 比某一位特定员工工资高 | 单行子查询，使用唯一字段 |
| 比某个查询结果高 | 单行子查询 |
| 比任意一个 Abel 高 | `> ANY` |
| 比所有 Abel 高 | `> ALL` |
| 比 Abel 最高工资高 | `> MAX()` |
| 比 Abel 最低工资高 | `> MIN()` |
| 同表中两笔记录互相比对 | 自连接 |
| 需要显示比较双方资料 | 自连接 |
| 自连接出现重复结果 | 检查基准资料是否唯一，必要时用 `DISTINCT` |
| 需要判断性能 | 使用 `EXPLAIN` / `EXPLAIN ANALYZE` |

## 常见回查问题

- 为什么同一个问题可以同时用自连接和子查询解决？
- “谁的工资比 Abel 高”这题的中间结果是什么？
- 自连接中的 `e1` 和 `e2` 分别代表什么？
- 子查询写法为什么更贴近“先查 Abel 工资，再比较”？
- 如果 `Abel` 不唯一，单行子查询会发生什么？
- 什么时候该用 `ANY`？
- 什么时候该用 `ALL`？
- 什么时候该用 `MAX()` 或 `MIN()`？
- 自连接为什么可能产生重复资料？
- 自连接一定比子查询快吗？
- 判断性能时为什么要看执行计划？
- 学习阶段应该先关注语法表达，还是先关注性能？

## 一句话抓核心

同一个查询问题往往不只一种写法；学习时要先确认题目语意和资料唯一性，再从子查询、自连接、`ANY`、`ALL`、聚合函数等写法中选择最能表达需求、最容易维护的方案；如果要比较性能，不能只看 SQL 外观，而要回到具体资料、索引和执行计划判断。
