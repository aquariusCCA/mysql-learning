# 5 抛一个思考题

> 所属章节：[第九章_子查询](./README.md)

> 建议回查情境：想比较自连接和子查询在同一题里的写法差异，或想建立“语法能写出来”与“实务中怎么选”之间的判断时

## 本节导读

这一页不是补新语法，而是用一个简单问题反过来训练思考：同一个需求，既可以写成自连接，也可以写成子查询，那么实际写 SQL 时该怎么判断。

重点不是背结论，而是要看到两件事：第一，同一个问题通常不只一种写法；第二，语法上能成立，不代表实务上就是唯一或最优选择。学习这一页时，可以先自己尝试写两种版本，再回头比较它们的表达方式与适用场景。

## 你会在这篇学到什么

- 同一个查询需求为什么可能有多种写法。
- 自连接与子查询如何分别表达同一个问题。
- 写得出来和写得合适之间，有哪些需要额外思考的点。
- 为什么不能把“哪种写法更好”当成绝对结论。

## 题目

谁的工资比 `Abel` 高？

## 两种常见写法

### 方式一：自连接

```sql
SELECT e2.last_name, e2.salary
FROM   employees e1,
       employees e2
WHERE  e1.last_name = 'Abel'
AND    e1.salary < e2.salary;
```

这条 SQL 的思路是：

- 先在 `employees e1` 中找到 `Abel`。
- 再把 `employees e2` 中工资比 `Abel` 高的员工筛出来。

也就是把同一张表当成两份数据来比较，因此称为自连接。

### 方式二：子查询

```sql
SELECT last_name, salary
FROM   employees
WHERE  salary > (
           SELECT salary
           FROM   employees
           WHERE  last_name = 'Abel'
       );
```

这条 SQL 的思路是：

- 先通过子查询找出 `Abel` 的工资。
- 再在外层查询中找出工资高于这个值的员工。

这种写法更接近“先求一个值，再拿这个值比较”的思维方式。

## 该怎么判断哪种更合适

这类题目常见的第一反应会是：哪一种方式更好？

可以先分开看：

- 从表达问题的角度，自连接和子查询都能正确解决这道题。
- 从可读性角度，如果你脑中先想到的是“先查 Abel 的工资”，子查询往往更直观。
- 从连接关系角度，如果你把它看成“同一张表中两笔记录互相比大小”，自连接也很自然。

## 常见经验与实务提醒

课程里常见的经验是：这类题目也可以改写成自连接，而某些数据库或某些执行计划下，自连接可能更容易被优化。

但这里不要把它记成绝对规则：

- 不能直接把“自连接一定比子查询快”当成固定结论。
- 实际执行效率会受到 DBMS、优化器、索引、数据量与改写方式影响。
- 在实务中，判断性能通常还是要看执行计划，而不是只看 SQL 外观。

更稳妥的学习方式是：

1. 先确认自己能写出两种版本。
2. 再比较哪一种更贴近题意、更容易维护。
3. 如果真的要讨论性能，再回到具体数据库环境和执行计划判断。

## 常见回查问题

- 为什么同一个问题可以同时用自连接和子查询解决？
- 这类题目什么时候子查询更直观？
- 自连接是否一定比子查询快？
- 学习阶段应该先关注语法表达，还是先关注性能？

## 一句话抓核心

同一个查询问题往往不只一种写法；学习时先建立多种表达能力，再用可读性、维护性和执行计划去判断哪一种更适合当前场景。
