## 本节导读

前面已经学习过单行子查询和多行子查询：

- 单行子查询重点在于：内层查询返回一个值。
- 多行子查询重点在于：内层查询返回一组值。

这一节要学习的是 **相关子查询**。

相关子查询的重点不在于它返回一行还是多行，而在于：

> 子查询是否依赖外层查询当前行的数据。

如果子查询中引用了外层查询的字段，那么子查询就不能独立执行。它必须随着外层查询当前处理的那一行资料，动态计算或判断结果。

例如：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

这条 SQL 中，子查询引用了外层的：

```sql
e1.department_id
```

因此它是相关子查询。

学习相关子查询时，不要只背：

> 相关子查询会重复执行。

更重要的是看懂：

1. 子查询引用了外层哪一个字段？
2. 外层每一行会把什么值传给内层？
3. 内层根据这个值计算或判断什么？
4. 外层再如何使用内层结果？

只要能回答这四个问题，相关子查询就会从“很绕”变成“逐行判断”。

## 关键字

- `相关子查询`：子查询引用了外层查询当前行的数据，因此依赖外层查询的子查询。
- `不相关子查询`：子查询可以独立执行，不需要外层当前行资料。
- `外层查询`：驱动整条 SQL 主体流程的查询。
- `内层查询`：嵌套在外层查询内部，并可能引用外层字段的查询。
- `外层字段引用`：子查询内部使用了外层表别名或外层字段，例如 `e2.department_id = e1.department_id`。
- `EXISTS`：只要子查询返回至少一行，就判定为 `TRUE`。
- `NOT EXISTS`：只有当子查询完全没有返回行时，才判定为 `TRUE`。
- `SELECT 1`：在 `EXISTS` 子查询中常见，表示只关心是否有资料，不关心具体返回值。
- `派生表`：`FROM` 子查询产生的临时结果表。
- `半连接`：优化器可能把某些 `EXISTS` 或 `IN` 查询改写成类似连接的执行方式。
- `反连接`：优化器可能把某些 `NOT EXISTS` 或反向匹配查询改写成用于排除对应资料的执行方式。

## 4.1 什么是相关子查询

相关子查询指的是：

> 子查询中引用了外层查询当前行的字段，因此子查询必须依赖外层查询当前行才能计算结果。

例如：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

这条 SQL 中，内层子查询是：

```sql
SELECT AVG(e2.salary)
FROM employees e2
WHERE e2.department_id = e1.department_id
```

其中：

```sql
e1.department_id
```

来自外层查询：

```sql
FROM employees e1
```

所以内层查询必须依赖外层当前员工的部门编号。

这就是相关子查询。

## 4.1.1 不相关子查询对比

不相关子查询可以单独执行。

例如：

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
);
```

子查询：

```sql
SELECT AVG(salary)
FROM employees
```

可以单独执行，因为它不需要外层任何字段。

相关子查询则不同。

例如：

```sql
SELECT AVG(e2.salary)
FROM employees e2
WHERE e2.department_id = e1.department_id;
```

这段子查询如果单独执行，会发现：

```sql
e1.department_id
```

没有来源。

因为 `e1` 是外层查询的表别名。  
所以这段子查询必须放在外层查询中才有意义。

## 4.1.2 相关子查询的核心判断

判断一段 SQL 是否是相关子查询，可以问：

> 子查询内部有没有使用外层查询的表别名或字段？

如果有，就是相关子查询。

例如：

```sql
WHERE e2.department_id = e1.department_id
```

假设：

- `e1` 定义在外层；
- `e2` 定义在内层；

那么子查询内部引用 `e1.department_id`，就代表它依赖外层当前行。

## 4.2 相关子查询的判断方法

看到一段子查询，可以按下面步骤判断。

## 4.2.1 第一步：找出外层查询表别名

例如：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

外层查询是：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
```

外层表别名是：

```sql
e1
```

## 4.2.2 第二步：检查子查询是否使用外层别名

内层查询是：

```sql
SELECT AVG(e2.salary)
FROM employees e2
WHERE e2.department_id = e1.department_id
```

这里出现了：

```sql
e1.department_id
```

因此内层查询引用了外层字段。

## 4.2.3 第三步：判断关联点

这条 SQL 的关联点是：

```sql
e2.department_id = e1.department_id
```

意思是：

> 内层只统计和外层当前员工同一个部门的员工工资。

所以，相关子查询真正要看的不是“里面有没有子查询”，而是：

> 内层查询如何通过外层当前行的字段建立关联条件。

## 4.2.4 相关子查询判断表

| 判断问题 | 如果答案是 yes |
| --- | --- |
| 子查询中是否出现外层表别名？ | 很可能是相关子查询 |
| 子查询中是否引用外层字段？ | 是相关子查询 |
| 子查询能否单独执行？ | 不能单独执行时，通常是相关子查询 |
| 子查询结果是否会随外层当前行改变？ | 是相关子查询 |
| 子查询只是计算固定值？ | 通常不是相关子查询 |

## 4.3 相关子查询的逻辑执行流程

相关子查询可以先用“逐行判断模型”理解。

以这条 SQL 为例：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

逻辑上可以理解为：

1. 外层查询先取出一位员工。
2. 取得这位员工的 `department_id`。
3. 内层查询根据这个 `department_id`，计算该部门平均工资。
4. 外层比较这位员工的工资是否大于该部门平均工资。
5. 如果大于，就保留这位员工。
6. 外层继续处理下一位员工，内层再根据下一位员工的部门重新计算。

## 4.3.1 用表格理解逐行判断

假设外层当前员工如下：

| 当前员工 | salary | department_id |
| --- | --- | --- |
| Alice | 6000 | 10 |

子查询就会变成：

```sql
SELECT AVG(e2.salary)
FROM employees e2
WHERE e2.department_id = 10;
```

如果部门 `10` 的平均工资是 `5500`，外层判断：

```sql
6000 > 5500
```

成立，所以 Alice 被保留。

下一位员工：

| 当前员工 | salary | department_id |
| --- | --- | --- |
| Bob | 4800 | 20 |

子查询就会变成：

```sql
SELECT AVG(e2.salary)
FROM employees e2
WHERE e2.department_id = 20;
```

如果部门 `20` 的平均工资是 `5200`，外层判断：

```sql
4800 > 5200
```

不成立，所以 Bob 不会被保留。

## 4.3.2 重要提醒：逐行执行是逻辑理解，不一定是最终执行计划

学习时，可以先把相关子查询理解成：

> 外层每处理一行，内层就跟着当前行重新计算一次。

这个模型非常适合帮助理解语意。

但要注意：

> 这不代表 MySQL 实际执行计划一定真的逐行重复执行子查询。

实际运行时，MySQL 优化器可能根据 SQL 写法、索引、统计信息和资料量，把某些相关子查询改写成连接、半连接、派生表或其他执行方式。

所以要分清楚：

| 角度 | 说明 |
| --- | --- |
| 逻辑理解 | 外层逐行，内层依赖当前行 |
| 实际执行 | 由 MySQL 优化器决定，可能被改写 |
| 性能判断 | 需要看 `EXPLAIN` 执行计划 |

学习阶段先用逐行模型理解；实务调优时再看执行计划。

## 4.4 经典题型：工资高于本部门平均工资的员工

题目：

> 查询员工中工资大于本部门平均工资的员工 `last_name`、`salary` 和 `department_id`。

这个题目非常适合用相关子查询，因为每一位员工都要和“自己部门”的平均工资比较。

## 4.4.1 方式一：相关子查询

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

### SQL 拆解

外层查询：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
```

代表：

> 逐一取出员工资料。

内层查询：

```sql
SELECT AVG(e2.salary)
FROM employees e2
WHERE e2.department_id = e1.department_id
```

代表：

> 计算外层当前员工所在部门的平均工资。

外层条件：

```sql
e1.salary > 当前部门平均工资
```

代表：

> 当前员工工资是否高于自己部门平均工资。

## 4.4.2 关联点

这条 SQL 的关联点是：

```sql
e2.department_id = e1.department_id
```

其中：

| 表别名 | 所在层级 | 角色 |
| --- | --- | --- |
| `e1` | 外层查询 | 当前员工 |
| `e2` | 内层查询 | 用来计算部门平均工资的员工集合 |

这段条件表示：

> 内层只统计和外层当前员工同部门的资料。

## 4.4.3 判断表

| 判断点 | 结果 |
| --- | --- |
| 子查询返回什么 | 当前员工所在部门的平均工资 |
| 返回几行几列 | `1` 行 `1` 列 |
| 是否引用外层字段 | 是，引用 `e1.department_id` |
| 属于哪种子查询 | 相关子查询 + 标量子查询 |
| 外层如何使用 | 用 `e1.salary > (...)` 比较 |

这个例子也说明：

> 相关子查询可以同时是标量子查询。

分类角度不同：

- 从返回结果看，它是标量子查询。
- 从是否依赖外层看，它是相关子查询。

## 4.5 相关子查询与 `FROM` 派生表 JOIN

同一个问题，也可以不用相关子查询，而是先算出每个部门的平均工资，再和员工表连接。

## 4.5.1 方式二：`FROM` 子查询产生派生表

```sql
SELECT e.last_name, e.salary, e.department_id, dept_avg.avg_salary
FROM employees e
JOIN (
    SELECT department_id, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
) AS dept_avg
ON e.department_id = dept_avg.department_id
WHERE e.salary > dept_avg.avg_salary;
```

这条 SQL 的思路是：

1. 先用子查询算出每个部门的平均工资。
2. 把这个结果当成一张派生表 `dept_avg`。
3. 再把员工表 `employees` 和 `dept_avg` 按部门编号连接。
4. 最后筛选工资高于本部门平均工资的员工。

## 4.5.2 派生表结果示意

内层查询：

```sql
SELECT department_id, AVG(salary) AS avg_salary
FROM employees
GROUP BY department_id
```

会产生类似结果：

| department_id | avg_salary |
| --- | --- |
| 10 | 5500 |
| 20 | 5200 |
| 30 | 6800 |

外层再连接：

```sql
e.department_id = dept_avg.department_id
```

就能让每位员工找到自己部门的平均工资。

## 4.5.3 两种写法对比

| 写法 | 思考方式 | 优点 | 注意事项 |
| --- | --- | --- | --- |
| 相关子查询 | 外层每位员工各自去算本部门平均工资 | 贴近“每一行和同组比较”的语意 | 初学时要看懂外层字段引用 |
| 派生表 JOIN | 先算好每个部门平均工资，再连接回来 | 结构清楚，常适合统计后再比较 | 派生表要取别名，连接条件要正确 |

## 4.5.4 什么时候优先想到相关子查询

当题目出现以下语意时，可以优先想到相关子查询：

- 每位员工和自己部门比较。
- 每笔订单和同一客户的订单比较。
- 每个商品和同类别商品比较。
- 每一行都要根据自己的某个字段去找对应资料。
- 要判断是否存在和当前行对应的记录。

例如：

> 查询工资高于本部门平均工资的员工。

关键字是：

```text
本部门
```

这代表每位员工的比较基准会随自己的部门不同而改变。

## 4.5.5 什么时候优先想到派生表 JOIN

当题目可以清楚拆成：

1. 先产生一张统计结果表；
2. 再和原表连接；

就可以优先考虑派生表 JOIN。

例如：

> 先算出每个部门平均工资，再查询高于本部门平均工资的员工。

这句话明显可以拆成：

```text
部门平均工资表 + 员工表
```

就很适合派生表 JOIN。

## 4.5.6 性能判断不要只看 SQL 外观

不要直接记：

> 相关子查询一定慢。  
> JOIN 一定快。

这不是绝对规则。

实际效率会受到：

- MySQL 版本；
- 优化器改写能力；
- 表资料量；
- 索引设计；
- 统计信息；
- 子查询是否能被合并或半连接优化；
- 查询是否需要重复聚合；

等因素影响。

实务上如果要比较性能，应该使用：

```sql
EXPLAIN
```

查看执行计划，而不是只靠 SQL 长相判断。

## 4.6 在 `ORDER BY` 中使用相关子查询

题目：

> 查询员工的 `employee_id`、`salary`，并按照 `department_name` 排序。

可以写成：

```sql
SELECT e.employee_id, e.salary
FROM employees e
ORDER BY (
    SELECT d.department_name
    FROM departments d
    WHERE d.department_id = e.department_id
);
```

## 4.6.1 这条 SQL 为什么是相关子查询

子查询：

```sql
SELECT d.department_name
FROM departments d
WHERE d.department_id = e.department_id
```

引用了外层的：

```sql
e.department_id
```

所以它是相关子查询。

它的作用是：

> 对外层每一位员工，根据他的部门编号找到部门名称，再用部门名称排序。

## 4.6.2 逻辑理解

对于每一位员工：

1. 外层取出员工的 `employee_id` 和 `salary`。
2. 子查询根据当前员工的 `department_id` 去 `departments` 表找 `department_name`。
3. `ORDER BY` 使用这个部门名称作为排序依据。

## 4.6.3 重要限制：排序子查询应返回单一值

这个子查询放在 `ORDER BY` 表达式位置，因此对每一位外层员工来说，最好只返回一个部门名称。

如果：

```sql
d.department_id = e.department_id
```

能通过主键或唯一约束保证只匹配一个部门，就没有问题。

如果子查询可能返回多行，就会出现问题。

所以这类写法要确认：

> 每一位员工的部门编号只会对应一个部门名称。

## 4.6.4 实务上也可以用 JOIN

同一题通常也可以写成 JOIN：

```sql
SELECT e.employee_id, e.salary
FROM employees e
JOIN departments d
ON d.department_id = e.department_id
ORDER BY d.department_name;
```

这通常更直观。

### 两种写法对比

| 写法 | 特点 |
| --- | --- |
| `ORDER BY (子查询)` | 展示相关子查询可以作为排序表达式 |
| `JOIN + ORDER BY` | 实务中通常更直观，也更容易阅读 |

学习时可以看懂子查询写法；实务中如果只是为了按关联表字段排序，JOIN 通常更常见。

## 4.7 在 `WHERE` 中使用相关子查询做计数过滤

题目：

> 如果 `employees` 表中的 `employee_id` 与 `job_history` 表中相同 `employee_id` 的记录数不少于 `2`，输出这些员工的 `employee_id`、`last_name` 和 `job_id`。

可以写成：

```sql
SELECT e.employee_id, e.last_name, e.job_id
FROM employees e
WHERE 2 <= (
    SELECT COUNT(*)
    FROM job_history j
    WHERE j.employee_id = e.employee_id
);
```

## 4.7.1 这条 SQL 的逻辑

对外层每一位员工：

1. 取得当前员工的 `employee_id`。
2. 子查询去 `job_history` 中统计这个员工有几笔历史记录。
3. 如果记录数大于等于 `2`，保留这位员工。

关联点是：

```sql
j.employee_id = e.employee_id
```

其中：

| 表别名 | 角色 |
| --- | --- |
| `e` | 外层当前员工 |
| `j` | 内层工作历史记录 |

## 4.7.2 为什么 `COUNT(*)` 可以搭配单行比较符

子查询：

```sql
SELECT COUNT(*)
FROM job_history j
WHERE j.employee_id = e.employee_id
```

无论有没有找到记录，`COUNT(*)` 都会返回一个数字。

例如：

| 情况 | `COUNT(*)` 结果 |
| --- | --- |
| 没有记录 | `0` |
| 有 1 笔记录 | `1` |
| 有 2 笔记录 | `2` |

所以它稳定返回 `1` 行 `1` 列，可以搭配：

```sql
2 <= (...)
```

这种单行比较。

## 4.7.3 也可以用 GROUP BY 改写

同一题也可以先在 `job_history` 中分组统计，再连接回来：

```sql
SELECT e.employee_id, e.last_name, e.job_id
FROM employees e
JOIN (
    SELECT employee_id, COUNT(*) AS history_count
    FROM job_history
    GROUP BY employee_id
    HAVING COUNT(*) >= 2
) AS jh
ON jh.employee_id = e.employee_id;
```

这两种写法都能表达需求。

| 写法 | 思考方式 |
| --- | --- |
| 相关子查询 | 对每位员工逐一统计历史记录数 |
| 派生表 JOIN | 先统计每位员工历史记录数，再连接员工表 |

## 4.8 `EXISTS` 与 `NOT EXISTS`

`EXISTS` 和 `NOT EXISTS` 是相关子查询中非常常见的用法。

它们的重点是：

> 不关心子查询返回什么值，只关心子查询有没有返回任何行。

## 4.8.1 `EXISTS` 的基本语意

```sql
WHERE EXISTS (
    子查询
)
```

表示：

> 只要子查询至少返回一行，条件就成立。

示例：查询至少有下单记录的客户。

```sql
SELECT c.id, c.name
FROM customers c
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

这里的逻辑是：

1. 外层取出一位客户。
2. 子查询去 `orders` 中找有没有该客户的订单。
3. 只要找到至少一笔订单，`EXISTS` 就成立。
4. 外层保留这位客户。

关联点：

```sql
o.customer_id = c.id
```

## 4.8.2 为什么常写 `SELECT 1`

在 `EXISTS` 中，子查询写：

```sql
SELECT 1
```

是因为：

> `EXISTS` 只关心有没有返回行，不关心返回的字段内容。

所以下面几种写法在语意上通常都可以：

```sql
SELECT 1
SELECT *
SELECT o.id
```

但为了表达“我只关心是否存在”，常见写法是：

```sql
SELECT 1
```

这样阅读者会更容易看出目的。

## 4.8.3 `NOT EXISTS` 的基本语意

```sql
WHERE NOT EXISTS (
    子查询
)
```

表示：

> 只有当子查询完全没有返回行时，条件才成立。

示例：查询没有下过订单的客户。

```sql
SELECT c.id, c.name
FROM customers c
WHERE NOT EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

逻辑是：

1. 外层取出一位客户。
2. 子查询检查 `orders` 中是否有该客户订单。
3. 如果找不到任何订单，`NOT EXISTS` 成立。
4. 外层保留这位客户。

## 4.8.4 `EXISTS` 与 `NOT EXISTS` 对比

| 关键字 | 子查询有返回行 | 子查询没有返回行 | 典型用途 |
| --- | --- | --- | --- |
| `EXISTS` | `TRUE` | `FALSE` | 查询有对应记录的资料 |
| `NOT EXISTS` | `FALSE` | `TRUE` | 查询没有对应记录的资料 |

## 4.8.5 `EXISTS` 常用于相关子查询

`EXISTS` 通常会搭配外层字段引用。

例如：

```sql
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
)
```

其中：

```sql
c.id
```

来自外层客户表。

如果没有这个关联条件，`EXISTS` 很可能会变成：

- 子查询只要整体有资料，所有外层行都成立；
- 子查询整体没资料，所有外层行都不成立。

所以写 `EXISTS` 时一定要检查：

> 子查询是否正确关联到外层当前行？

## 4.9 `EXISTS`、`IN`、`JOIN` 的比较

同一个需求，有时可以用 `EXISTS`、`IN` 或 `JOIN` 表达。

重点不是背哪一个绝对更好，而是理解它们的表达重点不同。

## 4.9.1 题型一：查询公司管理者

题目：

> 查询公司管理者的 `employee_id`、`last_name`、`job_id`、`department_id`。

管理者的意思是：

> 有其他员工的 `manager_id` 等于他的 `employee_id`。

### 方式一：使用 `EXISTS`

```sql
SELECT e1.employee_id, e1.last_name, e1.job_id, e1.department_id
FROM employees e1
WHERE EXISTS (
    SELECT 1
    FROM employees e2
    WHERE e2.manager_id = e1.employee_id
);
```

语意：

> 对每位员工，检查是否存在其他员工由他管理。

这很贴近“是否存在下属”的题意。

### 方式二：使用 JOIN

```sql
SELECT DISTINCT e1.employee_id, e1.last_name, e1.job_id, e1.department_id
FROM employees e1
JOIN employees e2
ON e2.manager_id = e1.employee_id;
```

这里要加：

```sql
DISTINCT
```

原因是：

> 如果一个管理者有多位下属，JOIN 后会产生多笔重复的管理者资料。

### 方式三：使用 IN

```sql
SELECT employee_id, last_name, job_id, department_id
FROM employees
WHERE employee_id IN (
    SELECT manager_id
    FROM employees
    WHERE manager_id IS NOT NULL
);
```

这里子查询先找出所有 `manager_id`，外层再查询这些员工。

加上：

```sql
WHERE manager_id IS NOT NULL
```

是为了让子查询结果更干净，也避免空值造成理解混乱。

## 4.9.2 三种写法对比

| 写法 | 表达重点 | 注意事项 |
| --- | --- | --- |
| `EXISTS` | 是否存在对应记录 | 子查询要正确关联外层字段 |
| `IN` | 外层值是否属于某个集合 | 注意子查询字段是否可能有 `NULL` |
| `JOIN` | 两张表之间的匹配关系 | 一对多时可能产生重复，需要 `DISTINCT` 或分组 |

### 学习建议

如果题目语意是：

> 有没有对应资料？

优先想到：

```sql
EXISTS
```

如果题目语意是：

> 某个字段是否属于一组结果？

优先想到：

```sql
IN
```

如果题目语意是：

> 需要把两张表的字段一起查出来？

优先想到：

```sql
JOIN
```

## 4.9.3 题型二：查询没有员工的部门

题目：

> 查询 `departments` 表中，没有任何员工的部门 `department_id` 和 `department_name`。

使用 `NOT EXISTS`：

```sql
SELECT d.department_id, d.department_name
FROM departments d
WHERE NOT EXISTS (
    SELECT 1
    FROM employees e
    WHERE e.department_id = d.department_id
);
```

逻辑：

1. 外层取出一个部门。
2. 子查询去 `employees` 中找这个部门有没有员工。
3. 如果找不到任何员工，`NOT EXISTS` 成立。
4. 保留这个部门。

## 4.9.4 为什么这个题适合 `NOT EXISTS`

因为题目的语意是：

> 不存在对应员工。

这正好对应：

```sql
NOT EXISTS
```

相比 `NOT IN`，`NOT EXISTS` 对 `NULL` 的问题通常更安全、语意也更直接。

也可以写成 `LEFT JOIN`：

```sql
SELECT d.department_id, d.department_name
FROM departments d
LEFT JOIN employees e
ON e.department_id = d.department_id
WHERE e.department_id IS NULL;
```

这也是常见写法。

| 写法 | 语意 |
| --- | --- |
| `NOT EXISTS` | 不存在对应员工 |
| `LEFT JOIN ... IS NULL` | 左表部门保留，找不到员工的留下 |
| `NOT IN` | 部门编号不在员工部门集合中，但要小心 `NULL` |

## 4.10 相关更新：`UPDATE` 中的相关子查询

相关子查询不只可以用于 `SELECT`，也可以用于 `UPDATE`。

基本结构：

```sql
UPDATE table1 alias1
SET column = (
    SELECT expression
    FROM table2 alias2
    WHERE alias2.column = alias1.column
);
```

重点是：

> 子查询引用了当前正在更新的那一行资料。

## 4.10.1 示例：更新每个员工的薪资为所在部门平均薪资

假设：

- `employees` 表包含 `id`、`name`、`department_id`、`salary`。
- `departments` 表包含 `id`、`avg_salary`。

可以写成：

```sql
UPDATE employees e
SET e.salary = (
    SELECT d.avg_salary
    FROM departments d
    WHERE d.id = e.department_id
);
```

这里的逻辑是：

1. 外层正在更新 `employees` 当前员工。
2. 子查询根据当前员工的 `department_id` 去 `departments` 找对应的 `avg_salary`。
3. 把这个值更新到当前员工的 `salary`。

关联点是：

```sql
d.id = e.department_id
```

## 4.10.2 注意：子查询必须返回单一值

因为：

```sql
SET e.salary = (...)
```

右边必须是一个可以赋值的单一值。

所以子查询必须保证每位员工只会找到一个部门平均工资。

如果子查询返回多行，就会出错。

因此要确认：

- `departments.id` 是否唯一；
- 关联条件是否写对；
- 子查询是否可能找不到资料；
- 找不到资料时是否会把字段更新成 `NULL`。

## 4.10.3 MySQL 更新时的额外注意

在 MySQL 中，如果你更新某张表，同时又在子查询中直接读取同一张表，某些写法可能会遇到限制或报错。

例如这类情境要小心：

```sql
UPDATE employees e
SET salary = (
    SELECT AVG(salary)
    FROM employees
);
```

不同版本和不同写法下，优化器处理方式可能不同。  
实务上若遇到目标表和子查询来源表相同的问题，可以考虑：

- 使用派生表包一层；
- 使用 JOIN 更新；
- 先把结果写入临时表；
- 查看实际报错讯息与执行计划。

学习阶段先掌握相关更新的结构即可；实务使用时要特别小心更新范围与子查询结果。

## 4.11 相关删除：`DELETE` 中的相关子查询

相关子查询也常用于 `DELETE`。

基本结构：

```sql
DELETE FROM table1 alias1
WHERE NOT EXISTS (
    SELECT 1
    FROM table2 alias2
    WHERE alias2.column = alias1.column
);
```

常见用途是：

> 删除没有对应资料的记录。

## 4.11.1 示例：删除没有订单的客户

假设：

- `customers` 表包含 `id`、`name`。
- `orders` 表包含 `id`、`customer_id`。

如果要删除没有订单的客户，可以写成：

```sql
DELETE FROM customers c
WHERE NOT EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

逻辑：

1. 外层准备删除 `customers` 中的某一位客户。
2. 子查询去 `orders` 中找这个客户是否有订单。
3. 如果没有任何订单，`NOT EXISTS` 成立。
4. 删除这位客户。

## 4.11.2 删除语句要特别小心

`DELETE` 比 `SELECT` 风险更高。

真正执行前，建议先把删除条件改成 `SELECT` 检查：

```sql
SELECT c.*
FROM customers c
WHERE NOT EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

确认查出来的资料就是你要删除的对象，再执行：

```sql
DELETE FROM customers c
WHERE NOT EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

### 实务提醒

执行 `UPDATE` 或 `DELETE` 前，建议：

1. 先用 `SELECT` 验证影响范围。
2. 确认 `WHERE` 条件没有漏写。
3. 确认关联条件正确。
4. 在事务中执行。
5. 必要时先备份资料。

## 4.12 写题判断流程

遇到相关子查询题，可以按下面步骤判断。

## 4.12.1 第一步：确认外层当前行是谁

例如题目：

> 查询工资高于本部门平均工资的员工。

外层当前行是：

```text
每一位员工
```

所以外层先写：

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
```

## 4.12.2 第二步：确认内层要根据当前行计算什么

题目说：

> 本部门平均工资。

这代表内层要根据外层当前员工的部门编号计算平均工资：

```sql
SELECT AVG(e2.salary)
FROM employees e2
WHERE e2.department_id = e1.department_id
```

## 4.12.3 第三步：找出关联条件

关联条件是：

```sql
e2.department_id = e1.department_id
```

它让内层查询知道：

> 要计算外层当前员工所在部门的平均工资。

## 4.12.4 第四步：确认子查询返回结果

这里子查询返回：

```text
当前部门平均工资
```

是 `1` 行 `1` 列。

所以外层可以写：

```sql
e1.salary > (...)
```

## 4.12.5 第五步：组合完整 SQL

```sql
SELECT e1.last_name, e1.salary, e1.department_id
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id
);
```

## 4.12.6 第六步：检查是否有更适合的写法

写完后可以再问：

1. 这题是否也能先分组统计再 JOIN？
2. 是否只是判断存在性？如果是，是否更适合 `EXISTS`？
3. 是否只是集合匹配？如果是，是否更适合 `IN`？
4. 是否需要避免重复资料？如果用 JOIN，是否要 `DISTINCT`？
5. 是否有 `NULL` 问题？
6. 是否需要看执行计划？

## 4.13 题目语言与写法对照表

| 题目语言 | 常见写法 |
| --- | --- |
| 高于本部门平均工资 | 相关子查询或派生表 JOIN |
| 每位员工与自己部门比较 | 相关子查询 |
| 每个客户是否有订单 | `EXISTS` |
| 每个客户是否没有订单 | `NOT EXISTS` |
| 没有员工的部门 | `NOT EXISTS` 或 `LEFT JOIN ... IS NULL` |
| 有下属的员工 | `EXISTS`、`IN` 或自连接 |
| 每笔资料要统计自己的关联记录数 | 相关子查询 + `COUNT(*)` |
| 先分组统计再和原表比较 | 派生表 JOIN |
| 按关联表字段排序 | JOIN 或 `ORDER BY` 相关子查询 |
| 删除没有对应资料的记录 | `DELETE + NOT EXISTS` |
| 更新为关联表中的某个值 | `UPDATE + 相关子查询` 或 JOIN 更新 |

## 4.14 常见误区

## 4.14.1 误区一：只记“相关子查询会重复执行”

这句话可以帮助理解，但不是判断标准。

真正的判断标准是：

> 子查询是否引用外层当前行字段。

如果有引用，就是相关子查询。

## 4.14.2 误区二：以为相关子查询一定实际逐行执行

学习时可以用逐行模型理解。  
但实际执行计划可能被 MySQL 优化器改写。

所以：

- 理解语意时，用逐行模型。
- 判断性能时，看 `EXPLAIN`。

## 4.14.3 误区三：忘记给表取别名

相关子查询中经常同一张表出现两次。

例如：

```sql
FROM employees e1
...
FROM employees e2
```

如果没有别名，就很难分清：

- 哪个是外层当前行；
- 哪个是内层用来比较或统计的资料集合。

## 4.14.4 误区四：关联条件写错

例如本来应该写：

```sql
e2.department_id = e1.department_id
```

却写成：

```sql
e2.employee_id = e1.department_id
```

字段语意不一致，结果就会错误。

相关子查询最重要的是：

> 关联条件必须表达正确的业务关系。

## 4.14.5 误区五：`EXISTS` 子查询忘记关联外层

错误示例：

```sql
SELECT c.id, c.name
FROM customers c
WHERE EXISTS (
    SELECT 1
    FROM orders o
);
```

如果 `orders` 表中有任何一笔资料，所有客户都会被查出来。

因为子查询没有关联：

```sql
o.customer_id = c.id
```

正确写法：

```sql
SELECT c.id, c.name
FROM customers c
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.id
);
```

## 4.14.6 误区六：用 JOIN 后忘记处理重复资料

例如查询管理者：

```sql
SELECT e1.employee_id, e1.last_name
FROM employees e1
JOIN employees e2
ON e2.manager_id = e1.employee_id;
```

如果一位管理者有多位下属，他会出现多次。

通常需要：

```sql
DISTINCT
```

或使用 `EXISTS` 避免重复。

## 4.14.7 误区七：把 `EXISTS` 当成关心返回值

`EXISTS` 不关心子查询返回什么字段。

下面写法的语意重点都是：

```sql
是否有返回行
```

所以常写：

```sql
SELECT 1
```

而不是为了真的取得数字 `1`。

## 4.14.8 误区八：更新和删除前没有先 SELECT 检查

相关更新和相关删除都可能影响多笔资料。

执行前最好先写成：

```sql
SELECT ...
WHERE ...
```

确认范围，再执行 `UPDATE` 或 `DELETE`。

## 4.15 本节总结

相关子查询的核心不是返回一行还是多行，而是：

> 子查询依赖外层当前行。

判断时先看：

1. 子查询中有没有使用外层表别名？
2. 子查询中有没有引用外层字段？
3. 内层查询是否必须根据外层当前行才能计算？
4. 关联条件是否写对？
5. 子查询结果是用来比较、统计，还是判断存在？
6. 是否可以改写成派生表 JOIN、`EXISTS`、`IN` 或 JOIN？
7. 如果涉及更新或删除，是否已经先用 `SELECT` 验证范围？

只要能找出外层当前行、内层计算目标和关联条件，相关子查询就会变得很清楚。

## 常见混淆点

- 相关子查询的判断标准是：子查询是否引用外层当前行字段。
- 相关子查询不一定只出现在 `WHERE`，也可以出现在 `SELECT`、`ORDER BY`、`UPDATE`、`DELETE` 等位置。
- 相关子查询可以同时是标量子查询，例如“高于本部门平均工资”。
- 相关子查询的逐行执行是逻辑理解模型，不一定等于最终执行计划。
- `EXISTS` / `NOT EXISTS` 关心的是有没有返回行，不关心返回字段值。
- `EXISTS` 子查询通常必须写好与外层当前行的关联条件。
- `FROM` 子查询和相关子查询都能解决某些同类问题，但思考方式不同。
- JOIN 写法可能产生重复资料，`EXISTS` 通常更贴近存在性判断。
- `NOT EXISTS` 通常适合表达“不存在对应资料”，也常比 `NOT IN` 更能避开 `NULL` 陷阱。
- `UPDATE` / `DELETE` 中使用相关子查询时，要特别注意影响范围。

## 一句话抓核心

相关子查询的核心是：子查询引用了外层查询当前行的字段，因此内层查询必须根据外层当前行动态计算、比较或判断是否存在对应资料；学习时用“外层逐行、内层依赖当前行”的模型理解语意，实务上再结合 `EXPLAIN` 判断 MySQL 实际执行方式与性能。
