## 本节导读

在真正写查询之前，有一个非常实际、也非常常见的问题：

> **我连这张表里有哪些字段都不确定，该怎么查？**

这时候最稳的做法，不是猜字段名，也不是凭印象硬写，而是先查看表结构。

在 MySQL 中，最常见的查看方式是：

- `DESCRIBE 表名;`
- `DESC 表名;`

它们的作用都是：  
**显示这张表的结构信息（元数据），而不是显示表中的业务数据。**

这一节的重点不是教你复杂语法，而是让你先掌握一个很重要的基本动作：

> **写查询前，如果对表结构不确定，先看表结构。**

## 你会在这篇学到什么

- 如何使用 `DESCRIBE` 或 `DESC` 查看表结构
- `DESCRIBE` 和 `DESC` 的关系
- 表结构结果中的 `Field`、`Type`、`Null`、`Key`、`Default`、`Extra` 分别代表什么
- 如何根据表结构快速判断字段类型、是否可为空、是否有主键或索引
- 为什么 `DESC` 有时表示“显示表结构”，有时又表示“降序”

- `4.1 为什么要先看表结构`：看它在写查询前的实际价值

## 关键字

- `DESCRIBE`：查看表结构的完整命令
- `DESC`：`DESCRIBE` 的简写形式
- `Field`：字段名
- `Type`：字段类型
- `Null`：字段是否允许存储 `NULL`
- `Key`：字段是否参与主键或索引
- `Default`：字段默认值
- `Extra`：字段的额外属性

## 4.1 为什么要先看表结构

初学 SQL 时，一个很常见的错误是：

- 记错字段名
- 不确定字段类型
- 不知道某个字段能不能存 `NULL`
- 不知道某列是不是主键或索引列

这时如果直接写查询，往往容易报错，或者写出不符合预期的 SQL。

所以更稳妥的习惯是：

> **不确定表长什么样时，先看表结构，再写查询。**

例如你准备写：

```sql
SELECT employee_id, last_name, salary
FROM employees;
```

如果你其实不确定：

* `employee_id` 有没有这列
* `salary` 是什么类型
* 有没有其他更适合的字段

那就应该先执行：

```sql
DESC employees;
```

先确认表结构，再继续写查询。

### 回查提示

写 SQL 时，如果你开始凭感觉猜字段名，就说明该先看表结构了。

## 4.2 使用 DESCRIBE 或 DESC

查看表结构最常见的写法有两种：

```sql
DESCRIBE employees;
```

或者简写成：

```sql
DESC employees;
```

两者作用相同，都是显示 `employees` 这张表的结构信息。

示例：

```sql
mysql> DESC employees;
+----------------+-------------+------+-----+---------+-------+
| Field          | Type        | Null | Key | Default | Extra |
+----------------+-------------+------+-----+---------+-------+
| employee_id    | int(6)      | NO   | PRI | 0       |       |
| first_name     | varchar(20) | YES  |     | NULL    |       |
| last_name      | varchar(25) | NO   |     | NULL    |       |
| email          | varchar(25) | NO   | UNI | NULL    |       |
| phone_number   | varchar(20) | YES  |     | NULL    |       |
| hire_date      | date        | NO   |     | NULL    |       |
| job_id         | varchar(10) | NO   | MUL | NULL    |       |
| salary         | double(8,2) | YES  |     | NULL    |       |
| commission_pct | double(2,2) | YES  |     | NULL    |       |
| manager_id     | int(6)      | YES  | MUL | NULL    |       |
| department_id  | int(4)      | YES  | MUL | NULL    |       |
+----------------+-------------+------+-----+---------+-------+
11 rows in set (0.00 sec)
```

### 这一结果到底是什么？

这里显示的不是员工资料本身，而是 **`employees` 表的结构定义**。
也就是说，它回答的是：

* 这张表有哪些字段
* 每个字段是什么类型
* 能不能是 `NULL`
* 有没有键或索引
* 默认值是什么
* 有没有额外属性

### 回查提示

如果你想看“表里存了什么数据”，用 `SELECT`；
如果你想看“表是怎么定义的”，用 `DESCRIBE` 或 `DESC`。

## 4.3 表结构结果说明

`DESCRIBE` 输出结果中，最重要的是这 6 列：

| 列名        | 含义          | 你应该怎么理解       |
| --------- | ----------- | ------------- |
| `Field`   | 字段名称        | 这张表有哪些列       |
| `Type`    | 字段类型        | 这列存什么类型的数据    |
| `Null`    | 是否允许 `NULL` | 能不能不填值        |
| `Key`     | 是否参与主键或索引   | 这列有没有键约束或索引信息 |
| `Default` | 默认值         | 没有显式赋值时，默认是什么 |
| `Extra`   | 额外属性        | 是否有额外说明，如自增   |

下面分别理解。

### 1）`Field`

表示字段名称，也就是你之后在 `SELECT`、`WHERE`、排序、分组时会用到的列名。

例如：

* `employee_id`
* `last_name`
* `salary`

如果字段名都不确定，就不应该直接开始写查询。

---

### 2）`Type`

表示该字段的数据类型。

例如：

* `int(6)`
* `varchar(20)`
* `date`
* `double(8,2)`

看这一列的意义在于：
你可以先知道这个字段大概存放什么类型的数据，例如：

* 是整数
* 是字符串
* 是日期
* 是小数

这会直接影响你后面写条件、运算和函数时的判断。

---

### 3）`Null`

表示这一列是否允许存储 `NULL`。

* `YES`：允许 `NULL`
* `NO`：不允许 `NULL`

这里要特别注意：

> `Null = YES` 的意思是“允许为空”，
> 不是“当前这一列的值一定是空”。

它描述的是**字段定义规则**，不是某一笔具体数据的当前状态。

---

### 4）`Key`

表示该字段是否和主键或索引有关。

这一列不会把所有数据库设计细节都讲完，但至少能让你快速判断：

* 这列是不是主键相关字段
* 这列有没有唯一性约束
* 这列是不是普通索引列

后面会单独说明常见取值。

---

### 5）`Default`

表示这个字段的默认值。

如果插入数据时没有给这个字段显式赋值，数据库可能会使用这里显示的默认值。

例如：

* `0`
* `NULL`

这里也要注意：

> `Default = NULL` 表示默认值是 `NULL`，
> 不是空字符串，也不是“没有这一列”。

---

### 6）`Extra`

表示额外属性。

最常见的例子是：

* `AUTO_INCREMENT`

这通常表示该列会自动递增，常见于主键字段。

### 回查提示

看 `DESCRIBE` 结果时，最实用的阅读顺序通常是：

1. `Field`：先确认字段名
2. `Type`：再确认数据类型
3. `Null`：确认能不能存空值
4. `Key`：看有没有主键或索引
5. `Default` / `Extra`：最后补充看默认值和额外属性

---

## 4.4 Key 列常见取值

`Key` 列最常见的值有三个：

* `PRI`
* `UNI`
* `MUL`

### 1）`PRI`

表示该列是 **主键（Primary Key）** 的一部分。

主键通常用来唯一标识一条记录。

例如：

* `employee_id`

如果看到某列是 `PRI`，通常说明这列很重要，后面写查询时也经常会拿来唯一定位一笔数据。

---

### 2）`UNI`

表示该列和 **唯一索引（Unique Index）** 有关。

它的重点在于：

> 这一列的值不应该重复。

例如邮箱字段常常可能设置成唯一：

* `email`

---

### 3）`MUL`

表示该列和 **普通索引** 有关，而且该列的值允许重复出现多次。

例如：

* `department_id`
* `manager_id`

这些列通常不是唯一值，因此看到 `MUL` 很正常。

### 回查提示

看到 `Key` 时，不需要一开始就把索引原理全部搞懂；
你先记住最基础的判断就够了：

* `PRI`：主键相关
* `UNI`：唯一索引相关
* `MUL`：普通索引相关

---

## 常见混淆点

* `DESCRIBE` / `DESC` 显示的是表结构，不是表中的业务数据
* `DESC employees;` 里的 `DESC` 是 `DESCRIBE` 的简写
* `ORDER BY salary DESC` 里的 `DESC` 表示降序排序
  这两个 `DESC` 长得一样，但意思完全不同
* `Null = YES` 表示“允许为空”，不代表当前值一定是 `NULL`
* `Default = NULL` 表示默认值是 `NULL`，不等于空字符串
* `Key` 列为空，不表示这个字段不能查询，只表示这里没有显示主键或索引相关信息

## 一句话抓核心

**显示表结构的核心，不是背命令本身，而是养成一个习惯：写查询前如果不确定表长什么样，先用 `DESCRIBE 表名;` 或 `DESC 表名;` 看清楚再写。**

## 小结

这一节你需要记住：

* `DESCRIBE 表名;` 和 `DESC 表名;` 都可以查看表结构
* 它们显示的是表的元数据，不是表中的业务记录
* `Field` 看字段名，`Type` 看类型，`Null` 看是否允许空值
* `Key` 中常见的 `PRI`、`UNI`、`MUL` 分别表示主键、唯一索引、普通索引相关信息
* `Default` 表示默认值，`Extra` 表示额外属性
* `DESC` 在不同语境下可能表示不同含义，要结合上下文判断
