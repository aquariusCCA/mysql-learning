## 本节导读

这一节开始正式进入 SQL 最核心的内容之一：**查询数据**。

在前一节中，你已经知道 SQL 的基本书写规则；从这一节开始，重点要转向一个更实际的问题：

> **我想从数据库里拿到我需要的数据，该怎么写？**

而最基础的答案，就是 `SELECT`。

学习 `SELECT` 时，最重要的不是一开始记很多花式写法，而是先建立一个非常稳定的骨架：

- **查什么**：由 `SELECT` 指定
- **从哪查**：由 `FROM` 指定

只要这两个核心先建立起来，后面的别名、去重、空值处理、着重号、查询常数，都会变成在这个骨架上逐步补充的细节，而不是零散的知识点。

- `SELECT ... FROM ...` 的最基础结构

- 如何查询指定列
- 如何使用列别名改善结果可读性
- `DISTINCT` 去重时，作用的是哪些列
- `NULL` 参与算术运算时为什么会让结果变成 `NULL`
- 对象名与关键字冲突时，为什么需要使用着重号
- 如何在查询结果中增加固定的常数列

## 关键字

- `SELECT`：指定要查询的列、表达式或常量
- `FROM`：指定数据来自哪张表
- `*`：表示查询表中的全部列
- `AS`：给列或表达式指定别名
- `DISTINCT`：对查询结果去重
- `NULL`：空值，参与运算时需要特别小心
- 着重号：`` ` ``，用于包裹特殊对象名
- 查询常数：在结果集中增加固定值列

## 3.1 SELECT 不带 FROM

很多初学者会以为：只要写 `SELECT`，就一定要查某张表。  
其实不是。

`SELECT` 在最基础的层面上，也可以直接查询：

- 常量
- 表达式
- 函数结果

例如：

```sql
SELECT 1;
SELECT 9 / 2;
SELECT 'hello';
```

这种写法不需要 `FROM`，因为它不是从表里取数据，而是让数据库直接返回你写出来的结果。

### 什么时候会用到这种写法？

最常见的场景是：

* 快速验证表达式结果
* 测试函数行为
* 临时确认某个字面值或计算结果

### 回查提示

如果你只是想快速测试一个表达式，而不是查表数据，可以直接写不带 `FROM` 的 `SELECT`。

---

## 3.2 SELECT ... FROM ... 的基本结构

真正从数据表中查询时，最核心的结构是：

```sql
SELECT 列名
FROM 表名;
```

你可以把它理解成一句非常直白的话：

* `SELECT`：我要看哪些内容
* `FROM`：这些内容来自哪张表

例如：

```sql
SELECT department_id, location_id
FROM departments;
```

这条语句的意思就是：

> 从 `departments` 表中，查询 `department_id` 和 `location_id` 这两列。

### 这一节最重要的核心

以后你学到再复杂的查询，都可以先回到这个基础框架来理解：

1. 先决定查什么
2. 再决定从哪查

这就是最基本的查询思维。

### 回查提示

写查询时，先不要急着补细节。先问自己两个问题：

1. 我到底要查哪些列？
2. 这些列来自哪张表？

## 3.3 查询全部列与指定列

这一小节主要处理一个非常常见的问题：

> 是不是直接写 `SELECT *` 就好了？

答案是：**能用，但不能当成默认习惯。**

### 1）查询全部列

使用 `*` 表示查询表中的全部列：

```sql
SELECT *
FROM departments;
```

这样写的优点是方便，尤其是在：

* 刚接触一张新表时
* 只是临时看看表里有哪些字段时
* 想快速检查数据时

### 2）查询指定列

更推荐的写法，是明确写出需要的列：

```sql
SELECT department_id, location_id
FROM departments;
```

这样做的好处是：

* 语义更明确
* 查询结果更干净
* 更利于维护
* 不容易因为表结构变化而影响既有代码

### 3）为什么开发中不建议默认用 `SELECT *`

因为实际开发里，你通常并不需要表中的所有字段。
如果默认使用 `SELECT *`，常见问题包括：

* 取回了本来不需要的列
* 查询结果不够明确
* 表结构一变，可能影响下游逻辑
* 阅读 SQL 时，不容易一眼看出真正关心的数据是什么

### 回查提示

`SELECT *` 适合临时查看数据，不适合在开发中作为默认写法。
真正写业务查询时，尽量明确列名。

## 3.4 列的别名

有时候，原始列名不够直观；有时候，查询结果其实是一个表达式。
这时就很适合使用**列别名**。

### 基本写法

```sql
SELECT last_name AS name
FROM employees;
```

也可以写成：

```sql
SELECT last_name name
FROM employees;
```

虽然 `AS` 可以省略，但在学习阶段或正式开发中，**建议保留 `AS`**，可读性会更好。

### 表达式也可以起别名

```sql
SELECT last_name AS name, salary * 12 AS annual_salary
FROM employees;
```

这里的 `annual_salary` 就是给计算结果起的名字。

### 为什么别名重要？

别名主要解决两个问题：

1. 让结果列更容易理解
2. 给表达式一个可读的名称

### 命名建议

* 尽量简短、清楚
* 不要为了省几个字符把名字写得很怪
* 如果没有必要，尽量不要把别名写得太花哨

### 回查提示

如果查询结果里出现一列计算值，而列名又难懂，优先考虑加别名。

## 3.5 去除重复行

默认情况下，查询会把符合条件的记录全部返回出来，**包括重复值**。

例如：

```sql
SELECT department_id
FROM employees;
```

如果多名员工属于同一个部门，那么结果里就会重复出现同样的 `department_id`。

### 使用 DISTINCT 去重

```sql
SELECT DISTINCT department_id
FROM employees;
```

这样可以得到不重复的部门编号。

### DISTINCT 的真正作用范围

这一点非常重要：

> `DISTINCT` 不是只对它后面最近的一列生效，
> 而是对 **整个查询出来的列组合** 去重。

例如：

```sql
SELECT DISTINCT department_id, salary
FROM employees;
```

这表示要对 `(department_id, salary)` 这组组合去重。
只有当两列的组合都完全相同，才会被视为重复。

### 一个很容易犯的错

错误写法：

```sql
SELECT salary, DISTINCT department_id
FROM employees;
```

因为 `DISTINCT` 应该放在选出列的最前面，而不是插在中间。

正确写法：

```sql
SELECT DISTINCT department_id, salary
FROM employees;
```

### 回查提示

如果你只想看“有哪些不同的部门”，那就只写：

```sql
SELECT DISTINCT department_id
FROM employees;
```

不要多带无关列，否则去重范围就变了。

## 3.6 空值参与运算

这一小节是初学者很容易踩坑的地方。

在 SQL 里，`NULL` 表示的是：

> **没有值 / 未知值**

它不是：

* `0`
* 空字符串
* 任意默认值

### NULL 参与算术运算时的常见结果

如果一个算术表达式中有 `NULL` 参与，结果通常会变成 `NULL`。

例如：

```sql
SELECT employee_id, salary, commission_pct,
       12 * salary * (1 + commission_pct) AS annual_sal
FROM employees;
```

如果某个员工的 `commission_pct` 是 `NULL`，那么这一行的 `annual_sal` 往往也会是 `NULL`。

### 为什么这很重要？

因为很多人看到“整列怎么突然算不出来”，会以为公式写错了。
其实问题常常不是公式，而是参与运算的字段里有空值。

### 需要先记住的核心

* `NULL` 不等于空字符串
* `NULL` 也不等于 `0`
* 只要表达式里涉及 `NULL`，结果就要特别小心检查

### 回查提示

如果计算结果整列或部分结果变成 `NULL`，先检查表达式中涉及的列是不是存在 `NULL`。

## 3.7 着重号

如果表名、字段名等对象名刚好和 SQL 关键字冲突，就可能导致语法报错。

例如，假设表名叫 `order`，直接写：

```sql
SELECT * FROM order;
```

就可能出错，因为 `ORDER` 本身也是 SQL 关键字的一部分。

### 正确写法

```sql
SELECT * FROM `order`;
```

这里的着重号 `` ` `` 作用是告诉 MySQL：

> 这不是关键字，这里是一个对象名。

### 什么时候会用到？

常见场景包括：

* 表名和关键字冲突
* 字段名和关键字冲突
* 对象名里包含特殊情况，需要明确标识

### 更好的做法

虽然可以用着重号解决，但更好的习惯通常是：

* 尽量不要直接用关键字当表名或字段名
* 从命名阶段就避开这种冲突

例如把 `order` 改成：

* `orders`
* `order_info`
* `customer_order`

通常会更稳。

### 回查提示

如果对象名看起来没问题、SQL 却报语法错误，检查它是不是和关键字冲突了。

## 3.8 查询常数

除了查询表中的列，SQL 还可以在结果集中直接放入一个固定值，也就是**查询常数**。

例如：

```sql
SELECT '尚硅谷' AS corporation, last_name
FROM employees;
```

这表示：

* `last_name` 来自 `employees` 表
* `corporation` 这一列不是来自表，而是每一行都固定显示 `'尚硅谷'`

### 查询常数有什么用？

最常见的用途有：

* 给结果集增加来源标记
* 数据导出时增加固定说明列
* 多份数据整合时标明数据来源

例如：

```sql
SELECT '公司A' AS source, last_name
FROM employees_a
UNION ALL
SELECT '公司B' AS source, last_name
FROM employees_b;
```

这样整合后的结果中，每一行都会带有来源标识。

### 回查提示

如果你需要在结果里标记“这批数据来自哪里”，可以直接在 `SELECT` 里加常数列，而不用等后处理再补。

## 常见混淆点

* `SELECT` 不一定非要配合 `FROM`；查询常量或表达式时可以不写 `FROM`
* `SELECT *` 能用，但不适合在开发中默认使用
* `DISTINCT` 作用于查询结果中选出的列组合，不是只对最近的一列生效
* `AS` 可以省略，但建议保留以提高可读性
* `NULL` 不等于 `0`，也不等于空字符串
* 着重号 `` ` `` 用来包裹对象名，不要和单引号、双引号混用
* 查询常数不是从表中取值，而是给结果集额外加上固定值列

## 小结

这一节你需要记住：

* `SELECT` 可以不带 `FROM` 查询常量、表达式或函数结果
* 从表中查询数据时，最基本结构是 `SELECT 列名 FROM 表名`
* `SELECT *` 适合临时查看，不适合开发中默认使用
* 列别名可以用 `AS` 指定，建议保留 `AS`
* `DISTINCT` 对查询出来的列组合去重
* `NULL` 参与算术运算时，结果通常会变成 `NULL`
* 对象名和关键字冲突时，可以使用着重号 `` ` `` 包裹
* 查询常数可以为结果集增加固定标记列
