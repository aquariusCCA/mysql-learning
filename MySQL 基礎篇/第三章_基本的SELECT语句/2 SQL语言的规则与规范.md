## 2.1 基本规则

这一小节先处理最容易在刚开始写 SQL 时出错的基础规则：语句怎么结束、关键字能不能拆写、标点怎样才算合法。

- SQL 可以写在一行或者多行。为了提高可读性，各子句分行写，必要时使用缩进。
- 每条命令以 `;` 或 `\g` 或 `\G` 结束。
- 关键字不能被缩写，也不能分行。

> **✏️关于标点符号**
> - 必须保证所有的 `()`、单引号、双引号是成对结束的。
> - 必须使用英文状态下的半角输入方式。
> - 字符串型和日期时间类型的数据可以使用单引号（`' '`）表示。
> - 列的别名尽量使用双引号（`" "`），而且不建议省略 `AS`。

### 回查提示

如果一条 SQL 在命令行中没有执行，而是继续等待输入，优先检查语句末尾是否缺少 `;`、`\g` 或 `\G`。

## 2.2 SQL 大小写规范（建议遵守）

这一小节重点不是记住“能不能大写”，而是理解不同环境下大小写处理不同，为什么统一风格能减少问题。

- **MySQL 在 Windows 环境下是大小写不敏感的。**
- **MySQL 在 Linux 环境下是大小写敏感的。**
- 数据库名、表名、表的别名、变量名是严格区分大小写的。
- 关键字、函数名、列名（或字段名）、列的别名（字段的别名）是忽略大小写的。

> **✏️推荐采用统一的书写规范：**
> - 数据库名、表名、表别名、字段名、字段别名等都小写。
> - SQL 关键字、函数名、绑定变量等都大写。

### 回查提示

为了减少跨系统迁移时的大小写问题，建议从一开始就保持统一风格：对象名使用小写，SQL 关键字和函数名使用大写。

## 2.3 注释

注释的作用不仅是给自己留说明，也是团队协作时帮助别人理解 SQL 语句的上下文。这里先记住最常见的三种写法。

- 单行注释：`#` 注释文字（MySQL 特有的方式）
- 单行注释：`--` 注释文字（`--` 后面必须包含一个空格）
- 多行注释：`/* 注释文字 */`

### 注释写法示例

```sql
# 这是 MySQL 支持的单行注释
-- 这是 SQL 中常见的单行注释，注意 -- 后面要有空格
/* 这是多行注释 */
```

## 2.4 命名规则（暂时了解）

命名规则这一小节先建立“不要给自己制造麻烦”的意识。重点不是背数字，而是先避开最常见的命名坑。

- 数据库、表名不得超过 30 个字符，变量名限制为 29 个。
- 必须只能包含 `A-Z`、`a-z`、`0-9`、`_` 共 63 个字符。
- 数据库名、表名、字段名等对象名中间不要包含空格。
- 同一个 MySQL 软件中，数据库不能同名；同一个库中，表不能重名；同一个表中，字段不能重名。
- 必须保证字段没有和保留字、数据库系统或常用方法冲突。如果坚持使用，请在 SQL 语句中使用 `` ` `` 包起来。
- 保持字段名和类型的一致性，在命名字段并为其指定数据类型的时候一定要保证一致性。假如数据类型在一个表里是整数，那在另一个表里可就别变成字符型了。

举例：

```sql
# 以下两句是一样的，不区分大小写
show databases;
SHOW DATABASES;

# 创建表格
# create table student info(...); # 表名错误，因为表名有空格
create table student_info(...);

# 其中 order 使用 `` 包裹，因为 order 和系统关键字或系统函数名等预定义标识符重名了
CREATE TABLE `order`(
    id INT,
    lname VARCHAR(20)
);

select id as "编号", `name` as "姓名" from t_stu; # 起别名时，as 都可以省略
select id as 编号, `name` as 姓名 from t_stu; # 如果字段别名中没有空格，那么可以省略 ""
select id as 编 号, `name` as 姓 名 from t_stu; # 错误，如果字段别名中有空格，那么不能省略 ""
```

### 回查提示

命名规则最容易出错的地方是空格、保留字和大小写。写表名、字段名时，优先使用小写字母和下划线，例如 `student_info`。

## 2.5 数据导入指令

如果你拿到的是课程提供的 `.sql` 文件，而不是一条条手敲 SQL，这一小节就是最常用的回查入口。

在命令行客户端登录 MySQL 后，使用 `source` 指令导入：

```bash
mysql> source d:\\mysqldb.sql
```

导入后，可以使用 `DESC 表名;` 等语句检查表结构是否已经成功导入。

```bash
mysql> desc employees;
+----------------+-------------+------+-----+---------+-------+
| Field          | Type        | Null | Key | Default | Extra |
+----------------+-------------+------+-----+---------+-------+
| employee_id    | int(6)      | NO   | PRI | 0       |       |
| first_name     | varchar(20) | YES  |     | NULL    |       |
| last_name      | varchar(25) | NO   |     | NULL    |       |
| email          | varchar(25) | NO   | UNI | NULL    |       |
| phone_number   | varchar(20) | YES  |     | NULL    |       |
| hire_date      | date        | NO   |     | NULL    |       |
| job_id         | varchar(10) | NO   | MUL | NULL    |       |
| salary         | double(8,2) | YES  |     | NULL    |       |
| commission_pct | double(2,2) | YES  |     | NULL    |       |
| manager_id     | int(6)      | YES  | MUL | NULL    |       |

+----------------+-------------+------+-----+---------+-------+

```

## 常见混淆点

- `--` 注释后面需要有空格，写成 `--注释内容` 容易出问题。
- Windows 下 MySQL 大小写通常不敏感，不代表 Linux 下也不敏感。
- SQL 关键字可以统一大写，但数据库名、表名、字段名建议统一小写，方便跨环境维护。
- 字段别名中如果包含空格，就不要省略双引号。
- 使用保留字作为表名或字段名时，需要用着重号包起来，但更好的做法是从命名上避开保留字。

## 一句话抓核心

SQL 规范的重点不是背格式，而是让语句在不同环境中稳定执行：命令要正确结束，大小写要统一，注释和标点要符合语法，对象命名要避开空格与保留字，脚本导入要在 MySQL 客户端中用 `source`。

## 小结

这一节你需要记住：

- SQL 可以分多行书写，但每条命令要用 `;`、`\g` 或 `\G` 结束。
- MySQL 在 Windows 下通常大小写不敏感，在 Linux 下部分对象名会严格区分大小写。
- 推荐使用统一风格：对象名小写，SQL 关键字和函数名大写。
- 注释可以使用 `#`、`-- ` 和 `/* ... */`，其中 `--` 后面要有空格。
- 数据库名、表名、字段名不要包含空格，也尽量不要和保留字冲突。
- 导入 SQL 文件时，可以在 MySQL 命令行客户端中使用 `source 文件路径`。
