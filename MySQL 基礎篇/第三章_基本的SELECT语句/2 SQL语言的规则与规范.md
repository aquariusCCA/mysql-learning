## 本节导读

这一节讲的不是“复杂查询技巧”，而是 **SQL 能不能稳定写对、顺利执行** 的基础规范。

很多初学者写 SQL 出错，并不是因为不会查询逻辑，而是因为这些基础细节没有处理好，例如：

- 语句结尾少了分号
- 注释写法不符合语法
- 标点用了中文全角
- 表名、字段名命名不规范
- 迁移到别的环境后，大小写问题开始出现
- 拿到 `.sql` 文件却不知道怎么导入

所以，这一节的目标不是让你死背格式，而是先建立一个稳定的认识：

> **SQL 规范的作用，是让语句更容易写对、更容易维护，也更不容易在不同环境中出问题。**

第一次学习时，建议按“语句怎么结束 -> 大小写怎么统一 -> 注释怎么写 -> 命名怎么避坑 -> 脚本怎么导入”的顺序阅读。  
之后如果实际写 SQL 遇到报错，也可以直接回到对应小节快速排查。

## 2.1 基本书写规则

这一小节先解决最常见、最容易让初学者报错的问题：  
**SQL 到底怎么写才算“格式正确、能顺利执行”。**

### 1）SQL 可以写成一行，也可以写成多行

SQL 不要求一定写在一行。  
为了提高可读性，通常会把不同子句分行书写，例如：

```sql
SELECT employee_id, last_name
FROM employees
WHERE department_id = 90;
```

这种写法比全部挤在一行更容易阅读和修改。

### 2）一条 SQL 语句通常要有结束符

在大多数情况下，一条 SQL 语句需要明确告诉系统“到这里结束了”。常见结束方式有：

* `;`
* `\g`
* `\G`（常见于 MySQL 命令行客户端，适合纵向显示结果）

示例：

```sql
SELECT * FROM employees;
SELECT * FROM employees\g
SELECT * FROM employees\G
```

> 日常学习和开发中，最常用、最稳定、最推荐优先记住的结束符是 `;`。

### 3）关键字不要随意缩写，也不要拆开

SQL 关键字要保持完整，例如：

* `SELECT` 不能乱写成其他缩略形式
* `ORDER BY`、`GROUP BY` 这类关键结构不要拆得让语义混乱

你可以换行排版，但不要把关键字本身写得不完整或随意变形。

### 4）标点和引号要特别小心

这一类问题看起来很小，但非常容易导致语法报错。

需要注意：

* `()` 要成对出现
* 单引号 `' '`、双引号 `" "` 要成对出现
* 尽量使用英文输入法下的半角标点
* 字符串、日期等字面值通常使用单引号
* 别名写法应以“清楚、稳定、少歧义”为优先，不要为了省几个字符把语句写得难读

例如：

```sql
SELECT 'Tom';
SELECT '2025-01-01';
```

### 回查提示

如果 SQL 在命令行里一直没有执行，而是继续等待你输入，先检查两件事：

1. 语句结尾是不是少了结束符
2. 引号或括号是不是没有成对闭合

## 2.2 大小写规范

这一小节的重点不是“SQL 能不能大写”，而是：

> **为什么你应该尽早养成统一的大小写习惯。**

### 1）SQL 关键字通常不区分大小写

例如下面两句，作用通常相同：

```sql
select * from employees;
SELECT * FROM employees;
```

所以从“语法能不能执行”的角度看，大小写通常不是关键问题。

### 2）真正需要小心的是数据库对象名

在实际使用中，更容易出问题的不是 `SELECT`、`FROM` 这些关键字，而是：

* 数据库名
* 表名
* 字段名
* 别名
* 以及某些和系统设置、运行环境有关的对象名称行为

也就是说：

* **关键字写大写还是小写，更多是风格问题**
* **对象名大小写是否统一，才更关系到可维护性和跨环境稳定性**

所以初学阶段最稳妥的做法不是死背复杂规则，而是先建立一个简单原则：

> **从一开始就统一风格，避免自己制造大小写混乱。**

### 3）推荐采用统一书写风格

建议：

* 数据库名、表名、字段名、表别名等统一使用小写
* SQL 关键字统一使用大写

例如：

```sql
SELECT employee_id, last_name
FROM employees
WHERE department_id = 90;
```

这样做的好处有三个：

1. 更容易阅读
2. 团队协作时风格更统一
3. 跨环境时更不容易因为命名不一致而踩坑

### 回查提示

学习阶段最重要的不是记住所有大小写细节，而是先养成统一习惯：
**对象名统一小写，关键字统一大写。**

## 2.3 注释

注释的作用不是让 SQL 变复杂，而是帮助你或团队成员快速理解这段语句为什么要这样写。

SQL 中常见的注释有三种：

### 1）`#` 单行注释

这是 MySQL 常见的写法：

```sql
# 查询 90 部门员工
SELECT * FROM employees;
```

### 2）`-- ` 单行注释

这种写法更通用，但要注意：

> `--` 后面必须至少有一个空格

正确示例：

```sql
-- 查询 90 部门员工
SELECT * FROM employees;
```

容易出问题的写法：

```sql
--查询 90 部门员工
```

### 3）`/* ... */` 多行注释

适合写较长说明：

```sql
/* 
这是一个多行注释
用于说明这段 SQL 的用途
*/
SELECT * FROM employees;
```

### 注释写法示例

```sql
# MySQL 常见单行注释

-- 这是另一种单行注释，注意 -- 后面要有空格

/* 这是多行注释 */
```

### 回查提示

如果你写了 `--注释内容` 却报错或行为异常，先检查是不是少了 `--` 后面的空格。

## 2.4 命名规则

命名规则的核心，不是背很多数字限制，而是先建立一个原则：

> **对象命名要让自己少出错，也要让别人看得懂。**

### 1）对象名不要包含空格

例如下面这种写法就很容易出问题：

```sql
create table student info(...);
```

更稳妥的写法是：

```sql
create table student_info(...);
```

也就是说，表名、字段名、数据库名等对象名，尽量不要包含空格。

### 2）尽量避开保留字

如果对象名和 SQL 关键字冲突，就容易让语句变得难写、难读、也更容易报错。

例如：

```sql
CREATE TABLE `order`(
    id INT,
    lname VARCHAR(20)
);
```

虽然可以用着重号 `` ` `` 包起来解决，但更好的做法通常是：

* 不要一开始就用容易冲突的名字
* 尽量选择更清楚、更业务化的命名

例如把 `order` 改成：

* `orders`
* `order_info`
* `customer_order`

通常会更稳定。

### 3）同一层级不要重名

例如：

* 同一个 MySQL 实例中，数据库名不能重复
* 同一个数据库中，表名不能重复
* 同一张表中，字段名不能重复

这不是写作风格问题，而是对象定义本身就不允许混乱。

### 4）命名风格尽量统一

推荐做法：

* 使用小写字母
* 单词之间使用下划线分隔
* 名称尽量见名知意
* 不要今天写 `userInfo`，明天又写 `user_info`，后天又写 `UserInfo`

例如：

* `student_info`
* `employee_salary`
* `order_detail`

这类命名通常都比随意混用大小写和空格更稳。

### 5）别名也要尽量清楚

列别名虽然可以省略 `AS`，但为了可读性，建议在学习阶段优先写完整。

例如：

```sql
SELECT employee_id AS id, last_name AS name
FROM employees;
```

如果别名中包含空格或特殊字符，就要特别注意写法，不要为了“看起来好看”把语句写得更容易出歧义。

### 回查提示

命名最容易出错的地方不是“美不美观”，而是：

* 有空格
* 和关键字冲突
* 同一项目里风格不统一
* 名称看不出含义

## 2.5 数据导入指令

如果你拿到的是课程提供的 `.sql` 文件，而不是手动一条条输入 SQL，那么最常见的做法是在 **MySQL 命令行客户端** 中使用 `source` 导入。

### 基本写法

```bash
mysql> source d:\\mysqldb.sql
```

执行后，MySQL 会按文件中的内容依序执行 SQL 语句。

### 导入后怎么确认是否成功

导入完成后，可以用查看表结构的方式确认：

```bash
mysql> DESC employees;
```

如果能正常看到字段信息，通常就说明相关表已经成功导入。

### 使用时要注意

* `source` 是在 **MySQL 命令行客户端里面** 执行，不是在系统命令列直接执行
* 路径要写对
* `.sql` 文件中的语句本身也必须是合法 SQL

### 回查提示

如果 `source` 执行失败，优先检查三件事：

1. 你是不是已经进入 MySQL 命令行客户端
2. 文件路径是不是写对了
3. SQL 文件本身有没有语法错误

## 常见混淆点

* `;` 是最常见的结束符，`\g`、`\G` 更多见于 MySQL 命令行客户端
* SQL 关键字大小写通常不是主要问题，但对象名风格不统一很容易埋坑
* `--` 注释后面必须有空格
* `#` 是 MySQL 常见注释写法，但它更偏向 MySQL 环境
* 表名、字段名中尽量不要有空格
* 如果对象名和保留字冲突，可以用着重号包裹，但更好的做法通常是重新命名
* `source` 是在 MySQL 客户端里执行的，不是普通系统终端命令
