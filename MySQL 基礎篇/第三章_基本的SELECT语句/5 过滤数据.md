## 本节导读

前面你已经学会了最基础的查询骨架：

- `SELECT`：决定查哪些列
- `FROM`：决定从哪张表查

但实际开发里，很少会直接把整张表的数据全部拿出来。  
更常见的情况是：

- 只看某个部门的员工
- 只看某种状态的订单
- 只看某个时间范围内的数据
- 只看符合某个条件的记录

这时候就需要用到 `WHERE`。

> **`WHERE` 的作用，是为查询加上条件，只保留满足条件的行。**

所以，这一节真正要建立的核心不是“`WHERE` 写在哪里”而已，而是：

> **查询时，列由 `SELECT` 控制，行由 `WHERE` 控制。**

只要这个观念先建立好，后面学比较运算、逻辑运算、模糊查询、范围查询、多条件筛选时，都会更顺。

- `SELECT ... FROM ... WHERE ...` 的基本结构
- 如何使用等值条件筛选记录
- 为什么 `SELECT` 和 `WHERE` 很容易被初学者混淆

## 5.1 为什么需要过滤数据

在学习最基本的 `SELECT ... FROM ...` 时，查询往往是“把某张表的数据取出来看”。  
但真正进入实际场景后，你通常不会关心整张表的所有记录。

例如：

- 你不一定想看所有员工，只想看 90 部门的员工
- 你不一定想看所有订单，只想看已付款的订单
- 你不一定想看所有交易，只想看某个时间范围内的记录

也就是说，实际查询的重点往往不是“能不能查出来”，而是：

> **能不能只查出我真正需要的那一部分。**

而 `WHERE`，就是专门解决这个问题的。

### 回查提示

如果你看到结果集太大、太杂、太多无关记录，第一反应应该是：  
**是不是该加 `WHERE` 条件了。**

---

## 5.2 WHERE 的基本作用

`WHERE` 的核心作用非常简单，但非常重要：

> **过滤掉不满足条件的行，只保留满足条件的行。**

注意这里的关键词是：

- **行**
- 不是列

也就是说：

- `SELECT` 决定结果中显示哪些字段
- `WHERE` 决定哪些记录有资格进入结果集

例如，一张 `employees` 表可能有很多员工记录。  
如果你写：

```sql
SELECT employee_id, last_name, department_id
FROM employees;
```

那么结果会返回所有员工，只是显示其中几列。

如果你再加上：

```sql
WHERE department_id = 90
```

那么意思就变成：

* 只看 `department_id = 90` 的员工
* 并且结果中显示 `employee_id`、`last_name`、`department_id` 这几列

### 回查提示

记忆 `WHERE` 时，不要只记“它写在 `FROM` 后面”，更要记住它在做什么：
**它负责筛选记录行。**

---

## 5.3 基本语法

最基础的条件查询结构如下：

```sql
SELECT 字段1, 字段2
FROM 表名
WHERE 过滤条件;
```

你可以把它拆成三部分来理解：

* `SELECT`：我要显示哪些列
* `FROM`：这些列来自哪张表
* `WHERE`：哪些行可以进入结果集

这是后面几乎所有条件查询的基础骨架。

### 位置一定要记住

`WHERE` 的位置是：

> **紧跟在 `FROM` 子句之后**

也就是说，最基本的顺序是：

1. `SELECT`
2. `FROM`
3. `WHERE`

### 回查提示

如果你写 SQL 时不确定顺序，就先背这个最基础的骨架：

```sql
SELECT ...
FROM ...
WHERE ...;
```

---

## 5.4 示例

下面这条语句的意思是：
从 `employees` 表中，查询部门编号为 `90` 的员工，并显示这几列：

* `employee_id`
* `last_name`
* `job_id`
* `department_id`

```sql
SELECT employee_id, last_name, job_id, department_id
FROM employees
WHERE department_id = 90;
```

### 如何拆开理解？

这条语句可以分成三部分：

| 子句                                                     | 作用                 |
| ------------------------------------------------------ | ------------------ |
| `SELECT employee_id, last_name, job_id, department_id` | 决定结果中显示哪些字段        |
| `FROM employees`                                       | 决定数据来自哪张表          |
| `WHERE department_id = 90`                             | 只保留部门编号等于 `90` 的记录 |

### 这一句最该看懂的地方

很多初学者会把这条语句看成一个整体去背。
但更好的理解方式是：

* 先确定显示哪些列
* 再确定从哪张表查
* 最后加条件，把不符合条件的记录排除掉

只要这样拆开，你以后面对更复杂的 SQL 也会更容易理解。

### 回查提示

如果查询结果比你预期多很多行，先单独检查 `WHERE` 条件有没有真的把范围缩小。

## 5.5 如何理解 SELECT 与 WHERE 的分工

这一小节是整篇最值得牢牢记住的地方。

很多初学者会把 `SELECT` 和 `WHERE` 混在一起，原因是两者都写在同一条 SQL 里。
但它们控制的对象根本不同：

### `SELECT` 控制列

它回答的是：

> **结果里要显示什么字段？**

例如：

```sql
SELECT employee_id, last_name
```

表示结果里只显示这两列。

---

### `WHERE` 控制行

它回答的是：

> **哪些记录可以留下来？**

例如：

```sql
WHERE department_id = 90
```

表示只有部门编号为 `90` 的记录才能进入结果集。

---

### 一个最值得记住的总结

> **`SELECT` 管列，`WHERE` 管行。**

把这句话记住，后面学习条件判断、多条件过滤、排序、分组时，会轻松很多。

### 回查提示

一旦你分不清某个子句在做什么，就回到这句：

* `SELECT` 管列
* `WHERE` 管行

---

## 常见混淆点

* `WHERE` 过滤的是**行**，不是列
* `SELECT` 后面列的是要显示的字段，不是过滤条件
* `WHERE` 应该写在 `FROM` 后面，而不是插在字段列表中间
* `department_id = 90` 是条件判断，表示只保留满足这个条件的记录
* 如果不写 `WHERE`，查询通常会返回表中所有符合其他子句要求的行
* `WHERE` 不是可有可无的装饰，它是控制结果范围的核心子句之一

* `WHERE` 子句应该写在 SQL 的哪个位置？
* `WHERE` 到底是在过滤行还是过滤列？
* `SELECT` 和 `WHERE` 分别控制什么？
* 想只查某个部门的员工，最基本的写法是什么？
* 为什么我的查询会返回这么多行？
* 不写 `WHERE` 会发生什么？

## 一句话抓核心

**过滤数据的核心，是在 `SELECT` 指定列、`FROM` 指定表之后，再用 `WHERE` 写条件，只保留满足条件的行。**

## 小结

这一节你需要记住：

* `WHERE` 用来给查询加过滤条件
* `WHERE` 的核心作用是过滤记录行
* `WHERE` 应该紧跟在 `FROM` 后面
* 最基础的写法是 `SELECT 字段 FROM 表名 WHERE 条件;`
* `SELECT` 决定显示哪些列，`WHERE` 决定保留哪些行
* 如果结果太多、范围太大，通常应该先检查是否需要补上 `WHERE`
