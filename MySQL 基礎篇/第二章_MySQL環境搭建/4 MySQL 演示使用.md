## 本节导读

这一节只解决一个核心问题：

**登录 MySQL 之后，怎样用一组最基础、最连贯的 SQL，完成“建库 → 建表 → 插入 → 查询 → 排查报错 → 清理对象”这一整条入门主线。**

对初学者来说，第一次练习 SQL 最容易混乱的，不是命令本身，而是下面这些问题：

- 知道单条命令，但不知道先后顺序
- 忘记先 `use` 数据库，结果报错
- 能建表，但不知道怎么确认表和数据库的默认字符集
- 插入中文时报错，却分不清是命令问题还是字符集问题
- 做完练习后，不知道怎么把表和数据库清理掉

所以这篇不会只是把几条 SQL 摆在一起，而是按更实用的顺序整理成：

**先看服务器里有哪些库 → 再创建自己的库 → 再切换到这个库 → 再建表、插入、查询 → 再看建表信息和字符集 → 最后学会删除对象。**

> ✏️ 说明：
>
> 这一节默认你已经完成前一节的安装与登录。  
> 如果你还不能进入 MySQL，请先回看 [3 MySQL 的登录](./3%20MySQL%20的登录.md)。

## 你会在这篇学到什么

- 如何查看 MySQL 服务器中现有的数据库。
- 如何创建数据库、切换默认数据库、查看数据库中的表。
- 如何创建数据表、插入数据、查询数据。
- 如何查看数据库和数据表的创建信息。
- 如何删除表和数据库。
- 遇到 `ERROR 1046 (3D000)` 时，为什么通常是因为没有先选中数据库。
- 遇到 `ERROR 1366 (HY000)` 时，为什么通常要从字符集角度排查。
- MySQL 5.7 与 MySQL 8.0 在默认字符集上的常见差异。

## 一句话先抓核心

**这一节最重要的不是单独记住某一条 SQL，而是建立“建库 → 切库 → 建表 → 写入 → 查询 → 检查字符集 → 清理对象”的完整顺序感。**

## 4.1 MySQL 的使用演示

这一部分会用一组最基础、最连贯的 SQL，带你从“什么都没有”走到“有一个数据库、有一张表、有几条数据”。

---

### 4.1.1 查看所有的数据库

```sql
show databases;
````

执行后，你通常会看到一些系统自带数据库，例如：

* `information_schema`
* `performance_schema`
* `sys`
* `mysql`

这些库主要是 MySQL 自己运行和管理时需要的系统库。

> ✏️ 初学阶段先不用深入研究这些系统库的内部结构。
> 你现在最重要的，是学会辨认“哪些是系统库，哪些是我自己创建的业务库”。

---

### 4.1.2 创建自己的数据库

基本语法：

```sql
create database 数据库名;
```

示例：

```sql
create database atguigudb;
```

### 这一步要先记住什么？

* 数据库名不能和现有数据库重名
* 创建成功后，数据库对象是“存在了”，但当前会话还**不一定已经切换进去**

也就是说，`create database` 和 `use 数据库名` 是两件不同的事。

---

### 4.1.3 使用自己的数据库

基本语法：

```sql
use 数据库名;
```

示例：

```sql
use atguigudb;
```

执行后，当前会话就会把 `atguigudb` 当成默认数据库。

### 为什么这一步特别重要？

因为很多后续命令，例如：

* `show tables;`
* `create table ...`
* `insert into ...`
* `select * from ...`

都默认是在“当前数据库”下面执行的。

如果你没有先执行 `use`，又没有写成 `数据库名.表名` 这种完整限定形式，就很容易报：

```text
ERROR 1046 (3D000): No database selected
```

这表示：**当前会话没有选中默认数据库。**

> ✏️ 记忆方式：
>
> `create database` 是“创建库”，
> `use 数据库名` 是“进入这个库”。

---

### 4.1.4 查看某个库的所有表格

当你已经切换到目标数据库后，可以执行：

```sql
show tables;
```

这会列出当前数据库中的所有表。

如果你想直接查看某个指定数据库中的表，也可以写成：

```sql
show tables from 数据库名;
```

例如：

```sql
show tables from atguigudb;
```

---

### 4.1.5 创建新的表

基本语法：

```sql
create table 表名称(
    字段名 数据类型,
    字段名 数据类型
);
```

示例：

```sql
create table student(
    id int,
    name varchar(20)
);
```

### 这一步要先记住什么？

* 字段定义之间用逗号分隔
* 最后一个字段后面不要多写逗号
* `varchar(20)` 表示这个字段最多存放 20 个字符

> ✏️ 初学阶段先把重点放在“表名、字段名、字段类型”这三个最基本元素上，不需要一开始就把主键、非空、默认值等细节全部塞进来。

---

### 4.1.6 查看一个表的数据

基本语法：

```sql
select * from 表名称;
```

示例：

```sql
select * from student;
```

### 这条命令的作用

它会把 `student` 表中当前所有字段、所有记录直接查出来。

对于刚建好的空表来说，执行后通常能看到：

* 表结构是存在的
* 但还没有任何数据

所以你可以把它当成最基础的“当前表里到底有什么”的查看命令。

---

### 4.1.7 添加一条记录

基本语法：

```sql
insert into 表名称 values(值列表);
```

示例：

```sql
insert into student values(1, '张三');
insert into student values(2, '李四');
```

插入完成后，可以再次执行：

```sql
select * from student;
```

来确认数据是否已经成功写入。

---

### 如果插入中文时报错怎么办？

例如你可能会看到类似下面的错误：

```text
mysql> insert into student values(1,'张三');
ERROR 1366 (HY000): Incorrect string value: '\\xD5\\xC5\\xC8\\xFD' for column 'name' at row 1

mysql> insert into student values(2,'李四');
ERROR 1366 (HY000): Incorrect string value: '\\xC0\\xEE\\xCB\\xC4' for column 'name' at row 1
```

这通常说明问题不在 `insert into` 语法本身，而更可能在：

* 当前连接字符集
* 数据库默认字符集
* 表默认字符集
* 字段字符集
* `my.ini` 中的默认字符集配置

也就是说，这类错误本质上更接近“编码配置问题”，而不是“SQL 写法问题”。

---

### 4.1.8 查看表的创建信息

基本语法：

```sql
show create table 表名称\G
```

示例：

```sql
show create table student\G
```

结果示例：

```text
*************************** 1. row ***************************
       Table: student
Create Table: CREATE TABLE `student` (
  `id` int(11) DEFAULT NULL,
  `name` varchar(20) DEFAULT NULL
) ENGINE=InnoDB DEFAULT CHARSET=latin1
1 row in set (0.00 sec)
```

### 为什么这一步特别重要？

因为这一步可以直接帮你确认：

* 表到底是不是你以为的那种结构
* 默认字符集到底是什么

上面的结果里，最值得注意的是：

```text
DEFAULT CHARSET=latin1
```

如果你的表默认字符集还是 `latin1`，那么插入中文时出现报错就很正常了。

---

### 4.1.9 查看数据库的创建信息

基本语法：

```sql
show create database 数据库名\G
```

示例：

```sql
show create database atguigudb\G
```

结果示例：

```text
*************************** 1. row ***************************
       Database: atguigudb
Create Database: CREATE DATABASE `atguigudb` /*!40100 DEFAULT CHARACTER SET latin1 */
1 row in set (0.00 sec)
```

### 这一条和上一条的区别是什么？

* `show create database`：看数据库级别的默认字符集
* `show create table`：看表级别的默认字符集

这两个层级不要混在一起。

也就是说：

* 数据库可以是 `latin1`
* 表也可以是 `latin1`
* 就算你后来改了 `my.ini`，旧数据库和旧表也不一定会自动变成 `utf8`

这也是为什么字符集问题常常不能只改配置文件就结束。

---

### 4.1.10 删除表格

基本语法：

```sql
drop table 表名称;
```

示例：

```sql
drop table student;
```

### 什么时候用这条命令？

当你只是想删掉表结构和表数据，但还想保留这个数据库时，就用 `drop table`。

---

### 4.1.11 删除数据库

基本语法：

```sql
drop database 数据库名;
```

示例：

```sql
drop database atguigudb;
```

### 什么时候用这条命令？

当你想把整个练习环境一起清掉时，就用 `drop database`。

> ⚠️ 注意：
>
> `drop database` 会删除整个数据库对象，连同里面的表一起消失。
> 如果这是正式数据环境，就不能随便执行。

## 4.2 MySQL 的编码设置

这一部分只解决一个问题：

**为什么同样的 SQL，插入英文没事，插入中文却报错？**

---

### 4.2.1 先理解：这通常不是 `insert` 写错了

如果你插入中文时看到：

```text
ERROR 1366 (HY000): Incorrect string value
```

高频原因通常有两类：

1. **当前连接字符集不匹配**
2. **数据库 / 表 / 字段本身的默认字符集不支持中文**

所以排查时不要只盯着 `insert into` 这一条 SQL，而要往更上层看。

---

### 4.2.2 MySQL 5.7 中常见的处理思路

先查看当前字符集与校对规则：

```sql
show variables like 'character_%';
show variables like 'collation_%';
```

如果确认默认字符集配置不符合预期，可以修改 `my.ini` 配置文件，例如：

```ini
[mysql]
default-character-set=utf8

[mysqld]
character-set-server=utf8
collation-server=utf8_general_ci
```

然后：

1. 停止 MySQL 服务
2. 修改 `my.ini`
3. 重新启动 MySQL 服务
4. 再次执行：

```sql
show variables like 'character_%';
show variables like 'collation_%';
```

确认配置是否已经生效。

> ⚠️ 注意：
>
> 修改配置文件时要谨慎，避免因为编辑器编码问题把 `my.ini` 改坏，导致 MySQL 服务无法正常启动。

---

### 4.2.3 当前连接字符集不匹配时怎么办？

如果问题主要出在“当前会话连接字符集”和“你输入中文时的实际环境”不一致，可以先尝试：

```sql
SET NAMES GBK;
```

### 这条命令的特点

* 它主要影响**当前连接**
* 关闭当前命令行窗口后，重新打开时可能还要重新设置
* 它更像“临时修正当前连接”，不是从根上修改数据库对象字符集

所以，如果你发现问题会反复出现，就不能只停在 `SET NAMES GBK;` 这一步。

---

### 4.2.4 为什么改了 `my.ini`，问题还是可能存在？

因为字符集是分层级的。

即使你已经修改了 `my.ini`，如果数据库和表是在修改之前创建的，那么它们可能仍然保留旧的字符集，例如：

* 数据库默认字符集还是 `latin1`
* 表默认字符集还是 `latin1`
* 某个字段甚至还保留旧编码

这时常见思路有两种：

#### 方式 1：删除旧对象后重新创建

适合当前只是练习环境，没有真实业务数据时。

#### 方式 2：使用 `ALTER` 语句逐层修改

适合已有对象不方便直接删掉时。

---

### 4.2.5 MySQL 5.7 和 MySQL 8.0 的默认字符集差异

这一点特别容易影响初学者的判断。

### 在 MySQL 5.7 及更早时期

默认字符集相关问题更容易出现。
如果数据库、表或连接层还是旧配置，就更容易在插入中文时报错。

### 在 MySQL 8.0 中

默认字符集改成了：

```text
utf8mb4
```

所以在默认安装和默认配置下，相较于 `MySQL 5.7`，通常更不容易遇到中文写入问题。

> ✏️ 但要注意：
>
> “更不容易遇到”不等于“绝对不会遇到”。
> 如果你是从旧版本升级、导入旧库、沿用旧配置，仍然可能碰到字符集问题。

## 常见风险点

### 风险 1：只记单条命令，不记执行顺序

这样最容易发生的就是：会 `create table`，却忘了先 `use 数据库名;`。

### 风险 2：报 `ERROR 1046` 时，以为是表名写错

其实很多时候只是因为当前没有选中数据库。

### 风险 3：插入中文报错时，只怀疑 `insert` 语法

这类问题更常见的根因是字符集，而不是 `insert` 本身。

### 风险 4：只改 `my.ini`，却不检查旧数据库和旧表

即使配置文件改了，已经存在的数据库和数据表也可能继续保留旧字符集。

### 风险 5：练习完直接 `drop database`，却没意识到会把整个库都删掉

在练习环境没问题，但在真实环境里一定要谨慎。

---

## 常见回查问题

### 1. 为什么我已经 `create database` 了，后面还是报错？

因为你可能只“创建了数据库”，但还没有：

```sql
use 数据库名;
```

### 2. `show tables;` 为什么执行不出来？

因为它默认查看的是“当前数据库”里的表。
如果你还没先 `use` 到某个数据库，就容易出问题。

### 3. `show create table` 和 `show create database` 有什么区别？

* `show create table`：看表级别定义和默认字符集
* `show create database`：看数据库级别定义和默认字符集

### 4. 插入中文报 `ERROR 1366`，第一步应该查什么？

优先查：

1. 当前连接字符集
2. 数据库默认字符集
3. 表默认字符集

### 5. `SET NAMES GBK;` 是不是永久生效？

不是。
它通常只影响当前连接。

### 6. 为什么 MySQL 8.0 比 5.7 更少遇到中文写入问题？

因为 `MySQL 8.0` 默认字符集已经改为 `utf8mb4`。

---

## 小结

这一节最重要的，不是把每条 SQL 死背下来，而是记住这条主线：

1. 先看服务器里有哪些数据库
2. 创建自己的数据库
3. 切换到这个数据库
4. 创建数据表
5. 插入并查询数据
6. 用 `show create table` 和 `show create database` 检查字符集
7. 练习完成后，再决定是否删除表和数据库

只要这条顺序不乱，初学阶段的大多数 SQL 演示操作都会变得很清楚。
