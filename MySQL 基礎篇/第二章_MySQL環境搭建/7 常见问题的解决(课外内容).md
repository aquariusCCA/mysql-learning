## 本节导读

这一节只解决一个核心问题：

**当 MySQL 环境已经搭好，但实际使用中出现报错或异常时，应该如何按现象快速定位问题，而不是乱试。**

对初学者来说，MySQL 最容易卡住的，不一定是知识点本身，而是下面这种情况：

- 明明已经安装好了，却还是登录不进去
- 明明命令写得没错，却提示 `mysql` 不是内部或外部命令
- 明明建了数据库，执行 SQL 却报 `No database selected`
- 明明 `insert` 语法没问题，插入中文却报字符集错误
- 明明改了 `my.ini`，乱码问题却还是没完全解决

所以这篇不会把问题写成一堆分散的小技巧，而是按更实用的方式整理成：

**先按“现象”找问题 → 再按“层级”排查原因 → 最后确认修改的是当前连接、配置文件、数据库、表，还是字段。**

你可以把这一节理解成：

**第二章的故障排查索引页。**

> ✏️ 说明：
>
> 这一节不是顺序阅读型内容。  
> 平时不需要从头读到尾；遇到具体问题时，直接按错误现象、错误编号或关键字跳到对应小节，会更高效。

## 你会在这篇学到什么

- 忘记 `root` 密码时，如何通过临时跳过权限表检查来重置密码。
- `mysql` 命令无法识别时，为什么通常要先检查 `Path` 环境变量。
- 遇到 `ERROR 1046 (3D000): No database selected` 时，为什么本质上是“当前会话没有默认数据库”。
- 命令行插入中文时报 `ERROR 1366 (HY000)` 时，如何从连接字符集、配置文件、数据库、表、字段几个层级去排查。
- 为什么修改 `my.ini` 后，已有数据库和表仍然可能保留旧字符编码。
- 修改数据库、表、字段字符编码时，分别应该使用哪些 `ALTER` 语句。

## 一句话先抓核心

**这一节最重要的，不是背“解决技巧”，而是建立“先看现象、再分层定位”的排查习惯。**

- `常见回查问题`：适合临时忘记时快速定位

## 关键字

- `root 密码重置`：忘记 `root` 密码时的处理思路

### 情况 1：我连 MySQL 都进不去，而且忘了 `root` 密码

重点看 `问题1`。  
因为这不是命令格式问题，也不是环境变量问题，而是账号认证问题。

### 情况 2：我在命令行里输入 `mysql`，系统根本不认识这个命令

因为这通常还没有进入“登录 MySQL”这一步，而是系统找不到 MySQL 客户端程序。

### 情况 4：我插入中文时报错

因为字符集问题往往不是单点问题，而是需要分层排查。

重点看 `问题5`。  
因为这很可能是“配置层”和“数据库对象层”没有分清。

## 问题1：root 用户密码忘记，重置的操作

### 现象

忘记 `root` 用户密码，导致无法通过正常密码登录 MySQL。

### 先理解本质

这类问题不是“命令写错”，而是：

**当前你无法通过正常认证流程进入 MySQL。**

所以处理思路通常不是一直重复试密码，而是先临时绕过权限检查，进入后再修改密码。

### 处理步骤

1. 通过任务管理器或者服务管理，先关闭 `mysqld` 服务进程。
2. 通过命令行使用特殊参数启动 `mysqld`：

   ```bash
   shell mysqld --defaults-file="D:\\ProgramFiles\\mysql\\MySQLServer5.7Data\\my.ini" --skip-grant-tables
   ```

3. 此时 `mysqld` 已启动，并且暂时跳过权限检查。

4. 另开一个客户端，使用 `mysql -uroot` 无密码登录。

5. 修改权限表：

   ```sql
   use mysql;
   update user set authentication_string=password('新密码') where user='root' and Host='localhost';
   flush privileges;
   ```

6. 再次关闭这个特殊启动的 `mysqld` 进程。

7. 按正常方式重新启动 MySQL 服务。

8. 使用新密码重新登录。

### 这一题最重要的理解

`--skip-grant-tables` 的作用不是“正常登录方式”，而是：

**为了紧急维护，临时跳过权限表检查。**

### 注意点

* 这种方式只适合在你确实忘记密码、无法正常登录时使用。
* 密码修改完成后，应关闭这种特殊启动方式，恢复正常服务启动。
* 如果你当前用的是后续版本环境，真正的密码修改语句可能会因版本不同而略有差异；但这篇的核心重点仍然是先理解“跳过权限表 → 进入系统库 → 修改认证信息 → 刷新权限”这条主线。

> ⚠️ 注意：
>
> `--skip-grant-tables` 会让权限检查暂时失效。
> 所以这一招是“应急排错用”，不是平常使用习惯。

## 问题2：mysql 命令报“不是内部或外部命令”

### 现象

在命令行输入 `mysql` 后，提示“不是内部或外部命令”。

### 先理解本质

这通常表示：

**系统还找不到 MySQL 的命令行程序。**

也就是说，此时问题通常不在：

* 用户名
* 密码
* 数据库服务本身

而更可能在：

* MySQL 的 `bin` 目录没有加入 `Path`
* 路径加错了
* 命令行窗口没有重新打开
* 你安装时改过目录，但自己忘了

### 处理方式

把 MySQL 安装目录下的 `bin` 目录加入环境变量 `Path` 即可。

![mysql 命令无法识别示例](./images/image-20210914093150145.png)

例如常见路径形态可能类似：

```text
C:\Program Files\MySQL\MySQL Server 8.0\bin
```

### 处理后怎么验证？

重新打开一个新的命令行窗口，再执行：

```bash
mysql --version
```

如果能正常显示版本，说明系统已经能找到 MySQL 客户端程序。

### 注意点

* 配置完成后，需要重新打开命令行窗口，让新的环境变量生效。
* 如果你不确定 `bin` 目录的位置，可以回看：

  * [2 MySQL 的下载、安装、配置](./2%20MySQL%20的下载、安装、配置.md)
  * [6 MySQL 目录结构与源码](./6%20MySQL%20目录结构与源码.md)

> ✏️ 记忆方式：
>
> * **命令不能执行**：优先想 `Path`
> * **命令能执行但登录失败**：再看用户名、密码、服务状态

## 问题3：错误 `ERROR 1046 (3D000)`，没有选择数据库

### 现象

执行表操作或数据操作时出现下面的错误：

```text
ERROR 1046 (3D000): No database selected
```

### 先理解本质

这表示：

**当前会话还没有选中默认数据库。**

这不是说 MySQL 没安装好，也不是说表一定不存在，而是你当前执行 SQL 时，系统不知道应该默认在哪一个数据库下面操作。

### 处理方式

### 解决方案 1：使用 `USE 数据库名;`

```sql
USE 数据库名;
```

执行后，后续 SQL 默认会针对这个数据库操作。

### 解决方案 2：在表对象前面加上数据库名

例如：

```sql
数据库名.表名
```

### 什么时候用哪一种？

* 如果后续很多 SQL 都在同一个数据库下执行，优先使用 `USE 数据库名;`
* 如果只是临时访问另一个数据库里的某一张表，用 `数据库名.表名` 更灵活

### 这一题最重要的理解

`ERROR 1046` 本质上是：

**上下文问题，不是数据库服务问题。**

也就是说，重点不是重装 MySQL，而是先补上当前会话缺失的默认数据库。

## 问题4：命令行客户端的字符集问题

### 现象

在命令行客户端插入中文数据时，出现字符集错误：

```text
mysql> INSERT INTO t_stu VALUES(1,'张三','男');
ERROR 1366 (HY000): Incorrect string value: '\\xD5\\xC5\\xC8\\xFD' for column 'sname' at row 1
```

### 先理解本质

这类问题常见原因不是 `insert into` 语法本身，而是：

**你输入中文时，当前连接、服务器配置、数据库对象之间的字符集不一致。**

### 常见原因

![字符集不一致示意](./images/1554912924219.png)

高频原因是：

* 服务器端认为当前客户端字符集是 `utf-8`
* 你实际输入数据时的环境却更接近 `GBK`

这就会导致：

**你以为自己在“写中文”，但服务器收到的是一串和预期编码不一致的字节。**

### 排查与处理

先查看当前字符集配置：

```sql
SHOW VARIABLES LIKE 'character_set_%';
```

![查看字符集配置](./images/1554912943186.png)

如果需要，也可以一起查看校对规则：

```sql
SHOW VARIABLES LIKE 'collation_%';
```

如果确认是当前连接字符集不匹配，可以先尝试设置当前连接的客户端字符集：

```sql
SET NAMES GBK;
```

![设置当前连接字符集](./images/1554912957353.png)

### 这一招解决的是什么？

`SET NAMES GBK;` 主要是在处理：

**当前连接层级** 的字符集问题。

也就是说，它更像是“先让这一次命令行会话讲同一种语言”。

### 注意点

* `SET NAMES GBK` 主要影响当前连接。
* 如果重新打开命令行客户端，可能需要重新设置。
* 如果问题反复出现，就不能只停留在连接级别，还要继续检查：

  * `my.ini`
  * 数据库默认字符集
  * 数据表默认字符集
  * 字段本身的字符编码

> ✏️ 记忆方式：
>
> `SET NAMES GBK;` 解决的是“这次连接怎么解释我输入的数据”，
> 不是永久修好所有库、所有表、所有字段。

## 问题5：修改数据库和表的字符编码

### 先理解本质

这一题最容易误解的地方是：

**很多人改了 `my.ini`，就以为所有旧数据库和旧表都会自动变成新编码。**

其实不会。

因为字符集问题至少可能分成下面几个层级：

1. 当前连接层
2. 配置文件层
3. 数据库层
4. 表层
5. 字段层

如果你没有把这几个层级分开，排错时就会反复卡住。

---

### 修改编码的基本流程

1. 停止 MySQL 服务
2. 修改 `my.ini` 文件
3. 重新启动服务

### 为什么修改 `my.ini` 之后问题可能还在？

如果数据库和数据表是在修改 `my.ini` 之前创建的，那么它们原有的字符编码仍然会保留。
例如原来是 `latin1`，并不会因为你改了配置文件就自动变成 `utf8`。

这时通常有两种思路：

* 删除旧对象后重新创建
* 使用 `ALTER` 语句逐层修改

---

### 示例：原来使用 latin1 创建数据库和表

```sql
mysql> create database 0728db charset Latin1;
Query OK, 1 row affected (0.00 sec)
```

```sql
mysql> use 0728db;
Database changed
```

```sql
mysql> create table student (id int , name varchar(20)) charset Latin1;
Query OK, 0 rows affected (0.02 sec)
```

```sql
mysql> show create table student\G
*************************** 1. row ***************************
       Table: student
Create Table: CREATE TABLE `student` (
  `id` int(11) NOT NULL,
  `name` varchar(20) DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=latin1
1 row in set (0.00 sec)
```

这说明：

* 数据库对象已经存在
* 而且当前表默认字符集确实是 `latin1`

---

### 示例：修改表和字段字符编码

先修改表的默认字符编码：

```sql
mysql> alter table student charset utf8;
Query OK, 0 rows affected (0.01 sec)
Records: 0  Duplicates: 0  Warnings: 0
```

再查看表结构：

```sql
mysql> show create table student\G
*************************** 1. row ***************************
       Table: student
Create Table: CREATE TABLE `student` (
  `id` int(11) NOT NULL,
  `name` varchar(20) CHARACTER SET latin1 DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8
1 row in set (0.00 sec)
```

这里最关键的点是：

**虽然表默认字符集已经改成 `utf8`，但字段 `name` 仍然保持 `latin1`。**

所以还需要继续单独修改字段：

```sql
mysql> alter table student modify name varchar(20) charset utf8;
Query OK, 0 rows affected (0.05 sec)
Records: 0  Duplicates: 0  Warnings: 0
```

再看结果：

```sql
mysql> show create table student\G
*************************** 1. row ***************************
       Table: student
Create Table: CREATE TABLE `student` (
  `id` int(11) NOT NULL,
  `name` varchar(20) DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8
1 row in set (0.00 sec)
```

### 这一段最重要的理解

* `ALTER TABLE ... CHARSET ...`：改的是**表级默认值**
* `ALTER TABLE ... MODIFY ... CHARSET ...`：改的是**字段级编码**

表默认值改了，不代表旧字段会自动一起改。

---

### 示例：修改数据库字符编码

先查看数据库创建信息：

```sql
mysql> show create database 0728db;
+--------+-----------------------------------------------------------------+
|Database| Create Database                                                 |
+--------+-----------------------------------------------------------------+
| 0728db | CREATE DATABASE `0728db` /*!40100 DEFAULT CHARACTER SET latin1 */ |
+--------+-----------------------------------------------------------------+
1 row in set (0.00 sec)
```

再修改数据库默认字符编码：

```sql
mysql> alter database 0728db charset utf8;
Query OK, 1 row affected (0.00 sec)
```

再次查看：

```sql
mysql> show create database 0728db;
+--------+-----------------------------------------------------------------+
|Database| Create Database                                                 |
+--------+-----------------------------------------------------------------+
| 0728db | CREATE DATABASE `0728db` /*!40100 DEFAULT CHARACTER SET utf8 */ |
+--------+-----------------------------------------------------------------+
1 row in set (0.00 sec)
```

### 这一段最重要的理解

这里改的是：

**数据库级默认字符编码。**

它会影响之后新建对象的默认行为，但不等于自动修复所有旧表、旧字段。

---

### 最后把三层修改关系记清楚

修改数据库默认字符编码、修改表默认字符编码、修改字段字符编码，是三个不同层级：

* `ALTER DATABASE`：改数据库级默认值
* `ALTER TABLE ... CHARSET ...`：改表级默认值
* `ALTER TABLE ... MODIFY ... CHARSET ...`：改字段级编码

已有字段不会因为表默认字符编码改变就自动改变。
所以排查乱码问题时，要分别确认：

* 当前连接
* 配置文件
* 数据库
* 表
* 字段

## 常见风险点

### 风险 1：一看到问题就开始重装 MySQL

很多问题根本不是安装坏掉，而是：

* `Path` 没配置好
* 当前会话没选数据库
* 当前连接字符集不匹配
* 旧数据库对象仍保留旧编码

### 风险 2：把“命令无法识别”误判成“密码错了”

如果系统连 `mysql` 这个命令都不认识，重点应先查环境变量，而不是账号密码。

### 风险 3：把 `ERROR 1046` 当成数据库损坏

它通常只是当前会话没有默认数据库，不是数据库本体坏掉。

### 风险 4：把 `SET NAMES GBK;` 当成永久解决方案

它解决的是当前连接层级，不是整个系统永久配置。

### 风险 5：改了 `my.ini` 就以为问题一定会全部消失

如果旧数据库、旧表、旧字段仍保留旧编码，乱码问题还是可能存在。

### 风险 6：只改表默认字符集，不改字段字符集

这会导致你以为“表已经是 utf8 了”，但实际某些旧字段仍然是 `latin1`。

## 常见回查问题

### 1. 忘记 `root` 密码时，应该用哪个参数临时跳过权限表检查？

使用：

```text
--skip-grant-tables
```

### 2. `mysql` 命令提示不是内部或外部命令时，应该先检查什么？

先检查 MySQL 的 `bin` 目录是否已经加入环境变量 `Path`。

### 3. `ERROR 1046 (3D000)` 表示什么？

表示当前会话没有选中默认数据库。

### 4. 解决 `ERROR 1046` 时，优先用哪条语句？

如果后续很多 SQL 都在同一个数据库下执行，优先使用：

```sql
USE 数据库名;
```

### 5. 插入中文时报 `ERROR 1366`，第一步应该查什么？

优先查当前连接字符集，再继续看数据库、表、字段的字符编码。

### 6. `SET NAMES GBK` 解决的是当前连接的问题，还是所有连接的问题？

它主要解决当前连接的问题。

### 7. 修改表默认字符编码后，为什么字段仍然可能保持 `latin1`？

因为表级默认值和字段级编码是两个不同层级；旧字段不会因为表默认值改变就自动改变。

### 8. 修改数据库、表、字段字符编码分别应该使用哪些语句？

* 数据库：`ALTER DATABASE`
* 表：`ALTER TABLE ... CHARSET ...`
* 字段：`ALTER TABLE ... MODIFY ... CHARSET ...`

---

## 小结

这一节最重要的，不是把每一种报错都死背下来，而是记住下面这条排查主线：

1. 先判断问题是发生在登录前、登录后，还是写入数据时
2. 如果命令无法执行，优先查 `Path`
3. 如果执行 SQL 报没有选择数据库，优先补上 `USE 数据库名;`
4. 如果插入中文报错，先看当前连接字符集，再看配置文件和数据库对象编码
5. 如果改了 `my.ini` 仍有问题，要继续分清数据库、表、字段三个层级

只要这条主线清楚，你排查 MySQL 初学阶段的大多数问题时，就不会一直乱试。
