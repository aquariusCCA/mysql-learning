## 本节导读

这一节集中整理 MySQL 初学阶段最常见的几类问题，例如忘记 `root` 密码、命令无法识别、没有选中数据库，以及字符集导致的中文乱码或报错。

它更像故障排查索引。平时不需要从头读到尾，遇到具体问题时，直接按错误现象或关键字跳到对应小节会更高效。

如果你想先看这些问题分别是在什么操作场景下出现的，可以搭配 [2 MySQL 的下载、安装、配置](./2%20MySQL%20的下载、安装、配置.md)、[3 MySQL 的登录](./3%20MySQL%20的登录.md) 和 [4 MySQL 演示使用](./4%20MySQL%20演示使用.md) 一起看。

## 你会在这篇学到什么

- 忘记 `root` 密码时，如何通过跳过权限表检查来重置密码。
- `mysql` 命令无法识别时，为什么通常要检查 `Path` 环境变量。
- 遇到 `ERROR 1046 (3D000): No database selected` 时，如何选择默认数据库。
- 命令行客户端插入中文时报 `ERROR 1366` 时，如何从字符集角度排查。
- 修改 `my.ini` 后，为什么已有数据库和表可能仍然保留旧字符编码。
- 修改数据库、表和字段字符编码时，分别应该使用哪些 `ALTER` 语句。

## 关键字

- `root密码重置`：忘记 `root` 密码时的处理思路
- `mysqld`：MySQL 服务进程，可用特殊参数启动
- `--skip-grant-tables`：跳过权限表校验，常用于紧急重置密码
- `PATH`：`mysql` 命令无法识别时需检查环境变量
- `ERROR 1046 (3D000)`：没有选择数据库
- `USE 数据库名`：切换当前默认数据库
- `character_set_%`：查看当前字符集相关配置
- `SET NAMES GBK`：设置当前连接的客户端字符集
- `my.ini`：MySQL 配置文件，可修改默认字符编码
- `latin1` `utf8`：数据库、表、字段常见字符编码
- `ALTER DATABASE`：修改数据库字符编码
- `ALTER TABLE`：修改表或字段字符编码

## 问题1：root 用户密码忘记，重置的操作

### 现象

忘记 `root` 用户密码，导致无法通过正常密码登录 MySQL。

### 处理步骤

1. 通过任务管理器或者服务管理，关掉 `mysqld` 服务进程。
2. 通过命令行和特殊参数开启 `mysqld`：

    ```bash
    shell mysqld --defaults-file="D:\\ProgramFiles\\mysql\\MySQLServer5.7Data\\my.ini" --skip-grant-tables
    ```

3. 此时 `mysqld` 服务进程已经打开，并且不需要权限检查。
4. 另开一个客户端，使用 `mysql -uroot` 无密码登录服务器。
5. 修改权限表：

    ```sql
    use mysql;
    update user set authentication_string=password('新密码') where user='root' and Host='localhost';
    flush privileges;
    ```

6. 通过任务管理器关掉 `mysqld` 服务进程。
7. 再次通过服务管理打开 MySQL 服务。
8. 使用修改后的新密码登录。

### 注意点

`--skip-grant-tables` 会跳过权限表检查，只适合用于紧急重置密码。密码修改完成后，应关闭这个特殊启动的服务进程，再按正常方式启动 MySQL 服务。

## 问题2：mysql 命令报「不是内部或外部命令」

### 现象

在命令行输入 `mysql` 后，提示「不是内部或外部命令」。

### 处理方式

如果输入 `mysql` 命令时报「不是内部或外部命令」，通常是系统还找不到 MySQL 的命令行程序。把 MySQL 安装目录下的 `bin` 目录配置到环境变量 `Path` 中即可。

![image-20210914093150145.png](./images/image-20210914093150145.png)

### 注意点

配置完成后，需要重新打开命令行窗口，让新的环境变量生效。

## 问题3：错误 ERROR：没有选择数据库就操作表格和数据

### 现象

执行表操作或数据操作时出现以下错误：

```text
ERROR 1046 (3D000): No database selected
```

这个错误表示当前会话还没有选中默认数据库。

### 处理方式

解决方案一：使用 `USE 数据库名;` 语句。执行后，接下来的语句默认针对这个数据库进行操作。

解决方案二：在所有表对象前面加上数据库名，例如 `数据库名.表名`。

### 注意点

如果后续 SQL 都是在同一个数据库下操作，优先使用 `USE 数据库名;`。如果只是在某条语句中临时访问其他数据库，可以使用 `数据库名.表名` 的写法。

## 问题4：命令行客户端的字符集问题

### 现象

在命令行客户端插入中文数据时，出现字符集错误：

```bash
mysql> INSERT INTO t_stu VALUES(1,'张三','男');
ERROR 1366 (HY000): Incorrect string value: '\\xD5\\xC5\\xC8\\xFD' for column 'sname' at row 1
```

### 原因

服务器端认为你的客户端字符集是 `utf-8`，而实际上你的客户端字符集是 `GBK`。

![1554912924219.png](./images/1554912924219.png)

### 排查与处理

查看所有字符集：

```sql
SHOW VARIABLES LIKE 'character_set_%';
```

![1554912943186.png](./images/1554912943186.png)

设置当前连接的客户端字符集：

```sql
SET NAMES GBK;
```

![1554912957353.png](./images/1554912957353.png)

### 注意点

`SET NAMES GBK` 主要影响当前连接。如果重新打开命令行客户端，可能需要重新设置，或进一步检查 `my.ini` 中的默认字符集配置。

## 问题5：修改数据库和表的字符编码

### 修改编码的基本流程

1. 先停止服务。
2. 修改 `my.ini` 文件。
3. 重新启动服务。

### 说明

如果是在修改 `my.ini` 之前建的库和表，那么库和表的编码还是原来的 `Latin1`。这时要么删除后重建，要么使用 `ALTER` 语句修改编码。

### 示例：原来使用 Latin1 创建数据库和表

```sql
mysql> create database 0728db charset Latin1;
Query OK, 1 row affected (0.00 sec)
```

```sql
mysql> use 0728db;
Database changed
```

```sql
mysql> create table student (id int , name varchar(20)) charset Latin1;
Query OK, 0 rows affected (0.02 sec)

mysql> show create table student\G
*************************** 1. row ***************************
       Table: student
Create Table: CREATE TABLE `student` (
  `id` int(11) NOT NULL,
  `name` varchar(20) DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=latin1
1 row in set (0.00 sec)
```

### 示例：修改表和字段字符编码

先修改表的默认字符编码：

```sql
mysql> alter table student charset utf8; #修改表字符编码为UTF8
Query OK, 0 rows affected (0.01 sec)
Records: 0  Duplicates: 0  Warnings: 0

mysql> show create table student\G
*************************** 1. row ***************************
       Table: student
Create Table: CREATE TABLE `student` (
  `id` int(11) NOT NULL,
  `name` varchar(20) CHARACTER SET latin1 DEFAULT NULL,  #字段仍然是latin1编码
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8
1 row in set (0.00 sec)
```

此时要注意：修改表的默认字符编码后，已有字段 `name` 仍然是 `latin1` 编码，所以还需要单独修改字段字符编码。

```sql
mysql> alter table student modify name varchar(20) charset utf8; #修改字段字符编码为UTF8
Query OK, 0 rows affected (0.05 sec)
Records: 0  Duplicates: 0  Warnings: 0

mysql> show create table student\G
*************************** 1. row ***************************
       Table: student
Create Table: CREATE TABLE `student` (
  `id` int(11) NOT NULL,
  `name` varchar(20) DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8
1 row in set (0.00 sec)
```

### 示例：修改数据库字符编码

```sql
mysql> show create database 0728db;;
+--------+-----------------------------------------------------------------+
|Database| Create Database                                                 |
+------+-------------------------------------------------------------------+
|0728db| CREATE DATABASE `0728db` /*!40100 DEFAULT CHARACTER SET latin1 */ |
+------+-------------------------------------------------------------------+
1 row in set (0.00 sec)

mysql> alter database 0728db charset utf8; #修改数据库的字符编码为utf8
Query OK, 1 row affected (0.00 sec)

mysql> show create database 0728db;
+--------+-----------------------------------------------------------------+
|Database| Create Database                                                 |
+--------+-----------------------------------------------------------------+
| 0728db | CREATE DATABASE `0728db` /*!40100 DEFAULT CHARACTER SET utf8 */ |
+--------+-----------------------------------------------------------------+
1 row in set (0.00 sec)
```

### 注意点

修改数据库默认字符编码、修改表默认字符编码、修改字段字符编码是三个不同层级。已有字段不会因为表默认字符编码改变就自动改变，所以排查乱码问题时要分别确认数据库、表和字段。

- 修改表默认字符编码后，为什么字段仍然可能保持 `latin1`？

## 一句话抓核心

排查 MySQL 初学阶段常见问题时，先按错误现象定位层级：登录问题看账号与权限，命令不可用看 `Path`，SQL 执行范围看当前数据库，中文乱码或报错看客户端、数据库、表和字段的字符集是否一致。

## 小结

这一节你需要记住：

- 忘记 `root` 密码时，可以临时使用 `--skip-grant-tables` 跳过权限表检查，但重置后要恢复正常启动方式。
- `mysql` 命令无法识别，通常需要把 MySQL 安装目录下的 `bin` 目录加入 `Path`。
- `ERROR 1046 (3D000): No database selected` 表示还没有选择默认数据库，可以使用 `USE 数据库名;`。
- 命令行插入中文时报字符集错误时，可以先用 `SHOW VARIABLES LIKE 'character_set_%';` 查看字符集，再视情况使用 `SET NAMES GBK;`。
- 修改 `my.ini` 不会自动改变已经存在的库、表和字段编码，必要时要用 `ALTER DATABASE`、`ALTER TABLE` 和字段级 `MODIFY` 分别处理。
