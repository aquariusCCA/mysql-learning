## 本节导读

这一节只解决一个核心问题：

**MySQL 安装完成后，常见目录和文件到底各自负责什么，以及源码目录应该用什么角度去看。**

对初学者来说，这一节最容易混乱的，不是目录名称本身，而是下面这些问题：

- 分不清**安装目录**和**数据目录**
- 知道 `mysql` 命令能执行，但不知道为什么和 `bin` 有关
- 知道 `my.ini` 很重要，却不知道它到底影响哪些问题
- 看到 `ProgramData`、`data`、数据库文件这些词时，容易混成一团
- 看到“源码”就以为这一节要开始深入读 MySQL 内核

其实更合理的理解应该是：

**这一节的重点不是“读懂 MySQL 源码”，而是先建立目录分工感。**

所以这篇不会只是列几个目录名字，而是按更实用的顺序整理成：

**先分清安装目录和数据目录 → 再认识 `bin`、`data`、`my.ini` → 再把这些目录和前面学过的环境变量、字符集、卸载重装问题串起来 → 最后对源码目录建立第一层印象。**

> ✏️ 说明：
>
> 这一节偏“结构认知型”，不是高频操作页。  
> 你不一定会天天回来看它，但当你遇到下面这些问题时，它会非常有用：
>
> - `mysql` 命令无法识别
> - 想知道数据库文件大致放在哪里
> - 想分清安装目录与数据目录
> - 修改字符集或配置文件时，不确定应该看哪里
> - 想开始理解 MySQL 源码大概按什么职责分目录

## 你会在这篇学到什么

- MySQL 安装目录中常见文件和目录的大致作用。
- 为什么 `bin` 目录常被加入 `Path` 环境变量。
- `data` 目录和用户创建的数据库文件有什么关系。
- `ProgramData` 路径为什么在卸载、重装和排查数据文件时经常出现。
- `my.ini` 在 MySQL 配置中的作用，以及它为什么常和字符集、服务配置、启动参数有关。
- 如何在下载页面选择 MySQL 源代码。
- MySQL 源码目录中 `sql`、`libmysql`、`mysql-test`、`mysys` 等子目录的大致含义。

**这一节最重要的，不是背目录名字，而是建立“程序文件看安装目录，数据库文件看数据目录，配置问题优先想到 `my.ini`，命令行执行和 `bin` 强相关”的整体认识。**

### 情况 4：我总是分不清安装目录和数据目录

重点看 `安装目录和数据目录要分开理解`。  
这正是这一节最重要、也最容易混淆的地方。

### 情况 5：我只是想先知道 MySQL 源码目录大概按什么分工

直接看 `6.2 MySQL 源代码获取` 和 `源码目录里常见的子目录`。

## 6.1 主要目录结构

这一部分先不追求“每个目录都背下来”，而是先抓住几个最常回查的位置。

| MySQL 的目录结构 | 说明 |
| --- | --- |
| `bin` 目录 | MySQL 的可执行文件目录，例如 `mysql.exe`、`mysqld.exe` |
| `data` 目录 | 数据文件相关目录，系统库和用户库文件通常与这里有关 |
| `my.ini` 文件 | MySQL 的主要配置文件 |
| `C:\ProgramData\MySQL\MySQL Server 8.0\data\` | Windows 下常见的数据目录位置 |
| 某些安装器辅助程序 | 安装和初始化时可能看到的相关可执行程序 |

> ✏️ 这里最重要的不是死记路径，而是先分清：
>
> - 哪些更偏“程序”
> - 哪些更偏“配置”
> - 哪些更偏“数据”

---

### `bin` 目录

`bin` 目录里放的是 MySQL 的可执行程序。

常见的例如：

- `mysql.exe`
- `mysqld.exe`

### 你现在最需要先记住什么？

#### 1）`mysql.exe` 更偏客户端命令

你在命令行里输入：

```bash
mysql -u root -p
````

本质上就是在调用 `mysql.exe`。

#### 2）`mysqld.exe` 更偏服务端程序

它对应的是 MySQL 服务本体，也就是数据库真正运行的那部分。

### 为什么 `bin` 会和环境变量 `Path` 扯上关系？

因为只有系统能找到这些可执行文件，你才可以在任意命令行窗口里直接输入：

```bash
mysql --version
```

或者：

```bash
mysql -u root -p
```

这也是为什么前面配置环境变量时，要把 MySQL 的 `bin` 目录加入 `Path`。

> ✏️ 记忆方式：
>
> * **命令能不能直接敲**，优先想到 `bin`
> * **`mysql` 命令找不到**，优先想到 `Path`
>
> 这和前面安装篇、常见问题篇是直接连在一起的。

---

### `data` 目录

`data` 目录主要和数据库文件本身有关。

你可以先这样理解：

* MySQL 自带的系统库会放在相关数据目录中
* 你自己创建的数据库，最终对应的数据文件也会落在数据目录相关位置

### 为什么这一点重要？

因为很多初学者遇到下面这些问题时，都会绕回这里：

* 数据库文件大概放在哪里
* 卸载后为什么还有旧数据
* 为什么重装后某些旧配置或旧库看起来还在
* 删除程序后，数据为什么没有一起消失

这些现象很多都和“程序删掉了，但数据目录还在”有关。

### Windows 下常见的数据目录位置

在 Windows 中，常见位置例如：

```text
C:\ProgramData\MySQL\MySQL Server 8.0\data\
```

> ⚠️ 注意：
>
> `ProgramData` 通常是隐藏目录。
> 如果你在资源管理器中没看到它，不代表它不存在。

### 当前阶段最重要的理解

你现在不用急着去记每一种数据文件的后缀和用途，
先建立下面这层认识就够了：

**数据库的“真实数据”更靠近 `data` / `ProgramData` 这条线，而不是安装程序所在目录。**

---

### `my.ini` 文件

`my.ini` 是 MySQL 的主要配置文件。

很多常见问题最后都会回到这里，例如：

* 默认字符集
* 服务相关配置
* 部分目录配置
* 启动参数

### 为什么 `my.ini` 特别值得记住？

因为它不是普通说明文件，而是会直接影响 MySQL 运行行为的配置文件。

也就是说：

* 你改了它，MySQL 的配置可能真的会变
* 你改错了它，MySQL 甚至可能无法正常启动

### 它和前面哪些章节直接相关？

#### 和第 2 篇安装篇有关

因为安装完成后，很多“环境整理”和“配置确认”的问题，最终会落到目录和配置文件层面。

#### 和第 4 篇演示使用有关

因为你在插入中文时报 `ERROR 1366` 时，除了看当前连接字符集，还常会继续追到 `my.ini`。

#### 和第 7 篇常见问题篇有关

因为字符集问题、配置修改、旧库旧表仍保留旧字符集这类排查，都会把 `my.ini` 当成关键节点之一。

### 当前阶段该怎样看待 `my.ini`？

先把它当成：

**“MySQL 的主要配置入口”**

而不是一开始就想把里面所有参数都看懂。

---

### 安装目录和数据目录要分开理解

这是一整节里最重要的地方之一。

初学时最容易混淆的是：

**安装目录不等于数据库数据目录。**

可以先用最简单的方式记住：

* **安装目录**：更偏向程序文件和可执行文件
* **数据目录**：更偏向数据库真实数据文件

### 为什么这个区分很重要？

因为你前面几篇学到的很多问题，其实都依赖这个区分：

#### 和环境变量有关

环境变量通常找的是可执行文件，所以重点看安装目录里的 `bin`。

#### 和卸载重装有关

卸载程序，不一定会删除数据目录。
所以如果你只删程序，旧数据和旧库可能还在。

#### 和字符集 / 配置问题有关

配置文件和数据库对象的关系也不能混成一层。
有时你改了 `my.ini`，但旧数据库和旧表仍然保留旧字符集，这时就要继续区分配置层、数据库层、表层、字段层。

> ✏️ 记忆方式：
>
> * **程序跑不跑得起来**，多半先想安装目录
> * **数据还在不在**，多半先想数据目录
> * **配置改哪里**，多半先想 `my.ini`

## 6.2 MySQL 源代码获取

这一部分只解决一个问题：

**如果我想看看 MySQL 源码，大致应该怎么进入这件事。**

### 第一步：下载类型不要选错

![MySQL 源码下载入口示意](./images/2402456-20220611182838668-2052351900.png)

如果你想下载 MySQL 源代码，不要继续使用默认的 Windows 安装包选项，而要在下载页面中选择：

```text
Source Code
```

然后再根据页面选项下载对应源码压缩包。

### 第二步：先建立正确期待

你现在看源码，不是为了立刻搞懂 MySQL 内核的所有实现细节，
而是为了先建立三件事：

1. MySQL 源码是可读的
2. 源码目录不是一整团，而是按职责分工
3. 以后想深入理解时，知道大致从哪里开始找

---

### MySQL 主要使用什么语言开发？

MySQL 主要使用：

```text
C++
```

开发。

所以当你打开源码目录时，本质上看到的是一套普通的 C/C++ 工程结构，而不是某种神秘格式。

> ✏️ 这也代表一件事：
>
> 如果你未来真的想往源码层理解 MySQL，C/C++ 基础会很重要。
> 但以当前这门课的阶段来说，你只需要先知道“源码目录是按职责组织的”。

---

### 源码目录里常见的子目录

以常见 MySQL 8 系列源码目录为例，先建立几个最基础的印象即可：

![MySQL 源码目录示意](./images/image-20211007154113052.png)

#### `sql`

MySQL 核心代码目录。

你可以把它理解成：

**最接近数据库核心逻辑的一层。**

如果以后要理解 SQL 解析、执行流程、数据库核心行为，这类目录通常会很重要。

#### `libmysql`

客户端程序 API 相关目录。

你可以先把它理解成：

**更偏客户端能力和接口相关的一层。**

#### `mysql-test`

测试工具目录。

顾名思义，它更偏向：

**测试、验证、回归检查** 这类用途。

#### `mysys`

操作系统相关函数和辅助函数目录。

你可以把它理解成：

**更偏基础支撑和工具性代码的一层。**

### 当前阶段该怎样看待这些目录？

不是现在就深入追源码，而是先建立这层印象：

**源码目录是按职责拆开的，不同目录关注的事情不同。**

这样以后你真的开始读源码时，就不会把整个源码树看成一整团。

---

### 如何查看源码？

源码可以直接用文本编辑器打开查看；
如果你已经有 C/C++ 开发环境，也可以用更适合代码导航的开发工具进一步阅读。

![查看 MySQL 源码示意](./images/image-20211007154213156.png)

但对当前阶段来说，最重要的不是工具，而是视角：

* 不求一开始全懂
* 先看目录名称和职责分工
* 先建立“哪个目录大概管什么”的地图感

## 常见风险点

### 风险 1：把安装目录和数据目录当成同一个东西

这是最高频误区。
程序文件和数据库真实数据文件，不应该放在同一个理解层面。

### 风险 2：只知道 `mysql` 命令能执行，却不知道为什么

如果不知道 `bin` 和 `Path` 的关系，后面遇到“命令无法识别”时会很容易卡住。

### 风险 3：把 `my.ini` 当成普通说明文件

它是配置文件，不只是说明文件。
改错时，可能直接影响 MySQL 启动和运行。

### 风险 4：一看到“源码”就想立刻深入实现细节

这一节的重点不是“读通源码”，而是先建立源码目录分工感。

### 风险 5：以为改了 `my.ini`，所有旧数据库和旧表都会自动跟着改变

并不会。
数据库、表、字段字符集是不同层级的问题，这一点在常见问题篇里会继续展开。

## 常见回查问题

### 1. `bin` 目录里通常放什么？为什么它和命令行登录有关？

它通常放 MySQL 的可执行程序，例如 `mysql.exe`、`mysqld.exe`。
命令行能直接执行 `mysql`，本质上和 `bin` 被加入 `Path` 有关。

### 2. `data` 目录和 `C:\ProgramData\MySQL\MySQL Server 8.0\data\` 有什么关系？

后者就是 Windows 下常见的数据目录位置之一。
你创建的数据库文件，通常会和这条数据目录线有关。

### 3. `my.ini` 是做什么用的？

它是 MySQL 的主要配置文件，常和字符集、服务配置、启动参数等问题有关。

### 4. 想下载 MySQL 源码时，下载页面应该选择哪个类型？

选择：

```text
Source Code
```

### 5. MySQL 源码中的 `sql`、`libmysql`、`mysql-test`、`mysys` 分别大致对应什么？

* `sql`：核心逻辑相关
* `libmysql`：客户端 API 相关
* `mysql-test`：测试相关
* `mysys`：操作系统相关函数和辅助函数
