# Study Notes Markdown

當使用者希望學習筆記保留教材型、教學導向、較詳細的說明，但又想讓這些筆記在之後更容易複習、瀏覽與搜尋時，使用這個 skill。

## 目標

不要把核心教學內容壓縮掉。重點是替既有內容補上外層結構，讓它更好用：

- 根目錄導航
- 章節導航
- 統一的筆記開頭格式
- 高價值內部連結
- 輕量型複習輔助

## 工作流程

1. 先檢查倉庫結構與章節分佈。
2. 除非使用者明確要求重寫，否則保留原本教材型正文。
3. 先判斷目前是大筆記還是小筆記。
4. 大筆記先拆分，再整理拆分後的小筆記；小筆記則直接整理。
5. 先補導航與結構，再考慮內容層面的微調。
6. 只加入高訊號的複習輔助，不要把筆記改成只有大綱的速記。
7. 優先做小而穩定、可重複套用的結構化修改，不做過度濃縮。

## 預設結構

### 根目錄 README

讓根目錄 `README.md` 成為整份筆記的入口頁：

- 簡短說明倉庫用途
- 目前進度
- 章節導航
- 如何使用這份筆記
- 參考資料放在導航之後，而不是之前

### 章節 README

每個主要章節資料夾都應有自己的 `README.md`，內容包含：

- 本章主要內容
- 建議閱讀順序
- 本章所有小節連結
- 哪些主題適合快速回查
- 返回根目錄 README 的連結

### 單篇筆記開頭

每篇筆記建議依照這個順序：

1. 標題
2. `所属章节`
3. `本节导读`
4. `关键字`
5. 正文教學內容

`本节导读` 應回答：

- 這一節主要解決什麼問題
- 這一節建議怎麼讀
- 之後什麼情況下適合回來重讀

保持精簡即可，通常 2 段短文字就夠。

## 大筆記拆分模式（先拆分，再整理）

當單一 Markdown 筆記過長、同時已經具備明顯標題階層時，可以啟用大筆記拆分模式。

### 適用情況

- 單篇筆記行數過長，影響閱讀、搜尋與維護
- 原文已經有穩定的標題階層，例如 `#`、`##`、`###`
- 使用者希望拆成多篇，但不希望改寫正文內容

### 拆分原則

- 預設保留原始大筆記，不直接覆寫原檔
- 依既有標題階層拆分，不靠語意重寫重組
- 正文逐段搬移，不改寫、不摘要、不潤飾
- 不修改原有列表、表格、引用、圖片、程式碼區塊
- 若原文有跨段參照，例如「下節會說明」或「上面提過」，預設保留，不自動改寫

### 兩階段流程

#### 第一階段：拆分

- 保留原始大筆記作為來源檔
- 依既有標題階層拆出多篇小筆記
- 拆分時只搬移正文內容，不改寫正文

#### 第二階段：整理拆分後的小筆記

- 允許對拆分後的小筆記補上 `所属章节`、`本节导读`、`关键字`
- 允許補上高價值內部連結與必要的返回導航
- 這些增補只能加在正文外層，不應重寫正文段落本身
- 若使用者明確要求純拆分版本，才跳過這一階段

### 拆分策略

- 先找出最適合的拆分層級
- 若檔案內有多個 `#` 主題，優先按 `#` 拆分
- 若只有一個 `#`，但有多個 `##`，則按 `##` 拆分
- 子筆記內保留原本更低層級的標題結構

### 輸出方式

- 原始檔案保留為來源檔
- 另建一個拆分資料夾放置子筆記與章節導航
- 導航頁負責串接拆分後的小筆記
- 子筆記正文以原文片段為主
- 拆分完成後，可再依單篇筆記開頭模板補齊外層結構

### 拆分目錄命名

- 正式落地時，拆分目錄預設使用原筆記檔名去掉 `.md` 後的名稱
- 例如 `2 链表.md` 拆分後，正式目錄應命名為 `2 链表/`
- 若是臨時測試、過渡輸出，或需要與原始檔案並存，可在目錄名後加上 `_拆分`
- 例如測試輸出可命名為 `2 链表_拆分/`

### 導航頁用途

拆分後可新增一個 `README.md` 作為入口，提供：

- 本組筆記說明
- 建議閱讀順序
- 拆分後小節連結
- 返回上層章節的連結

### 圖片與資產處理

- 若原始筆記的圖片已集中在該章的 `images/` 目錄，拆分後的小筆記預設沿用原章節圖片，不複製資產
- 拆分後應改用新的相對路徑引用圖片，例如子筆記位於 `2 链表/01 概述.md` 時，章節圖片可寫成 `../images/example.png`
- 不要為了拆分方便，就在每個子筆記目錄下再複製一份 `images/`
- 只有在拆分後整組筆記需要獨立搬移、獨立發布，或明確要與原章節解耦時，才考慮在拆分目錄下建立自己的 `images/`
- 若原始圖片本身就在錯誤位置、缺失，或引用已失效，先保留原引用關係並標註問題，不要在拆分時臨時重組資產

## 小筆記整理模式

當筆記本身已經足夠獨立、長度可控時，不需要先拆分。

- 直接在原筆記上補齊 `所属章节`、`本节导读`、`关键字`
- 視需要補高價值內部連結與文末 `延伸阅读`
- 保留教材型正文，不用摘要取代正文

## 內部連結規則

只有在連結真的能幫助讀者決定「下一步該看哪裡」時才加。

適合加連結的目標：

- 前置概念
- 自然的下一篇
- 從報錯案例跳到故障排查頁
- 從理論頁跳到實作頁

避免：

- 每個重複名詞都加連結
- 一句短句裡塞太多連結
- 已經有明顯章節導航時，還在附近重複補無意義連結

優先使用兩種模式：

### 正文中的跳轉連結

當一句話自然指向另一篇筆記時使用。

例如：

- DB / DBMS / SQL -> MySQL 介紹
- RDBMS vs NoSQL -> 表關係設計
- 指令報錯案例 -> 常見問題排查
- SQL 概述 -> SQL 實作示範頁

### 文末的 `延伸阅读`

適合放在核心筆記或概念總覽型筆記的文末。

通常控制在 2 到 4 個連結。

## 撰寫原則

- 保留教材型說明。
- 不要用單薄摘要取代詳細解釋。
- 複習輔助是外層補強，不是正文替代品。
- 標題與命名方式要和現有倉庫保持一致。
- 優先使用相對路徑連結。

## 預期成果

套用這個 skill 後，筆記應同時支援兩種使用模式：

- 第一次學習：可以從頭到尾順讀
- 之後複習：可以透過章節 README、關鍵字與內部連結跳讀
