# Study Notes Markdown

當使用者希望保留教材型、教學導向、較詳細的正文內容，但又希望筆記在後續更容易理解、瀏覽、搜尋、跳讀、回查與複習時，使用這個 skill。

這個 skill 的核心不是把內容濃縮成摘要，而是讓原本的筆記「更好用」。

---

# 核心目標

不要刪掉核心教學內容。  
不要把詳細說明改成只有提綱。  
不要為了整齊而犧牲理解。

重點是替既有內容補上穩定的外層結構，讓筆記同時支援：

- 第一次學習時可順讀
- 複習時可快速定位
- 搜尋時容易命中
- 跳讀時不容易迷路
- 之後擴充時維持一致性

---

# 成功標準

整理完成後，筆記應盡量達成以下效果：

1. 使用者一打開就知道這篇在講什麼
2. 使用者知道這篇適合怎麼讀
3. 使用者知道這篇和前後知識點的關係
4. 使用者之後想回查時，能在短時間內找到答案
5. 使用者複習時，不必重讀整篇才能抓到重點位置
6. 使用者在倉庫中不容易迷路

---

# 核心原則

## 1. 保留教材型正文

- 除非使用者明確要求重寫，否則保留原本教學型正文
- 不用摘要取代正文
- 不把段落改成只剩大綱式速記
- 允許做的是加上外層結構、導航、標示與少量輔助

## 2. 先補結構，再碰內容

整理順序優先是：

1. 倉庫入口與章節導航
2. 單篇筆記開頭格式統一
3. 必要的內部連結
4. 輕量型複習輔助
5. 最後才是小幅內容微調

## 3. 小改穩改，不過度濃縮

- 優先做可重複套用的結構化修改
- 不做大量語意重寫
- 不追求把每篇都整理成考前速記
- 不追求每段都加總結，避免破壞原文節奏

## 4. 理解優先於整齊

若形式上的整齊會讓內容更難懂，應保留原本較易理解的寫法。  
整理的目的是讓筆記更容易吸收，而不是只讓版面變漂亮。

---

# 工作流程

1. 檢查整體倉庫結構與章節分布
2. 判斷目前是大筆記還是小筆記
3. 若為大筆記，先拆分，再整理拆分後的小筆記
4. 若為小筆記，直接整理
5. 先補 README / 導航 / 開頭格式
6. 再補高價值內部連結
7. 最後再視需要補複習輔助

---

# 大筆記與小筆記的判斷

## 視為大筆記的情況

符合以下任一情況時，可視為大筆記：

- 單篇過長，閱讀與搜尋成本高
- 同時涵蓋多個明顯子主題
- 已有穩定的標題階層，例如 `#`、`##`、`###`
- 使用者在單篇中不容易快速定位內容

## 視為小筆記的情況

符合以下情況時，可直接整理：

- 單篇主題明確且範圍集中
- 長度可控
- 適合獨立閱讀與獨立回查
- 不拆分也不影響理解與維護

---

# 根目錄 README

讓根目錄 `README.md` 成為整份筆記的入口頁，而不是只放標題。

根目錄 README 應優先包含：

1. 倉庫用途簡介
2. 目前學習進度
3. 章節導航
4. 如何使用這份筆記
5. 需要快速回查的入口
6. 參考資料

## 根目錄 README 建議內容

- 這份筆記在學什麼
- 目前已整理到哪裡
- 各章節連結
- 建議閱讀順序
- 適合初學順讀的章節
- 適合複習跳讀的章節
- 參考資料放在導航之後，而不是之前

---

# 章節 README

每個主要章節資料夾都應有自己的 `README.md`。

章節 README 應包含：

- 本章主要內容
- 本章和整體學習地圖的關係
- 建議閱讀順序
- 本章所有小節連結
- 哪些主題適合快速回查
- 返回根目錄 README 的連結

## 章節 README 的用途

章節 README 不只是列表頁，它應該回答：

- 這章在學什麼
- 先讀哪篇比較不會卡
- 哪幾篇是概念基礎
- 哪幾篇是常回查頁
- 這章讀完後通常接哪一章

---

# 單篇筆記開頭格式

每篇筆記應優先補齊以下開頭結構：

1. 標題
2. 所属章节
3. 本节导读
4. 关键字
5. 建議回查情境
6. 正文教學內容

## 建議模板

```md
# 標題

> 所属章节：資料結構 / 鏈表  
> 关键字：單向鏈表、雙向鏈表、節點、插入、刪除、遍歷  
> 建議回查情境：忘了某個概念、想快速確認差異、遇到實作卡點時

## 本节导读

這一節主要在說明什麼、解決什麼問題。  
第一次閱讀時建議先看哪一部分，再看哪一部分。

當你之後忘記核心概念、分不清相近名詞，或想快速回查實作思路時，適合回到這一篇。

## 你會在這篇學到什麼

- 概念 A
- 概念 B
- 概念 C

## 正文
```
