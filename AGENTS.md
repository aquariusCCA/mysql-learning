# Repository Guidelines

## 專案結構與模組組織
此倉庫以課程筆記為主，並非應用程式專案。請以 `README.md` 作為頂層索引，整理學習資源入口。內容應放在主題資料夾下，例如 `MySQL 基礎篇/` 與 `MySQL 高級篇/`。每一冊建議以章節為單位建立子資料夾，例如 `MySQL 基礎篇/第一章_數據庫概述/`，並將該章圖片集中放在本章的 `images/` 子資料夾中，確保 Markdown 可使用相對路徑引用。

## 建置、測試與開發指令
本倉庫目前沒有建置流程，也沒有套件管理器。提交前請使用以下輕量檢查方式：

- `rg --files`：快速列出所有已追蹤的筆記與資產檔案。
- `git diff --check`：檢查尾隨空白與錯誤的合併標記。
- `git status --short`：查看本次實際變更了哪些筆記或圖片。

送出前請先在編輯器中預覽 Markdown，尤其是新增表格、提示區塊或圖片時。

## 撰寫風格與命名慣例
文件請使用 UTF-8 編碼的 Markdown，標題層級清楚，段落保持精簡。命名請盡量沿用現有模式：

- 章節資料夾：`第X章_主題`
- 課程文件：`序號 空格標題.md`，或遵循該章既有格式
- 圖片檔案：使用具描述性的檔名，並放在該章的 `images/` 資料夾內

圖片與其他資產優先使用相對路徑，例如 `./images/example.png`。文字風格請保持教學導向，並與相鄰章節語氣一致。

## 測試指引
目前沒有自動化測試，也沒有覆蓋率要求，因此需進行人工驗證：

- 確認 Markdown 預覽中的標題層級與排版正確。
- 確認所有圖片與連結都能從編輯後的文件正常開啟。
- 確認新檔案放在正確的章節資料夾，而不是誤放在倉庫根目錄。

## Commit 與 Pull Request 規範
請遵循目前提交紀錄中的格式：`docs(<scope>): <summary>`，例如 `docs(第一章_數據庫概述): 增加数据库定义说明`。`scope` 優先使用章節資料夾名稱。

Pull Request 應簡要說明此次修改的章節或內容，註明是否有重新命名或移動檔案，並列出新增的圖片或外部參考資料。只有在排版呈現或資產位置需要特別確認時才附上截圖。
